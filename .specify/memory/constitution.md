# NiuSX 个人技术博客项目宪章

## 核心原则

### I. 静态优先架构

发布的网站必须由版本控制中的 Jekyll 源文件生成静态 HTML、CSS、JavaScript 和资源文件。
核心内容和导航不得依赖应用服务器、数据库、登录系统或运行时 API 才能渲染和访问。
每次变更都必须能够通过 `bundle exec jekyll build` 完成干净构建。

`_site/` 等生成目录是构建产物，不得作为源代码的真实来源。只有在通过宪章修订说明静态方案
不足以满足需求后，项目才可以引入服务端功能。

**理由**：静态部署使个人博客成本低、可迁移、易维护，并且能够稳定运行在 GitHub Pages 上。

### II. 优先复用 Minimal Mistakes

实现功能时必须优先使用 Minimal Mistakes 提供的配置、布局、include、数据文件和 Sass 扩展点。
当配置项、主题覆盖或小型自定义 include 可以满足需求时，不得创建平行的组件体系。

不得在没有必要的情况下复制或大范围修改主题源码。自定义行为必须隔离、命名清晰，并在改变
主题默认行为时补充说明。不得仅为了重建 Jekyll 或 Minimal Mistakes 已经提供的能力而新增前端
框架或额外依赖。

**理由**：优先复用主题能够降低维护成本，并保留未来升级主题的可行路径。

### III. 内容与阅读体验优先

文章发现和长文阅读必须优先于装饰效果。页面必须保持标题、日期、分类、标签、摘要、标题层级、
代码块、表格、图片和参考资料的可读性与语义结构。新增文章和页面必须使用有效的 front matter
以及稳定的站内链接。

导航、搜索、归档和文章详情页必须清晰易懂，不得使用营销式文案、侵入式遮罩、自动播放媒体或
与正文争夺注意力的装饰。技术结论和操作步骤必须具体到可以被验证。

**理由**：博客的长期价值来自持续积累的技术知识，而不是短期视觉效果。

### IV. GitHub Pages 部署完整性

所有路由、资源、规范链接、订阅源、站点地图和站内链接都必须在配置的 GitHub Pages `url`
和 `baseurl` 下正常工作。模板必须使用 `relative_url` 或 `absolute_url` 等 Jekyll URL 过滤器，
不得假设网站部署在域名根路径。

GitHub Actions 部署流程必须从已提交的源代码构建网站，不得依赖开发者本机文件。部署凭据必须遵循
最小权限原则，且不得提交到仓库。涉及 `_config.yml`、插件、路径或构建依赖的变更，必须按照生产
部署方式进行验证。

**理由**：本地可访问的页面并不等于完成；如果页面在 `/blog` 部署路径下失效，功能就是不完整的。

### V. 默认可访问、响应式与轻量

核心内容和导航必须在移动端、桌面端、键盘操作和常见浏览器中可用。变更后的界面必须使用语义化
HTML、清晰可见的焦点状态、有效的替代文本、足够的颜色对比度，并且不能只依赖颜色传达状态。

JavaScript、图片、字体和第三方嵌入必须保持在功能所需的最小范围内。即使可选脚本或外部资源
加载失败，核心阅读和导航仍必须可用。可避免的布局偏移、页面级横向溢出和阻塞性资源体积都
必须视为缺陷处理。

**理由**：技术博客必须能够在受限设备和不稳定网络环境下保持可读和可访问。

## 平台与架构约束

- 项目是使用 `minimal-mistakes-jekyll` 主题和 Bundler 构建的 Jekyll 静态网站。
- 生产环境是 GitHub Pages，目前通过 `/blog` 作为站点的基础路径发布。
- 文章必须放在 `_posts/`，静态页面必须放在 `_pages/`；共享展示逻辑应按职责放在 `_layouts/`、
  `_includes/`、`_sass/`、`assets/` 和 `_data/`。
- 站点级行为和元数据必须放在 `_config.yml`；当默认配置可以表达相同规则时，不得在多个页面重复配置。
- `_site/` 和缓存目录是生成产物。源代码变更必须发生在 Jekyll 源码目录中，不得直接修改生成的 HTML。
- 插件和依赖必须兼容项目配置的 Jekyll 版本以及 GitHub Actions 构建环境。新增依赖必须说明必要性
  并完成构建验证。
- 外部服务只能作为可选增强。外部服务失败时，不得阻断文章、归档、导航或本地资源的访问。

## 内容与变更质量门禁

每个功能规格、实施计划和代码实现都必须根据适用情况完成以下检查：

1. **静态兼容性**：功能不包含未声明的服务端、数据库或运行时 API 要求。
2. **主题复用**：计划明确说明复用了哪些 Minimal Mistakes 扩展点，并解释任何自定义替代方案。
3. **内容完整性**：受影响文章或页面的 front matter、标题、代码、图片、元数据和站内链接有效。
4. **部署路径**：URL 和资源在 `url`、`baseurl` 以及 GitHub Pages 输出路径下均可用。
5. **响应式与可访问性**：变更后的界面在窄屏和键盘操作下仍然可用。
6. **构建可复现**：变更包含一次干净的 `bundle exec jekyll build` 检查。
7. **范围控制**：新增依赖、脚本和自定义组件是满足需求的最小方案。

违反门禁时，必须在实施计划的“复杂度追踪”中记录违反原因以及被拒绝的简单替代方案。未记录的
门禁违反项会阻塞实现。

## 开发与发布流程

- 功能开发必须从明确的规格开始，规格包含可独立验证的用户场景和可度量的成功标准。
- 实施计划必须列出受影响的 Jekyll 内容、模板、样式、配置、资源和工作流路径。
- 任务必须按用户可见结果组织，并在适用时包含构建、路由、响应式和可访问性验证。
- 本地验证应从与变更最相关的最小检查开始，并在完成前执行 `bundle exec jekyll build`。
- 修改 `_config.yml`、`Gemfile`、`Gemfile.lock`、GitHub Actions 或部署路径时，必须明确验证生产影响。
- 不得通过手动修改 `_site/` 生成内容来修复源代码缺陷。
- 当变更引入长期有效的设计、内容、部署或维护规则时，必须同步更新文档。

## 治理

本宪章是项目规格、计划、任务生成、实现和评审的最高项目级依据。与本宪章冲突的本地约定必须被
修改，或通过修订流程明确说明例外原因。

修订必须包含：

1. 对变更内容和动机的书面说明。
2. 对模板、运行时文档、构建行为和部署流程的影响评估。
3. 本文档的语义化版本更新。
4. 当依赖模板受到影响时，在同一变更中同步更新 Spec Kit 模板。
5. 验证没有未解释的占位符或互相矛盾的规则。

版本遵循语义化版本规范：

- **主版本 MAJOR**：以不向后兼容的方式删除或重新定义核心原则。
- **次版本 MINOR**：新增原则，或实质性扩大强制治理规则和质量门禁。
- **修订版本 PATCH**：只澄清文字，不改变强制行为。

每个功能计划必须在研究前完成“宪章检查”，并在设计完成后再次检查。每次实现评审都必须在宣布
完成前验证适用的质量门禁。例外只对当前功能有效，除非通过修订流程纳入宪章。
