## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供[具体读者能力]。
- **FR-002**：功能必须保留有效的 Jekyll front matter 和稳定的站内 URL。
- **FR-003**：功能必须复用最合适且范围最小的 Minimal Mistakes 布局、include、配置项或 Sass 扩展点。
- **FR-004**：在适用时，功能必须提供[空状态、错误状态或不可用状态]。
- **FR-005**：功能必须在配置的 GitHub Pages 部署路径下可用。

### 静态站点与部署需求（必填）

- **SD-001**：核心内容必须在没有服务端、数据库、登录或运行时 API 的情况下由静态输出提供。
- **SD-002**：站内链接和资源必须使用适合部署路径的 URL 处理方式，并在配置的 `url` 和 `baseurl` 下可用。
- **SD-003**：功能必须能够通过 `bundle exec jekyll build` 可重复构建。
- **SD-004**：任何新增依赖、外部脚本、嵌入内容或构建插件都必须说明必要性、失败影响和 GitHub Pages 兼容性。

### 可访问性与响应式需求（修改界面时必填）

- **UI-001**：修改后的标记必须使用语义化结构、可见的键盘焦点，以及适用的标签或替代文本。
- **UI-002**：功能在窄屏下必须保持可读，且页面不能出现意外的整体横向滚动。
- **UI-003**：状态和含义不得只通过颜色表达。

## 成功标准（必填）

### 可度量结果

- **SC-001**：读者可以在生成的静态输出中完成主要旅程，不依赖运行时服务端或客户端数据请求。
- **SC-002**：功能完成后 `bundle exec jekyll build` 成功执行。
- **SC-003**：修改后的页面在桌面端和移动端宽度下可用，不出现破坏布局的溢出。
- **SC-004**：站内链接、资源、元数据及受影响的归档/搜索路径在配置的 GitHub Pages `baseurl` 下可解析。
- **SC-005**：修改后的界面中，与键盘和屏幕阅读器相关的内容仍然可发现。

## 假设

- 除非功能明确提出宪章修订，否则项目继续使用 Jekyll 和 Minimal Mistakes。
- 文章继续以带 front matter 的 Markdown 文件保存，不引入数据库内容。
- GitHub Pages 继续作为生产部署平台。
- 可选第三方服务可能失败，但不能阻断核心文章阅读。
