# 任务：[功能名称]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档

**前置条件**：必须有 `plan.md` 和 `spec.md`；仅在需要时增加 `research.md`、`data-model.md`、
`quickstart.md` 或 `contracts/`

**测试**：除非规格明确要求，自动化测试是可选的；但适用的 Jekyll 构建、路由、响应式、可访问性和资源检查必须完成。

**组织方式**：任务按用户可见结果分组，并且必须包含准确的仓库路径。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可以并行执行，因为文件和前置条件互不重叠。
- **[Story]**：用户故事标识，例如 `US1`、`US2` 或 `US3`。
- 每个任务描述必须包含准确的文件或目录路径。

## 路径约定

- `_posts/`：Markdown 文章和 front matter。
- `_pages/`：静态页面、归档、搜索和关于页面。
- `_layouts/`：页面和文章布局覆盖。
- `_includes/`：可复用 Liquid 片段。
- `_data/`：导航和结构化站点数据。
- `_sass/`：主题变量和项目样式覆盖。
- `assets/`：样式表、脚本、图片和字体。
- `_config.yml`、`Gemfile`、`Gemfile.lock`：构建和主题配置。
- `.github/workflows/`：GitHub Pages 构建和部署工作流。
- `docs/`：运行和维护文档。

## 依赖关系与执行顺序

### 阶段依赖

- **准备工作**：无依赖；负责识别最小可行扩展点。
- **基础能力**：依赖准备工作，并阻塞所有用户故事。
- **用户故事**：依赖基础能力；互不依赖的故事可以并行。
- **收尾工作**：依赖所有请求的用户故事完成。

### 每个用户故事内部

- 先处理内容和 front matter，再处理页面展示。
- 先处理共享布局和 include，再处理页面专用覆盖。
- 结构确定后再处理样式和资源。
- 实现完成后验证生成路由和构建结果。
- 只有完成规定的状态和部署路径检查，用户故事才算完成。

### 可并行机会

- 不修改同一共享布局或配置时，独立的 Markdown 文章、静态页面、数据条目和 Sass 文件可以并行。
- 共享的 `_config.yml`、`_layouts/`、`_includes/` 和工作流文件同一时间应只由一个负责人修改。

## 实施策略

### 先完成最小可用版本

1. 完成准备工作和基础能力阶段。
2. 实现 P1 读者结果。
3. 执行干净构建，并在生产部署路径下检查生成页面。
4. 验证通过后再添加低优先级用户故事。

### 增量交付

1. 增加一个可以独立使用的读者结果。
2. 验证内容、URL、响应式行为、可访问性和构建输出。
3. 在不破坏已验证路由的前提下增加下一个故事。
4. 不通过直接编辑 `_site/` 产物修复源代码问题。

## 说明

- `[P]` 任务不得共享会被同时修改的文件，也不得依赖未完成的任务。
- `[Story]` 标签用于保持任务与规格之间的可追溯性。
- 即使规格没有要求自动化测试，构建验证仍然是必需的。
- 除非先修订宪章，不得添加后端模型、服务、接口、迁移或认证任务。
- 避免模糊任务；每个实现任务都必须写明具体路径和可观察结果。
