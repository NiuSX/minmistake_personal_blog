# 2. 设计规范文档 `DESIGN.md`

`DESIGN.md` 可以理解为写给 AI 的“设计说明书”。它的作用让 AI 每次生成 UI 时都有稳定的风格边界。

`DESIGN.md` 主要解决三个问题：

- 风格一致：颜色、字体、圆角、间距不要每次都变。
- 方向明确：告诉 AI 这个产品是后台、工具、App，还是官网。
- 限制清楚：明确哪些风格不要出现。

`DESIGN.md` 更像是代码领域的 `constitution.md`

>注：代码领域有 Spec Kit 这类“规格驱动开发”工具，但 UI 领域目前还没有同等统一的标准。

**`Design.md`资源网站**:
- [awesome-design-md-cn](https://fchangjun.github.io/awesome-design-md-cn/index.html)   
- [awesome-design-md-cn中文版的Github地址](https://github.com/fchangjun/awesome-design-md-cn)
- [awesome-design-md-cn原英文版的Github地址](https://github.com/VoltAgent/awesome-design-md)

如果要搭建一套更完整的 UI 规格，可以这样拆：

| 文档 | 作用 |
| --- | --- |
| `PRODUCT_BRIEF.md` | 产品目标、用户、场景、核心任务 |
| `UX_SPEC.md` | 用户路径、页面流程、信息架构 |
| `DESIGN.md` | 视觉风格、设计原则、组件气质 |
| `SCREEN_SPEC.md` | 每个页面的模块、状态、响应式规则 |
| `COMPONENT_SPEC.md` | 组件规范，例如按钮、输入框、表格、卡片 |
| `TOKENS.json` | 颜色、字号、间距、圆角等设计变量 |
| `HANDOFF.md` | 设计交付和代码实现说明 |

个人项目可以使用简化版的`design.md`,最小可用组合是：`PRODUCT_BRIEF.md` + `DESIGN.md` + `SCREEN_SPEC.md`

简化版`design.md`模板:

```markdown
# DESIGN.md

## 产品定位
这是一个面向个人开发者的博客后台，帮助用户快速写文章、管理草稿、查看数据。

## 目标用户
- 个人开发者
- 技术写作者
- 非专业内容运营者

## 视觉风格
- 清爽、克制、专业。
- 信息密度适中，适合长时间使用。
- 不要营销落地页风格。
- 不要大面积渐变、玻璃拟态、过度阴影。
- 卡片圆角不超过 8px。

## 色彩
- 背景使用浅灰白。
- 主色使用偏蓝绿色，表达理性和效率。
- 文本使用深灰，不使用纯黑。
- 边框轻，不要厚重。

## 布局
- 桌面端使用左侧导航 + 顶部工具栏 + 主内容区。
- 移动端导航收起到底部或抽屉。
- 重要操作放在右上角或主内容区顶部。

## 组件
- 按钮要有明确主次。
- 表单必须有 label、帮助文本和错误提示。
- 列表要支持空状态、加载状态和错误状态。
- 图标使用简洁线性风格。

## 文案
- 简短、直接，不要营销腔。
- 按钮使用动词，例如“新建文章”“保存草稿”。
```

# 4. 精细调整：利用 Figma 完成

这里的“精细调整”指的是：把 AI 生成的原型放到 Figma 中，进行人工可控的视觉修正、组件整理、设计系统沉淀和交付准备。

AI 适合快速生成方向，但 Figma 更适合做这些细节：

- 对齐
- 间距
- 字体层级
- 颜色变量
- 组件复用
- Auto Layout
- 设计 Token
- 响应式规则
- 原型连线
- 交付标注

简单说：

> AI 负责“先做出来”，Figma 负责“把它做规范”。

## 4.1 把 AI 原型导入 Figma

根据工具不同，导入方式可能不同：

- 如果 AI 工具支持 Figma 导出，优先导出到 Figma。
- 如果只生成图片，可以截图导入 Figma，然后照着重建关键组件。
- 如果生成 HTML/CSS，可以用插件或手动方式还原结构。
- 如果生成 React 代码，可以先在浏览器预览，再截图回 Figma 做视觉校准。

导入后不要急着改颜色，先做整理：

- 删除重复图层。
- 给页面和 Frame 正确命名。
- 把同一类页面放在一起。
- 把不同状态分组，例如 Default、Empty、Loading、Error。
- 检查页面尺寸是否符合目标设备。

## 4.2 用 Figma 修正布局和间距

AI 初稿最常见的问题是“整体还行，细节不齐”。在 Figma 里优先检查：

- 页面左右边距是否统一。
- 卡片之间的间距是否统一。
- 标题、正文、辅助文字是否有稳定层级。
- 图标和文字是否垂直居中。
- 表格列宽是否合理。
- 按钮高度是否一致。

建议先定义一套基础间距：

| 用途 | 建议值 |
| --- | --- |
| 小元素间距 | 4px / 8px |
| 表单内部间距 | 8px / 12px |
| 卡片内部间距 | 16px / 20px |
| 卡片之间间距 | 16px / 24px |
| 页面边距 | 24px / 32px |

非专业设计者可以记住一个原则：

> 间距不怕简单，怕不统一。

## 4.3 用 Figma 建立颜色和字体变量

如果只是一次性页面，可以直接调颜色；但如果要做一组页面，最好把颜色和字体沉淀成变量。

颜色建议至少分成：

- Background：页面背景
- Surface：卡片和面板背景
- Border：边框
- Text Primary：主要文本
- Text Secondary：辅助文本
- Primary：主操作色
- Danger：危险操作
- Success：成功状态
- Warning：警告状态

字体建议至少分成：

- Page Title
- Section Title
- Body
- Caption
- Button
- Table Header

这样后续改风格时，不需要一个个图层手动改，只要调整变量或样式即可。

## 4.4 用 Figma 组件化常用元素

AI 生成的页面里经常会出现很多“看起来一样但其实不一样”的按钮、卡片、输入框。进入 Figma 后，要把这些整理成组件。

优先组件化：

- Button
- Input
- Select
- Checkbox
- Tabs
- Card
- Table
- Sidebar
- Topbar
- Empty State
- Toast
- Modal

每个组件至少考虑这些状态：

- Default
- Hover
- Active
- Disabled
- Loading
- Error

组件化的目的不是为了显得专业，而是为了减少后续修改成本。比如主按钮高度要从 40px 改成 36px，如果没有组件，可能要改几十个按钮；有组件时只需要改一次。

## 4.5 用 Auto Layout 整理响应式

AI 生成的设计经常像一张静态海报，放大缩小后容易乱。Figma 精修时要用 Auto Layout 把页面变成更接近真实前端布局的结构。

重点处理：

- 横向排列的按钮组。
- 卡片列表。
- 表单项。
- 顶部导航。
- 侧边栏。
- 表格工具栏。
- 空状态内容。

检查响应式时至少看三个尺寸：

- 桌面端：1440px
- 平板或小桌面：1024px
- 移动端：375px / 390px

如果是后台系统，移动端可以不追求完整体验，但至少不能出现文字重叠、按钮超出、表格完全不可读。

## 4.6 用 Figma 做原型连线

AI 能生成页面，但页面之间的交互关系最好在 Figma 里明确下来。

常见连线：

- 仪表盘 -> 文章列表
- 文章列表 -> 编辑文章
- 编辑文章 -> 保存草稿提示
- 编辑文章 -> 发布成功弹窗
- 数据卡片 -> 数据详情
- 设置 -> 保存成功 Toast

原型连线的作用是让别人能像使用真实产品一样走一遍流程，而不是只看孤立页面截图。

## 4.7 Figma 精修检查清单

进入代码前，建议按这个清单检查：

- 页面结构是否清楚。
- 主操作按钮是否明显。
- 同类组件是否一致。
- 字号层级是否稳定。
- 间距是否统一。
- 图标风格是否一致。
- 表单是否有 label 和错误提示。
- 是否包含空状态、加载状态、错误状态。
- 移动端是否没有明显错位。
- 是否已经沉淀关键组件。
- 是否有颜色、字体、间距规范。
- 原型流程是否能走通。

# 5. 转为代码

Figma 精细调整完成后，再进入代码阶段。这个阶段可以继续用 AI，但 AI 的角色要变成“前端实现助手”，而不是“设计师”。

转代码有两条路线。

## 5.1 设计优先路线

```text
AI 原型 -> Figma 精修 -> Figma 组件/标注 -> 前端实现
```

这条路线适合：

- 团队协作。
- 需要设计稿评审。
- 页面较多。
- 后续还会持续迭代。
- 有设计系统或组件库。

## 5.2 代码优先路线

```text
AI 原型 -> Figma 快速校准 -> AI 生成代码 -> 浏览器截图回看 -> 继续修正
```

这条路线适合：

- 个人项目。
- 内部工具。
- MVP 验证。
- Landing Page。
- 简单后台页面。

## 5.3 代码生成提示词

```text
请根据 Figma 精修后的设计实现 React + TypeScript + Tailwind 页面。

要求：
- 优先复用项目已有组件。
- 拆分为 Sidebar、Topbar、MetricCard、ArticleTable、EmptyState 等组件。
- 使用语义化 HTML。
- 支持响应式布局。
- 按钮包含 hover、active、disabled、loading 状态。
- 表单包含 label、helper text、error message。
- 表格在移动端改成卡片列表。
- 不要引入新的 UI 库。
- 不要修改无关文件。
```

如果使用 Codex、Cursor 或 Claude Code，最好再补一句：

```text
请先阅读现有项目结构、样式规范和组件库，再制定实现计划。
实现时保持最小改动，完成后运行 lint 和测试。
```

# 6. 非UI设计师设计UI流程

1. 用 AI 整理产品 Brief。
2. 写一个简化版 `DESIGN.md`。
3. 让 AI 生成页面流程和页面清单。
4. 让 AI 生成低保真原型。
5. 让 AI 生成 3 到 5 个高保真方向。
6. 选择一个方向，让 AI 补齐多页面和状态。
7. 把 AI 原型导入 Figma。
8. 在 Figma 中整理页面、修正间距、建立颜色和字体变量。
9. 在 Figma 中组件化按钮、输入框、卡片、表格、导航等元素。
10. 在 Figma 中做响应式检查和原型连线。
11. 用 Figma 设计稿或截图指导 AI 生成前端代码。
12. 在浏览器中检查实现效果，再回到 Figma 或代码中修正。

这套流程里，AI 和 Figma 的分工很清楚：

| 阶段 | 主要工具 | 目标 |
| --- | --- | --- |
| 需求整理 | AI | 把想法变成清晰 Brief |
| 设计约束 | AI + DESIGN.md | 固定风格和边界 |
| 原型设计 | AI | 快速生成流程、页面和初稿 |
| 精细调整 | Figma | 修正视觉细节，建立组件和规范 |
| 代码实现 | AI 编码工具 + 前端项目 | 把设计落地为可运行页面 |

# 7. 常见误区

## 7.1 误区一：把 AI 原型当最终设计稿

AI 原型适合快速探索，但它通常会有细节问题，例如间距不统一、组件不一致、响应式不足、图层混乱。真正要交付时，仍然需要 Figma 精修。

## 7.2 误区二：在 AI 阶段过度纠结细节

AI 阶段应该重点看方向、结构和流程。如果一直让 AI 改“按钮再小一点、间距再窄一点”，效率反而会下降。这类细节更适合在 Figma 里调整。

## 7.3 误区三：Figma 只当截图容器

Figma 的价值不是把 AI 图片放进去，而是：

- 整理设计结构。
- 统一组件。
- 建立变量。
- 做响应式。
- 连接原型。
- 形成可交付规范。

## 7.4 误区四：直接复制 AI 代码上线

AI 生成的代码可能能跑，但不一定适合维护。上线前至少要检查：

- 组件拆分是否合理。
- 样式是否复用项目变量。
- 是否处理加载、错误、空状态。
- 是否支持响应式。
- 是否符合无障碍要求。
- 是否和现有项目结构一致。

# 8. 最终总结

非 UI 设计者利用 AI 做 UI，最重要的是不要把所有事情都交给一个工具。

更合理的分工是：

```text
AI：生成原型、探索方向、补齐页面和状态
Figma：精细调整、组件化、变量化、原型连线和设计交付
代码工具：把设计转成可运行页面
```

最终要记住这条主线：

```text
想法 -> Brief -> DESIGN.md -> AI 原型 -> Figma 精修 -> 代码实现
```

AI 让非设计者更快拥有一个可以讨论的初稿；Figma 让这个初稿变得统一、可靠、可交付；代码工具再把设计变成真正能运行的产品界面。
