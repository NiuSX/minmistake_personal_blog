# 10. 工程化与生产实践

## 从 Demo 到生产的差异

Demo 关注“看起来能跑”。生产系统关注：

- 稳定性。
- 可观测性。
- 可评测性。
- 权限与安全。
- 成本。
- 延迟。
- 用户体验。
- 运维和回滚。

Agent 的工程难点不在于调一次模型 API，而在于把不确定的模型行为放进可控的软件系统。

## 推荐服务架构

```text
Frontend / Chat UI / API
  ↓
Backend API
  ↓
Task Orchestrator
  ↓
Agent Runtime
  ↓
Model Gateway
  ↓
Tool Gateway
  ↓
Databases / Vector Store / External APIs
  ↓
Logs / Trace / Metrics / Eval Store
```

## Model Gateway

模型网关统一处理：

- 多模型路由。
- API Key 管理。
- 超时和重试。
- 限流。
- 成本统计。
- 输入输出日志。
- 模型版本记录。
- 降级策略。

好处是应用层不用散落大量模型提供方细节。

## Tool Gateway

工具网关统一处理：

- 工具注册。
- schema 管理。
- 权限校验。
- 参数校验。
- 审计日志。
- 幂等。
- 限流。
- 错误标准化。

工具网关是 Agent 安全的关键边界。

## 状态持久化

生产 Agent 不应只依赖内存变量。需要保存：

- 会话状态。
- 任务状态。
- 执行计划。
- 工具结果摘要。
- 审批状态。
- 错误信息。
- 成本统计。

这样才能支持刷新页面、后台执行、失败恢复和审计。

## 队列与异步任务

长任务应进入队列：

- 用户提交任务。
- API 返回 task_id。
- 后台 Worker 执行。
- 前端轮询或订阅进度。
- 完成后通知用户。

常见基础设施：

- Redis Queue。
- Celery。
- Sidekiq。
- Temporal。
- Kafka。
- 云厂商队列服务。

## 错误处理

错误类型：

- 用户输入错误。
- 权限错误。
- 模型格式错误。
- 模型拒答。
- 工具超时。
- 工具返回空。
- 外部系统错误。
- 预算耗尽。
- 安全拦截。

每类错误都应有明确处理策略，而不是统一返回“失败”。

## 重试策略

可以重试：

- 临时网络错误。
- 模型限流。
- 工具超时。
- 结构化输出格式轻微错误。

不应盲目重试：

- 权限不足。
- 参数明显非法。
- 高风险写操作。
- 用户取消。

重试必须有上限。

## 缓存

可缓存内容：

- 文档检索结果。
- embedding。
- 静态工具结果。
- 模型中间摘要。
- 路由分类结果。

不宜缓存：

- 带敏感权限的数据。
- 高实时性数据。
- 个性化结果。
- 会影响外部状态的工具结果。

## 成本控制

成本来自：

- 模型输入 token。
- 模型输出 token。
- 工具调用。
- 向量检索。
- rerank。
- 多 Agent 并行。
- 评测和反思。

控制方式：

- 设置单任务预算。
- 使用小模型做路由和分类。
- 压缩上下文。
- 限制最大轮次。
- 缓存可复用结果。
- 对高成本任务提前告知用户。

## 延迟优化

常见方法：

- 并行检索。
- 流式输出。
- 小模型快速路由。
- 工具结果分页。
- 提前加载用户上下文。
- 避免不必要反思。
- 将长任务异步化。

## 版本管理

必须记录：

- 模型版本。
- Prompt 版本。
- 工具 schema 版本。
- RAG 索引版本。
- Agent 运行时版本。
- 安全策略版本。

否则线上问题很难复现。

## 发布流程

建议流程：

1. 本地样例验证。
2. 离线评测集验证。
3. 小流量灰度。
4. 监控关键指标。
5. 扩大流量。
6. 保留快速回滚。

## 用户体验

Agent UX 应关注：

- 让用户知道任务进度。
- 长任务不要卡死界面。
- 需要确认时说明影响。
- 失败时给出可行动建议。
- 输出结果可复制、可追踪、可编辑。
- 不把大量内部日志暴露给用户。

## 生产上线清单

- 有明确任务边界。
- 有权限和租户隔离。
- 有工具 schema 和参数校验。
- 有最大步数、时间和成本限制。
- 有 trace 和日志。
- 有离线评测集。
- 有安全测试。
- 有人工审批机制。
- 有灰度和回滚。
- 有监控告警。
- 有数据脱敏策略。
- 有用户反馈入口。
