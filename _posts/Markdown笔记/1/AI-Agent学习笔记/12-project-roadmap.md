# 12. 项目路线与实战建议

## 总体路线

学习 Agent 最快的方式是做一个真实项目。建议按四个阶段推进：

1. 最小可用 Agent。
2. 工具增强 Agent。
3. 带 RAG 和记忆的 Agent。
4. 可评测、可观测、可上线的 Agent。

每个阶段都要留下可运行代码、测试样例和失败记录。

## 项目一：个人研究助手

### 目标

输入一个主题，Agent 自动搜索资料、整理来源、归纳观点并生成报告。

### 核心能力

- 查询改写。
- 搜索工具。
- 来源去重。
- 摘要。
- 引用。
- 反事实检查。

### 进阶能力

- 区分事实和推断。
- 标注来源可信度。
- 生成大纲后让用户确认。
- 支持长任务后台运行。

### 评测点

- 是否引用真实来源。
- 是否混入未支持结论。
- 是否覆盖主要观点。
- 是否能处理检索不到的情况。

## 项目二：企业知识库问答

### 目标

基于内部文档回答员工问题，并给出引用。

### 核心能力

- 文档解析。
- chunk 切分。
- 向量检索。
- 关键词检索。
- rerank。
- 权限过滤。
- 引用输出。

### 进阶能力

- 多版本文档处理。
- 问题改写。
- 答案置信度。
- 找不到证据时拒答。
- 用户反馈闭环。

### 评测点

- 检索召回率。
- 引用准确率。
- 答案正确率。
- 无证据拒答率。

## 项目三：代码库助手

### 目标

Agent 能阅读代码、解释模块、定位问题、修改小 bug 并运行测试。

### 核心能力

- 文件搜索。
- 代码阅读。
- 任务规划。
- 修改文件。
- 运行测试。
- 生成变更说明。

### 进阶能力

- 支持 Git diff。
- 自动选择相关测试。
- 修改前生成计划。
- 修改后自检。
- 不触碰无关文件。

### 评测点

- 是否理解代码结构。
- 是否最小化修改。
- 是否通过测试。
- 是否避免破坏用户已有改动。

## 项目四：工单处理 Agent

### 目标

自动分类工单、补全信息、查询知识库、生成回复草稿，并在必要时转人工。

### 核心能力

- 意图分类。
- 优先级判断。
- 信息抽取。
- 知识库检索。
- 回复生成。
- 人工转接。

### 进阶能力

- SLA 规则。
- 用户情绪识别。
- 工单去重。
- 内部备注。
- 质量抽检。

### 评测点

- 分类准确率。
- 回复可用率。
- 人工转接合理性。
- 是否泄露内部信息。

## 项目五：运营数据分析助手

### 目标

用户用自然语言提问，Agent 查询数据库、生成图表和解释。

### 核心能力

- SQL 生成。
- 查询权限控制。
- 结果解释。
- 图表建议。
- 异常值分析。

### 进阶能力

- SQL 审批。
- 只读数据库账号。
- 查询成本限制。
- 数据口径解释。
- 自动生成分析报告。

### 评测点

- SQL 正确性。
- 权限遵守。
- 数值准确性。
- 是否解释数据口径。

## 推荐实现顺序

### 第一步：写死工具，跑通循环

先实现最小 ReAct：

```text
用户输入 → 模型选择动作 → 调用工具 → 模型总结
```

只接入一两个低风险工具。

### 第二步：加入结构化状态

保存 task_id、plan、steps、tool_results 和 final_answer。

### 第三步：加入评测集

不要等项目大了再评测。早期就建立 20 到 50 条样例。

### 第四步：加入 trace

记录每次模型调用、工具调用和状态变化。

### 第五步：加入权限和审批

任何写操作都先做审批。

### 第六步：优化成本和体验

加入缓存、上下文压缩、流式输出和异步任务。

## 一周学习计划

### 第一天

阅读概念、架构和工具章节，手写一个最小工具调用 Agent。

### 第二天

加入结构化输出和错误处理。

### 第三天

接入 RAG，完成带引用的知识库问答。

### 第四天

加入计划、状态和最大步数。

### 第五天

加入 trace、评测集和回放。

### 第六天

加入权限、审批和注入测试。

### 第七天

整理项目 README、架构图、失败案例和改进计划。

## 最终作品要求

一个合格的 Agent 实战项目应包含：

- 清晰的任务边界。
- 可运行代码。
- 至少 3 个工具。
- 至少 20 条评测样例。
- trace 记录。
- 错误处理。
- 最大步数和成本限制。
- 高风险操作审批。
- README 和架构说明。
