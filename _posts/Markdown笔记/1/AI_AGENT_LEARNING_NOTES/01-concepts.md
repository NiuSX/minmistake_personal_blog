# 01. AI Agent 概念基础

## 什么是 AI Agent

AI Agent 是一种以大模型为核心推理组件的软件系统。它接收目标或任务，在上下文、工具、记忆和规则的约束下，通过多步观察、思考、行动和反馈来完成任务。

一个完整 Agent 至少包含以下要素：

- 目标：用户想完成的任务或系统预设的任务。
- 环境：Agent 可观察和影响的外部世界，例如网页、数据库、代码仓库、工单系统。
- 模型：负责理解、推理、决策、生成计划和解释结果。
- 工具：模型可以调用的外部能力，例如搜索、计算、代码执行、API、文件读写。
- 状态：当前任务进度、历史消息、工具返回结果、临时变量。
- 策略：何时规划、何时调用工具、何时停止、何时请求人工输入。
- 约束：权限、预算、时间、合规、安全边界。
- 反馈：工具结果、用户反馈、评测器反馈或环境变化。

## Agent 与普通 LLM 应用的区别

普通 LLM 应用通常是单轮或少轮问答。输入是用户问题，输出是文本或结构化结果。它的行为更多由 Prompt 和模型能力决定。

Agent 应用通常具有执行循环。它会根据中间结果动态决定下一步，可能调用多个工具，可能修改计划，也可能在不确定时请求澄清。

| 对比项 | 普通 LLM 应用 | AI Agent |
| --- | --- | --- |
| 目标 | 回答问题或生成内容 | 完成任务 |
| 执行方式 | 一次或少数几次模型调用 | 多步推理和行动 |
| 工具使用 | 可选，通常较简单 | 核心能力之一 |
| 状态管理 | 多依赖会话历史 | 显式保存任务状态 |
| 不确定性处理 | 输出解释或免责声明 | 重试、验证、请求人工确认 |
| 工程重点 | Prompt、检索、输出质量 | 控制流、权限、评测、可观测性 |

## Agent 的基本循环

经典 Agent 循环可以抽象为：

```text
接收目标
  ↓
理解任务
  ↓
制定计划
  ↓
选择动作
  ↓
调用工具或生成结果
  ↓
观察反馈
  ↓
更新状态
  ↓
判断是否完成
```

如果未完成，则继续下一轮；如果遇到风险、信息不足或预算不足，则暂停并请求用户输入。

## ReAct 模式

ReAct 是 Reasoning and Acting 的缩写，强调模型在推理和行动之间交替进行。它的核心思想是：

- 先根据任务和上下文思考下一步。
- 再选择一个动作，例如调用工具。
- 观察工具返回结果。
- 根据观察结果继续推理。

在工程中，不一定要把模型的完整思考过程暴露给用户。更推荐保存机器可读的 trace 和简短决策摘要，而不是展示内部推理细节。

## Agent 的能力边界

Agent 擅长：

- 目标明确但步骤不完全固定的任务。
- 需要结合多个工具或数据源的任务。
- 需要边执行边根据结果调整的任务。
- 需要自然语言理解、分类、摘要、生成或决策的任务。
- 低到中风险的自动化流程。

Agent 不擅长：

- 强实时、强确定性、低延迟任务。
- 高风险且不能容忍错误的操作。
- 权限边界不清晰的自动化。
- 数据质量差且没有验证机制的分析。
- 需要大量精确数学证明或复杂状态穷举的任务。

## 什么时候不该用 Agent

以下场景更适合传统程序、规则引擎或固定工作流：

- 任务流程完全确定，输入输出结构稳定。
- 错误成本极高，且无法设置人工审批。
- 工具调用成本高，但收益不明确。
- 只需要一次分类、抽取或摘要。
- 系统还没有日志、回滚、权限和审计基础设施。

判断标准很简单：如果用普通代码能稳定解决，就不要为了“Agent 化”而引入 Agent。

## Agent 的典型类型

### 任务型 Agent

目标是完成具体任务，例如“帮我整理会议纪要并创建待办事项”。这类 Agent 通常依赖工具调用和状态管理。

### 对话型 Agent

目标是长期服务用户，例如客服助手、学习助手、健康管理助手。它强调上下文、用户画像、记忆和安全边界。

### 研究型 Agent

目标是收集资料、比较观点、形成报告。它强调检索、来源可信度、引用、反事实检查和长任务管理。

### 代码型 Agent

目标是阅读、修改、测试和解释代码。它需要文件系统、终端、版本控制、测试框架和严格的变更边界。

### 操作型 Agent

目标是对真实系统执行操作，例如创建订单、修改配置、发送邮件。这类 Agent 风险最高，必须有权限控制、审批、审计和回滚机制。

## Agent 的成熟度等级

### L0：纯问答

没有工具，没有状态，只基于输入生成文本。

### L1：带工具的问答

模型可以调用一个或多个工具，但调用通常是一次性的。

### L2：单 Agent 多步执行

系统具备执行循环，能多轮调用工具并维护任务状态。

### L3：可控工作流 Agent

Agent 被嵌入确定性工作流中，关键节点由代码控制，模型负责语义判断、生成和异常处理。

### L4：多 Agent 协作

多个 Agent 分工协作，可能包含规划者、执行者、审查者、工具专家等角色。

### L5：自治系统

Agent 可长期运行、主动监控环境并触发任务。此类系统需要非常严格的治理机制，生产落地难度最高。

## Agent 工程的核心矛盾

Agent 的价值来自灵活性，但风险也来自灵活性。工程化的目标不是让模型“想做什么就做什么”，而是在清晰边界中释放模型能力。

常见平衡点：

- 自主性与可控性。
- 通用能力与任务稳定性。
- 长上下文与成本延迟。
- 多工具能力与权限风险。
- 多步规划与错误累积。
- 用户体验与安全审批。
