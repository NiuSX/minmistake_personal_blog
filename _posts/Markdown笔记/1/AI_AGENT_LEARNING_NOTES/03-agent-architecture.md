# 03. Agent 架构

## 标准架构视图

一个生产级 Agent 系统通常可以拆成以下层次：

```text
用户入口
  ↓
会话与身份层
  ↓
任务路由层
  ↓
Agent 控制器
  ↓
模型调用层
  ↓
工具执行层
  ↓
数据与记忆层
  ↓
评测、观测、安全、审计
```

不要把 Agent 理解为一个大函数。更准确的理解是：它是一个带模型决策节点的分布式应用。

## 主要组件

### 用户入口

可能是网页、聊天窗口、IDE 插件、CLI、浏览器扩展、企业 IM 或 API。入口层负责接收请求、展示进度、处理确认和反馈。

### 身份与权限

Agent 必须知道“代表谁执行”。同一个工具在不同用户身份下权限不同。例如同样是查询 CRM，销售只能查自己的客户，管理员可以查所有客户。

### 任务路由器

路由器判断请求类型：

- 普通问答。
- 知识库问答。
- 工具执行。
- 长任务。
- 高风险操作。
- 无法处理的请求。

路由可以由规则、分类模型或 LLM 完成。高风险场景建议规则优先，LLM 辅助。

### Agent 控制器

控制器是 Agent 的核心运行时，负责：

- 初始化任务状态。
- 选择策略。
- 调用模型。
- 解析模型动作。
- 执行工具。
- 更新状态。
- 判断停止条件。
- 处理异常。

### 模型调用层

模型调用层封装不同模型提供方，统一处理：

- 模型参数。
- 重试。
- 超时。
- 速率限制。
- token 统计。
- 结构化输出。
- 内容安全过滤。

### 工具执行层

工具执行层负责真正操作外部系统。它不应该盲目信任模型参数，而要校验 schema、权限和业务规则。

### 记忆与状态层

状态是当前任务运行所需的信息。记忆是跨任务、跨会话可复用的信息。二者应分开设计。

### 观测与审计层

生产系统必须记录：

- 用户请求。
- 模型输入输出摘要。
- 工具调用参数。
- 工具返回结果摘要。
- 错误和重试。
- 成本和耗时。
- 人工审批记录。

敏感字段应脱敏或加密保存。

## 单 Agent 架构

单 Agent 架构适合大多数早期项目。

优点：

- 实现简单。
- 调试容易。
- 责任清晰。
- 成本较低。

缺点：

- 复杂任务容易上下文膨胀。
- 一个 Prompt 承担过多规则。
- 难以并行。

适用场景：

- 单领域助手。
- 企业知识库问答。
- 代码库小范围修改。
- 自动化办公任务。

## Router + Specialist 架构

这种架构使用一个路由器把任务分配给不同专家 Agent 或处理器。

```text
用户请求
  ↓
Router
  ├─ FAQ Agent
  ├─ RAG Agent
  ├─ Data Analysis Agent
  ├─ Code Agent
  └─ Human Support
```

优点：

- 每个 Agent 更专注。
- Prompt 更短。
- 权限更容易隔离。
- 便于独立评测。

缺点：

- 路由错误会影响结果。
- 跨 Agent 状态共享复杂。
- 运维复杂度上升。

## Planner + Executor 架构

Planner 负责拆解任务，Executor 负责执行步骤。

```text
目标
  ↓
Planner 生成计划
  ↓
Executor 执行步骤
  ↓
Verifier 校验结果
  ↓
必要时返回 Planner 修订
```

适合：

- 研究报告。
- 代码修改。
- 数据分析。
- 多步骤业务流程。

工程要点：

- 计划必须结构化。
- 每一步要有完成条件。
- Executor 不应无限扩展任务范围。
- Verifier 应能拒绝不合格结果。

## Workflow + Agent 架构

生产系统常用工作流包裹 Agent。

```text
固定流程节点 A
  ↓
LLM 判断或生成
  ↓
固定流程节点 B
  ↓
工具调用
  ↓
人工审批
  ↓
固定流程节点 C
```

这种模式的优点是可控。模型只在需要语义能力的节点出现，而不是支配整个流程。

## Event-driven Agent 架构

事件驱动 Agent 监听外部事件：

- 新邮件到达。
- 工单状态变化。
- 监控报警触发。
- 文档更新。
- 用户上传文件。

处理流程：

```text
事件进入队列
  ↓
规则过滤
  ↓
Agent 分析
  ↓
生成建议或执行动作
  ↓
记录审计
```

此类系统必须设置去重、幂等、限流和人工审批。

## Agent 状态机

一个可控 Agent 可以设计成状态机：

```text
CREATED
  ↓
NEED_CLARIFICATION
  ↓
PLANNING
  ↓
EXECUTING
  ↓
WAITING_APPROVAL
  ↓
VERIFYING
  ↓
COMPLETED / FAILED / CANCELLED
```

状态机的好处：

- 易于恢复。
- 易于展示进度。
- 易于审计。
- 易于限制非法跳转。

## 停止条件

Agent 必须有明确停止条件：

- 达到最终答案。
- 达到最大步数。
- 达到最大 token 或成本。
- 达到最大执行时间。
- 连续失败次数过多。
- 用户取消。
- 需要人工确认。
- 检测到风险。

没有停止条件的 Agent 很容易陷入循环调用工具。

## 架构设计检查清单

- 是否区分了模型决策和系统控制？
- 是否有显式任务状态？
- 是否有工具权限校验？
- 是否有最大步数、最大耗时和最大成本？
- 是否支持失败恢复？
- 是否记录 trace？
- 是否能回放一次任务？
- 是否能解释最终答案来自哪些工具或资料？
- 是否能在风险操作前暂停？
