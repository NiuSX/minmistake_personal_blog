# 04. 工具调用、Function Calling 与 MCP

## 工具为什么重要

LLM 的内部知识可能过时，也无法直接操作外部系统。工具让 Agent 能够：

- 获取实时信息。
- 查询私有数据。
- 执行计算。
- 调用业务 API。
- 读写文件。
- 操作浏览器或应用。
- 调用搜索、数据库、代码执行器等能力。

没有工具的 Agent 只能“说”。有工具的 Agent 才可能“做”。

## 工具的基本结构

一个工具通常包含：

- 名称：模型用于选择工具。
- 描述：说明工具适合什么场景。
- 输入 schema：参数名、类型、必填项、枚举和约束。
- 输出 schema：返回值结构。
- 权限策略：谁能调用，能访问哪些资源。
- 副作用说明：是否会修改外部系统。
- 错误语义：失败时返回什么。

示例：

```json
{
  "name": "search_docs",
  "description": "在内部技术文档中检索相关内容。",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": { "type": "string" },
      "top_k": { "type": "integer", "minimum": 1, "maximum": 10 }
    },
    "required": ["query"]
  }
}
```

## 好工具的设计原则

### 单一职责

一个工具只做一类事情。不要设计万能工具，例如 `execute_anything`。万能工具难以描述、难以授权、难以审计。

### 参数清晰

参数应尽量结构化。避免让模型把大段自然语言塞给工具再由工具猜测意图。

### 返回值可读

工具返回结果应该短、结构化、可解释。过长结果会污染上下文。

### 显式错误

错误信息应区分：

- 参数错误。
- 权限不足。
- 资源不存在。
- 外部服务超时。
- 业务规则不允许。

### 副作用可控

有副作用的工具，例如发邮件、转账、删除文件、修改配置，必须支持审批、幂等和审计。

## Function Calling

Function Calling 是让模型选择函数并生成参数的一种机制。运行时根据模型输出调用真实函数。

基本流程：

```text
用户请求
  ↓
模型判断需要调用工具
  ↓
模型生成工具名和参数
  ↓
运行时校验参数
  ↓
运行时执行工具
  ↓
工具结果返回模型
  ↓
模型继续回答或继续调用工具
```

关键点：模型只是建议调用什么工具和传什么参数，真正执行前必须由代码校验。

## 工具调用的权限模型

权限至少分三层：

- 工具级权限：用户是否能使用这个工具。
- 资源级权限：用户能访问哪些数据或对象。
- 动作级权限：用户能读、写、删除还是审批。

示例：

```text
用户可以调用 query_invoice 工具。
用户只能查询自己部门的发票。
用户不能调用 approve_payment。
```

## 读工具与写工具

工具可以按风险分为：

- 只读工具：搜索、查询、读取文件。
- 低风险写工具：创建草稿、生成待办、保存临时结果。
- 高风险写工具：发送外部消息、删除数据、修改权限、支付、发布内容。

高风险写工具必须采用 Human-in-the-loop。

## 工具结果压缩

工具返回结果不应无节制塞进上下文。常见做法：

- 返回结构化摘要。
- 返回命中文档 ID 和片段。
- 大文件只返回分页或片段。
- 把完整结果存储在外部，只把引用放入上下文。
- 对表格数据先聚合再返回。

## MCP 的定位

MCP，全称 Model Context Protocol，是一种连接 AI 应用与外部上下文、工具和数据源的开放协议。它的目标是让模型应用以统一方式接入文件系统、数据库、浏览器、业务系统和开发工具等能力。

可以把 MCP 理解为“AI 应用的工具和上下文适配层”。它不等于 Agent 框架，也不等于模型本身，而是连接 Agent 和外部系统的一种协议。

## MCP 的基本角色

MCP 通常包含：

- Host：用户直接使用的 AI 应用，例如桌面助手、IDE、聊天应用。
- Client：Host 内部用于连接某个 MCP Server 的组件。
- Server：暴露工具、资源和提示模板的服务。
- Tools：可被模型调用的动作。
- Resources：可读取的上下文数据。
- Prompts：可复用的提示模板。

## MCP 与 Function Calling 的关系

Function Calling 更偏模型 API 层面的工具调用能力。

MCP 更偏应用和外部工具之间的协议层。MCP Server 可以把能力暴露给 Host，Host 再把这些能力以模型可理解的方式提供给 LLM。

关系可以理解为：

```text
LLM Function Calling
  ↓
Agent Runtime
  ↓
MCP Client
  ↓
MCP Server
  ↓
外部系统
```

## 什么时候适合使用 MCP

适合：

- 需要接入多个工具源。
- 希望工具能力在不同 AI 应用间复用。
- 要把本地文件、数据库、浏览器或开发工具暴露给 Agent。
- 团队希望统一工具协议，而不是每个应用重复封装。

不一定需要：

- 只有一两个内部 API。
- 工具能力只服务一个简单应用。
- 系统安全边界还没有设计清楚。
- 只需要普通 RAG 检索。

## 工具调用失败处理

Agent 必须处理工具失败：

- 参数不合法：让模型修正参数或向用户澄清。
- 权限不足：停止并说明权限问题。
- 服务超时：重试有限次数。
- 结果为空：换查询策略或说明未找到。
- 返回冲突：请求人工判断或使用验证器。

不要让模型无限重试。

## 工具测试

工具测试至少包含：

- schema 校验。
- 权限测试。
- 正常输入。
- 缺失字段。
- 非法字段。
- 边界值。
- 外部服务失败。
- 高风险操作审批。

## 工具设计检查清单

- 工具名称是否清晰？
- 工具描述是否说明了适用和不适用场景？
- 参数 schema 是否足够严格？
- 返回结果是否短且结构化？
- 是否区分只读和写操作？
- 是否有权限校验？
- 是否有幂等键？
- 是否记录审计日志？
- 是否限制调用频率？
- 是否能在失败时返回可恢复错误？
