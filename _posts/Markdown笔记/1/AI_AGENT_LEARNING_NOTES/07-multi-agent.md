# 07. 多智能体系统

## 什么是多智能体

多智能体系统是由多个 Agent 协同完成任务的系统。每个 Agent 可以有不同角色、工具、记忆和权限。

常见角色：

- Planner：规划任务。
- Researcher：收集资料。
- Coder：修改代码。
- Reviewer：审查结果。
- Critic：提出反例和风险。
- Executor：执行工具。
- Coordinator：协调多个 Agent。

## 什么时候需要多 Agent

适合使用多 Agent：

- 任务天然需要分工。
- 需要相互审查。
- 可以并行处理多个子任务。
- 不同角色需要不同工具或权限。
- 希望用辩论、投票或竞争提升质量。

不适合使用多 Agent：

- 单 Agent 已能稳定完成。
- 任务很短。
- 成本和延迟敏感。
- 缺乏清晰评测标准。
- 角色只是 Prompt 上的装饰，没有实际分工。

## 常见协作模式

### Supervisor 模式

一个主管 Agent 分配任务给多个子 Agent。

```text
Supervisor
  ├─ Research Agent
  ├─ Code Agent
  └─ Review Agent
```

优点是控制清晰。缺点是 Supervisor 可能成为瓶颈。

### Debate 模式

多个 Agent 对同一问题给出观点，再由 Judge 汇总。

适合：

- 方案评审。
- 风险分析。
- 复杂判断。

不适合：

- 事实查询。
- 简单任务。
- 对延迟敏感的场景。

### Pipeline 模式

任务按阶段传递：

```text
Researcher → Writer → Editor → Verifier
```

适合内容生成、报告整理和代码修改。

### Blackboard 模式

多个 Agent 共享一个工作区，读取和写入中间结果。

适合探索性任务，但需要严格的写入规范和冲突控制。

### Market / Voting 模式

多个 Agent 生成候选结果，通过投票、打分或规则选择最佳结果。

适合开放式生成和方案比较。

## 多 Agent 通信

通信内容应结构化：

```json
{
  "from": "researcher",
  "to": "writer",
  "task_id": "report-001",
  "summary": "已完成三家竞品资料收集",
  "artifacts": ["doc-1", "doc-2"],
  "open_questions": ["第三家竞品价格页不可访问"]
}
```

不要让 Agent 之间无边界地聊天。通信越自由，越难评测和调试。

## 共享状态

共享状态应包含：

- 当前任务目标。
- 角色职责。
- 已完成产物。
- 待处理问题。
- 决策记录。
- 冲突记录。
- 版本号。

共享状态要避免被某个 Agent 随意覆盖。可以采用 append-only 日志，再由 Coordinator 汇总。

## 角色漂移

角色漂移是指 Agent 偏离自己的职责。例如 Reviewer 开始重写全部内容，Researcher 开始做最终决策。

缓解方式：

- 每个 Agent 只暴露必要工具。
- 每个角色 Prompt 明确职责边界。
- Coordinator 校验输出类型。
- 使用结构化消息。
- 设置最大轮次。

## 冲突解决

多 Agent 可能给出冲突结论。解决方式：

- 优先证据来源更可靠的一方。
- 使用规则判断。
- 调用外部验证工具。
- 交给 Judge Agent。
- 请求人工决策。

对事实问题，不应通过简单投票决定。多个 Agent 都可能错。

## 成本与延迟

多 Agent 会显著增加：

- 模型调用次数。
- 上下文长度。
- 工具调用次数。
- Trace 复杂度。
- 调试成本。

因此多 Agent 系统必须设置预算。

## 多 Agent 评测

评测维度：

- 最终任务成功率。
- 子任务完成率。
- Agent 间通信是否有效。
- 轮次是否过多。
- 是否出现重复工作。
- 是否出现角色漂移。
- 是否正确处理冲突。
- 成本是否可接受。

## 实用建议

- 先实现单 Agent，再拆多 Agent。
- 拆分依据应是职责、工具和权限，而不是为了显得复杂。
- 每个 Agent 都要有可测试输入输出。
- Coordinator 应掌握停止条件。
- 高风险工具不要暴露给所有 Agent。
- 多 Agent 最终要汇聚到一个明确结果。

## 检查清单

- 是否真的需要多 Agent？
- 每个 Agent 的职责是否清晰？
- 是否限制每个 Agent 的工具权限？
- 通信是否结构化？
- 是否设置最大轮次？
- 是否有冲突解决机制？
- 是否能追踪每个 Agent 的贡献和错误？
