# 连接、超时、重试与保活学习笔记

最后整理：2026-06-14

Last researched：2026-06-14

很多网络问题不是“协议不懂”，而是超时、重试、连接池、Keepalive、心跳、NAT 超时、代理超时之间没有设计清楚。无论 HTTP、RPC、MQTT、WebSocket、数据库连接还是工业网关，都需要认真处理连接生命周期。

## 学习目标

- 分清连接超时、读超时、写超时、请求总超时、空闲超时。
- 理解 TCP Keepalive 和应用层心跳的区别。
- 理解重试的收益、风险、幂等性和退避策略。
- 能排查长连接断开、连接池耗尽、TIME_WAIT/CLOSE_WAIT、端口耗尽问题。

## 常见超时类型

| 超时 | 含义 | 典型问题 |
|---|---|---|
| DNS 超时 | 域名解析等待时间 | DNS 服务异常导致请求没开始 |
| Connect Timeout | 建立 TCP/QUIC 连接等待时间 | 网络不通、防火墙丢包、服务不可达 |
| TLS Handshake Timeout | TLS 握手等待时间 | 证书、算法、服务端负载、中间设备问题 |
| Write Timeout | 写请求数据等待时间 | 对端接收慢、发送缓冲阻塞 |
| Read Timeout | 等响应数据等待时间 | 服务处理慢、上游卡住 |
| Request Timeout | 整个请求生命周期上限 | 避免局部阶段无限拖延 |
| Idle Timeout | 连接空闲多久被关闭 | 连接池、代理、NAT、负载均衡常见 |
| Keepalive Timeout | 保活探测间隔和失败次数 | 发现半开连接 |

生产系统不要只设置一个模糊的 timeout，要明确每个阶段的含义。

## TCP Keepalive 与应用心跳

| 对比 | TCP Keepalive | 应用层心跳 |
|---|---|---|
| 所在层 | 传输层 | 应用层 |
| 是否携带业务语义 | 否 | 可以 |
| 默认间隔 | 操作系统默认可能很长 | 应用自己定义 |
| 能否证明业务可用 | 不能 | 可以部分证明 |
| 常见用途 | 发现死连接、保持 NAT 映射 | WebSocket、MQTT、RPC、设备在线状态 |

应用层心跳可以证明“协议栈和应用逻辑仍在工作”，但太频繁会增加负载。

## 重试设计

适合重试：

- 网络瞬断；
- DNS 临时失败；
- 连接复用中的旧连接失效；
- HTTP 502/503/504；
- 幂等请求失败。

谨慎重试：

- 非幂等写操作，例如扣款、下单、执行设备动作；
- 服务端已经收到请求但客户端没收到响应；
- 大规模故障下所有客户端同时重试；
- 工业控制命令可能重复执行。

## 幂等性

| 操作 | 是否通常幂等 | 说明 |
|---|---|---|
| GET 查询 | 是 | 不应改变服务端状态 |
| PUT 设置资源为某状态 | 通常是 | 重复设置同一状态结果一致 |
| DELETE 删除资源 | 通常是 | 删除不存在可视为最终一致 |
| POST 创建订单 | 通常不是 | 需要幂等键 |
| Modbus 写单寄存器为固定值 | 通常是 | 重复写同值结果一致 |
| Modbus 写触发动作寄存器 | 可能不是 | 重复写可能重复执行动作 |

常见做法：

- 请求携带 Idempotency-Key。
- 业务使用唯一命令 ID。
- 服务端记录已处理请求。
- 响应丢失时客户端查询最终状态，而不是盲目重放。

## 退避与抖动

重试间隔不要固定同时发生，否则故障时容易形成重试风暴。

```text
第一次失败：100ms 后重试
第二次失败：200ms 后重试
第三次失败：400ms 后重试
继续指数增长，上限 5s 或 30s
每次加入随机抖动 jitter
```

原则：

- 设置最大重试次数。
- 设置总超时时间。
- 区分可重试和不可重试错误。
- 服务端限流时尊重 `Retry-After`。
- 对消息队列使用死信队列或延迟队列。

## 连接池

连接池复用 TCP/TLS 连接，减少握手成本，但会引入生命周期问题。

| 参数 | 含义 |
|---|---|
| Max Connections | 最大连接数 |
| Max Idle Connections | 最大空闲连接数 |
| Idle Timeout | 空闲连接保留时间 |
| Max Lifetime | 连接最大寿命 |
| Acquire Timeout | 从连接池获取连接的等待时间 |

常见问题：

- 连接池太小导致请求排队。
- 连接池太大压垮服务端或耗尽端口。
- 空闲连接被负载均衡关闭，客户端复用时失败。
- 没有读完响应体导致连接无法复用。
- DNS 变更后旧连接长期连到旧后端。

## TIME_WAIT、CLOSE_WAIT、端口耗尽

| 状态 | 常见含义 | 问题判断 |
|---|---|---|
| TIME_WAIT 很多 | 主动关闭方保留连接状态 | 不一定是问题，高并发短连接下常见 |
| CLOSE_WAIT 很多 | 对端已关闭，本地应用未关闭 socket | 常见应用资源泄漏 |
| SYN_SENT 很多 | 连接对外建立中 | 目标不通或防火墙丢弃 |
| ESTABLISHED 很多 | 正常连接或泄漏 | 结合应用连接池判断 |

端口耗尽常见于大量短连接、未复用连接、NAT 出口端口不足、连接关闭慢。

## NAT、代理和负载均衡超时

长连接路径中任何一处空闲超时低于应用心跳间隔，都可能断开连接。

建议：

- 应用心跳间隔小于链路中最短 idle timeout。
- 服务端能识别半开连接。
- 客户端能自动重连并恢复订阅/状态。
- 重要操作有请求 ID 和补偿查询。

## 常见问题

| 现象 | 可能原因 | 排查方向 |
|---|---|---|
| 偶发请求第一次失败，重试成功 | 连接池复用到已被关闭的空闲连接 | 调整 idle timeout，增加健康检查 |
| 大量 CLOSE_WAIT | 应用未关闭 socket/响应体 | 查代码资源释放 |
| 大量 TIME_WAIT | 短连接高频主动关闭 | 开启连接复用，调整架构 |
| WebSocket 每 60s 断开 | 代理 idle timeout | 调整代理超时或心跳 |
| MQTT 设备离线 | Keep Alive 与 NAT 超时不匹配 | 缩短 Keep Alive 或应用心跳 |
| 重试导致服务雪崩 | 无退避、无熔断、全量客户端同时重试 | 指数退避、限流、熔断 |
| 工业命令重复执行 | 非幂等动作被重试 | 命令 ID、状态查询、去重 |

## 排查命令

```bash
ss -tanp
ss -s
cat /proc/sys/net/ipv4/tcp_keepalive_time
tcpdump -i any -nn host <target>
curl -v --connect-timeout 3 --max-time 10 https://example.com
```

Windows：

```powershell
netstat -ano
Test-NetConnection example.com -Port 443
```
