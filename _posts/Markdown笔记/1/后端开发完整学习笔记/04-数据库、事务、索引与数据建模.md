## 讲义导读：把后端当成一条请求生命线

这一章讲的是 **数据库、事务、索引与数据建模**。阅读时不要只背框架名、组件名或面试题答案，而要把每个概念放回一条请求生命线里：请求如何进入系统，如何被认证和校验，业务规则在哪里执行，数据如何保持一致，慢操作如何异步化，故障如何被观测，变更如何安全上线。后端学习的目标不是堆技术栈，而是能设计、实现、排查和维护一个长期运行的业务系统。

### 一句话先懂

数据库学习的核心是数据状态：表结构表达事实，事务保护状态变化，索引加速查询，但每个选择都会影响一致性、性能和演进成本。

### 通俗类比

数据库像账本和仓库合体：账本要记得准，仓库要找得快，出入库要成套完成，不能只改库存不记流水。

类比只是帮助建立第一印象。回到工程上，要把类比里的入口、调度、仓库、通道、监控和维护分别对应到 API、业务层、数据库、缓存、消息队列、可观测性和部署运维。后端概念只有放进真实链路，才知道它解决的是正确性、性能、安全、可靠性、可维护性还是成本问题。

### 本章学习主线

1. **先看职责**：这个概念负责处理请求链路里的哪一段，输入和输出是什么。
2. **再看边界**：它不负责什么，哪些问题应该交给数据库、缓存、队列、网关、客户端或运维平台。
3. **然后看失败**：超时、重复、乱序、并发、脏数据、权限绕过、容量耗尽时会发生什么。
4. **接着看验证**：怎样用单元测试、集成测试、压测、日志、指标、trace 或故障演练证明设计可靠。
5. **最后看演进**：需求变更、流量增长、团队协作和版本升级时，这个设计是否还能维护。

### 概念怎么学才不容易忘

遇到后端概念时，建议按 白话职责 -> 链路位置 -> 最小例子 -> 常见事故 -> 观测信号 -> 修复策略 六步理解。比如缓存不是加速器这么简单，还要看命中率、TTL、一致性、热点 key 和失效策略；消息队列不是异步这么简单，还要看确认、重试、幂等、积压和补偿；JWT 不是登录态这么简单，还要看签名、过期、撤销、泄露和权限边界。

### 最小实践任务

为订单、订单项、支付单设计表结构和索引，写出创建订单事务边界，并解释哪些查询会走索引、哪些会慢。

实践时要故意设计失败场景：重复请求、数据库超时、缓存失效、消息重复、权限不足、发布回滚、下游服务不可用。后端能力往往不是在正常路径里体现，而是在异常路径里体现。

## 1. 数据库在后端中的位置

数据库是后端系统的权威数据源。缓存可以丢，消息可以重试，服务可以重启，但核心业务数据必须可靠保存。

后端依赖数据库完成：

- 保存业务状态。
- 查询业务对象。
- 保证唯一约束。
- 执行事务。
- 进行并发控制。
- 支持审计和追溯。
- 支持报表和统计。
- 支持备份恢复。

数据库设计错误会长期影响系统。代码可以重构，接口可以加版本，缓存可以重建，但错误的数据模型和混乱的数据状态往往最难修。

## 2. 关系型数据库

关系型数据库使用表、行、列表达结构化数据，并通过 SQL 查询。

### 2.1 核心概念

| 概念 | 含义 |
| --- | --- |
| Table | 表，表示一类数据 |
| Row | 行，一条记录 |
| Column | 列，一个字段 |
| Primary Key | 主键，唯一标识一行 |
| Foreign Key | 外键，表达表间关系 |
| Index | 索引，加速查询 |
| Constraint | 约束，保证数据合法 |
| Transaction | 事务，一组原子操作 |
| View | 视图，虚拟查询结果 |

### 2.2 适合场景

关系型数据库适合：

- 结构清晰的数据。
- 强一致业务。
- 复杂查询。
- 多条件筛选。
- 需要事务的场景。
- 需要约束保障的核心数据。

例如订单、支付、账户、库存、用户、权限、合同、审批流程通常适合关系型数据库。

### 2.3 不适合场景

关系型数据库不适合：

- 大量非结构化文件，例如图片、视频。
- 高维向量检索。
- 超大规模日志原文检索。
- 极高吞吐且关系很弱的事件流。
- 字段结构极度动态且查询不确定的数据。

这些场景可以考虑对象存储、搜索引擎、时序数据库、文档数据库、列式数据库等。

## 3. NoSQL

NoSQL 不是一种数据库，而是一类非关系型数据存储。

| 类型 | 数据模型 | 适合场景 |
| --- | --- | --- |
| Key-Value | key 到 value | 缓存、会话、计数 |
| Document | JSON-like 文档 | 内容、配置、用户画像 |
| Wide Column | 行键 + 列族 | 大规模写入、日志、时序 |
| Graph | 点和边 | 社交关系、路径分析 |
| Search Engine | 倒排索引 | 全文搜索、日志检索 |
| Time Series | 时间戳 + 指标 | 监控、IoT、行情 |
| Object Storage | 对象和元数据 | 图片、视频、文件、备份 |

选择 NoSQL 时，不要只看“性能高”。必须确认：

- 查询模式是否适配。
- 一致性模型是什么。
- 是否支持事务。
- 是否支持二级索引。
- 分片如何做。
- 运维和备份如何做。
- 数据增长后成本如何。

## 4. 数据建模

数据建模不是简单建表，而是把业务对象、关系、状态和约束表达成可持久化结构。

### 4.1 建模步骤

1. 找核心实体：用户、订单、商品、支付、库存。
2. 找实体关系：一对一、一对多、多对多。
3. 找业务状态：订单状态、支付状态、审核状态。
4. 找不变量：金额不能为负、订单号唯一、库存不能超卖。
5. 找查询场景：详情、列表、搜索、统计、导出。
6. 找写入场景：创建、更新、删除、批量导入。
7. 找数据增长：哪些表增长快，是否需要归档。
8. 找一致性边界：哪些数据必须同事务更新。

### 4.2 主键设计

主键用于唯一标识记录。

常见主键：

| 类型 | 优点 | 缺点 |
| --- | --- | --- |
| 自增 ID | 简单、索引局部性好 | 分布式生成困难，暴露规模 |
| UUID | 去中心化生成 | 长、索引随机、可读性差 |
| 雪花类 ID | 趋势递增、分布式 | 依赖时钟和机器 ID |
| 业务 ID | 业务可读 | 变更风险高，不一定稳定 |

建议：

- 核心表主键应稳定不可变。
- 不要用可能变化的字段做主键，例如手机号、邮箱。
- 对外暴露 ID 时考虑枚举风险。
- 分布式系统要提前规划 ID 生成。

### 4.3 唯一约束

唯一约束是后端幂等和数据正确性的最后防线。

场景：

- 用户名唯一。
- 手机号唯一。
- 订单号唯一。
- 支付流水号唯一。
- 同一用户同一活动只能领一次券。

不要只靠应用层查询后判断是否存在，因为并发下两个请求可能同时通过检查。最终必须有数据库唯一约束兜底。

## 5. 范式与反范式

### 5.1 范式

范式用于减少数据冗余和更新异常。

简单理解：

- 一个事实只存一处。
- 表达清楚实体关系。
- 避免重复字段导致不一致。

例如用户昵称如果在评论表里也冗余一份，用户改昵称后历史评论是否同步就是业务问题。

### 5.2 反范式

反范式是为了查询性能或业务便利主动冗余数据。

例子：

- 订单保存商品名称和价格快照。
- 评论冗余用户昵称和头像。
- 报表表提前汇总统计结果。

反范式不是坏事，但要明确：

- 冗余字段谁负责更新？
- 允许不一致多久？
- 如何修复不一致？
- 是否需要定期对账？

## 6. 索引

索引是数据库性能的核心。

### 6.1 索引为什么快

没有索引时，数据库可能需要扫描整张表。索引通过额外数据结构帮助数据库快速定位记录。

代价：

- 占用存储空间。
- 写入、更新、删除时需要维护索引。
- 索引过多会拖慢写入。
- 不合适的索引可能完全用不上。

### 6.2 常见索引类型

| 类型 | 用途 |
| --- | --- |
| 主键索引 | 按主键查找 |
| 唯一索引 | 加速查询并保证唯一 |
| 普通索引 | 加速常见查询 |
| 联合索引 | 多列组合查询 |
| 覆盖索引 | 查询字段都在索引中，减少回表 |
| 全文索引 | 文本搜索 |
| 空间索引 | 地理空间查询 |

### 6.3 联合索引

联合索引顺序非常重要。

假设索引：

```text
(tenant_id, status, created_at)
```

适合：

```text
WHERE tenant_id = ? AND status = ? ORDER BY created_at
WHERE tenant_id = ?
```

不一定适合：

```text
WHERE status = ?
WHERE created_at > ?
```

这和数据库索引结构、最左前缀、选择性、执行计划有关。实际要看具体数据库执行计划。

### 6.4 索引设计原则

- 为高频查询建索引。
- 为高选择性字段建索引。
- 让索引服务过滤、排序、分页。
- 避免对频繁变化字段建太多索引。
- 关注联合索引顺序。
- 定期清理无用索引。
- 使用执行计划验证。

### 6.5 常见索引失效原因

不同数据库实现不同，但常见问题包括：

- 对索引列使用函数导致无法利用索引。
- 隐式类型转换。
- 前缀通配模糊查询。
- OR 条件不合理。
- 统计信息过期。
- 查询返回比例太高，数据库选择全表扫描。
- 联合索引不满足前缀。

## 7. 事务

事务保证一组操作作为整体执行。

### 7.1 ACID

| 特性 | 含义 |
| --- | --- |
| Atomicity 原子性 | 全部成功或全部失败 |
| Consistency 一致性 | 事务前后满足约束 |
| Isolation 隔离性 | 并发事务相互隔离到一定程度 |
| Durability 持久性 | 提交后数据持久保存 |

### 7.2 事务不是越大越好

大事务问题：

- 锁持有时间长。
- 回滚成本高。
- 日志量大。
- 阻塞其他事务。
- 主从复制延迟。
- 容易超时。

事务中不要做：

- 慢 HTTP 调用。
- 等待用户输入。
- 大文件处理。
- 长时间批量任务。
- 不可回滚的外部副作用。

## 8. 隔离级别与并发异常

### 8.1 常见并发异常

| 异常 | 含义 |
| --- | --- |
| 脏读 | 读到未提交数据 |
| 不可重复读 | 同事务两次读同一行结果不同 |
| 幻读 | 同事务两次范围查询出现新行或少行 |
| 丢失更新 | 两个事务覆盖彼此修改 |
| 写偏斜 | 多事务分别合法，合起来破坏约束 |

### 8.2 隔离级别

| 隔离级别 | 特点 |
| --- | --- |
| Read Uncommitted | 可能读到未提交数据，很少用于业务系统 |
| Read Committed | 只能读提交数据，但同事务两次读可能不同 |
| Repeatable Read | 同事务中重复读更稳定 |
| Serializable | 最接近串行执行，并发性能最低 |

不同数据库对隔离级别实现不完全相同。学习和排障时要查具体数据库官方文档。

## 9. MVCC

MVCC 是多版本并发控制。它通过保存数据的多个版本，让读写可以更好并发。

直觉：

- 读请求看到某个时间点的一致快照。
- 写请求创建新版本。
- 老版本在没有事务需要后被清理。

MVCC 的好处：

- 读不阻塞写。
- 写不阻塞普通快照读。
- 提升并发。

代价：

- 需要维护多版本。
- 需要清理旧版本。
- 长事务会阻止旧版本回收。
- 存储可能膨胀。

## 10. 锁

数据库锁用于并发控制。

常见锁：

| 锁 | 含义 |
| --- | --- |
| 行锁 | 锁定某些行 |
| 表锁 | 锁定整张表 |
| 间隙锁 | 锁定索引范围间隙 |
| 意向锁 | 表示将要锁某些行 |
| 共享锁 | 多个事务可读 |
| 排他锁 | 独占修改 |

后端排查锁问题要关注：

- 哪个事务持锁。
- 持锁多久。
- 等待锁的 SQL。
- 是否有大事务。
- 是否有缺失索引导致锁范围扩大。

## 11. 慢查询

慢查询常见原因：

- 没有索引。
- 索引不合适。
- 返回数据太多。
- 深分页。
- N+1 查询。
- 复杂 join。
- 排序或分组无法利用索引。
- 锁等待。
- 数据量增长后执行计划变化。

### 11.1 深分页

`OFFSET 100000 LIMIT 20` 可能很慢，因为数据库要跳过大量记录。

优化：

- 使用游标分页。
- 使用主键或时间作为 seek 条件。
- 限制最大页数。
- 对导出任务异步化。

### 11.2 N+1 查询

示例：

1. 查询 100 个订单。
2. 每个订单再查一次用户。
3. 总共 101 次查询。

优化：

- 批量查询。
- join。
- 预加载。
- 缓存。

## 12. 读写分离

读写分离通过主库写、从库读扩展读取能力。

优点：

- 减轻主库读压力。
- 提升读吞吐。

问题：

- 主从延迟。
- 刚写完读从库可能读不到。
- 事务内读写一致性复杂。
- 故障切换后路由要更新。

解决读己之写：

- 写后短时间读主库。
- 根据版本或时间判断。
- 业务允许最终一致。

## 13. 分库分表

当单库单表无法承载数据量或写入压力时，考虑分库分表。

### 13.1 垂直拆分

按业务或字段拆：

- 用户库。
- 订单库。
- 支付库。
- 商品库。

优点：

- 业务边界清晰。
- 减少单库压力。

问题：

- 跨库查询复杂。
- 事务边界变化。

### 13.2 水平拆分

按分片键把同一类数据拆到多个库或表。

分片键选择：

- 高频查询条件。
- 分布均匀。
- 不易变化。
- 能减少跨分片查询。

常见分片键：

- 用户 ID。
- 租户 ID。
- 订单 ID。
- 时间。

### 13.3 分库分表代价

- 跨分片分页难。
- 跨分片 join 难。
- 全局唯一 ID。
- 分布式事务。
- 扩容迁移。
- 热点分片。
- 运维复杂。

不要过早分库分表。先考虑索引、缓存、归档、读写分离、业务拆分。

## 14. 数据备份与恢复

备份不是目的，恢复才是目的。

### 14.1 备份类型

| 类型 | 含义 |
| --- | --- |
| 全量备份 | 备份全部数据 |
| 增量备份 | 备份变化部分 |
| 逻辑备份 | 导出 SQL 或逻辑格式 |
| 物理备份 | 备份数据文件 |
| 时间点恢复 | 恢复到某个时间点 |

### 14.2 RPO 与 RTO

| 指标 | 含义 |
| --- | --- |
| RPO | 最多能接受丢失多少数据 |
| RTO | 最长能接受恢复多久 |

例如 RPO 5 分钟表示最多能接受丢 5 分钟数据；RTO 30 分钟表示故障后 30 分钟内要恢复服务。

### 14.3 备份检查

- 是否定期备份。
- 是否异地保存。
- 是否加密。
- 是否定期恢复演练。
- 是否记录恢复步骤。
- 是否监控备份失败。

没有恢复演练的备份不能算可靠备份。

## 15. 数据治理

后端数据长期运行后会遇到：

- 数据膨胀。
- 历史数据查询慢。
- 冷热数据混杂。
- 敏感数据合规。
- 删除和归档策略不清。
- 报表查询影响在线业务。

治理方法：

- 冷热分离。
- 历史归档。
- 分区表。
- 数据脱敏。
- 数据生命周期策略。
- 报表库或数仓。
- 审计日志。

## 16. 数据库设计检查清单

建表前检查：

- 主键是否稳定？
- 是否需要唯一约束？
- 是否需要状态字段？
- 状态是否合法可控？
- 是否需要创建时间、更新时间？
