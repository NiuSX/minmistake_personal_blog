# 04-数据库、事务、索引与数据建模

> 本文目标：系统理解后端数据层知识，包括关系型数据库、NoSQL、数据建模、主键、索引、事务、隔离级别、MVCC、锁、慢查询、分库分表、备份恢复和数据治理。本文不绑定具体数据库产品，但会用通用数据库概念讲解。

## 1. 数据库在后端中的位置

数据库是后端系统的权威数据源。缓存可以丢，消息可以重试，服务可以重启，但核心业务数据必须可靠保存。

后端依赖数据库完成：

- 保存业务状态。
- 查询业务对象。
- 保证唯一约束。
- 执行事务。
- 进行并发控制。
- 支持审计和追溯。
- 支持报表和统计。
- 支持备份恢复。

数据库设计错误会长期影响系统。代码可以重构，接口可以加版本，缓存可以重建，但错误的数据模型和混乱的数据状态往往最难修。

## 2. 关系型数据库

关系型数据库使用表、行、列表达结构化数据，并通过 SQL 查询。

### 2.1 核心概念

| 概念 | 含义 |
| --- | --- |
| Table | 表，表示一类数据 |
| Row | 行，一条记录 |
| Column | 列，一个字段 |
| Primary Key | 主键，唯一标识一行 |
| Foreign Key | 外键，表达表间关系 |
| Index | 索引，加速查询 |
| Constraint | 约束，保证数据合法 |
| Transaction | 事务，一组原子操作 |
| View | 视图，虚拟查询结果 |

### 2.2 适合场景

关系型数据库适合：

- 结构清晰的数据。
- 强一致业务。
- 复杂查询。
- 多条件筛选。
- 需要事务的场景。
- 需要约束保障的核心数据。

例如订单、支付、账户、库存、用户、权限、合同、审批流程通常适合关系型数据库。

### 2.3 不适合场景

关系型数据库不适合：

- 大量非结构化文件，例如图片、视频。
- 高维向量检索。
- 超大规模日志原文检索。
- 极高吞吐且关系很弱的事件流。
- 字段结构极度动态且查询不确定的数据。

这些场景可以考虑对象存储、搜索引擎、时序数据库、文档数据库、列式数据库等。

## 3. NoSQL

NoSQL 不是一种数据库，而是一类非关系型数据存储。

| 类型 | 数据模型 | 适合场景 |
| --- | --- | --- |
| Key-Value | key 到 value | 缓存、会话、计数 |
| Document | JSON-like 文档 | 内容、配置、用户画像 |
| Wide Column | 行键 + 列族 | 大规模写入、日志、时序 |
| Graph | 点和边 | 社交关系、路径分析 |
| Search Engine | 倒排索引 | 全文搜索、日志检索 |
| Time Series | 时间戳 + 指标 | 监控、IoT、行情 |
| Object Storage | 对象和元数据 | 图片、视频、文件、备份 |

选择 NoSQL 时，不要只看“性能高”。必须确认：

- 查询模式是否适配。
- 一致性模型是什么。
- 是否支持事务。
- 是否支持二级索引。
- 分片如何做。
- 运维和备份如何做。
- 数据增长后成本如何。

## 4. 数据建模

数据建模不是简单建表，而是把业务对象、关系、状态和约束表达成可持久化结构。

### 4.1 建模步骤

1. 找核心实体：用户、订单、商品、支付、库存。
2. 找实体关系：一对一、一对多、多对多。
3. 找业务状态：订单状态、支付状态、审核状态。
4. 找不变量：金额不能为负、订单号唯一、库存不能超卖。
5. 找查询场景：详情、列表、搜索、统计、导出。
6. 找写入场景：创建、更新、删除、批量导入。
7. 找数据增长：哪些表增长快，是否需要归档。
8. 找一致性边界：哪些数据必须同事务更新。

### 4.2 主键设计

主键用于唯一标识记录。

常见主键：

| 类型 | 优点 | 缺点 |
| --- | --- | --- |
| 自增 ID | 简单、索引局部性好 | 分布式生成困难，暴露规模 |
| UUID | 去中心化生成 | 长、索引随机、可读性差 |
| 雪花类 ID | 趋势递增、分布式 | 依赖时钟和机器 ID |
| 业务 ID | 业务可读 | 变更风险高，不一定稳定 |

建议：

- 核心表主键应稳定不可变。
- 不要用可能变化的字段做主键，例如手机号、邮箱。
- 对外暴露 ID 时考虑枚举风险。
- 分布式系统要提前规划 ID 生成。

### 4.3 唯一约束

唯一约束是后端幂等和数据正确性的最后防线。

场景：

- 用户名唯一。
- 手机号唯一。
- 订单号唯一。
- 支付流水号唯一。
- 同一用户同一活动只能领一次券。

不要只靠应用层查询后判断是否存在，因为并发下两个请求可能同时通过检查。最终必须有数据库唯一约束兜底。

## 5. 范式与反范式

### 5.1 范式

范式用于减少数据冗余和更新异常。

简单理解：

- 一个事实只存一处。
- 表达清楚实体关系。
- 避免重复字段导致不一致。

例如用户昵称如果在评论表里也冗余一份，用户改昵称后历史评论是否同步就是业务问题。

### 5.2 反范式

反范式是为了查询性能或业务便利主动冗余数据。

例子：

- 订单保存商品名称和价格快照。
- 评论冗余用户昵称和头像。
- 报表表提前汇总统计结果。

反范式不是坏事，但要明确：

- 冗余字段谁负责更新？
- 允许不一致多久？
- 如何修复不一致？
- 是否需要定期对账？

## 6. 索引

索引是数据库性能的核心。

### 6.1 索引为什么快

没有索引时，数据库可能需要扫描整张表。索引通过额外数据结构帮助数据库快速定位记录。

代价：

- 占用存储空间。
- 写入、更新、删除时需要维护索引。
- 索引过多会拖慢写入。
- 不合适的索引可能完全用不上。

### 6.2 常见索引类型

| 类型 | 用途 |
| --- | --- |
| 主键索引 | 按主键查找 |
| 唯一索引 | 加速查询并保证唯一 |
| 普通索引 | 加速常见查询 |
| 联合索引 | 多列组合查询 |
| 覆盖索引 | 查询字段都在索引中，减少回表 |
| 全文索引 | 文本搜索 |
| 空间索引 | 地理空间查询 |

### 6.3 联合索引

联合索引顺序非常重要。

假设索引：

```text
(tenant_id, status, created_at)
```

适合：

```text
WHERE tenant_id = ? AND status = ? ORDER BY created_at
WHERE tenant_id = ?
```

不一定适合：

```text
WHERE status = ?
WHERE created_at > ?
```

这和数据库索引结构、最左前缀、选择性、执行计划有关。实际要看具体数据库执行计划。

### 6.4 索引设计原则

- 为高频查询建索引。
- 为高选择性字段建索引。
- 让索引服务过滤、排序、分页。
- 避免对频繁变化字段建太多索引。
- 关注联合索引顺序。
- 定期清理无用索引。
- 使用执行计划验证。

### 6.5 常见索引失效原因

不同数据库实现不同，但常见问题包括：

- 对索引列使用函数导致无法利用索引。
- 隐式类型转换。
- 前缀通配模糊查询。
- OR 条件不合理。
- 统计信息过期。
- 查询返回比例太高，数据库选择全表扫描。
- 联合索引不满足前缀。

## 7. 事务

事务保证一组操作作为整体执行。

### 7.1 ACID

| 特性 | 含义 |
| --- | --- |
| Atomicity 原子性 | 全部成功或全部失败 |
| Consistency 一致性 | 事务前后满足约束 |
| Isolation 隔离性 | 并发事务相互隔离到一定程度 |
| Durability 持久性 | 提交后数据持久保存 |

### 7.2 事务不是越大越好

大事务问题：

- 锁持有时间长。
- 回滚成本高。
- 日志量大。
- 阻塞其他事务。
- 主从复制延迟。
- 容易超时。

事务中不要做：

- 慢 HTTP 调用。
- 等待用户输入。
- 大文件处理。
- 长时间批量任务。
- 不可回滚的外部副作用。

## 8. 隔离级别与并发异常

### 8.1 常见并发异常

| 异常 | 含义 |
| --- | --- |
| 脏读 | 读到未提交数据 |
| 不可重复读 | 同事务两次读同一行结果不同 |
| 幻读 | 同事务两次范围查询出现新行或少行 |
| 丢失更新 | 两个事务覆盖彼此修改 |
| 写偏斜 | 多事务分别合法，合起来破坏约束 |

### 8.2 隔离级别

| 隔离级别 | 特点 |
| --- | --- |
| Read Uncommitted | 可能读到未提交数据，很少用于业务系统 |
| Read Committed | 只能读提交数据，但同事务两次读可能不同 |
| Repeatable Read | 同事务中重复读更稳定 |
| Serializable | 最接近串行执行，并发性能最低 |

不同数据库对隔离级别实现不完全相同。学习和排障时要查具体数据库官方文档。

## 9. MVCC

MVCC 是多版本并发控制。它通过保存数据的多个版本，让读写可以更好并发。

直觉：

- 读请求看到某个时间点的一致快照。
- 写请求创建新版本。
- 老版本在没有事务需要后被清理。

MVCC 的好处：

- 读不阻塞写。
- 写不阻塞普通快照读。
- 提升并发。

代价：

- 需要维护多版本。
- 需要清理旧版本。
- 长事务会阻止旧版本回收。
- 存储可能膨胀。

## 10. 锁

数据库锁用于并发控制。

常见锁：

| 锁 | 含义 |
| --- | --- |
| 行锁 | 锁定某些行 |
| 表锁 | 锁定整张表 |
| 间隙锁 | 锁定索引范围间隙 |
| 意向锁 | 表示将要锁某些行 |
| 共享锁 | 多个事务可读 |
| 排他锁 | 独占修改 |

后端排查锁问题要关注：

- 哪个事务持锁。
- 持锁多久。
- 等待锁的 SQL。
- 是否有大事务。
- 是否有缺失索引导致锁范围扩大。

## 11. 慢查询

慢查询常见原因：

- 没有索引。
- 索引不合适。
- 返回数据太多。
- 深分页。
- N+1 查询。
- 复杂 join。
- 排序或分组无法利用索引。
- 锁等待。
- 数据量增长后执行计划变化。

### 11.1 深分页

`OFFSET 100000 LIMIT 20` 可能很慢，因为数据库要跳过大量记录。

优化：

- 使用游标分页。
- 使用主键或时间作为 seek 条件。
- 限制最大页数。
- 对导出任务异步化。

### 11.2 N+1 查询

示例：

1. 查询 100 个订单。
2. 每个订单再查一次用户。
3. 总共 101 次查询。

优化：

- 批量查询。
- join。
- 预加载。
- 缓存。

## 12. 读写分离

读写分离通过主库写、从库读扩展读取能力。

优点：

- 减轻主库读压力。
- 提升读吞吐。

问题：

- 主从延迟。
- 刚写完读从库可能读不到。
- 事务内读写一致性复杂。
- 故障切换后路由要更新。

解决读己之写：

- 写后短时间读主库。
- 根据版本或时间判断。
- 业务允许最终一致。

## 13. 分库分表

当单库单表无法承载数据量或写入压力时，考虑分库分表。

### 13.1 垂直拆分

按业务或字段拆：

- 用户库。
- 订单库。
- 支付库。
- 商品库。

优点：

- 业务边界清晰。
- 减少单库压力。

问题：

- 跨库查询复杂。
- 事务边界变化。

### 13.2 水平拆分

按分片键把同一类数据拆到多个库或表。

分片键选择：

- 高频查询条件。
- 分布均匀。
- 不易变化。
- 能减少跨分片查询。

常见分片键：

- 用户 ID。
- 租户 ID。
- 订单 ID。
- 时间。

### 13.3 分库分表代价

- 跨分片分页难。
- 跨分片 join 难。
- 全局唯一 ID。
- 分布式事务。
- 扩容迁移。
- 热点分片。
- 运维复杂。

不要过早分库分表。先考虑索引、缓存、归档、读写分离、业务拆分。

## 14. 数据备份与恢复

备份不是目的，恢复才是目的。

### 14.1 备份类型

| 类型 | 含义 |
| --- | --- |
| 全量备份 | 备份全部数据 |
| 增量备份 | 备份变化部分 |
| 逻辑备份 | 导出 SQL 或逻辑格式 |
| 物理备份 | 备份数据文件 |
| 时间点恢复 | 恢复到某个时间点 |

### 14.2 RPO 与 RTO

| 指标 | 含义 |
| --- | --- |
| RPO | 最多能接受丢失多少数据 |
| RTO | 最长能接受恢复多久 |

例如 RPO 5 分钟表示最多能接受丢 5 分钟数据；RTO 30 分钟表示故障后 30 分钟内要恢复服务。

### 14.3 备份检查

- 是否定期备份。
- 是否异地保存。
- 是否加密。
- 是否定期恢复演练。
- 是否记录恢复步骤。
- 是否监控备份失败。

没有恢复演练的备份不能算可靠备份。

## 15. 数据治理

后端数据长期运行后会遇到：

- 数据膨胀。
- 历史数据查询慢。
- 冷热数据混杂。
- 敏感数据合规。
- 删除和归档策略不清。
- 报表查询影响在线业务。

治理方法：

- 冷热分离。
- 历史归档。
- 分区表。
- 数据脱敏。
- 数据生命周期策略。
- 报表库或数仓。
- 审计日志。

## 16. 数据库设计检查清单

建表前检查：

- 主键是否稳定？
- 是否需要唯一约束？
- 是否需要状态字段？
- 状态是否合法可控？
- 是否需要创建时间、更新时间？
- 是否需要软删除？
- 是否有租户隔离字段？
- 查询条件是什么？
- 需要哪些索引？
- 数据量增长如何？
- 是否需要归档？
- 是否包含敏感信息？
- 是否需要审计？

上线前检查：

- 慢查询是否压测过？
- 执行计划是否合理？
- 索引是否过多或缺失？
- 大事务是否存在？
- 数据迁移是否可回滚？
- 是否有备份？

## 17. 本章小结

数据库是后端系统最重要的基础。掌握数据库不能只会写 SQL，还要理解数据建模、约束、索引、事务、隔离级别、锁、备份、归档和容量增长。很多后端稳定性问题最终都能追溯到数据层设计。

```text
步骤：最小可复现流程
现象：成功输出、失败输出、日志、指标或测量数据
分析：为什么会出现该现象，和哪些概念相关
结论：可复用的规则、命令、配置或设计取舍
风险：边界条件、性能、安全、兼容性或维护成本
下一步：继续实验、补充资料或应用到项目
```

## 讲义级补充：如何真正学懂《04-数据库、事务、索引与数据建模》

> 适用位置：后端开发完整学习笔记\04-数据库、事务、索引与数据建模.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

后端知识的核心是让业务数据在并发、故障、流量波动和安全威胁下仍然保持正确。学习时要把接口、数据、事务、缓存、队列、日志和运维放在同一张图里看。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 接口：请求如何进入系统，参数如何校验，错误如何返回。
- 数据：如何建模、持久化、索引、缓存和迁移。
- 并发：事务、锁、幂等、重试和一致性如何处理。
- 运维：日志、指标、告警、容量、发布和回滚如何保障。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把后端系统类比成餐厅后厨：API 是点餐窗口，数据库是库存账本，缓存是备好的常用食材，消息队列是排队小票，日志和监控是摄像头与账单。高峰期能否稳定服务，取决于整套流程是否有秩序。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《04-数据库、事务、索引与数据建模》时，可以设计一个下单或登录接口：先写正常流程，再补参数校验、幂等、事务、日志、限流和错误返回。这个小例子能覆盖后端工程里最常见的一组真实问题。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 从一个业务请求开始，画出 API、鉴权、服务逻辑、数据库、缓存、队列和日志的路径。
- 逐个分析并发、事务、幂等、超时、重试、限流和降级。
- 用压测、监控和故障演练验证设计，而不是只看单元测试通过。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。
6. 反例与排错：展示错误做法会导致什么现象，再说明如何定位和修复。
7. 总结迁移：最后说明它和相邻知识点的区别、联系以及后续该学什么。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。
- 记录遇到的第一个错误、定位过程和最终修复方法。
- 完成后写 5 行复盘：我原来以为是什么，实际是什么，下次会如何更快处理。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 接口能跑就认为完成，忽略并发、幂等和事务边界。
- 缓存和数据库一致性没有策略，故障时数据互相打架。
- 没有日志、指标和 trace，出问题只能猜。

遇到这些问题时，不要急着背更多资料。更有效的办法是回到一个最小例子，把输入、状态变化、输出和验证方式重新走一遍。

### 9. 自测题

1. 用一句话说明本讲主题解决的核心问题。
2. 列出本讲最重要的 3 个概念，并说明它们的关系。
3. 举一个生活类比，再指出这个类比在哪些地方不严谨。
4. 写出一个最小实践任务的验收标准。
5. 如果结果不符合预期，你会优先检查哪 3 个环节？为什么？
6. 本讲和相邻章节的知识边界是什么？哪些问题应该交给其他章节解决？

### 10. 复习口诀

先问场景，再看输入；先拆结构，再走流程；先做小例，再谈优化；先会排错，再做规模化。
