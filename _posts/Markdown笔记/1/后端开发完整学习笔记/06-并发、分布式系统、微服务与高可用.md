# 06-并发、分布式系统、微服务与高可用

> 本文目标：理解后端系统在高并发、多节点、多服务环境下会遇到的问题。内容覆盖并发、线程池、连接池、背压、分布式理论、CAP、BASE、一致性模型、复制、分片、微服务、服务治理、分布式事务、高可用、超时、重试、熔断、限流和容灾。

## 1. 并发为什么是后端核心问题

后端系统不是一次只处理一个请求。真实系统同时面对大量用户请求、定时任务、消息消费、后台任务、第三方回调和内部服务调用。

并发会带来：

- 资源竞争。
- 数据竞争。
- 锁等待。
- 队列堆积。
- 连接池耗尽。
- 下游过载。
- 重试放大。
- 缓存击穿。
- 状态不一致。

并发不是简单“开更多线程”。线程、连接、队列、数据库、缓存、消息队列、外部服务都有容量上限。没有上限的并发最终会变成排队、超时、内存耗尽和雪崩。

## 2. 并发、并行、同步、异步

| 概念 | 含义 | 后端例子 |
| --- | --- | --- |
| 并发 | 同一时间段处理多个任务 | 服务同时处理多个 HTTP 请求 |
| 并行 | 同一时刻多个任务真正执行 | 多核 CPU 同时执行多个请求 |
| 同步 | 调用方等待结果 | API 直接查数据库再返回 |
| 异步 | 调用方不等待最终结果 | 下单后发消息异步通知 |
| 阻塞 | 执行单元等待资源 | 等数据库响应 |
| 非阻塞 | 不等待资源，可继续做别的事 | 事件循环处理 IO |

后端优化时要区分：

- CPU 密集任务：主要消耗计算资源。
- IO 密集任务：主要等待网络、磁盘、数据库、第三方。

CPU 密集任务线程过多会增加上下文切换。IO 密集任务可以支持更高并发，但不能超过下游容量。

## 3. 线程池

线程池用于复用线程和限制并发。

关键参数：

| 参数 | 作用 |
| --- | --- |
| 核心线程数 | 常驻工作线程 |
| 最大线程数 | 允许的最高线程数 |
| 队列大小 | 暂存等待任务 |
| 空闲回收时间 | 非核心线程多久回收 |
| 拒绝策略 | 过载时如何处理 |
| 线程名称 | 排障识别 |

### 3.1 常见错误

- 无界队列导致内存爆。
- 无限创建线程导致系统崩溃。
- 所有任务共用同一个线程池。
- 阻塞 IO 和 CPU 计算混在一起。
- 拒绝策略没有业务处理。
- 没有指标监控。

### 3.2 线程池隔离

不同任务应隔离：

- 核心业务线程池。
- 慢外部调用线程池。
- 消息消费线程池。
- 定时任务线程池。
- 后台导出线程池。

隔离的目标是避免一个慢依赖拖垮整个应用。

## 4. 连接池

数据库、Redis、HTTP 客户端、RPC 客户端通常都有连接池。

连接池指标：

- 最大连接数。
- 活跃连接数。
- 空闲连接数。
- 获取连接等待时间。
- 获取连接超时次数。
- 连接创建失败。

连接池太小会排队，太大可能压垮下游。连接池大小必须结合：

- 应用实例数量。
- 下游最大连接数。
- 单请求占用连接时间。
- 峰值并发。
- 超时时间。

## 5. 背压

背压是系统处理不过来时，让上游减速或拒绝。

没有背压：

- 队列越来越长。
- 请求越来越慢。
- 超时越来越多。
- 重试越来越多。
- 内存耗尽。
- 整体雪崩。

背压手段：

- 有界队列。
- 限流。
- 拒绝策略。
- 暂停消费。
- 降级。
- 返回 429 或 503。

## 6. 分布式系统的本质

分布式系统是多个节点通过网络协作。难点来自：

- 网络可能延迟。
- 网络可能丢包。
- 节点可能宕机。
- 时钟不一致。
- 消息可能重复。
- 消息可能乱序。
- 数据存在多个副本。
- 局部成功和局部失败并存。

单机中你可以假设函数调用要么成功要么抛异常；分布式中远程调用可能出现“不知道成功还是失败”的状态。

## 7. CAP

CAP 描述网络分区下的一致性和可用性取舍。

| 字母 | 含义 |
| --- | --- |
| C Consistency | 所有节点看到一致数据 |
| A Availability | 每个请求都得到非错误响应 |
| P Partition Tolerance | 网络分区时系统仍运行 |

关键理解：

- 分布式系统无法避免网络分区。
- 分区发生时，必须在一致性和可用性之间取舍。
- CAP 不是简单“三选二”的标签游戏。

例子：

- 支付账户系统更偏一致性，宁可拒绝部分请求也不能算错钱。
- 社交点赞数可以偏可用性，短暂不一致可接受。

## 8. BASE 与最终一致

BASE 包括：

- Basically Available：基本可用。
- Soft State：允许中间状态。
- Eventually Consistent：最终一致。

最终一致适合：

- 积分到账。
- 搜索索引更新。
- 推荐特征更新。
- 订单状态同步到报表。
- 用户资料同步到缓存。

最终一致不是“不管一致”。它需要：

- 可靠事件。
- 重试。
- 幂等。
- 补偿。
- 对账。
- 监控。

## 9. 一致性模型

| 模型 | 含义 |
| --- | --- |
| 强一致 | 读一定看到最新写入 |
| 线性一致 | 操作像按真实时间全局排序 |
| 顺序一致 | 所有节点看到相同顺序 |
| 因果一致 | 有因果关系的操作顺序一致 |
| 最终一致 | 没有新写入后最终收敛 |
| 读己之写 | 用户能看到自己刚写的数据 |

业务设计常常需要的是局部一致，而不是全局强一致。

例子：

- 用户修改头像后，自己应该立即看到新头像。
- 其他用户晚几秒看到可以接受。
- 订单支付成功后订单状态不能回退。

## 10. 复制

复制是把数据保存到多个节点。

目的：

- 提高可用性。
- 提高读能力。
- 容灾。
- 降低地域访问延迟。

模式：

| 模式 | 特点 |
| --- | --- |
| 主从复制 | 主写从读，简单常见 |
| 多主复制 | 多节点可写，冲突复杂 |
| 同步复制 | 写入等待副本确认，一致性好但慢 |
| 异步复制 | 主提交后异步复制，性能好但可能丢 |
| 仲裁复制 | 多数派确认 |

问题：

- 复制延迟。
- 主从切换。
- 脑裂。
- 读到旧数据。
- 冲突解决。

## 11. 分片

分片把数据拆到多个节点。

分片目的：

- 提升容量。
- 提升写入能力。
- 降低单节点压力。

分片键选择：

- 分布均匀。
- 高频查询条件。
- 不易变化。
- 避免热点。

分片问题：

- 跨分片查询。
- 跨分片事务。
- 全局排序。
- 全局唯一 ID。
- 扩容迁移。
- 热点分片。

## 12. 分布式锁

分布式锁用于跨进程互斥。

适合：

- 防止多实例重复执行定时任务。
- 控制同一资源短时间只有一个处理者。
- 减少缓存击穿回源。

注意：

- 必须设置过期时间。
- 解锁要校验持有者身份。
- 锁过期不代表业务完成。
- 网络抖动可能导致锁语义复杂。
- 分布式锁不能替代数据库唯一约束。

最终正确性要靠：

- 唯一约束。
- 状态机。
- 乐观锁。
- 幂等。

## 13. 微服务

微服务是把系统按业务能力拆成多个可独立部署的服务。

### 13.1 单体、模块化单体、微服务

| 架构 | 优点 | 缺点 |
| --- | --- | --- |
| 单体 | 简单，部署方便，事务直接 | 变大后边界混乱 |
| 模块化单体 | 单部署但模块清晰 | 需要纪律维护边界 |
| 微服务 | 独立部署、团队自治、可独立扩展 | 分布式复杂度高 |

不要为了“看起来高级”过早微服务。微服务适合：

- 团队规模较大。
- 业务边界清晰。
- 不同模块发布频率差异大。
- 不同模块扩展需求不同。
- 有足够运维和治理能力。

### 13.2 服务拆分

拆分依据：

- 业务能力。
- 数据所有权。
- 团队边界。
- 变更频率。
- 性能瓶颈。
- 安全边界。

错误拆分：

- 按 Controller、Service、DAO 技术层拆。
- 一张表一个服务。
- 多个服务共享同一个数据库。
- 服务之间循环调用。

## 14. 服务治理

微服务需要治理能力。

### 14.1 服务发现

服务发现解决服务实例在哪里。

模式：

- 客户端发现。
- 服务端发现。
- DNS 发现。
- 注册中心。

服务实例要上报：

- 地址。
- 端口。
- 健康状态。
- 版本。
- 元数据。

### 14.2 API 网关

网关职责：

- 路由。
- TLS。
- 认证。
- 限流。
- 熔断。
- 灰度。
- 日志。
- 协议转换。

风险：

- 网关变成业务大单体。
- 网关配置错误影响全局。
- 网关本身必须高可用。

### 14.3 配置中心

配置中心管理：

- 数据库地址。
- 开关。
- 超时。
- 限流规则。
- 灰度规则。
- 密钥引用。

要求：

- 变更审计。
- 灰度发布。
- 回滚。
- 权限控制。
- 客户端容错。

## 15. 分布式事务

跨服务、跨数据库的强事务很难。

### 15.1 2PC

两阶段提交：

1. 准备阶段。
2. 提交阶段。

问题：

- 阻塞。
- 协调者单点。
- 可用性差。
- 长事务。

### 15.2 Saga

Saga 把长事务拆成多个本地事务，每一步有补偿。

例子：下单流程。

```text
创建订单 -> 锁库存 -> 创建支付单 -> 通知用户
```

如果创建支付单失败：

```text
释放库存 -> 取消订单
```

Saga 关键：

- 每步本地事务可靠。
- 补偿操作明确。
- 状态可追踪。
- 支持重试。
- 支持人工介入。

### 15.3 TCC

TCC：

- Try：预留资源。
- Confirm：确认提交。
- Cancel：取消释放。

适合资源预留场景，但业务侵入较强。

## 16. 高可用

高可用的目标是故障发生时服务仍能在可接受范围内运行。

### 16.1 可用性指标

| 可用性 | 年不可用时间约 |
| --- | --- |
| 99% | 3.65 天 |
| 99.9% | 8.76 小时 |
| 99.99% | 52.6 分钟 |
| 99.999% | 5.26 分钟 |

高可用需要：

- 冗余。
- 健康检查。
- 自动故障转移。
- 限流。
- 降级。
- 备份。
- 监控。
- 演练。

## 17. 超时

每个远程调用都必须有超时。

没有超时会导致：

- 线程长期占用。
- 连接池耗尽。
- 队列堆积。
- 请求雪崩。

超时要分层：

- 客户端请求总超时。
- 网关超时。
- 服务内部调用超时。
- 数据库查询超时。
- 消息处理超时。

超时预算要从用户请求总时长反推。

## 18. 重试

重试只适合临时错误。

可重试：

- 网络抖动。
- 短暂超时。
- 临时 503。
- 限流后按 Retry-After 等待。

不应重试：

- 参数错误。
- 权限错误。
- 业务规则失败。
- 非幂等写操作。

重试要求：

- 最大次数。
- 指数退避。
- 随机抖动。
- 幂等。
- 总超时限制。

## 19. 熔断

熔断是在下游持续失败时快速失败，保护系统。

状态：

- Closed：正常。
- Open：熔断。
- Half-Open：试探恢复。

熔断适合：

- 下游错误率高。
- 下游超时严重。
- 线程池被慢调用占满。

熔断后要有 fallback：

- 返回缓存。
- 返回默认值。
- 提示稍后重试。
- 隐藏非核心模块。

## 20. 限流

限流控制请求速率或并发。

算法：

| 算法 | 特点 |
| --- | --- |
| 固定窗口 | 简单但边界突刺 |
| 滑动窗口 | 更平滑 |
| 漏桶 | 匀速处理 |
| 令牌桶 | 允许短暂突发 |
| 并发限制 | 控制同时处理数 |

限流维度：

- IP。
- 用户。
- 租户。
- 接口。
- 资源 ID。
- 应用。

## 21. 降级

降级是牺牲非核心能力保核心链路。

例子：

- 商品详情不展示推荐。
- 评论返回旧缓存。
- 排行榜返回昨天快照。
- 搜索关闭复杂排序。
- 图片处理降级为默认图。

降级要提前设计，不要故障时临时拍脑袋。

## 22. 容灾

容灾关注故障时如何恢复服务。

术语：

| 术语 | 含义 |
| --- | --- |
| RPO | 最多丢多少数据 |
| RTO | 多久恢复 |
| 同城双活 | 同城多机房 |
| 异地灾备 | 异地备份 |
| 多活 | 多区域同时提供服务 |

容灾难点：

- 数据复制延迟。
- 流量切换。
- 跨地域延迟。
- 数据冲突。
- 成本高。
- 演练复杂。

## 23. 故障演练

常见演练：

- 关闭一个服务实例。
- 模拟数据库慢查询。
- 模拟 Redis 不可用。
- 模拟 MQ 积压。
- 模拟第三方超时。
- 模拟网络延迟。
- 模拟磁盘满。

演练目标不是制造事故，而是验证：

- 监控是否发现。
- 告警是否有效。
- 降级是否生效。
- 文档是否可用。
- 团队是否知道怎么处理。

## 24. 本章小结

并发和分布式系统的核心是控制不确定性。你不能假设网络可靠、节点不宕机、消息不重复、请求不超时、下游永远健康。成熟后端系统必须设计超时、重试、幂等、限流、熔断、降级、隔离、补偿和观测。

```text
现象：成功输出、失败输出、日志、指标或测量数据
分析：为什么会出现该现象，和哪些概念相关
结论：可复用的规则、命令、配置或设计取舍
风险：边界条件、性能、安全、兼容性或维护成本
下一步：继续实验、补充资料或应用到项目
```

## 讲义级补充：如何真正学懂《06-并发、分布式系统、微服务与高可用》

> 适用位置：后端开发完整学习笔记\06-并发、分布式系统、微服务与高可用.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

后端知识的核心是让业务数据在并发、故障、流量波动和安全威胁下仍然保持正确。学习时要把接口、数据、事务、缓存、队列、日志和运维放在同一张图里看。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 接口：请求如何进入系统，参数如何校验，错误如何返回。
- 数据：如何建模、持久化、索引、缓存和迁移。
- 并发：事务、锁、幂等、重试和一致性如何处理。
- 运维：日志、指标、告警、容量、发布和回滚如何保障。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把后端系统类比成餐厅后厨：API 是点餐窗口，数据库是库存账本，缓存是备好的常用食材，消息队列是排队小票，日志和监控是摄像头与账单。高峰期能否稳定服务，取决于整套流程是否有秩序。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《06-并发、分布式系统、微服务与高可用》时，可以设计一个下单或登录接口：先写正常流程，再补参数校验、幂等、事务、日志、限流和错误返回。这个小例子能覆盖后端工程里最常见的一组真实问题。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 从一个业务请求开始，画出 API、鉴权、服务逻辑、数据库、缓存、队列和日志的路径。
- 逐个分析并发、事务、幂等、超时、重试、限流和降级。
- 用压测、监控和故障演练验证设计，而不是只看单元测试通过。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。
6. 反例与排错：展示错误做法会导致什么现象，再说明如何定位和修复。
7. 总结迁移：最后说明它和相邻知识点的区别、联系以及后续该学什么。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。
- 记录遇到的第一个错误、定位过程和最终修复方法。
- 完成后写 5 行复盘：我原来以为是什么，实际是什么，下次会如何更快处理。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 接口能跑就认为完成，忽略并发、幂等和事务边界。
- 缓存和数据库一致性没有策略，故障时数据互相打架。
- 没有日志、指标和 trace，出问题只能猜。

遇到这些问题时，不要急着背更多资料。更有效的办法是回到一个最小例子，把输入、状态变化、输出和验证方式重新走一遍。

### 9. 自测题

1. 用一句话说明本讲主题解决的核心问题。
2. 列出本讲最重要的 3 个概念，并说明它们的关系。
3. 举一个生活类比，再指出这个类比在哪些地方不严谨。
4. 写出一个最小实践任务的验收标准。
5. 如果结果不符合预期，你会优先检查哪 3 个环节？为什么？
6. 本讲和相邻章节的知识边界是什么？哪些问题应该交给其他章节解决？

### 10. 复习口诀

先问场景，再看输入；先拆结构，再走流程；先做小例，再谈优化；先会排错，再做规模化。
