# 08-可观测性、性能容量与故障排查

> 本文目标：理解后端系统上线后如何观察、衡量和排查问题。内容覆盖日志、指标、链路追踪、Profiling、RED、USE、四大黄金信号、SLO、告警、压测、容量规划、性能优化和故障复盘。

## 1. 可观测性是什么

可观测性是通过系统输出推断系统内部状态的能力。

主要信号：

- Logs：日志，记录事件。
- Metrics：指标，观察趋势。
- Traces：链路追踪，查看请求路径。
- Profiles：性能剖析，分析资源消耗。

没有可观测性，系统出问题时只能猜。成熟后端系统必须从设计阶段就考虑日志、指标和追踪。

## 2. 日志

### 2.1 日志的作用

日志用于回答：

- 发生了什么？
- 谁触发的？
- 影响哪个业务对象？
- 请求走到了哪一步？
- 失败原因是什么？
- 是否发生重试、降级、超时？

### 2.2 日志级别

| 级别 | 用途 |
| --- | --- |
| DEBUG | 调试细节，生产慎用 |
| INFO | 关键业务事件 |
| WARN | 可恢复异常或异常趋势 |
| ERROR | 请求失败或需要处理的问题 |
| FATAL | 进程级严重错误 |

### 2.3 结构化日志

推荐结构化：

```json
{
  "time": "2026-06-16T10:00:00Z",
  "level": "INFO",
  "traceId": "trace_001",
  "requestId": "req_001",
  "event": "payment_callback_received",
  "orderId": "order_1",
  "paymentId": "pay_1",
  "status": "SUCCESS"
}
```

结构化日志便于检索、聚合和告警。

### 2.4 日志脱敏

不能记录：

- 密码。
- Token。
- 私钥。
- 身份证完整值。
- 银行卡完整值。
- 支付敏感信息。

可记录脱敏值：

```text
phone=138****1234
card=6222********1234
```

## 3. 指标

指标适合观察趋势和告警。

### 3.1 指标类型

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| Counter | 单调递增计数 | 请求总数 |
| Gauge | 当前值 | 队列长度、内存使用 |
| Histogram | 分布 | 请求耗时 |
| Summary | 摘要 | 分位数 |

### 3.2 服务指标

每个核心服务应有：

- 请求量。
- 错误率。
- P50/P95/P99。
- 并发数。
- 线程池活跃数。
- 队列长度。
- 连接池等待时间。
- 下游调用耗时。
- 缓存命中率。
- 消息积压。

## 4. 链路追踪

链路追踪用于查看一次请求跨服务经过了哪里。

术语：

| 术语 | 含义 |
| --- | --- |
| Trace | 一次完整请求链路 |
| Span | 一个操作片段 |
| Trace ID | 链路 ID |
| Span ID | 片段 ID |
| Parent Span | 父片段 |

追踪能回答：

- 哪个服务最慢？
- 哪个数据库查询最慢？
- 是否发生重试？
- 是否走了降级？
- 请求在哪个节点失败？

### 4.1 Trace Context

跨服务传递 trace 信息需要统一上下文。W3C Trace Context 定义了 `traceparent` 等标准 Header。

后端要确保：

- 入口生成 trace ID。
- 调用下游传递 trace ID。
- 日志中包含 trace ID。
- 异步消息也尽量传递 trace 信息。

## 5. Profiling

Profiling 用于分析资源消耗。

常见类型：

- CPU Profile。
- Memory Profile。
- Lock Profile。
- IO Profile。
- Allocation Profile。

适用场景：

- CPU 高但不知道哪里消耗。
- 内存增长。
- GC 压力大。
- 锁竞争严重。
- 序列化或压缩耗时高。

Profiling 要注意对生产性能的影响，通常采用采样和短时间抓取。

## 6. 四大黄金信号

Google SRE 提出的四大黄金信号：

| 信号 | 含义 |
| --- | --- |
| Latency | 请求延迟 |
| Traffic | 流量 |
| Errors | 错误 |
| Saturation | 饱和度 |

### 6.1 Latency

延迟要区分：

- 成功请求延迟。
- 失败请求延迟。
- 下游调用延迟。
- 排队等待延迟。

不要只看平均值，要看 P95/P99。

### 6.2 Traffic

流量包括：

- QPS。
- 请求体大小。
- 响应体大小。
- 消息生产速度。
- 消息消费速度。

### 6.3 Errors

错误包括：

- 5xx。
- 业务失败。
- 超时。
- 下游错误。
- 消息消费失败。

### 6.4 Saturation

饱和度表示资源接近极限：

- CPU run queue。
- 线程池队列。
- 连接池等待。
- 磁盘 IO 队列。
- MQ 积压。
- Redis 延迟。

## 7. RED 方法

RED 适合服务接口：

| 指标 | 含义 |
| --- | --- |
| Rate | 请求速率 |
| Errors | 错误率 |
| Duration | 请求耗时 |

每个重要 API 都应该有 RED 指标。

## 8. USE 方法

USE 适合资源排查：

| 指标 | 含义 |
| --- | --- |
| Utilization | 使用率 |
| Saturation | 饱和度 |
| Errors | 错误 |

资源表：

| 资源 | Utilization | Saturation | Errors |
| --- | --- | --- | --- |
| CPU | 使用率 | run queue | throttling |
| 内存 | used/RSS/heap | swap/GC | OOM |
| 磁盘 | busy/throughput | await/queue | IO error |
| 网络 | 带宽 | 丢包/重传 | reset |
| 线程池 | active | queue | reject |
| DB 连接池 | active | pending | timeout |

## 9. SLI、SLO、SLA

| 概念 | 含义 |
| --- | --- |
| SLI | 服务等级指标，如成功率、延迟 |
| SLO | 服务等级目标，如 99.9% 请求小于 300ms |
| SLA | 对外承诺，违约可能有赔偿 |

SLO 让团队围绕用户体验管理可靠性。

### 9.1 错误预算

错误预算 = 允许失败的空间。

如果服务 SLO 是 99.9%，意味着一定周期内允许 0.1% 请求失败。错误预算耗尽后，团队应减少高风险发布，优先修稳定性问题。

## 10. 告警

好的告警应该：

- 基于用户影响。
- 可行动。
- 有明确责任人。
- 包含上下文。
- 能减少噪声。

坏告警：

- CPU 60% 就报警。
- 每分钟报一次无意义错误。
- 没有服务名和接口名。
- 没有排查链接。
- 无人处理。

告警内容建议：

- 服务。
- 接口。
- 区域。
- 版本。
- 指标值。
- 阈值。
- 持续时间。
- 可能原因。
- 排查手册链接。

## 11. 性能指标

| 指标 | 含义 |
| --- | --- |
| RT | 响应时间 |
| QPS | 每秒请求数 |
| TPS | 每秒事务数 |
| P50 | 中位数延迟 |
| P95/P99 | 尾延迟 |
| Error Rate | 错误率 |
| Throughput | 吞吐 |
| Saturation | 饱和度 |

平均值很容易骗人。P99 高说明少量用户体验很差，也可能预示系统接近瓶颈。

## 12. 性能优化流程

```text
明确目标 -> 建立基线 -> 定位瓶颈 -> 小步优化 -> 压测验证 -> 灰度上线 -> 持续观察
```

不要凭感觉优化。优化必须有指标证明。

常见瓶颈：

- 慢 SQL。
- 缓存命中率低。
- 下游慢。
- 线程池耗尽。
- 连接池耗尽。
- 锁竞争。
- 序列化慢。
- 日志过多。
- GC 压力。
- 网络抖动。

## 13. 压测

压测前定义：

- 目标 QPS。
- 目标 P95/P99。
- 数据规模。
- 请求比例。
- 是否包含登录。
- 是否包含写入。
- 是否包含第三方依赖。
- 缓存是否预热。
- 可接受错误率。

### 13.1 压测类型

| 类型 | 目的 |
| --- | --- |
| 基准测试 | 测单组件能力 |
| 负载测试 | 验证目标负载 |
| 压力测试 | 找极限和拐点 |
| 稳定性测试 | 长时间运行 |
| 峰值测试 | 模拟突发 |
| 故障演练 | 验证异常行为 |

### 13.2 压测注意

- 测试数据要接近真实。
- 不要污染生产数据。
- 注意缓存预热状态。
- 注意第三方依赖是否可压。
- 观察服务端和客户端两端指标。
- 记录版本和配置。

## 14. 容量规划

容量规划要估算：

- 峰值 QPS。
- 平均和峰值响应大小。
- 数据增长速度。
- 存储保留周期。
- 缓存容量。
- 队列积压容量。
- 数据库连接数。
- 单实例吞吐。
- 故障时剩余节点承载能力。

常见问题：

- 只按平均流量估算。
- 没考虑节假日峰值。
- 没考虑故障少节点后的容量。
- 没考虑数据增长。
- 没考虑日志和备份占用。

## 15. 故障排查方法

### 15.1 接口变慢

排查：

1. 是所有接口慢还是部分接口慢？
2. 是所有用户慢还是某地区慢？
3. 是平均慢还是 P99 慢？
4. 应用 CPU 是否高？
5. 线程池是否排队？
6. 数据库是否慢？
7. 缓存命中率是否下降？
8. 下游是否超时？
9. 最近是否发布？

### 15.2 错误率升高

排查：

- 4xx 还是 5xx？
- 是否某个版本导致？
- 是否某个下游失败？
- 是否限流触发？
- 是否依赖证书、DNS、网络变化？
- 是否请求参数异常增加？

### 15.3 数据库压力高

排查：

- 慢 SQL。
- 锁等待。
- 连接池等待。
- QPS 突增。
- 缓存失效。
- 批量任务。
- 新发布 SQL。

### 15.4 消息积压

排查：

- 生产速度是否突增？
- 消费者是否减少？
- 单条消费是否变慢？
- 下游是否失败？
- 是否有毒消息反复重试？
- 分区是否热点？

## 16. 故障复盘

复盘要回答：

- 发生了什么？
- 影响范围多大？
- 用户影响是什么？
- 什么时候发现？
- 为什么监控发现或没发现？
- 根因是什么？
- 临时措施是什么？
- 长期修复是什么？
- 如何避免再次发生？

复盘不要只追责个人，要改进系统。

## 17. 本章小结

可观测性是后端系统的眼睛。性能优化和故障排查都必须基于日志、指标、链路追踪和 Profiling。成熟系统不是不出故障，而是能更快发现、更快定位、更小范围影响、更可靠恢复。

## 讲义级补充：如何真正学懂《08-可观测性、性能容量与故障排查》

> 适用位置：后端开发完整学习笔记\08-可观测性、性能容量与故障排查.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

后端知识的核心是让业务数据在并发、故障、流量波动和安全威胁下仍然保持正确。学习时要把接口、数据、事务、缓存、队列、日志和运维放在同一张图里看。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 接口：请求如何进入系统，参数如何校验，错误如何返回。
- 数据：如何建模、持久化、索引、缓存和迁移。
- 并发：事务、锁、幂等、重试和一致性如何处理。
- 运维：日志、指标、告警、容量、发布和回滚如何保障。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把后端系统类比成餐厅后厨：API 是点餐窗口，数据库是库存账本，缓存是备好的常用食材，消息队列是排队小票，日志和监控是摄像头与账单。高峰期能否稳定服务，取决于整套流程是否有秩序。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《08-可观测性、性能容量与故障排查》时，可以设计一个下单或登录接口：先写正常流程，再补参数校验、幂等、事务、日志、限流和错误返回。这个小例子能覆盖后端工程里最常见的一组真实问题。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 从一个业务请求开始，画出 API、鉴权、服务逻辑、数据库、缓存、队列和日志的路径。
- 逐个分析并发、事务、幂等、超时、重试、限流和降级。
- 用压测、监控和故障演练验证设计，而不是只看单元测试通过。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。
6. 反例与排错：展示错误做法会导致什么现象，再说明如何定位和修复。
7. 总结迁移：最后说明它和相邻知识点的区别、联系以及后续该学什么。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。
- 记录遇到的第一个错误、定位过程和最终修复方法。
- 完成后写 5 行复盘：我原来以为是什么，实际是什么，下次会如何更快处理。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 接口能跑就认为完成，忽略并发、幂等和事务边界。
- 缓存和数据库一致性没有策略，故障时数据互相打架。
- 没有日志、指标和 trace，出问题只能猜。

遇到这些问题时，不要急着背更多资料。更有效的办法是回到一个最小例子，把输入、状态变化、输出和验证方式重新走一遍。

### 9. 自测题

1. 用一句话说明本讲主题解决的核心问题。
2. 列出本讲最重要的 3 个概念，并说明它们的关系。
3. 举一个生活类比，再指出这个类比在哪些地方不严谨。
4. 写出一个最小实践任务的验收标准。
5. 如果结果不符合预期，你会优先检查哪 3 个环节？为什么？
6. 本讲和相邻章节的知识边界是什么？哪些问题应该交给其他章节解决？

### 10. 复习口诀

先问场景，再看输入；先拆结构，再走流程；先做小例，再谈优化；先会排错，再做规模化。
