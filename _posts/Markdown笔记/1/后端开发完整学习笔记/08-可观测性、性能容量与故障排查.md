# 08-可观测性、性能容量与故障排查

> 本文目标：理解后端系统上线后如何观察、衡量和排查问题。内容覆盖日志、指标、链路追踪、Profiling、RED、USE、四大黄金信号、SLO、告警、压测、容量规划、性能优化和故障复盘。

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把后端当成一条请求生命线

这一章讲的是 **可观测性、性能容量与故障排查**。阅读时不要只背框架名、组件名或面试题答案，而要把每个概念放回一条请求生命线里：请求如何进入系统，如何被认证和校验，业务规则在哪里执行，数据如何保持一致，慢操作如何异步化，故障如何被观测，变更如何安全上线。后端学习的目标不是堆技术栈，而是能设计、实现、排查和维护一个长期运行的业务系统。

### 一句话先懂

可观测性的目标是出问题时能快速回答：谁慢了、谁错了、影响多大、从什么时候开始、下一步该查哪里。

### 通俗类比

像医院急诊分诊：日志是病历，指标是生命体征，链路追踪是检查路线，告警是分诊铃，压测和容量规划是提前演练。

类比只是帮助建立第一印象。回到工程上，要把类比里的入口、调度、仓库、通道、监控和维护分别对应到 API、业务层、数据库、缓存、消息队列、可观测性和部署运维。后端概念只有放进真实链路，才知道它解决的是正确性、性能、安全、可靠性、可维护性还是成本问题。

### 本章学习主线

1. **先看职责**：这个概念负责处理请求链路里的哪一段，输入和输出是什么。
2. **再看边界**：它不负责什么，哪些问题应该交给数据库、缓存、队列、网关、客户端或运维平台。
3. **然后看失败**：超时、重复、乱序、并发、脏数据、权限绕过、容量耗尽时会发生什么。
4. **接着看验证**：怎样用单元测试、集成测试、压测、日志、指标、trace 或故障演练证明设计可靠。
5. **最后看演进**：需求变更、流量增长、团队协作和版本升级时，这个设计是否还能维护。

### 概念怎么学才不容易忘

遇到后端概念时，建议按 白话职责 -> 链路位置 -> 最小例子 -> 常见事故 -> 观测信号 -> 修复策略 六步理解。比如缓存不是加速器这么简单，还要看命中率、TTL、一致性、热点 key 和失效策略；消息队列不是异步这么简单，还要看确认、重试、幂等、积压和补偿；JWT 不是登录态这么简单，还要看签名、过期、撤销、泄露和权限边界。

### 最小实践任务

为一个下单接口设计日志字段、RED 指标、数据库慢查询指标、消息积压指标、trace span 和告警阈值。

实践时要故意设计失败场景：重复请求、数据库超时、缓存失效、消息重复、权限不足、发布回滚、下游服务不可用。后端能力往往不是在正常路径里体现，而是在异常路径里体现。

### 读完本章应该能做到

- 用自己的话解释本章概念在后端请求链路中的位置。
- 画出最小流程图，标清入口、处理、存储、副作用、返回和观测点。
- 说出至少三个常见失败模式，以及对应的日志、指标或 trace 信号。
- 给出一个可落地的小设计，并说明它的事务、幂等、安全和回滚边界。
- 能解释日志、指标、链路追踪、RED/USE、P95/P99、SLO、压测、容量估算和故障排查顺序。

> 本节是讲义化阅读入口，后续正文中的协议、架构、数据库、缓存、消息、安全、运维和案例都应围绕这条请求生命线来理解。

## 1. 可观测性是什么

可观测性是通过系统输出推断系统内部状态的能力。

主要信号：

- Logs：日志，记录事件。
- Metrics：指标，观察趋势。
- Traces：链路追踪，查看请求路径。
- Profiles：性能剖析，分析资源消耗。

没有可观测性，系统出问题时只能猜。成熟后端系统必须从设计阶段就考虑日志、指标和追踪。

## 2. 日志

### 2.1 日志的作用

日志用于回答：

- 发生了什么？
- 谁触发的？
- 影响哪个业务对象？
- 请求走到了哪一步？
- 失败原因是什么？
- 是否发生重试、降级、超时？

### 2.2 日志级别

| 级别 | 用途 |
| --- | --- |
| DEBUG | 调试细节，生产慎用 |
| INFO | 关键业务事件 |
| WARN | 可恢复异常或异常趋势 |
| ERROR | 请求失败或需要处理的问题 |
| FATAL | 进程级严重错误 |

### 2.3 结构化日志

推荐结构化：

```json
{
  "time": "2026-06-16T10:00:00Z",
  "level": "INFO",
  "traceId": "trace_001",
  "requestId": "req_001",
  "event": "payment_callback_received",
  "orderId": "order_1",
  "paymentId": "pay_1",
  "status": "SUCCESS"
}
```

结构化日志便于检索、聚合和告警。

### 2.4 日志脱敏

不能记录：

- 密码。
- Token。
- 私钥。
- 身份证完整值。
- 银行卡完整值。
- 支付敏感信息。

可记录脱敏值：

```text
phone=138****1234
card=6222********1234
```

## 3. 指标

指标适合观察趋势和告警。

### 3.1 指标类型

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| Counter | 单调递增计数 | 请求总数 |
| Gauge | 当前值 | 队列长度、内存使用 |
| Histogram | 分布 | 请求耗时 |
| Summary | 摘要 | 分位数 |

### 3.2 服务指标

每个核心服务应有：

- 请求量。
- 错误率。
- P50/P95/P99。
- 并发数。
- 线程池活跃数。
- 队列长度。
- 连接池等待时间。
- 下游调用耗时。
- 缓存命中率。
- 消息积压。

## 4. 链路追踪

链路追踪用于查看一次请求跨服务经过了哪里。

术语：

| 术语 | 含义 |
| --- | --- |
| Trace | 一次完整请求链路 |
| Span | 一个操作片段 |
| Trace ID | 链路 ID |
| Span ID | 片段 ID |
| Parent Span | 父片段 |

追踪能回答：

- 哪个服务最慢？
- 哪个数据库查询最慢？
- 是否发生重试？
- 是否走了降级？
- 请求在哪个节点失败？

### 4.1 Trace Context

跨服务传递 trace 信息需要统一上下文。W3C Trace Context 定义了 `traceparent` 等标准 Header。

后端要确保：

- 入口生成 trace ID。
- 调用下游传递 trace ID。
- 日志中包含 trace ID。
- 异步消息也尽量传递 trace 信息。

## 5. Profiling

Profiling 用于分析资源消耗。

常见类型：

- CPU Profile。
- Memory Profile。
- Lock Profile。
- IO Profile。
- Allocation Profile。

适用场景：

- CPU 高但不知道哪里消耗。
- 内存增长。
- GC 压力大。
- 锁竞争严重。
- 序列化或压缩耗时高。

Profiling 要注意对生产性能的影响，通常采用采样和短时间抓取。

## 6. 四大黄金信号

Google SRE 提出的四大黄金信号：

| 信号 | 含义 |
| --- | --- |
| Latency | 请求延迟 |
| Traffic | 流量 |
| Errors | 错误 |
| Saturation | 饱和度 |

### 6.1 Latency

延迟要区分：

- 成功请求延迟。
- 失败请求延迟。
- 下游调用延迟。
- 排队等待延迟。

不要只看平均值，要看 P95/P99。

### 6.2 Traffic

流量包括：

- QPS。
- 请求体大小。
- 响应体大小。
- 消息生产速度。
- 消息消费速度。

### 6.3 Errors

错误包括：

- 5xx。
- 业务失败。
- 超时。
- 下游错误。
- 消息消费失败。

### 6.4 Saturation

饱和度表示资源接近极限：

- CPU run queue。
- 线程池队列。
- 连接池等待。
- 磁盘 IO 队列。
- MQ 积压。
- Redis 延迟。

## 7. RED 方法

RED 适合服务接口：

| 指标 | 含义 |
| --- | --- |
| Rate | 请求速率 |
| Errors | 错误率 |
| Duration | 请求耗时 |

每个重要 API 都应该有 RED 指标。

## 8. USE 方法

USE 适合资源排查：

| 指标 | 含义 |
| --- | --- |
| Utilization | 使用率 |
| Saturation | 饱和度 |
| Errors | 错误 |

资源表：

| 资源 | Utilization | Saturation | Errors |
| --- | --- | --- | --- |
| CPU | 使用率 | run queue | throttling |
| 内存 | used/RSS/heap | swap/GC | OOM |
| 磁盘 | busy/throughput | await/queue | IO error |
| 网络 | 带宽 | 丢包/重传 | reset |
| 线程池 | active | queue | reject |
| DB 连接池 | active | pending | timeout |

## 9. SLI、SLO、SLA

| 概念 | 含义 |
| --- | --- |
| SLI | 服务等级指标，如成功率、延迟 |
| SLO | 服务等级目标，如 99.9% 请求小于 300ms |
| SLA | 对外承诺，违约可能有赔偿 |

SLO 让团队围绕用户体验管理可靠性。

### 9.1 错误预算

错误预算 = 允许失败的空间。

如果服务 SLO 是 99.9%，意味着一定周期内允许 0.1% 请求失败。错误预算耗尽后，团队应减少高风险发布，优先修稳定性问题。

## 10. 告警

好的告警应该：

- 基于用户影响。
- 可行动。
- 有明确责任人。
- 包含上下文。
- 能减少噪声。

坏告警：

- CPU 60% 就报警。
- 每分钟报一次无意义错误。
- 没有服务名和接口名。
- 没有排查链接。
- 无人处理。

告警内容建议：

- 服务。
- 接口。
- 区域。
- 版本。
- 指标值。
- 阈值。
- 持续时间。
- 可能原因。
- 排查手册链接。

## 11. 性能指标

| 指标 | 含义 |
| --- | --- |
| RT | 响应时间 |
| QPS | 每秒请求数 |
| TPS | 每秒事务数 |
| P50 | 中位数延迟 |
| P95/P99 | 尾延迟 |
| Error Rate | 错误率 |
| Throughput | 吞吐 |
| Saturation | 饱和度 |

平均值很容易骗人。P99 高说明少量用户体验很差，也可能预示系统接近瓶颈。

## 12. 性能优化流程

```text
明确目标 -> 建立基线 -> 定位瓶颈 -> 小步优化 -> 压测验证 -> 灰度上线 -> 持续观察
```

不要凭感觉优化。优化必须有指标证明。

常见瓶颈：

- 慢 SQL。
- 缓存命中率低。
- 下游慢。
- 线程池耗尽。
- 连接池耗尽。
- 锁竞争。
- 序列化慢。
- 日志过多。
- GC 压力。
- 网络抖动。

## 13. 压测

压测前定义：

- 目标 QPS。
- 目标 P95/P99。
- 数据规模。
- 请求比例。
- 是否包含登录。
- 是否包含写入。
- 是否包含第三方依赖。
- 缓存是否预热。
- 可接受错误率。

### 13.1 压测类型

| 类型 | 目的 |
| --- | --- |
| 基准测试 | 测单组件能力 |
| 负载测试 | 验证目标负载 |
| 压力测试 | 找极限和拐点 |
| 稳定性测试 | 长时间运行 |
| 峰值测试 | 模拟突发 |
| 故障演练 | 验证异常行为 |

### 13.2 压测注意

- 测试数据要接近真实。
- 不要污染生产数据。
- 注意缓存预热状态。
- 注意第三方依赖是否可压。
- 观察服务端和客户端两端指标。
- 记录版本和配置。

## 14. 容量规划

容量规划要估算：

- 峰值 QPS。
- 平均和峰值响应大小。
- 数据增长速度。
- 存储保留周期。
- 缓存容量。
- 队列积压容量。
- 数据库连接数。
- 单实例吞吐。
- 故障时剩余节点承载能力。

常见问题：

- 只按平均流量估算。
- 没考虑节假日峰值。
- 没考虑故障少节点后的容量。
- 没考虑数据增长。
- 没考虑日志和备份占用。

## 15. 故障排查方法

### 15.1 接口变慢

排查：

1. 是所有接口慢还是部分接口慢？
2. 是所有用户慢还是某地区慢？
3. 是平均慢还是 P99 慢？
4. 应用 CPU 是否高？
5. 线程池是否排队？
6. 数据库是否慢？
7. 缓存命中率是否下降？
8. 下游是否超时？
9. 最近是否发布？

### 15.2 错误率升高

排查：

- 4xx 还是 5xx？
- 是否某个版本导致？
- 是否某个下游失败？
- 是否限流触发？
- 是否依赖证书、DNS、网络变化？
- 是否请求参数异常增加？

### 15.3 数据库压力高

排查：

- 慢 SQL。
- 锁等待。
- 连接池等待。
- QPS 突增。
- 缓存失效。
- 批量任务。
- 新发布 SQL。

### 15.4 消息积压

排查：

- 生产速度是否突增？
- 消费者是否减少？
- 单条消费是否变慢？
- 下游是否失败？
- 是否有毒消息反复重试？
- 分区是否热点？

## 16. 故障复盘

复盘要回答：

- 发生了什么？
- 影响范围多大？
- 用户影响是什么？
- 什么时候发现？
- 为什么监控发现或没发现？
- 根因是什么？
- 临时措施是什么？
- 长期修复是什么？
- 如何避免再次发生？

复盘不要只追责个人，要改进系统。

## 17. 本章小结

可观测性是后端系统的眼睛。性能优化和故障排查都必须基于日志、指标、链路追踪和 Profiling。成熟系统不是不出故障，而是能更快发现、更快定位、更小范围影响、更可靠恢复。

## 2026 后端资料与工程核对补充

后端基础概念相对稳定，但规范、组件版本和安全风险会持续变化。复现实践前，建议记录运行时版本、框架版本、数据库版本、缓存和消息队列版本、容器镜像、Kubernetes 版本、云厂商组件、配置文件、迁移脚本和压测环境。不要只记录代码提交，还要记录依赖和运行条件。

学习后端时建议优先核对官方规范和项目文档：HTTP 语义看 RFC 9110，HTTP 缓存看 RFC 9111，API 安全风险看 OWASP API Security Top 10，观测体系看 OpenTelemetry，数据库事务和索引看对应数据库官方文档，容器编排看 Kubernetes 官方文档。社区文章适合补充事故经验和踩坑案例，但不能替代规范和官方文档。
