# 08-可观测性、性能容量与故障排查

> 本文目标：理解后端系统上线后如何观察、衡量和排查问题。内容覆盖日志、指标、链路追踪、Profiling、RED、USE、四大黄金信号、SLO、告警、压测、容量规划、性能优化和故障复盘。

## 1. 可观测性是什么

可观测性是通过系统输出推断系统内部状态的能力。

主要信号：

- Logs：日志，记录事件。
- Metrics：指标，观察趋势。
- Traces：链路追踪，查看请求路径。
- Profiles：性能剖析，分析资源消耗。

没有可观测性，系统出问题时只能猜。成熟后端系统必须从设计阶段就考虑日志、指标和追踪。

## 2. 日志

### 2.1 日志的作用

日志用于回答：

- 发生了什么？
- 谁触发的？
- 影响哪个业务对象？
- 请求走到了哪一步？
- 失败原因是什么？
- 是否发生重试、降级、超时？

### 2.2 日志级别

| 级别 | 用途 |
| --- | --- |
| DEBUG | 调试细节，生产慎用 |
| INFO | 关键业务事件 |
| WARN | 可恢复异常或异常趋势 |
| ERROR | 请求失败或需要处理的问题 |
| FATAL | 进程级严重错误 |

### 2.3 结构化日志

推荐结构化：

```json
{
  "time": "2026-06-16T10:00:00Z",
  "level": "INFO",
  "traceId": "trace_001",
  "requestId": "req_001",
  "event": "payment_callback_received",
  "orderId": "order_1",
  "paymentId": "pay_1",
  "status": "SUCCESS"
}
```

结构化日志便于检索、聚合和告警。

### 2.4 日志脱敏

不能记录：

- 密码。
- Token。
- 私钥。
- 身份证完整值。
- 银行卡完整值。
- 支付敏感信息。

可记录脱敏值：

```text
phone=138****1234
card=6222********1234
```

## 3. 指标

指标适合观察趋势和告警。

### 3.1 指标类型

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| Counter | 单调递增计数 | 请求总数 |
| Gauge | 当前值 | 队列长度、内存使用 |
| Histogram | 分布 | 请求耗时 |
| Summary | 摘要 | 分位数 |

### 3.2 服务指标

每个核心服务应有：

- 请求量。
- 错误率。
- P50/P95/P99。
- 并发数。
- 线程池活跃数。
- 队列长度。
- 连接池等待时间。
- 下游调用耗时。
- 缓存命中率。
- 消息积压。

## 4. 链路追踪

链路追踪用于查看一次请求跨服务经过了哪里。

术语：

| 术语 | 含义 |
| --- | --- |
| Trace | 一次完整请求链路 |
| Span | 一个操作片段 |
| Trace ID | 链路 ID |
| Span ID | 片段 ID |
| Parent Span | 父片段 |

追踪能回答：

- 哪个服务最慢？
- 哪个数据库查询最慢？
- 是否发生重试？
- 是否走了降级？
- 请求在哪个节点失败？

### 4.1 Trace Context

跨服务传递 trace 信息需要统一上下文。W3C Trace Context 定义了 `traceparent` 等标准 Header。

后端要确保：

- 入口生成 trace ID。
- 调用下游传递 trace ID。
- 日志中包含 trace ID。
- 异步消息也尽量传递 trace 信息。

## 5. Profiling

Profiling 用于分析资源消耗。

常见类型：

- CPU Profile。
- Memory Profile。
- Lock Profile。
- IO Profile。
- Allocation Profile。

适用场景：

- CPU 高但不知道哪里消耗。
- 内存增长。
- GC 压力大。
- 锁竞争严重。
- 序列化或压缩耗时高。

Profiling 要注意对生产性能的影响，通常采用采样和短时间抓取。

## 6. 四大黄金信号

Google SRE 提出的四大黄金信号：

| 信号 | 含义 |
| --- | --- |
| Latency | 请求延迟 |
| Traffic | 流量 |
| Errors | 错误 |
| Saturation | 饱和度 |

### 6.1 Latency

延迟要区分：

- 成功请求延迟。
- 失败请求延迟。
- 下游调用延迟。
- 排队等待延迟。

不要只看平均值，要看 P95/P99。

### 6.2 Traffic

流量包括：

- QPS。
- 请求体大小。
- 响应体大小。
- 消息生产速度。
- 消息消费速度。

### 6.3 Errors

错误包括：

- 5xx。
- 业务失败。
- 超时。
- 下游错误。
- 消息消费失败。

### 6.4 Saturation

饱和度表示资源接近极限：

- CPU run queue。
- 线程池队列。
- 连接池等待。
- 磁盘 IO 队列。
- MQ 积压。
- Redis 延迟。

## 7. RED 方法

RED 适合服务接口：

| 指标 | 含义 |
| --- | --- |
| Rate | 请求速率 |
| Errors | 错误率 |
| Duration | 请求耗时 |

每个重要 API 都应该有 RED 指标。

## 8. USE 方法

USE 适合资源排查：

| 指标 | 含义 |
| --- | --- |
| Utilization | 使用率 |
| Saturation | 饱和度 |
| Errors | 错误 |

资源表：

| 资源 | Utilization | Saturation | Errors |
| --- | --- | --- | --- |
| CPU | 使用率 | run queue | throttling |
| 内存 | used/RSS/heap | swap/GC | OOM |
| 磁盘 | busy/throughput | await/queue | IO error |
| 网络 | 带宽 | 丢包/重传 | reset |
| 线程池 | active | queue | reject |
| DB 连接池 | active | pending | timeout |

## 9. SLI、SLO、SLA

| 概念 | 含义 |
| --- | --- |
| SLI | 服务等级指标，如成功率、延迟 |
| SLO | 服务等级目标，如 99.9% 请求小于 300ms |
| SLA | 对外承诺，违约可能有赔偿 |

SLO 让团队围绕用户体验管理可靠性。

### 9.1 错误预算

错误预算 = 允许失败的空间。

如果服务 SLO 是 99.9%，意味着一定周期内允许 0.1% 请求失败。错误预算耗尽后，团队应减少高风险发布，优先修稳定性问题。

## 10. 告警

好的告警应该：

- 基于用户影响。
- 可行动。
- 有明确责任人。
- 包含上下文。
- 能减少噪声。

坏告警：

- CPU 60% 就报警。
- 每分钟报一次无意义错误。
- 没有服务名和接口名。
- 没有排查链接。
- 无人处理。

告警内容建议：

- 服务。
- 接口。
- 区域。
- 版本。
- 指标值。
- 阈值。
- 持续时间。
- 可能原因。
- 排查手册链接。

## 11. 性能指标

| 指标 | 含义 |
| --- | --- |
| RT | 响应时间 |
| QPS | 每秒请求数 |
| TPS | 每秒事务数 |
| P50 | 中位数延迟 |
| P95/P99 | 尾延迟 |
| Error Rate | 错误率 |
| Throughput | 吞吐 |
| Saturation | 饱和度 |

平均值很容易骗人。P99 高说明少量用户体验很差，也可能预示系统接近瓶颈。

## 12. 性能优化流程

```text
明确目标 -> 建立基线 -> 定位瓶颈 -> 小步优化 -> 压测验证 -> 灰度上线 -> 持续观察
```

不要凭感觉优化。优化必须有指标证明。

常见瓶颈：

- 慢 SQL。
- 缓存命中率低。
- 下游慢。
- 线程池耗尽。
- 连接池耗尽。
- 锁竞争。
- 序列化慢。
- 日志过多。
- GC 压力。
- 网络抖动。

## 13. 压测

压测前定义：

- 目标 QPS。
- 目标 P95/P99。
- 数据规模。
- 请求比例。
- 是否包含登录。
- 是否包含写入。
- 是否包含第三方依赖。
- 缓存是否预热。
- 可接受错误率。

### 13.1 压测类型

| 类型 | 目的 |
| --- | --- |
| 基准测试 | 测单组件能力 |
| 负载测试 | 验证目标负载 |
| 压力测试 | 找极限和拐点 |
| 稳定性测试 | 长时间运行 |
| 峰值测试 | 模拟突发 |
| 故障演练 | 验证异常行为 |

### 13.2 压测注意

- 测试数据要接近真实。
- 不要污染生产数据。
- 注意缓存预热状态。
- 注意第三方依赖是否可压。
- 观察服务端和客户端两端指标。
- 记录版本和配置。

## 14. 容量规划

容量规划要估算：

- 峰值 QPS。
- 平均和峰值响应大小。
- 数据增长速度。
- 存储保留周期。
- 缓存容量。
- 队列积压容量。
- 数据库连接数。
- 单实例吞吐。
- 故障时剩余节点承载能力。

常见问题：

- 只按平均流量估算。
- 没考虑节假日峰值。
- 没考虑故障少节点后的容量。
- 没考虑数据增长。
- 没考虑日志和备份占用。

## 15. 故障排查方法

### 15.1 接口变慢

排查：

1. 是所有接口慢还是部分接口慢？
2. 是所有用户慢还是某地区慢？
3. 是平均慢还是 P99 慢？
4. 应用 CPU 是否高？
5. 线程池是否排队？
6. 数据库是否慢？
7. 缓存命中率是否下降？
8. 下游是否超时？
9. 最近是否发布？

### 15.2 错误率升高

排查：

- 4xx 还是 5xx？
- 是否某个版本导致？
- 是否某个下游失败？
- 是否限流触发？
- 是否依赖证书、DNS、网络变化？
- 是否请求参数异常增加？

### 15.3 数据库压力高

排查：

- 慢 SQL。
- 锁等待。
- 连接池等待。
- QPS 突增。
- 缓存失效。
- 批量任务。
- 新发布 SQL。

### 15.4 消息积压

排查：

- 生产速度是否突增？
- 消费者是否减少？
- 单条消费是否变慢？
- 下游是否失败？
- 是否有毒消息反复重试？
- 分区是否热点？

## 16. 故障复盘

复盘要回答：

- 发生了什么？
- 影响范围多大？
- 用户影响是什么？
- 什么时候发现？
- 为什么监控发现或没发现？
- 根因是什么？
- 临时措施是什么？
- 长期修复是什么？
- 如何避免再次发生？

复盘不要只追责个人，要改进系统。

## 17. 本章小结

可观测性是后端系统的眼睛。性能优化和故障排查都必须基于日志、指标、链路追踪和 Profiling。成熟系统不是不出故障，而是能更快发现、更快定位、更小范围影响、更可靠恢复。
