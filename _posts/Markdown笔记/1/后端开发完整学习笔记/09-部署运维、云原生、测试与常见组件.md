# 09-部署运维、云原生、测试与常见组件

> 本文目标：理解后端系统如何从代码变成稳定运行的线上服务。内容覆盖 12-Factor、环境配置、容器、Kubernetes、CI/CD、发布策略、测试体系、反向代理、负载均衡、搜索引擎、对象存储、定时任务和常见运维问题。

## 1. 后端交付链路

后端不是写完代码就结束。真实交付链路包括：

```text
需求 -> 设计 -> 开发 -> 测试 -> 构建 -> 安全扫描 -> 部署 -> 灰度 -> 监控 -> 回滚/全量 -> 复盘
```

每一步都可能影响系统质量。

成熟后端团队追求：

- 构建可重复。
- 配置可管理。
- 发布可灰度。
- 故障可回滚。
- 运行可观察。
- 数据可恢复。

## 2. 12-Factor App

12-Factor 是云原生应用的重要方法论。

核心思想：

| 原则 | 含义 |
| --- | --- |
| Codebase | 一个代码库，多环境部署 |
| Dependencies | 显式声明依赖 |
| Config | 配置与代码分离 |
| Backing services | 后端服务作为附加资源 |
| Build release run | 构建、发布、运行分离 |
| Processes | 应用进程无状态 |
| Port binding | 通过端口提供服务 |
| Concurrency | 通过进程模型扩展 |
| Disposability | 快速启动和优雅关闭 |
| Dev/prod parity | 开发、预发、生产尽量一致 |
| Logs | 日志作为事件流 |
| Admin processes | 管理任务一次性运行 |

这些原则的核心是让应用更容易自动化部署和水平扩展。

## 3. 环境与配置

常见环境：

- 本地开发。
- 测试。
- 预发。
- 灰度。
- 生产。

配置分类：

- 普通配置：端口、日志级别。
- 环境配置：数据库地址、缓存地址。
- 业务配置：开关、阈值。
- 敏感配置：密钥、Token、证书。

配置管理原则：

- 不同环境配置隔离。
- 敏感配置进入 Secret。
- 配置变更可审计。
- 配置支持回滚。
- 配置启动时校验。

## 4. 容器

容器把应用和依赖打包成镜像，提供一致运行环境。

### 4.1 容器解决什么

- 本地和线上环境不一致。
- 依赖安装复杂。
- 部署步骤不稳定。
- 多应用隔离。
- 快速扩缩容。

### 4.2 镜像

镜像是只读模板。镜像设计关注：

- 基础镜像安全。
- 镜像大小。
- 依赖版本。
- 构建可重复。
- 非 root 用户运行。
- 不把密钥写进镜像。

### 4.3 容器运行注意

- 容器不是虚拟机。
- 容器文件系统可能随重启丢失。
- 日志应输出到标准输出。
- 应用要支持优雅关闭。
- 要设置 CPU 和内存限制。

## 5. Kubernetes

Kubernetes 是容器编排平台。

### 5.1 核心概念

| 概念 | 含义 |
| --- | --- |
| Pod | 最小调度单元 |
| Deployment | 管理无状态副本 |
| StatefulSet | 管理有状态服务 |
| Service | 稳定访问入口 |
| Ingress | HTTP/HTTPS 入口 |
| ConfigMap | 非敏感配置 |
| Secret | 敏感配置 |
| Namespace | 逻辑隔离 |
| HPA | 自动水平扩缩容 |
| Probe | 健康检查 |

### 5.2 健康检查

常见探针：

- Liveness Probe：判断是否需要重启。
- Readiness Probe：判断是否可以接流量。
- Startup Probe：给慢启动应用更多时间。

错误配置风险：

- Readiness 太早成功，未初始化完成就接流量。
- Liveness 太激进，应用正常慢启动却被反复杀。
- 健康接口依赖数据库导致数据库短暂抖动时全服务被摘。

### 5.3 优雅关闭

应用收到终止信号后应：

1. 停止接收新请求。
2. 等待正在处理的请求完成。
3. 停止消费消息。
4. 释放资源。
5. 退出进程。

如果不支持优雅关闭，滚动发布时可能丢请求或重复消费。

## 6. CI/CD

CI/CD 目标是让发布自动化、可重复、可追踪。

典型流水线：

```text
代码提交 -> 静态检查 -> 单元测试 -> 集成测试 -> 构建镜像 -> 安全扫描 -> 部署测试 -> 灰度 -> 全量
```

### 6.1 CI

CI 关注：

- 编译。
- 测试。
- 代码质量。
- 依赖检查。
- 安全扫描。

### 6.2 CD

CD 关注：

- 发布自动化。
- 环境一致。
- 灰度策略。
- 回滚能力。
- 发布记录。

## 7. 发布策略

| 策略 | 含义 | 优点 | 风险 |
| --- | --- | --- | --- |
| 滚动发布 | 分批替换实例 | 资源省 | 新旧版本共存 |
| 蓝绿发布 | 两套环境切换 | 回滚快 | 资源成本高 |
| 金丝雀发布 | 小流量验证 | 风险低 | 需要流量控制 |
| 灰度发布 | 按用户/区域/比例放量 | 精细控制 | 规则复杂 |
| 特性开关 | 代码发布和功能发布分离 | 快速关闭 | 开关治理成本 |

发布前要确认：

- 数据库变更是否兼容。
- API 是否兼容。
- 配置是否正确。
- 回滚是否可行。
- 监控和告警是否覆盖。

## 8. 数据库变更发布

数据库变更风险很高。

安全做法：

- 先加字段，后用字段，再清旧字段。
- 字段新增优先兼容旧代码。
- 大表变更避开高峰。
- 建索引评估锁和耗时。
- 数据迁移分批执行。
- 保留回滚方案。

不安全做法：

- 直接删除线上字段。
- 改字段类型不兼容旧代码。
- 大表一次性全量更新。
- 上线时执行长事务。

## 9. 测试体系

### 9.1 测试类型

| 类型 | 目的 |
| --- | --- |
| 单元测试 | 验证小逻辑 |
| 集成测试 | 验证模块和依赖 |
| 契约测试 | 验证服务间接口 |
| 端到端测试 | 验证完整流程 |
| 回归测试 | 防旧功能破坏 |
| 性能测试 | 验证容量 |
| 安全测试 | 发现漏洞 |
| 混沌测试 | 验证故障韧性 |

### 9.2 测试金字塔

合理结构：

- 大量单元测试。
- 适量集成测试。
- 少量关键端到端测试。

端到端测试价值高但慢、脆弱、维护成本高，不适合覆盖所有细节。

### 9.3 测试数据

测试数据要覆盖：

- 正常数据。
- 空值。
- 边界值。
- 非法值。
- 重复请求。
- 并发冲突。
- 权限差异。
- 大数据量。
- 外部服务失败。

## 10. 反向代理

反向代理代表服务端接收请求，再转发到后端。

职责：

- TLS 终止。
- 路由。
- 负载均衡。
- 静态资源。
- 压缩。
- 缓存。
- 限流。
- 请求头处理。

常见问题：

- 请求体大小限制。
- 超时配置不合理。
- 转发 Header 缺失。
- WebSocket 未正确升级。
- 上传大文件失败。

## 11. 负载均衡

负载均衡把流量分发到多个实例。

算法：

| 算法 | 特点 |
| --- | --- |
| 轮询 | 简单 |
| 加权轮询 | 实例能力不同 |
| 最少连接 | 适合长连接 |
| 一致性哈希 | 会话亲和或缓存亲和 |
| 随机 | 简单有效 |

负载均衡要配合健康检查，否则会把流量发给坏实例。

## 12. 搜索引擎

搜索引擎用于全文检索和复杂查询。

核心概念：

- 文档。
- 倒排索引。
- 分词。
- 相关性评分。
- 分片。
- 副本。
- 索引刷新。

适合：

- 商品搜索。
- 文章搜索。
- 日志检索。
- 多字段过滤。
- 高亮。

注意：

- 搜索索引不是权威数据源。
- 索引更新通常有延迟。
- 权限过滤要谨慎。
- 复杂聚合查询成本高。

## 13. 对象存储

对象存储用于文件。

概念：

- Bucket。
- Object。
- Key。
- Metadata。
- Presigned URL。
- Lifecycle。

常见上传模式：

1. 客户端向后端申请上传凭证。
2. 后端生成临时 URL。
3. 客户端直传对象存储。
4. 后端保存文件元数据。

优点：

- 减轻后端带宽压力。
- 文件存储可靠。
- 可配合 CDN。

注意：

- 文件类型校验。
- 大小限制。
- 权限控制。
- 防盗链。
- 生命周期清理。

## 14. 定时任务

定时任务用于周期性处理：

- 订单超时关闭。
- 数据同步。
- 报表生成。
- 清理过期数据。
- 重试补偿。

多实例问题：

- 同一个任务可能重复执行。
- 节点宕机任务中断。
- 长任务超过周期。

设计要求：

- 幂等。
- 分布式锁或分片。
- 执行记录。
- 失败告警。
- 可重试。
- 可人工触发。

## 15. 运维常见问题

| 问题 | 可能原因 |
| --- | --- |
| 发布后大量 502 | 服务未就绪、健康检查错误、端口错误 |
| 发布后请求变慢 | 新版本慢查询、连接池变化、缓存失效 |
| Pod 反复重启 | 内存限制、探针错误、启动失败 |
| 磁盘满 | 日志未轮转、临时文件堆积 |
| 配置未生效 | 配置中心缓存、环境变量错误 |
| 回滚失败 | 数据库变更不兼容 |

## 16. 本章小结

部署运维能力决定代码能否稳定变成服务。后端工程师不一定要成为专职运维，但必须理解容器、配置、发布、健康检查、灰度、回滚、测试和常见组件，否则很难对线上结果负责。
