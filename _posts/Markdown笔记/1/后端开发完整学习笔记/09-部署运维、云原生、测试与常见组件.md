## 讲义导读：把后端当成一条请求生命线

这一章讲的是 **部署运维、云原生、测试与常见组件**。阅读时不要只背框架名、组件名或面试题答案，而要把每个概念放回一条请求生命线里：请求如何进入系统，如何被认证和校验，业务规则在哪里执行，数据如何保持一致，慢操作如何异步化，故障如何被观测，变更如何安全上线。后端学习的目标不是堆技术栈，而是能设计、实现、排查和维护一个长期运行的业务系统。

### 一句话先懂

能上线不等于能运行，后端交付要覆盖构建、配置、测试、镜像、安全扫描、灰度、回滚、资源限制和运行监控。

### 通俗类比

像把餐厅开到连锁门店：菜谱要标准化，原料要可追踪，试营业要可回滚，消防和卫生要检查，每家店的客流和库存都要监控。

类比只是帮助建立第一印象。回到工程上，要把类比里的入口、调度、仓库、通道、监控和维护分别对应到 API、业务层、数据库、缓存、消息队列、可观测性和部署运维。后端概念只有放进真实链路，才知道它解决的是正确性、性能、安全、可靠性、可维护性还是成本问题。

### 本章学习主线

1. **先看职责**：这个概念负责处理请求链路里的哪一段，输入和输出是什么。
2. **再看边界**：它不负责什么，哪些问题应该交给数据库、缓存、队列、网关、客户端或运维平台。
3. **然后看失败**：超时、重复、乱序、并发、脏数据、权限绕过、容量耗尽时会发生什么。
4. **接着看验证**：怎样用单元测试、集成测试、压测、日志、指标、trace 或故障演练证明设计可靠。
5. **最后看演进**：需求变更、流量增长、团队协作和版本升级时，这个设计是否还能维护。

### 概念怎么学才不容易忘

遇到后端概念时，建议按 白话职责 -> 链路位置 -> 最小例子 -> 常见事故 -> 观测信号 -> 修复策略 六步理解。比如缓存不是加速器这么简单，还要看命中率、TTL、一致性、热点 key 和失效策略；消息队列不是异步这么简单，还要看确认、重试、幂等、积压和补偿；JWT 不是登录态这么简单，还要看签名、过期、撤销、泄露和权限边界。

### 最小实践任务

为一个服务写发布清单：配置项、数据库迁移、镜像版本、健康检查、灰度策略、回滚方案、测试范围和监控看板。

实践时要故意设计失败场景：重复请求、数据库超时、缓存失效、消息重复、权限不足、发布回滚、下游服务不可用。后端能力往往不是在正常路径里体现，而是在异常路径里体现。

### 读完本章应该能做到

- 用自己的话解释本章概念在后端请求链路中的位置。
- 画出最小流程图，标清入口、处理、存储、副作用、返回和观测点。
- 说出至少三个常见失败模式，以及对应的日志、指标或 trace 信号。
- 给出一个可落地的小设计，并说明它的事务、幂等、安全和回滚边界。
- 能解释 CI/CD、Docker、Kubernetes、Service、Ingress、ConfigMap、Secret、健康检查、测试金字塔和发布风险控制。

## 1. 后端交付链路

后端不是写完代码就结束。真实交付链路包括：

```text
需求 -> 设计 -> 开发 -> 测试 -> 构建 -> 安全扫描 -> 部署 -> 灰度 -> 监控 -> 回滚/全量 -> 复盘
```

每一步都可能影响系统质量。

成熟后端团队追求：

- 构建可重复。
- 配置可管理。
- 发布可灰度。
- 故障可回滚。
- 运行可观察。
- 数据可恢复。

## 2. 12-Factor App

12-Factor 是云原生应用的重要方法论。

核心思想：

| 原则 | 含义 |
| --- | --- |
| Codebase | 一个代码库，多环境部署 |
| Dependencies | 显式声明依赖 |
| Config | 配置与代码分离 |
| Backing services | 后端服务作为附加资源 |
| Build release run | 构建、发布、运行分离 |
| Processes | 应用进程无状态 |
| Port binding | 通过端口提供服务 |
| Concurrency | 通过进程模型扩展 |
| Disposability | 快速启动和优雅关闭 |
| Dev/prod parity | 开发、预发、生产尽量一致 |
| Logs | 日志作为事件流 |
| Admin processes | 管理任务一次性运行 |

这些原则的核心是让应用更容易自动化部署和水平扩展。

## 3. 环境与配置

常见环境：

- 本地开发。
- 测试。
- 预发。
- 灰度。
- 生产。

配置分类：

- 普通配置：端口、日志级别。
- 环境配置：数据库地址、缓存地址。
- 业务配置：开关、阈值。

- 配置支持回滚。
- 配置启动时校验。

## 4. 容器

容器把应用和依赖打包成镜像，提供一致运行环境。

### 4.1 容器解决什么

- 本地和线上环境不一致。
- 依赖安装复杂。
- 部署步骤不稳定。
- 多应用隔离。
- 快速扩缩容。

### 4.2 镜像

镜像是只读模板。镜像设计关注：

- 基础镜像安全。
- 镜像大小。
- 依赖版本。
- 构建可重复。
- 非 root 用户运行。
- 不把密钥写进镜像。

### 4.3 容器运行注意

- 容器不是虚拟机。
- 容器文件系统可能随重启丢失。
- 日志应输出到标准输出。
- 应用要支持优雅关闭。
- 要设置 CPU 和内存限制。

## 5. Kubernetes

Kubernetes 是容器编排平台。

### 5.1 核心概念

| 概念 | 含义 |
| --- | --- |
| Pod | 最小调度单元 |
| Deployment | 管理无状态副本 |
| StatefulSet | 管理有状态服务 |
| Service | 稳定访问入口 |
| Ingress | HTTP/HTTPS 入口 |
| ConfigMap | 非敏感配置 |
| Secret | 敏感配置 |
| Namespace | 逻辑隔离 |
| HPA | 自动水平扩缩容 |
| Probe | 健康检查 |

### 5.2 健康检查

常见探针：

- Liveness Probe：判断是否需要重启。
- Readiness Probe：判断是否可以接流量。
- Startup Probe：给慢启动应用更多时间。

错误配置风险：

- Readiness 太早成功，未初始化完成就接流量。
- Liveness 太激进，应用正常慢启动却被反复杀。
- 健康接口依赖数据库导致数据库短暂抖动时全服务被摘。

### 5.3 优雅关闭

应用收到终止信号后应：

1. 停止接收新请求。
2. 等待正在处理的请求完成。
3. 停止消费消息。
4. 释放资源。
5. 退出进程。

如果不支持优雅关闭，滚动发布时可能丢请求或重复消费。

## 6. CI/CD

CI/CD 目标是让发布自动化、可重复、可追踪。

典型流水线：

```text
代码提交 -> 静态检查 -> 单元测试 -> 集成测试 -> 构建镜像 -> 安全扫描 -> 部署测试 -> 灰度 -> 全量
```

### 6.1 CI

CI 关注：

- 编译。
- 测试。
- 代码质量。
- 依赖检查。
- 安全扫描。

### 6.2 CD

CD 关注：

- 发布自动化。
- 环境一致。
- 灰度策略。
- 回滚能力。
- 发布记录。

## 7. 发布策略

| 策略 | 含义 | 优点 | 风险 |
| --- | --- | --- | --- |
| 滚动发布 | 分批替换实例 | 资源省 | 新旧版本共存 |
| 蓝绿发布 | 两套环境切换 | 回滚快 | 资源成本高 |
| 金丝雀发布 | 小流量验证 | 风险低 | 需要流量控制 |
| 灰度发布 | 按用户/区域/比例放量 | 精细控制 | 规则复杂 |
| 特性开关 | 代码发布和功能发布分离 | 快速关闭 | 开关治理成本 |

发布前要确认：

- 数据库变更是否兼容。
- API 是否兼容。
- 配置是否正确。
- 回滚是否可行。
- 监控和告警是否覆盖。

## 8. 数据库变更发布

数据库变更风险很高。

安全做法：

- 先加字段，后用字段，再清旧字段。
- 字段新增优先兼容旧代码。
- 大表变更避开高峰。
- 建索引评估锁和耗时。
- 数据迁移分批执行。
- 保留回滚方案。

不安全做法：

- 直接删除线上字段。
- 改字段类型不兼容旧代码。
- 大表一次性全量更新。
- 上线时执行长事务。

## 9. 测试体系

### 9.1 测试类型

| 类型 | 目的 |
| --- | --- |
| 单元测试 | 验证小逻辑 |
| 集成测试 | 验证模块和依赖 |
| 契约测试 | 验证服务间接口 |
| 端到端测试 | 验证完整流程 |
| 回归测试 | 防旧功能破坏 |
| 性能测试 | 验证容量 |
| 安全测试 | 发现漏洞 |
| 混沌测试 | 验证故障韧性 |

### 9.2 测试金字塔

合理结构：

- 大量单元测试。
- 适量集成测试。
- 少量关键端到端测试。

端到端测试价值高但慢、脆弱、维护成本高，不适合覆盖所有细节。

### 9.3 测试数据

测试数据要覆盖：

- 正常数据。
- 空值。
- 边界值。
- 非法值。
- 重复请求。
- 并发冲突。
- 权限差异。
- 大数据量。
- 外部服务失败。

## 10. 反向代理

反向代理代表服务端接收请求，再转发到后端。

职责：

- TLS 终止。
- 路由。
- 负载均衡。
- 静态资源。
- 压缩。
- 缓存。
- 限流。
- 请求头处理。

常见问题：

- 请求体大小限制。
- 超时配置不合理。
- 转发 Header 缺失。
- WebSocket 未正确升级。
- 上传大文件失败。

## 11. 负载均衡

负载均衡把流量分发到多个实例。

算法：

| 算法 | 特点 |
| --- | --- |
| 轮询 | 简单 |
| 加权轮询 | 实例能力不同 |
| 最少连接 | 适合长连接 |
| 一致性哈希 | 会话亲和或缓存亲和 |
| 随机 | 简单有效 |

负载均衡要配合健康检查，否则会把流量发给坏实例。

## 12. 搜索引擎

搜索引擎用于全文检索和复杂查询。

核心概念：

- 文档。
- 倒排索引。
- 分词。
- 相关性评分。
- 分片。
- 副本。
- 索引刷新。

适合：

- 商品搜索。
- 文章搜索。
- 日志检索。
- 多字段过滤。
- 高亮。

注意：

- 搜索索引不是权威数据源。
- 索引更新通常有延迟。
- 权限过滤要谨慎。
- 复杂聚合查询成本高。

## 13. 对象存储

对象存储用于文件。

概念：

- Bucket。
- Object。
- Key。
- Metadata。
- Presigned URL。
- Lifecycle。

常见上传模式：

1. 客户端向后端申请上传凭证。
2. 后端生成临时 URL。
3. 客户端直传对象存储。
4. 后端保存文件元数据。

优点：

- 减轻后端带宽压力。
- 文件存储可靠。
- 可配合 CDN。

注意：

- 文件类型校验。
- 大小限制。
- 权限控制。
- 防盗链。
- 生命周期清理。

## 14. 定时任务

定时任务用于周期性处理：

- 订单超时关闭。
- 数据同步。
- 报表生成。
- 清理过期数据。
- 重试补偿。

多实例问题：

- 同一个任务可能重复执行。
- 节点宕机任务中断。
- 长任务超过周期。

设计要求：

- 幂等。
- 分布式锁或分片。
- 执行记录。
- 失败告警。
- 可重试。
- 可人工触发。

## 15. 运维常见问题

| 问题 | 可能原因 |
| --- | --- |
| 发布后大量 502 | 服务未就绪、健康检查错误、端口错误 |
| 发布后请求变慢 | 新版本慢查询、连接池变化、缓存失效 |
| Pod 反复重启 | 内存限制、探针错误、启动失败 |
| 磁盘满 | 日志未轮转、临时文件堆积 |
| 配置未生效 | 配置中心缓存、环境变量错误 |
| 回滚失败 | 数据库变更不兼容 |

### 精讲扩展 1：部署运维、云原生、测试与常见组件 的接口设计、领域建模 与工程化理解

学习 $topic 时，不能只把它当成一个孤立知识点来背诵，而要把它放到 $category 的完整问题链条里理解。一个知识点通常同时包含概念定义、适用边界、输入输出、运行过程、常见异常和工程取舍。真正掌握它，意味着看到一个具体场景时，能够判断它解决什么问题、依赖哪些前提、失败时会出现什么现象，以及应该用什么手段验证自己的判断。

### 精讲扩展 6：部署运维、云原生、测试与常见组件 的并发控制、高可用 与工程化理解

学习 $topic 时，不能只把它当成一个孤立知识点来背诵，而要把它放到 $category 的完整问题链条里理解。一个知识点通常同时包含概念定义、适用边界、输入输出、运行过程、常见异常和工程取舍。真正掌握它，意味着看到一个具体场景时，能够判断它解决什么问题、依赖哪些前提、失败时会出现什么现象，以及应该用什么手段验证自己的判断。

从 $a 的角度看，最重要的是先建立清晰的对象模型。也就是明确系统里有哪些参与者、它们之间如何连接、数据或控制信号如何流动、哪些环节是同步的、哪些环节是异步的、哪些状态是临时状态、哪些状态需要长期保存。很多初学问题并不是公式不会、API 不熟，而是对象边界不清：把配置当成状态，把结果当成过程，把局部现象当成全局规律。写笔记时建议始终追问：这个概念的主体是谁，输入是什么，输出是什么，中间约束是什么，错误会在哪里暴露。

从 $b 的角度看，流程比单点知识更关键。一个成熟方案通常不是单个技巧，而是一组步骤：先确定目标，再拆分约束，然后选择工具，最后通过测试和复盘确认效果。比如在实际项目中，不能只问“怎么实现”，还要问“为什么要这样实现”“有没有更简单的替代方案”“边界条件是什么”“数据量、并发量、实时性、可靠性变化后还能不能工作”。这种流程意识能够避免把学习停留在教程层面，也能让后续排错有明确路线。

$topic 的 $c 往往决定它在真实项目中的稳定性。理论上可行的方案，到了工程环境中会受到数据质量、硬件条件、依赖版本、网络环境、团队协作、部署方式和维护成本影响。写代码或做设计时，应该把正常路径和异常路径同时考虑：正常情况下如何运行，输入为空怎么办，超时怎么办，重复执行怎么办，部分成功怎么办，版本升级后兼容性怎么办，日志和指标如何证明系统确实按预期工作。

进一步看 $d，它通常对应性能、可靠性或可维护性的核心矛盾。很多技术选择并没有绝对正确答案，只有是否适合当前约束。例如追求极致性能可能牺牲可读性，追求高度抽象可能增加调试成本，追求快速交付可能留下技术债，追求完全通用可能让简单场景变复杂。高质量笔记应该把这些取舍写出来，而不是只给一个“推荐方案”。推荐方案背后的条件越清楚，迁移到新场景时越不容易误用。

最后从 $e 的角度进行复盘，可以把知识从“看懂”推进到“会用”。建议为 $topic 建立三个层次的检查：第一层是概念检查，确认术语、流程和边界没有混淆；第二层是实践检查，确认能够独立完成一个最小案例；第三层是工程检查，确认这个案例在异常、规模、性能和维护方面经得起追问。每次学习完一个章节，都可以用这三层检查反向补齐笔记。

#### 典型场景拆解

在真实场景中，$topic 通常会经历“需求出现、方案选择、实现落地、问题暴露、持续优化”几个阶段。需求出现时，要先判断这个需求属于基础能力、性能优化、体验改进、可靠性建设还是长期架构演进。不同类型的需求对方案的评价标准不同：基础能力看正确性，性能优化看指标，体验改进看路径是否顺滑，可靠性建设看故障时能否降级和恢复，架构演进看未来变化是否容易吸收。

方案选择阶段，最容易犯的错误是直接套用熟悉工具。更稳妥的方式是列出约束：数据规模、时延要求、资源预算、团队熟悉度、运维能力、安全要求、可测试性和长期维护成本。只有把约束列清楚，才能解释为什么选择当前方案。否则方案看似高级，实际可能只是增加了复杂度。

实现落地阶段，要把 $a 和 $b 拆成可验证的小步骤。每一步都应该有明确的输入、输出和检查方式。对学习笔记而言，这意味着不能只有大段概念，还应该补充流程图式的文字描述、伪代码、命令示例、参数解释、错误现象和排查路径。这样以后复习时，笔记不仅能帮助理解，也能直接指导实践。

问题暴露阶段，要优先区分“理解错误、实现错误、环境错误、数据错误、依赖错误、边界条件错误”。很多复杂问题之所以难排，是因为一开始就把问题归因到错误层级。例如把配置问题当成算法问题，把权限问题当成代码问题，把数据分布变化当成模型失效，把硬件噪声当成软件逻辑错误。好的排查顺序应该从可观测事实开始，而不是从猜测开始。

持续优化阶段，不应只追求把当前问题压下去，还要沉淀成规则。比如记录触发条件、影响范围、定位方法、最终修复、预防措施和可监控指标。这样下一次出现类似问题时，团队可以复用经验，而不是重新从零排查。
