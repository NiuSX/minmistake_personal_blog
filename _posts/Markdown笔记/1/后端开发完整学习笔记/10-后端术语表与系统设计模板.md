# 10-后端术语表与系统设计模板

> 本文目标：提供后端常见术语速查、系统设计思考模板、场景分析模板和面试/工程表达模板。适合作为复习和设计前检查文档。

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把后端当成一条请求生命线

这一章讲的是 **后端术语表与系统设计模板**。阅读时不要只背框架名、组件名或面试题答案，而要把每个概念放回一条请求生命线里：请求如何进入系统，如何被认证和校验，业务规则在哪里执行，数据如何保持一致，慢操作如何异步化，故障如何被观测，变更如何安全上线。后端学习的目标不是堆技术栈，而是能设计、实现、排查和维护一个长期运行的业务系统。

### 一句话先懂

术语表和设计模板的作用是把零散概念变成可复用的提问框架，避免系统设计时漏掉关键边界。

### 通俗类比

像工程验收表：不是为了填表而填表，而是提醒你检查地基、消防、水电、承重、逃生路线和维护入口。

类比只是帮助建立第一印象。回到工程上，要把类比里的入口、调度、仓库、通道、监控和维护分别对应到 API、业务层、数据库、缓存、消息队列、可观测性和部署运维。后端概念只有放进真实链路，才知道它解决的是正确性、性能、安全、可靠性、可维护性还是成本问题。

### 本章学习主线

1. **先看职责**：这个概念负责处理请求链路里的哪一段，输入和输出是什么。
2. **再看边界**：它不负责什么，哪些问题应该交给数据库、缓存、队列、网关、客户端或运维平台。
3. **然后看失败**：超时、重复、乱序、并发、脏数据、权限绕过、容量耗尽时会发生什么。
4. **接着看验证**：怎样用单元测试、集成测试、压测、日志、指标、trace 或故障演练证明设计可靠。
5. **最后看演进**：需求变更、流量增长、团队协作和版本升级时，这个设计是否还能维护。

### 概念怎么学才不容易忘

遇到后端概念时，建议按 白话职责 -> 链路位置 -> 最小例子 -> 常见事故 -> 观测信号 -> 修复策略 六步理解。比如缓存不是加速器这么简单，还要看命中率、TTL、一致性、热点 key 和失效策略；消息队列不是异步这么简单，还要看确认、重试、幂等、积压和补偿；JWT 不是登录态这么简单，还要看签名、过期、撤销、泄露和权限边界。

### 最小实践任务

用模板设计一个短链接、评论或秒杀系统，逐项写出需求、接口、数据模型、缓存、消息、安全、观测和容量估算。

实践时要故意设计失败场景：重复请求、数据库超时、缓存失效、消息重复、权限不足、发布回滚、下游服务不可用。后端能力往往不是在正常路径里体现，而是在异常路径里体现。

### 读完本章应该能做到

- 用自己的话解释本章概念在后端请求链路中的位置。
- 画出最小流程图，标清入口、处理、存储、副作用、返回和观测点。
- 说出至少三个常见失败模式，以及对应的日志、指标或 trace 信号。
- 给出一个可落地的小设计，并说明它的事务、幂等、安全和回滚边界。
- 能用固定模板完成一次后端系统设计复盘，并把术语和真实工程问题对应起来。

> 本节是讲义化阅读入口，后续正文中的协议、架构、数据库、缓存、消息、安全、运维和案例都应围绕这条请求生命线来理解。

## 10. 系统设计通用模板

设计任何后端系统时，可以按下面顺序思考。

### 10.1 需求与边界

- 这个系统解决什么问题？
- 核心用户是谁？
- 核心业务流程是什么？
- 哪些功能必须实时？
- 哪些功能可以异步？
- 哪些数据是强一致？
- 哪些数据可以最终一致？
- 预期流量和数据规模是多少？

### 10.2 API 设计

- 有哪些资源？
- 使用 REST、RPC、GraphQL 还是事件？
- 请求字段有哪些？
- 响应字段有哪些？
- 错误码有哪些？
- 是否需要分页？
- 是否需要幂等？
- 是否需要版本？
- 是否需要 Webhook？

### 10.3 数据设计

- 核心实体有哪些？
- 表如何设计？
- 主键是什么？
- 唯一约束是什么？
- 状态字段是什么？
- 需要哪些索引？
- 查询模式是什么？
- 数据增长如何？
- 是否需要归档？

### 10.4 事务与一致性

- 哪些操作必须同事务？
- 哪些操作可以最终一致？
- 是否需要状态机？
- 是否需要乐观锁？
- 是否需要唯一约束？
- 是否需要分布式事务？
- 是否可以用 Saga？

### 10.5 缓存

- 哪些数据适合缓存？
- 缓存 key 如何设计？
- TTL 多久？
- 写入后如何失效？
- 穿透怎么处理？
- 击穿怎么处理？
- 雪崩怎么处理？
- 热点 key 怎么处理？

### 10.6 异步与消息

- 哪些步骤可以异步？
- 消息主题是什么？
- 生产失败怎么办？
- 消费失败怎么办？
- 重复消费怎么办？
- 顺序是否重要？
- 积压怎么办？
- 是否需要死信队列？

### 10.7 安全

- 是否需要登录？
- 权限模型是什么？
- 是否有对象级权限？
- 是否有敏感数据？
- 是否需要审计？
- 是否需要签名？
- 是否需要限流？
- 是否防重放？

### 10.8 稳定性

- 远程调用超时多少？
- 是否需要重试？
- 重试是否幂等？
- 是否需要熔断？
- 是否有降级？
- 资源是否隔离？
- 是否有容量预估？

### 10.9 可观测性

- 记录哪些日志？
- 指标有哪些？
- 是否接入链路追踪？
- 关键告警是什么？
- 如何定位慢请求？
- 如何定位数据不一致？

### 10.10 发布与运维

- 是否需要数据库迁移？
- 是否兼容旧版本？
- 是否能灰度？
- 是否能回滚？
- 配置如何管理？
- 是否需要容灾？
- 备份如何做？

## 11. 场景模板：设计登录系统

需要考虑：

- 用户名密码或第三方登录。
- 密码哈希存储。
- 登录失败限流。
- Session 或 Token。
- Token 过期和刷新。
- 设备管理。
- 退出登录。
- 异地登录提醒。
- 审计日志。
- CSRF 或 Token 存储安全。

## 12. 场景模板：设计订单系统

需要考虑：

- 商品快照。
- 价格计算。
- 库存校验。
- 订单状态机。
- 幂等下单。
- 支付回调。
- 超时取消。
- 消息通知。
- 售后退款。
- 对账。

## 13. 场景模板：设计秒杀系统

需要考虑：

- 页面静态化。
- CDN。
- 网关限流。
- 用户资格校验。
- Redis 预扣库存。
- MQ 削峰。
- 异步创建订单。
- 数据库唯一约束。
- 防刷。
- 排队状态查询。

## 14. 场景模板：设计文件上传

需要考虑：

- 文件大小限制。
- 文件类型校验。
- 直传对象存储。
- 上传凭证有效期。
- 文件元数据。
- 权限控制。
- 病毒扫描。
- CDN 分发。
- 生命周期清理。

## 15. 后端面试表达模板

回答系统设计题时：

1. 先澄清需求和规模。
2. 给出核心 API。
3. 设计数据模型。
4. 说明核心流程。
5. 说明缓存和异步。
6. 说明一致性和幂等。
7. 说明高可用和降级。
8. 说明安全。
9. 说明监控和排查。
10. 说明取舍和演进。

## 16. 后端学习复盘模板

学完一个技术后写：

- 它解决什么问题？
- 它的核心概念是什么？
- 它适合什么场景？
- 它不适合什么场景？
- 它的常见坑是什么？
- 它和类似技术有什么区别？
- 它在系统里如何监控？
- 它故障时如何降级？

## 17. 本章小结

术语表帮助快速建立后端语言体系，设计模板帮助把知识用于真实场景。后端学习最终要落到“能设计、能实现、能上线、能排查、能复盘”。

## 2026 后端资料与工程核对补充

后端基础概念相对稳定，但规范、组件版本和安全风险会持续变化。复现实践前，建议记录运行时版本、框架版本、数据库版本、缓存和消息队列版本、容器镜像、Kubernetes 版本、云厂商组件、配置文件、迁移脚本和压测环境。不要只记录代码提交，还要记录依赖和运行条件。

学习后端时建议优先核对官方规范和项目文档：HTTP 语义看 RFC 9110，HTTP 缓存看 RFC 9111，API 安全风险看 OWASP API Security Top 10，观测体系看 OpenTelemetry，数据库事务和索引看对应数据库官方文档，容器编排看 Kubernetes 官方文档。社区文章适合补充事故经验和踩坑案例，但不能替代规范和官方文档。
