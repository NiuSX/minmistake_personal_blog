# 10. 机器人建模、仿真与数字孪生

## 学习目标

学完本章，你应该能：

- 理解 URDF、xacro、SDF、MJCF 等机器人模型格式的作用。
- 知道视觉模型、碰撞模型、惯量模型的区别。
- 理解仿真中的传感器、物理引擎、控制器和插件。
- 能分析“仿真能跑，真机不行”的常见原因。
- 能建立一个可用于算法验证的机器人仿真模型。

## 1. 为什么建模和仿真重要

企业项目不可能所有问题都在真机上试：

- 真机调试成本高。
- 机器人可能损坏设备。
- 安全风险高。
- 很多极端场景难以重复。

仿真的价值：

- 低成本验证算法。
- 自动化回归测试。
- 复现故障。
- 训练任务策略。
- 生成合成数据。
- 在硬件未到位前推进软件开发。

## 2. 模型文件解决什么问题

机器人模型描述：

- 机器人有哪些连杆。
- 连杆之间如何连接。
- 关节类型和限制。
- 外观如何显示。
- 碰撞体是什么形状。
- 质量、惯量、摩擦等物理属性。
- 传感器安装在哪里。

常见格式：

| 格式 | 主要用途 |
|---|---|
| URDF | ROS 生态中常用机器人描述格式 |
| xacro | URDF 的宏模板，方便复用和参数化 |
| SDF | Gazebo 使用更强的仿真描述格式 |
| MJCF | MuJoCo 使用的模型格式 |

## 3. URDF 核心概念

URDF 由 link 和 joint 构成。

### 3.1 Link

link 是刚体部件。

通常包含：

- visual：显示模型。
- collision：碰撞模型。
- inertial：质量和惯量。

### 3.2 Joint

joint 连接两个 link。

常见类型：

- fixed：固定连接。
- revolute：有限角度旋转。
- continuous：无限旋转。
- prismatic：直线移动。
- floating：六自由度浮动。

### 3.3 Joint origin

joint origin 非常关键，它定义父 link 到子 link 的变换。

常见错误：

- 方向写反。
- 单位用错。
- roll/pitch/yaw 顺序理解错误。
- joint axis 和实际运动方向不一致。

## 4. visual、collision、inertial

### 4.1 Visual

visual 用于显示。

可以很精细：

- CAD 网格。
- 材质。
- 颜色。

但 visual 复杂不代表仿真准确。

### 4.2 Collision

collision 用于碰撞检测。

建议比 visual 简单：

- box。
- cylinder。
- sphere。
- 简化 mesh。

碰撞模型过复杂会降低仿真速度。

### 4.3 Inertial

inertial 描述：

- 质量。
- 质心。
- 惯量矩阵。

惯量错误会导致：

- 仿真抖动。
- 控制不稳定。
- 机器人倒下。
- 机械臂轨迹执行异常。

## 5. SDF 和 Gazebo

SDF 比 URDF 更适合仿真，能表达：

- 世界。
- 光源。
- 地形。
- 传感器。
- 插件。
- 物理参数。
- 多机器人场景。

Gazebo 常用于：

- 移动机器人导航仿真。
- 传感器仿真。
- 控制器验证。
- ROS 2 集成测试。

## 6. 传感器模型

常见仿真传感器：

- 2D LiDAR。
- 3D LiDAR。
- RGB camera。
- depth camera。
- IMU。
- GPS。
- 接触传感器。
- 力/力矩传感器。

建模时要关注：

- 发布频率。
- 噪声。
- 延迟。
- 分辨率。
- 视场角。
- 坐标系。
- 数据 topic。

传感器仿真过于理想，会导致算法到真机后不稳定。

## 7. 物理引擎

物理引擎模拟：

- 刚体运动。
- 碰撞。
- 摩擦。
- 约束。
- 关节。

常见问题：

- 时间步长过大导致不稳定。
- 惯量矩阵不合理导致发散。
- 碰撞体穿模。
- 摩擦参数不真实。
- 控制器频率和仿真步长不匹配。

## 8. 控制器和插件

仿真机器人需要控制接口。

常见方式：

- ROS 2 control。
- Gazebo plugin。
- 自定义 driver 模拟节点。

理想架构：

```text
算法节点 -> ROS 2 控制接口 -> 仿真控制器/真机控制器
```

这样算法层不用关心底层是真机还是仿真。

## 9. 数字孪生

数字孪生不是“有一个模型”这么简单。

更完整的数字孪生包括：

- 几何模型。
- 物理模型。
- 传感器模型。
- 控制模型。
- 场景模型。
- 数据回放。
- 参数版本管理。
- 和真机行为对齐的验证流程。

企业项目中，数字孪生用于：

- 回归测试。
- 故障复现。
- 新算法验证。
- 场景覆盖。
- 客户现场模拟。

## 10. 仿真到真机差异

| 差异 | 影响 |
|---|---|
| 质量/惯量不准 | 控制效果不同 |
| 摩擦不准 | 轮滑、刹车、转向不同 |
| 传感器无噪声 | 感知算法过度乐观 |
| 无通信延迟 | 控制到真机后振荡 |
| 碰撞模型简化过度 | 真实环境中碰撞 |
| 地面过于理想 | 真机里程计漂移更大 |

## 11. 建模检查清单

- 所有 link 名称清晰。
- joint parent/child 正确。
- 坐标轴方向符合 REP 103/105 习惯。
- visual 和 collision 分离。
- inertial 不为空且数值合理。
- 关节限制符合真机。
- 传感器 TF 正确。
- topic 名称和真机一致。
- 仿真控制接口和真机接口尽量一致。

## 12. 练习

1. 写一个最小 URDF：底盘 + 两个轮子 + 雷达。
2. 用 xacro 把左右轮参数化。
3. 在 RViz 中检查 TF 和模型。
4. 在 Gazebo 中加入 LiDAR 和 IMU。
5. 对同一套 Nav2 参数分别跑仿真和真机，记录差异。
