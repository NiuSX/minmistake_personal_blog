# AI Agent 架构万字精讲学习笔记

## 1. 一句话理解 Agent 架构

AI Agent 架构研究的是：如何把大语言模型放进一个可执行、可观测、可评测、可控制的系统里，让它不只是回答问题，而是能够围绕目标持续感知环境、制定计划、调用工具、读取结果、更新状态、处理失败，并最终交付可验证的任务成果。

如果普通 LLM 应用像一个“问答接口”，那么 Agent 更像一个“带模型决策能力的自动化系统”。模型只是 Agent 的核心部件之一，真正的 Agent 架构还包括入口层、身份权限、任务路由、上下文管理、工具注册、执行循环、状态存储、记忆系统、规划器、验证器、评测系统、日志追踪、安全护栏和人工介入机制。

可以用一个公式概括：

```text
AI Agent = Model + Goal + Context + Tools + State + Policy + Feedback Loop
```

其中 Model 负责理解和生成，Goal 负责确定目标，Context 负责提供事实，Tools 负责连接外部世界，State 负责保存进度，Policy 负责约束行为，Feedback Loop 负责让 Agent 根据结果调整下一步。缺少这些结构，模型再强也只是一次性生成器，很难稳定完成真实任务。

## 2. 为什么要学习 Agent 架构

很多人学习 Agent 时容易从框架开始，比如 LangGraph、OpenAI Agents SDK、AutoGen、CrewAI、LlamaIndex、Microsoft Agent Framework。框架当然重要，但只学框架 API 容易停留在“会写 demo”。真正重要的是理解框架背后的架构问题：

- 模型怎么知道下一步该做什么？
- 工具怎么被描述、选择、调用和校验？
- 多轮任务的状态存在哪里？
- 长上下文如何裁剪和压缩？
- 什么时候该让模型自由规划，什么时候该用确定性工作流？
- 工具失败后如何恢复？
- 高风险操作如何审批？
- 如何判断 Agent 做对了？
- 如何记录 trace，复盘失败？
- 多 Agent 为什么经常变复杂，什么时候值得用？

Agent 架构学习的目标不是背名词，而是能设计系统。你应该能拿到一个需求后判断：这只是普通 LLM 调用，还是需要 RAG，还是需要工具调用，还是需要完整 Agent，还是需要多 Agent 工作流。也应该能画出入口、模型、工具、状态、记忆、评测、安全和运维之间的关系。

## 3. Agent 与普通 LLM 应用的区别

普通 LLM 应用通常是：

```text
用户输入 -> 拼 Prompt -> 模型生成 -> 返回结果
```

这种模式适合翻译、摘要、改写、简单问答、格式转换。它的问题是缺乏行动能力和持续状态。模型输出之后，系统就结束了。

Agent 应用通常是：

```text
用户目标
-> 识别任务
-> 获取上下文
-> 规划下一步
-> 调用工具
-> 观察结果
-> 更新状态
-> 验证是否完成
-> 未完成则继续循环
-> 输出结果和证据
```

Agent 的核心差异有五个。

第一，目标导向。Agent 不只是回答当前问题，而是围绕目标推进任务。例如“帮我修复博客导航下拉问题”不是一次回答，而是需要搜索文件、阅读代码、修改样式、构建验证。

第二，工具使用。Agent 可以调用外部工具，如文件系统、终端、浏览器、数据库、API、搜索、代码解释器、MCP Server。工具让模型从“语言系统”变成“行动系统”。

第三，状态管理。Agent 需要知道任务执行到哪一步，已经读过哪些文件，哪些操作成功，哪些失败，下一步是什么。

第四，反馈循环。Agent 会根据工具结果调整计划。测试失败就读日志再修复，检索不到资料就换查询，权限不足就询问用户。

第五，约束与验证。Agent 需要权限边界、安全策略和结果验证。没有这些，Agent 只是会乱动的模型。

## 4. Agent 架构总览

一个完整 Agent 系统可以拆成以下层次：

```text
用户入口层
  ↓
身份、会话与权限层
  ↓
任务理解与路由层
  ↓
Agent Runtime / 控制器
  ↓
模型调用层
  ↓
上下文与记忆层
  ↓
工具注册与执行层
  ↓
状态机与工作流层
  ↓
验证、评测、观测、安全层
  ↓
最终输出与反馈层
```

这些层不一定在代码里完全分开，但架构设计时必须能分清职责。很多 Agent 项目混乱，就是因为把模型、工具、状态、权限、业务逻辑和日志全部揉成一个大函数。短期 demo 可以跑，长期一定难维护。

## 5. 用户入口层

用户入口层负责接收请求和展示结果。入口可以是网页、聊天窗口、IDE 插件、命令行、企业 IM、客服系统、浏览器扩展、移动 App 或 API。

入口层不仅是 UI。它还负责：

- 接收用户输入。
- 展示 Agent 的执行进度。
- 展示工具调用状态。
- 收集用户确认。
- 处理中断和取消。
- 展示引用、证据和最终结果。
- 收集反馈。

对于长任务，入口层不能只等最终答案。用户需要看到“正在检索资料”“正在运行测试”“等待审批”“验证失败正在重试”等状态。否则用户会觉得系统卡住了。生产 Agent 应尽量支持流式输出、进度事件和可取消任务。

## 6. 身份、会话与权限层

Agent 必须知道自己代表谁在行动。身份决定权限，权限决定工具和数据边界。

例如同一个“查询客户信息”工具，在不同身份下结果不同：

- 普通销售只能查自己负责的客户。
- 区域经理可以查本区域客户。
- 管理员可以查全部客户。
- 外部用户不能查内部客户。

如果 Agent 不处理身份和权限，工具调用就可能越权。身份层要解决：

- 当前用户是谁。
- 当前会话属于哪个组织、项目、租户。
- 用户拥有哪些角色。
- 当前任务是否需要额外授权。
- 工具调用是否允许。
- 数据返回是否需要脱敏。

会话层负责保存当前交互上下文。它不等同于长期记忆。会话记录当前任务的对话、状态、工具结果和用户确认；长期记忆保存跨会话可复用的偏好和事实。

权限层应尽量在代码和策略系统中实现，而不是只靠提示词告诉模型“不要越权”。模型可以参与判断，但不能成为唯一权限边界。

## 7. 任务理解与路由层

任务理解与路由层负责判断用户到底想做什么，以及应该走哪条处理路径。

常见任务类型包括：

- 普通问答。
- 知识库问答。
- 文件修改。
- 代码生成。
- 数据分析。
- API 操作。
- 高风险业务操作。
- 需要澄清的模糊需求。
- 无法处理或应该拒绝的请求。

路由可以用规则、分类模型、LLM 或混合方式。高风险场景建议规则优先，LLM 辅助。例如涉及删除、付款、退款、生产部署、数据库写入时，不应该完全依赖模型判断风险。

路由层的输出最好结构化：

```yaml
task_type: code_modification
risk_level: medium
requires_context: true
requires_tools: true
requires_approval: false
suggested_agent: coding_agent
```

这种结构化结果可以给后续控制器使用。

## 8. Agent Runtime / 控制器

Agent Runtime 是整个 Agent 的运行核心。它不一定是一个框架，也可以是你自己写的控制循环。它负责管理模型调用、工具调用、状态更新、停止条件、错误处理和输出。

一个最小 Runtime 的职责包括：

- 初始化任务状态。
- 选择系统提示词和模型。
- 注入上下文。
- 调用模型。
- 解析模型输出。
- 判断是否需要工具。
- 调用工具。
- 将工具结果反馈给模型。
- 更新状态。
- 判断是否完成。
- 处理失败和重试。
- 输出最终结果。

OpenAI Agents SDK 的官方说明把 Agent 看作配置了 instructions、tools，并可带 handoffs、guardrails、structured outputs 等运行时行为的 LLM；OpenAI 的 Agents 指南也强调，当应用需要自己拥有编排、工具执行、审批和状态时，适合使用 Agents SDK。LangGraph 官方文档则强调 persistence、human-in-the-loop、short-term memory 和 long-term memory 等能力。这些框架表达不同，但核心都是围绕 Runtime 解决长任务可靠执行。

Runtime 最重要的设计原则是：模型负责语义判断，代码负责控制边界。不要把所有控制逻辑都交给模型。最大步骤数、最大成本、超时、工具权限、审批策略、状态持久化、回滚策略，都应该由 Runtime 管。

## 9. 模型调用层

模型调用层负责封装不同模型提供方和调用方式。它看起来只是 API wrapper，但在 Agent 架构中非常重要。

模型调用层需要处理：

- 模型选择。
- 参数配置。
- 重试和退避。
- 超时。
- 速率限制。
- token 统计。
- 成本统计。
- 结构化输出。
- 流式输出。
- 多模态输入。
- 安全过滤。
- fallback 模型。

Agent 不一定只用一个模型。常见做法是：

- 用强模型处理复杂规划和关键决策。
- 用便宜模型做分类、路由、摘要、格式校验。
- 用专用 embedding 模型做检索。
- 用视觉模型处理截图和图片。
- 用代码模型处理代码生成。

模型调用层还要支持版本管理。Prompt、模型、工具 schema、检索策略都可能变化。没有版本记录，线上问题很难复盘。

## 10. Prompt 与指令层

Prompt 在 Agent 架构中仍然重要，但它不是全部。Prompt 层包括系统指令、开发者指令、工具说明、输出格式、少样本示例、任务上下文和安全约束。

好的 Agent Prompt 应该明确：

- Agent 的角色。
- 当前任务目标。
- 可用工具和使用规则。
- 禁止事项。
- 输出格式。
- 失败时如何处理。
- 什么时候请求用户。
- 什么时候停止。

坏 Prompt 常见问题：

- 角色太宽泛。
- 工具使用规则不清。
- 没有停止条件。
- 没有错误处理。
- 把权限规则只写在 Prompt 里。
- 输出格式不稳定。

Prompt 应该和 Runtime 配合。比如 Prompt 说“高风险操作需要审批”，Runtime 也必须真的拦截高风险工具。否则模型一旦忘记规则，就会越界。

## 11. 上下文管理层

上下文管理层负责让模型在正确时间看到正确信息。Agent 的上下文不只是聊天历史，还包括用户目标、系统规则、工具说明、检索结果、文件内容、数据库 schema、历史摘要、状态信息和中间观察。

上下文管理的核心问题不是“怎么塞更多”，而是“怎么给得刚好”。上下文过少，模型会猜；上下文过多，模型会被噪声干扰，成本和延迟也会上升。

上下文管理包含：

- 文件选择。
- 文档检索。
- RAG。
- 重排。
- 摘要。
- 历史压缩。
- 引用保留。
- 上下文预算控制。
- 可信度标注。
- 外部内容隔离。

例如一个代码 Agent 不应该一开始读取整个仓库，而应该先搜索错误关键词、接口名、测试名，再读取相关文件。一个企业知识 Agent 不应该把所有文档塞给模型，而应该检索、重排、截取、标注来源，并在没有证据时拒绝编造。

## 12. 工具注册与执行层

工具是 Agent 的行动能力。工具可以是函数、API、数据库、文件系统、浏览器、搜索、代码解释器、终端、MCP Server、内部服务。

工具层通常包括：

- Tool Registry：工具注册表。
- Tool Schema：参数结构。
- Tool Description：给模型看的工具说明。
- Permission Policy：权限策略。
- Execution Engine：工具执行器。
- Result Normalizer：结果标准化。
- Audit Logger：审计日志。

好工具应该职责单一、参数清晰、输出结构化、错误可解释、权限明确。不要暴露一个万能工具让模型随便做所有事。万能工具虽然灵活，但风险极高，也难评测。

例如不要只暴露：

```text
run(command)
```

更好的方式是根据任务暴露窄工具：

```text
search_files(pattern, path)
read_file(path, start_line, end_line)
apply_patch(patch)
run_tests(command, timeout)
query_readonly_sql(query, max_rows)
```

MCP 的官方规范把 Tools 定义为让模型与外部系统交互的能力，例如查询数据库、调用 API 或执行计算。MCP 的价值在于标准化工具、资源和提示的连接方式，但 MCP 只解决连接问题，不自动解决权限、安全、编排和评测。接入 MCP Server 时仍然需要 Harness 和 Agent Runtime 管控。

## 13. 状态管理层

状态是当前任务执行所需的结构化信息。状态管理是 Agent 与普通聊天的重要区别。

状态可以包含：

- 用户目标。
- 当前阶段。
- 计划列表。
- 已完成步骤。
- 待完成步骤。
- 已读取文件。
- 已调用工具。
- 工具结果摘要。
- 错误记录。
- 验证结果。
- 成本和耗时。
- 用户审批状态。

状态最好结构化保存，而不是只让模型在上下文中“记住”。模型上下文会被裁剪，模型也可能遗忘或误解。结构化状态能让系统恢复、暂停、回放、调试和评测。

LangGraph 的 persistence 文档强调，持久化能让 Agent 在失败、中断、长时间运行和跨交互记忆中继续工作；它将短期记忆通过 checkpointer 保存，将长期记忆通过 store 保存。无论你是否使用 LangGraph，这个思想都值得借鉴：状态必须能持久化，不能只存在模型回复里。

## 14. 记忆系统

记忆是 Agent 架构中容易混淆的概念。记忆不是聊天历史的同义词，也不是把所有东西都存起来。

可以分为三类：

短期工作记忆：当前任务内使用，例如当前计划、工具结果、用户刚刚确认的约束。

长期用户记忆：跨任务复用，例如用户偏好、常用格式、项目路径、团队规范。

领域知识记忆：业务知识、术语、流程、接口说明、历史决策。

记忆系统要有写入策略。不是所有模型生成的内容都应该进入记忆。建议只保存：

- 用户明确要求保存的信息。
- 多次出现且稳定的偏好。
- 经过验证的项目事实。
- 可复用的流程和约定。

不要保存：

- 密码、令牌、身份证等敏感信息。
- 临时任务细节。
- 未确认推测。
- 可能很快过期的数据。
- 外部恶意内容。

记忆读取也要受权限和任务相关性控制。长期记忆如果无差别注入上下文，会污染当前任务。

## 15. 规划模块

规划模块负责把目标拆成可执行步骤。它可以由模型生成，也可以由代码模板或工作流定义。

常见规划模式：

### 15.1 隐式规划

模型在内部判断下一步，不显式输出计划。这适合简单任务，但可观测性差。

### 15.2 显式计划

模型先输出步骤列表，再执行。适合中等复杂任务。例如：

```text
1. 搜索导航组件
2. 阅读样式文件
3. 修改下拉菜单 hover 区域
4. 构建验证
5. 汇报结果
```

### 15.3 Plan and Execute

Planner 生成计划，Executor 执行每一步，Verifier 检查结果。适合复杂任务。

### 15.4 动态重规划

执行中根据工具结果修改计划。比如测试失败后加入“读取错误日志”和“修复失败用例”。

规划不是越详细越好。计划过粗无法执行，计划过细容易僵化。好的计划应该有目标、动作、依赖、完成条件和验证方式。

## 16. 执行循环

Agent 的执行循环通常是 ReAct 变体：

```text
Reason -> Act -> Observe -> Update -> Repeat
```

Reason：模型判断当前状态和下一步。

Act：选择工具或输出答案。

Observe：读取工具结果。

Update：更新任务状态。

Repeat：继续直到完成或停止。

执行循环需要停止条件，否则 Agent 可能无限循环。停止条件包括：

- 达成目标。
- 验证通过。
- 达到最大步数。
- 达到最大成本。
- 达到超时时间。
- 遇到不可恢复错误。
- 需要用户输入。
- 被安全策略拦截。

停止条件应该由 Runtime 强制执行，而不是完全依赖模型自觉。

## 17. 工作流与状态机

很多生产 Agent 并不是完全自治，而是 Workflow + Agent 的混合架构。

Workflow 负责确定性流程，Agent 负责语义判断和灵活处理。例如：

```text
接收请求
-> 权限检查
-> 分类
-> 检索资料
-> Agent 生成草案
-> 校验格式
-> 人工审批
-> 发布
```

其中权限检查、检索、校验、审批、发布可以是确定性节点；生成草案和异常解释可以由 Agent 完成。

Microsoft Agent Framework 的官方资料也强调，可以用 workflow 将 agent reasoning 与业务逻辑结合，支持条件分支、并行 fan out、结果汇聚、checkpoint 和长流程恢复。这个方向说明生产系统不会把所有事情交给模型，而是用工作流约束模型能力。

状态机适合明确阶段的任务。例如代码 Agent：

```text
understanding -> inspecting -> planning -> editing -> verifying -> reporting
```

每个阶段允许的工具不同。`inspecting` 可以读文件，`editing` 可以写文件，`verifying` 可以运行测试，`reporting` 输出总结。状态机能防止模型跳步。

## 18. 验证器与评审器

Verifier 是 Agent 架构中的质量控制模块。它负责判断结果是否满足目标。

验证方式包括：

- 规则校验。
- JSON schema 校验。
- 单元测试。
- 构建检查。
- SQL 结果核对。
- 引用检查。
- 格式检查。
- 第二模型评审。
- 人工审查。

模型自评可以作为辅助，但不能作为唯一验证。代码要跑测试，数据要对账，链接要检查，文档要校验结构。Verifier 越依赖外部客观信号，系统越可靠。

评审器可以是另一个 Agent，也可以是规则系统。多 Agent 架构中常见 Critic Agent，但要注意 Critic 也可能错。关键任务最好结合确定性验证。

## 19. 可观测性与 Trace

Agent 架构必须可观测。没有 trace，失败时无法知道模型看到什么、调用了什么工具、为什么做出某个决策。

一次完整 trace 应包含：

- 用户输入。
- 会话和身份信息。
- Prompt 模板版本。
- 模型版本。
- 注入上下文摘要。
- 模型输出。
- 工具调用名称和参数。
- 工具返回结果。
- 状态转移。
- 验证结果。
- 错误和重试。
- 成本和耗时。
- 人工审批记录。

OpenAI Agents SDK 文档提到 tracing 会记录 agent run 中的 LLM generations、tool calls、handoffs、guardrails 等事件。这个思想非常关键：Agent 不是单次黑箱调用，而是一串可追踪事件。

Trace 用于调试、评测样本生成、成本分析、安全审计和用户解释。生产 Agent 没有 trace，就像后端服务没有日志。

## 20. 评测体系

评测是 Agent 架构成熟的标志。没有评测，就无法判断改模型、改 prompt、改工具、改检索策略后系统是变好还是变坏。

Agent 评测比普通问答评测更复杂，因为它要评估过程和结果。

结果指标：

- 任务成功率。
- 答案准确率。
- 格式正确率。
- 引用正确率。
- 测试通过率。
- 用户满意度。

过程指标：

- 工具调用是否正确。
- 是否读取必要上下文。
- 是否越权。
- 是否跳过验证。
- 是否多次无效重试。
- 是否按需请求用户。

运行指标：

- 平均耗时。
- 平均成本。
- 平均步骤数。
- 失败恢复率。
- 人工接管率。

评测集应来自真实任务，包括正常样例、边界样例、失败样例、恶意样例和权限不足样例。不要只用理想输入测试 Agent。

## 21. 安全与 Guardrails

Agent 有工具能力，因此安全风险比普通聊天更高。Guardrails 负责在输入、输出和工具调用上设置护栏。

输入护栏：

- 检查违法或危险请求。
- 检查 prompt injection。
- 检查敏感信息。
- 检查任务是否超出能力范围。

输出护栏：

- 防止泄露敏感信息。
- 检查格式。
- 检查合规话术。
- 检查是否包含无证据断言。

工具护栏：

- 检查路径是否允许。
- 检查 SQL 是否只读。
- 检查命令是否白名单。
- 检查 API 操作是否需要审批。
- 检查参数是否超限。

高风险工具必须由代码层拦截。不要只靠 prompt 让模型“谨慎”。LangChain 的 Human-in-the-loop 文档也提到，可以对写文件、执行 SQL 等工具调用设置策略，让模型提出操作后暂停，等待人工决定。

## 22. 人类在环

Human-in-the-loop 是生产 Agent 的重要组成。它不是失败补丁，而是治理机制。

需要人类介入的场景：

- 用户需求不清。
- 权限不足。
- 高风险操作。
- 模型置信度低。
- 验证失败。
- 需要业务判断。
- 涉及金钱、法律、合规、生产环境。

人类介入可以有多种形式：

- 批准。
- 拒绝。
- 修改参数。
- 补充信息。
- 选择方案。
- 接管任务。

系统应记录人工决策，作为审计和后续优化依据。

## 23. 多 Agent 架构

多 Agent 架构让多个 Agent 分工协作。它适合任务天然需要角色分离、并行处理、独立评审或专业路由的场景。

常见模式：

### 23.1 Router + Specialists

一个路由 Agent 将任务分配给专家 Agent。

```text
Router
├── FAQ Agent
├── Code Agent
├── Data Agent
└── Support Agent
```

OpenAI Agents SDK 的 handoffs 文档说明，handoff 可让一个 agent 将任务委派给另一个专门 agent，例如客服系统中订单状态、退款、FAQ 由不同 agent 处理。

### 23.2 Planner + Executor + Verifier

Planner 规划，Executor 执行，Verifier 审查。适合复杂任务。

### 23.3 Supervisor + Workers

Supervisor 管理多个 worker，分配子任务并汇总结果。适合研究、数据处理和多步骤协作。

### 23.4 Debate / Critique

多个 Agent 提出观点，Critic 评审。适合分析任务，但成本高，且不保证正确。

多 Agent 风险包括成本高、延迟高、状态同步困难、责任边界不清、循环讨论、错误互相放大。因此不要为了“高级”而多 Agent。单 Agent 加好工具和验证，往往更稳。

## 24. Agent 与 RAG 的关系

RAG 是 Agent 的知识获取能力之一，不等于 Agent。

普通 RAG：

```text
问题 -> 检索 -> 拼上下文 -> 回答
```

Agentic RAG：

```text
问题
-> 判断需要哪些信息
-> 多轮检索
-> 根据结果改写查询
-> 调用工具分析
-> 汇总证据
-> 输出答案和引用
```

Agentic RAG 的优势是灵活，能处理复杂问题。但它也更难评测和控制。需要管理检索次数、查询改写、引用、无答案策略和上下文污染。

RAG 架构关键点：

- 文档切分。
- 元数据。
- embedding。
- 召回。
- 重排。
- 引用。
- 答案生成。
- 无答案拒答。
- 评测。

## 25. Agent 与 MCP 的关系

MCP 是连接外部工具和资源的协议，不是 Agent 架构全部。Anthropic 对 MCP 的介绍中强调，它是连接数据源和 AI 工具的开放标准；MCP 官方 Tools 规范说明，工具让模型与外部系统交互，例如查询数据库、调用 API 或执行计算。

在 Agent 架构中，MCP 位于工具层和资源层：

```text
Agent Runtime
  ↓
MCP Client
  ↓
MCP Server
  ├── Tools
  ├── Resources
  └── Prompts
```

MCP 的价值：

- 工具接入标准化。
- 资源暴露标准化。
- 多客户端复用。
- 降低集成成本。

MCP 的风险：

- 工具权限过大。
- 服务器来源不可信。
- prompt injection。
- 本地命令执行风险。
- 工具描述过多导致上下文膨胀。

所以接入 MCP 时仍要做权限、沙箱、审计和审批。

## 26. 生产级 Agent 架构示例：代码助手

一个代码 Agent 可以这样设计：

```text
IDE / CLI / Chat
  ↓
任务解析
  ↓
代码上下文检索
  ↓
Agent Runtime
  ├── 搜索文件
  ├── 读取代码
  ├── 修改文件
  ├── 运行测试
  └── 读取错误日志
  ↓
验证器
  ↓
结果总结
```

关键状态：

- 当前需求。
- 相关文件。
- 修改计划。
- 已修改文件。
- 测试命令。
- 测试结果。
- 剩余风险。

关键护栏：

- 不回退用户未授权改动。
- 不执行危险删除命令。
- 修改前先读文件。
- 修改后必须构建或测试。
- 无法验证时明确说明。

评测样例：

- 修复小 bug。
- 修改 UI 文案。
- 添加配置项。
- 测试失败后修复。
- 工作区有未提交改动。
- 用户需求模糊需要提问。

## 27. 生产级 Agent 架构示例：企业知识助手

企业知识助手架构：

```text
企业 IM / Web
  ↓
身份权限
  ↓
问题分类
  ↓
知识库检索
  ↓
引用重排
  ↓
模型生成
  ↓
引用校验
  ↓
回答 / 拒答
```

关键点：

- 权限先于检索。
- 检索结果必须带来源。
- 没有证据时拒答。
- 外部文档内容不能覆盖系统指令。
- 用户反馈进入评测集。

评测指标：

- 答案准确率。
- 引用正确率。
- 无答案拒答率。
- 权限违规率。
- 用户追问率。

## 28. 生产级 Agent 架构示例：数据分析助手

数据分析 Agent 架构：

```text
用户问题
  ↓
权限检查
  ↓
schema 检索
  ↓
SQL 草案生成
  ↓
SQL 安全检查
  ↓
只读查询
  ↓
结果解释
  ↓
图表或报告
```

关键护栏：

- 默认只读。
- 限制最大行数。
- 禁止跨权限查询。
- SQL 执行前 explain 或规则检查。
- 聚合结果要说明口径。
- 不确定字段含义时询问。

常见失败：

- 字段理解错误。
- 时间口径错误。
- 权限越界。
- 抽样当全量。
- 忽略空值和重复。
- 结果解释过度。

## 29. 从 Demo 到生产的演进路线

第一阶段：普通 LLM 应用。先用 prompt 和结构化输出解决简单任务。

第二阶段：工具调用。加入少量低风险工具，让模型能查资料或执行计算。

第三阶段：单 Agent Runtime。加入循环、状态和停止条件。

第四阶段：上下文工程。加入 RAG、文件选择、摘要和上下文压缩。

第五阶段：验证闭环。加入测试、格式校验、引用校验和失败恢复。

第六阶段：安全治理。加入权限、guardrails、审批、审计。

第七阶段：评测观测。加入 trace、指标、离线评测集和线上反馈。

第八阶段：工作流和多 Agent。只有在任务复杂度需要时再引入。

## 30. Agent 架构设计清单

设计一个 Agent 前，先回答：

- 这个 Agent 的目标是什么？
- 输入是什么？
- 输出是什么？
- 哪些事情不做？
- 哪些工具可用？
- 哪些工具高风险？
- 需要哪些上下文？
- 状态保存在哪里？
- 是否需要长期记忆？
- 是否需要规划？
- 是否需要工作流？
- 停止条件是什么？
- 如何验证结果？
- 如何处理失败？
- 是否需要人工审批？
- 如何记录 trace？
- 如何评测？
- 如何控制成本？
- 如何防止越权和注入？

如果这些问题都没有答案，就不要急着写代码。

## 31. 常见架构误区

误区一：有工具调用就是 Agent。一次 function calling 只能说明模型能用工具，不代表有状态、规划和反馈循环。

误区二：Agent 越自主越好。生产系统更需要可控，而不是无限自主。

误区三：所有流程都交给模型。权限、预算、审批、回滚、停止条件应由代码控制。

误区四：多 Agent 一定更强。多 Agent 经常增加复杂度，先把单 Agent 做稳。

误区五：RAG 能解决所有幻觉。RAG 也会检索错、排序错、引用错。

误区六：只看最终答案。Agent 的中间过程同样需要评测。

误区七：不做 trace。没有 trace 的 Agent 无法生产运维。

误区八：记忆越多越好。记忆过多会污染上下文并带来隐私风险。

## 32. 学习路线

第一步，掌握 LLM 基础：token、上下文窗口、结构化输出、多模态、模型调用。

第二步，掌握 Prompt 和上下文：系统指令、任务提示、RAG、摘要、压缩。

第三步，掌握工具调用：function calling、工具 schema、错误处理、权限。

第四步，实现最小 ReAct Agent：Reason、Act、Observe、Repeat。

第五步，加入状态管理：计划、步骤、工具结果、验证结果。

第六步，加入验证：测试、规则校验、引用校验、人工评审。

第七步，加入可观测性：trace、日志、成本、延迟、回放。

第八步，加入安全：guardrails、权限、审批、prompt injection 防护。

第九步，学习工作流：状态机、图编排、checkpoint、恢复。

第十步，学习多 Agent：handoff、router、supervisor、planner/executor/verifier。

最后，做一个真实项目。不要只停留在 demo。选择一个你每天会用的任务，比如博客维护、代码助手、知识库问答、数据分析助手，把它做成有状态、有工具、有验证、有 trace 的 Agent。

## 33. 总结

Agent 架构的本质，是把模型放进一个可靠的执行系统。模型负责语言理解、推理和生成；架构负责上下文、工具、状态、权限、验证、观测和恢复。

一个优秀 Agent 不是“模型很强 + Prompt 很长”，而是具备以下能力：

- 能理解目标。
- 能获取必要上下文。
- 能选择合适工具。
- 能保存状态。
- 能根据结果调整。
- 能验证输出。
- 能在风险操作前请求审批。
- 能记录 trace。
- 能被评测和迭代。

Agent 架构学习最终要落到工程判断：什么时候用简单 LLM，什么时候用 RAG，什么时候用工具调用，什么时候用完整 Agent，什么时候用工作流，什么时候用多 Agent。只有能做这种取舍，才算真正理解 Agent。
