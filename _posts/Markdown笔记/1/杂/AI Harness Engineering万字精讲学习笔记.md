# AI Harness Engineering 万字精讲学习笔记

## 1. 先用一句话理解 Harness Engineering

AI Harness Engineering 可以翻译为“AI 驱动系统的执行工程”或“智能体外壳工程”。它研究的不是模型本身怎么训练，而是模型外面那一整套让模型能够稳定做事的工程系统：提示词、上下文、工具、权限、文件系统、状态机、记忆、任务编排、错误恢复、评测、观测、安全策略、人类审批、成本控制、发布流程，全部都属于 Harness 的范围。

一个简单公式是：

```text
AI Agent = Model + Harness
```

Model 是能力来源，Harness 是能力落地系统。模型会理解语言、生成文本、推理和规划，但模型本身不天然知道当前项目有哪些文件、哪些工具能用、哪些操作危险、执行到哪一步、失败后如何恢复、怎样判断结果正确、什么时候必须找人审批。Harness 就是把这些外部条件组织起来，让模型从“会说话”变成“能可靠完成任务”的工程结构。

如果把模型比作大脑，那么 Harness 就像身体、工具箱、记忆系统、工作台、流程制度、监控仪表和安全护栏。没有 Harness，模型只能一次性回答；有了 Harness，模型可以读取资料、调用工具、修改文件、运行测试、观察结果、修正计划、保存状态、接受人类反馈，并在约束范围内完成长链路任务。

## 2. 为什么 Harness Engineering 变得重要

早期使用大模型时，很多人关注 Prompt Engineering，也就是怎样写提示词让模型回答更好。后来大家发现，很多问题不是提示词写得不够漂亮，而是模型缺少正确上下文，于是 Context Engineering 变得重要。再往后，当 AI 开始执行真实任务，尤其是写代码、操作文件、查询数据库、调用 API、参与工作流时，新的问题出现了：即使提示词和上下文都不错，系统仍然可能失败。

失败原因包括：

- 模型不知道哪些工具可用。
- 工具调用参数错误。
- 上下文太长，关键事实被挤掉。
- 多步骤任务执行到一半丢失状态。
- 出错后没有回滚和恢复。
- 生成结果没有被测试验证。
- 高风险操作没有审批。
- 生产环境没有 trace，无法复盘错误。
- 成本和延迟失控。
- 用户权限与数据边界没有隔离。
- 评测只看单轮问答，没有覆盖真实任务链路。

这些都不是单纯 Prompt 能解决的问题。Prompt 解决“怎么表达任务”，Context 解决“给模型什么信息”，Harness 解决“整个 AI 系统如何稳定执行”。当 AI 从聊天机器人走向工作流自动化、代码代理、数据分析助手、企业知识助手、客服系统、自动运营系统时，Harness Engineering 就变成核心工程能力。

## 3. Prompt Engineering、Context Engineering 和 Harness Engineering 的关系

可以把三者看成嵌套关系：

```text
Harness Engineering
└── Context Engineering
    └── Prompt Engineering
```

Prompt Engineering 关注指令表达。它研究角色设定、任务描述、输出格式、示例、约束语句、提示模板。比如“你是一名资深代码审查工程师，请优先指出 bug，并按严重程度排序”。Prompt 的核心问题是：如何让模型理解当前要做什么。

Context Engineering 关注信息组织。它研究检索、RAG、上下文压缩、文件选择、历史摘要、记忆注入、上下文窗口管理。比如代码代理在修改文件前要读取哪些文件，知识问答系统要检索哪些文档，长会话要保留哪些事实。Context 的核心问题是：如何让模型在正确时间拿到正确事实。

Harness Engineering 关注执行系统。它研究工具注册、权限、状态、编排、观测、评测、验证、回滚、安全、人类审批、成本控制。比如一个代码代理修改文件后必须运行测试，失败后要读取错误日志并修复；一个数据代理执行 SQL 前必须做权限检查；一个客服 Agent 给退款前必须让人审批。Harness 的核心问题是：如何让模型在真实环境中持续做对。

简单任务里，Prompt 往往最重要。知识密集任务里，Context 更重要。长链路生产任务里，Harness 最重要。

## 4. Harness Engineering 的核心目标

Harness Engineering 的目标不是让模型变得“更聪明”，而是让 AI 系统变得“更可靠”。它追求以下目标：

第一，可执行。系统不只输出建议，还能连接工具和环境，真正完成任务。

第二，可控。模型只能在授权范围内操作，不该访问的数据不能访问，不该执行的命令不能执行。

第三，可恢复。任务失败后能定位原因、重试、回滚、重新规划，而不是直接崩溃。

第四，可观测。每次模型调用、工具调用、状态变化、错误和决策都能被记录，方便调试和复盘。

第五，可评测。系统能用真实任务集衡量效果，而不是只凭单次演示判断。

第六，可扩展。随着工具、业务流程、用户角色、模型版本变化，Harness 能逐步演进。

第七，可生产化。系统能处理成本、延迟、并发、安全、权限、版本发布和运维问题。

## 5. Harness 的六层架构

一个完整的 AI Harness 可以拆成六层：

```text
1. 信息边界层
2. 上下文管理层
3. 工具系统层
4. 执行编排层
5. 状态与记忆层
6. 验证观测与安全层
```

这六层不是严格线性，而是互相配合。信息边界决定模型能知道什么，上下文管理决定模型当前看到什么，工具系统决定模型能做什么，执行编排决定模型按什么流程做，状态与记忆决定任务如何延续，验证观测与安全层决定系统是否可信。

### 5.1 信息边界层

信息边界层负责定义 AI 能接触哪些信息、不能接触哪些信息。它包括系统提示词、角色规则、项目规范、数据隔离、租户隔离、权限模型、敏感信息屏蔽、上下文注入规则。

在企业场景里，信息边界非常关键。一个销售助手不能看到财务敏感数据；一个客服助手不能看到不属于当前用户的订单；一个代码助手不能把私有仓库内容发送到不受信任的外部服务；一个数据分析 Agent 不能绕过行级权限查询全库数据。

信息边界层要解决三个问题：

- 谁在使用系统。
- 这个用户能看什么。
- 这个任务需要什么最小信息。

好的 Harness 不会把所有资料都塞给模型，而是按权限和任务注入最小必要上下文。最小必要原则非常重要，因为上下文越多，不仅成本越高，也越容易引入泄露和干扰。

### 5.2 上下文管理层

上下文管理层负责把正确的信息放进模型上下文。它包含 RAG、文件选择、摘要、压缩、历史会话管理、检索排序、引用注入、缓存、上下文预算控制。

上下文管理不是“越多越好”。模型上下文窗口再大，也不能无限装入所有内容。更现实的目标是：在当前任务的当前步骤，把最相关、最可信、最必要的信息放进去。

例如代码 Agent 在修 bug 时，不应该读取整个仓库，而应该先定位报错栈、相关测试、相关模块、邻近实现和接口定义。写文档时，不应该读取所有历史文档，而应该读取当前主题的规范、样例和最新版本。数据问答时，不应该把整张表给模型，而应该提供 schema、字段含义、示例行和查询约束。

上下文管理层常见机制包括：

- 检索：从文档、代码、数据库或知识库中找相关片段。
- 重排：用规则或模型重新排序候选内容。
- 摘要：把长历史压缩成任务状态。
- 分块：将大文档拆成可检索片段。
- 引用：保留来源，方便结果可追溯。
- 缓存：复用稳定上下文，降低成本。
- 剪裁：移除无关历史，防止上下文污染。

### 5.3 工具系统层

工具系统层负责让模型与外部世界交互。工具可以是函数、API、命令行、文件系统、数据库、浏览器、搜索引擎、MCP Server、内部服务、自动化脚本。

工具系统要解决的不只是“能不能调用”，还包括：

- 工具名称和描述是否清楚。
- 参数 schema 是否严格。
- 调用结果是否结构化。
- 错误信息是否可供模型理解。
- 工具是否有权限控制。
- 工具是否可取消、可超时、可重试。
- 工具是否有审计日志。
- 工具是否能模拟或沙箱运行。

工具设计是 Harness 的核心。很多 Agent 失败不是模型不会推理，而是工具抽象太差。比如一个工具叫 `run`，描述是“运行东西”，模型很难安全使用；如果工具叫 `run_readonly_sql`，参数包含 `query`、`timeout_seconds`、`max_rows`，并且明确只允许 SELECT，模型就更容易稳定调用。

工具应尽量满足单一职责。一个工具只做一件清楚的事。不要把“查询、修改、删除、发送通知”全部塞进一个万能工具。万能工具自由度过高，安全风险也高。

### 5.4 执行编排层

执行编排层负责多步骤任务如何推进。它可以是简单循环，也可以是状态机、工作流引擎、图结构、多 Agent 协作、计划器、任务队列。

单轮聊天通常不需要复杂编排。但真实任务经常是长链路。例如：

```text
读取需求 -> 搜索相关文件 -> 制定计划 -> 修改代码 -> 运行测试 -> 读取错误 -> 修复 -> 再测试 -> 总结结果
```

这个过程不是一个 prompt 能稳定完成的。Harness 需要管理每一步的输入、输出、状态和失败处理。

常见编排模式包括：

- ReAct 循环：模型思考下一步，调用工具，观察结果，再继续。
- Plan and Execute：先生成计划，再逐步执行。
- 状态机：每个节点代表固定阶段，按条件跳转。
- 图编排：不同节点处理不同任务，支持分支和回路。
- 多 Agent 协作：规划 Agent、执行 Agent、审查 Agent、工具 Agent 分工。
- 人类在环：高风险节点暂停，等待审批。

编排层的关键不是让流程复杂，而是让流程可控。简单任务用简单循环即可；高风险任务用明确状态机；需要并行或复杂依赖时再使用图结构。

### 5.5 状态与记忆层

状态与记忆层负责保存任务中间结果、长期偏好、用户信息、执行进度和历史经验。

状态和记忆不是一回事。状态是当前任务的进度，例如已经读取哪些文件、当前计划是什么、测试结果如何、下一步要做什么。记忆是跨任务复用的信息，例如用户喜欢的输出风格、项目约定、常用路径、业务规则、历史决策。

状态管理解决“任务做到哪了”。记忆系统解决“以后还要记住什么”。

没有状态管理，长任务容易断片。模型可能重复做已经做过的事，忘记刚才的错误，或者在多轮对话中丢失目标。没有记忆系统，AI 每次都像第一次接触用户，需要重复解释偏好和背景。

状态与记忆层要注意边界。不是所有东西都应该记住。应该记住稳定、明确、可复用、用户允许保存的信息；不应该记住临时敏感信息、一次性上下文、未经确认的推测。

### 5.6 验证观测与安全层

验证观测与安全层是 Harness 从玩具走向生产的关键。

验证负责回答：结果对不对。代码是否通过测试，SQL 是否只读，文档是否符合格式，答案是否有来源，数据是否和原表一致，工具调用是否产生预期效果。

观测负责回答：系统刚才发生了什么。模型输入是什么，输出是什么，调用了哪些工具，耗时多少，失败在哪一步，重试几次，成本多少，用户是否接管。

安全负责回答：系统是否越界。是否访问了不该访问的数据，是否执行了危险命令，是否泄露隐私，是否被 prompt injection 诱导，是否绕过审批。

没有这一层，Agent 看起来能跑，但无法可信运行。

## 6. Harness 的典型运行循环

一个通用 Agent Harness 可以抽象成如下循环：

```text
接收任务
-> 识别意图和风险
-> 获取必要上下文
-> 制定或更新计划
-> 选择工具或直接回答
-> 执行工具
-> 观察结果
-> 更新状态
-> 验证阶段结果
-> 如果未完成则继续
-> 输出结果和证据
```

这背后有几个关键点。

第一，每一步都应该有状态更新。不能只让模型在上下文里“记得”，因为上下文可能被压缩或截断。

第二，工具调用后必须观察结果。观察不是简单把结果塞回模型，而是要结构化地告诉模型成功、失败、错误类型、输出摘要和可用证据。

第三，验证不应该只在最后做。很多任务应该阶段性验证。例如代码修改后先运行相关测试，数据处理后先检查行数，文件生成后先检查路径和编码。

第四，失败后要有策略。不同失败对应不同恢复方式：参数错误可以重试，权限错误要请求用户，测试失败要调试，外部服务超时可以退避重试，危险操作要中止或审批。

## 7. 工具调用设计详解

工具调用是 Harness 的“手”。模型通过工具对外部环境产生影响。工具设计好坏直接决定 Agent 是否可靠。

### 7.1 好工具的特征

好工具应该具备以下特征：

- 名称清楚。
- 描述准确。
- 参数少而明确。
- 输入输出结构化。
- 错误信息可解释。
- 权限边界明确。
- 幂等性尽量好。
- 支持超时。
- 支持审计。
- 可以在沙箱中测试。

例如一个文件读取工具可以设计成：

```text
read_file(path: string, start_line?: number, end_line?: number)
```

输出包括：

```text
success: true
path: ...
content: ...
truncated: false
```

一个执行命令工具可以设计成：

```text
run_command(command: string, working_directory: string, timeout_ms: number)
```

但这个工具风险较高，因此 Harness 需要限制命令、工作目录、超时、环境变量和权限。对于生产系统，更推荐把常见命令封装成更窄的工具，例如 `run_tests`、`build_project`、`format_code`，而不是直接暴露万能 shell。

### 7.2 工具描述要为模型服务

工具描述不是给人看的注释，而是给模型选择工具时用的。描述应回答：

- 这个工具做什么。
- 什么时候用。
- 什么时候不要用。
- 参数如何填写。
- 输出代表什么。

如果工具描述模糊，模型会误用。比如：

```text
search: search things
```

这很差。更好的描述是：

```text
search_project_files: Search text patterns in the current project files. Use this before editing code to locate definitions, references, configuration, or existing examples. Do not use it for internet search.
```

### 7.3 工具结果要可供下一步推理

工具输出不能只是原始大段文本。Harness 应该尽量提供结构化结果。例如运行测试后，不仅返回 stdout，还应提取：

- exit_code
- passed_count
- failed_count
- failure_files
- first_error
- duration
- log_path

这样模型更容易判断下一步。

### 7.4 工具权限分级

工具可以按风险分级：

低风险工具：读取文件、搜索文本、读取文档、查询只读数据、计算。

中风险工具：写文件、格式化代码、运行测试、调用非生产 API。

高风险工具：删除文件、部署、发送邮件、修改数据库、支付退款、访问敏感数据、执行任意命令。

Harness 应该对不同风险工具设置不同策略。低风险可以自动执行；中风险可以记录审计；高风险通常需要审批、沙箱、双确认或策略判断。

## 8. 上下文工程在 Harness 中的位置

Context Engineering 是 Harness 的组成部分。很多 Agent 失败不是因为模型差，而是上下文给错了。

### 8.1 上下文选择

上下文选择要围绕任务阶段进行。比如代码任务的不同阶段需要不同上下文：

需求分析阶段：用户需求、相关 issue、项目说明。

定位阶段：搜索结果、文件结构、错误日志。

修改阶段：目标文件、邻近代码、接口定义、测试样例。

验证阶段：测试命令、失败日志、构建配置。

总结阶段：修改文件、测试结果、剩余风险。

如果所有阶段都塞同一堆上下文，模型会被干扰。

### 8.2 上下文压缩

长任务中，历史会越来越长。Harness 需要压缩上下文。压缩不是简单摘要聊天记录，而是提取任务状态：

```text
目标：修复导航下拉 hover 断开问题
已读文件：_includes/masthead.html, _sass/minimal-mistakes/_navigation.scss
已修改：消除 submenu margin 空隙
验证：bundle exec jekyll build 通过
剩余：用户要求首页背景也变浅绿
```

这种状态摘要比完整聊天历史更有用。

### 8.3 防止上下文污染

上下文污染指无关、过期、错误、恶意或低质量信息进入模型上下文，影响决策。

常见污染来源：

- 过期文档。
- 用户误传的错误资料。
- 检索召回的无关片段。
- 被 prompt injection 污染的网页。
- 历史会话中的旧目标。
- 工具输出中的噪声。

Harness 应该对上下文做来源标注、时间标注、可信度标注和任务相关性筛选。特别是网页、邮件、工单、外部文档中可能包含“忽略之前指令”这类注入内容，不能让模型把它们当系统指令。

## 9. 状态机与工作流编排

Harness 的核心能力之一是把 Agent 从“随机对话”变成“可控流程”。状态机是一种非常实用的编排方法。

### 9.1 为什么需要状态机

很多任务都有阶段。例如代码修改：

```text
intake -> inspect -> plan -> edit -> verify -> report
```

每个阶段允许的动作不同。`inspect` 阶段应该读文件，不应该直接修改；`edit` 阶段可以写文件；`verify` 阶段应该运行测试；`report` 阶段应该总结。状态机可以让这些边界更明确。

没有状态机时，模型可能跳步。例如没看代码就改文件，没验证就报告完成，测试失败却忽略。状态机让 Harness 能检查“当前阶段该做什么”。

### 9.2 常见状态设计

通用 Agent 状态可以包含：

- `created`：任务创建。
- `understanding`：理解需求。
- `gathering_context`：收集上下文。
- `planning`：计划。
- `executing`：执行。
- `waiting_for_tool`：等待工具。
- `observing`：观察工具结果。
- `verifying`：验证。
- `waiting_for_user`：等待用户输入或审批。
- `recovering`：失败恢复。
- `completed`：完成。
- `failed`：失败。
- `cancelled`：取消。

不同应用可以裁剪。不要为了显得高级而引入过多状态。状态数量越多，维护成本越高。

### 9.3 状态转移条件

状态机的关键是转移条件。比如：

```text
planning -> executing：计划已生成，且无审批需求
executing -> verifying：工具执行成功，且产生可验证结果
verifying -> completed：验证通过
verifying -> recovering：验证失败但可修复
executing -> waiting_for_user：缺少权限或信息
recovering -> executing：已生成修复方案
```

状态转移条件越清楚，Agent 行为越稳定。

## 10. 记忆系统设计

记忆是 Harness 中最容易被滥用的部分。很多系统一开始就想“让 AI 记住一切”，结果很快变成隐私风险和噪声来源。

### 10.1 记忆分类

记忆可以分为短期记忆、长期记忆、情景记忆、语义记忆和操作记忆。

短期记忆保存当前任务状态。例如当前修复的 bug、已经读过的文件、测试失败信息。

长期记忆保存跨会话稳定信息。例如用户偏好、项目路径、团队规范。

情景记忆保存过去事件。例如某次部署失败的原因、某个历史决策。

语义记忆保存抽象知识。例如业务术语、架构说明、领域规则。

操作记忆保存做事方法。例如“这个项目修改样式后运行 `bundle exec jekyll build`”。

### 10.2 什么应该记住

应该记住：

- 用户明确要求保存的偏好。
- 稳定的项目规范。
- 可复用的业务规则。
- 多次出现的工作流程。
- 经过验证的事实。

不应该记住：

- 密码、令牌、身份证号等敏感信息。
- 用户临时情绪。
- 未确认推测。
- 一次性任务细节。
- 可能很快过期的数据。

### 10.3 记忆写入要有门槛

记忆系统不能让模型随便写。更好的做法是：

1. 模型提出候选记忆。
2. Harness 判断是否符合规则。
3. 必要时让用户确认。
4. 写入结构化存储。
5. 后续使用时标注来源和更新时间。

记忆不是越多越好。高质量记忆应该少、准、稳定。

## 11. 验证闭环

验证闭环是 Harness Engineering 的灵魂。没有验证，Agent 的结果只是“看起来像对”。

### 11.1 验证的类型

代码验证：

- 单元测试。
- 集成测试。
- 构建。
- 类型检查。
- lint。
- 快照测试。
- 端到端测试。

数据验证：

- 行数核对。
- 字段类型核对。
- 空值和异常值检查。
- 聚合结果复算。
- 抽样检查。
- 与来源数据对账。

文本验证：

- 格式是否符合要求。
- 引用是否存在。
- 是否遗漏章节。
- 是否有重复段落。
- 是否包含不确定事实。
- 是否满足目标读者。

工具验证：

- API 返回状态。
- 副作用是否发生。
- 操作是否幂等。
- 是否触发审计记录。
- 是否符合权限边界。

### 11.2 验证要靠外部信号

模型自我判断“我做对了”不可靠。Harness 应尽量使用外部信号验证。比如代码要跑测试，链接要请求检查，JSON 要用解析器校验，SQL 要 explain 或只读执行，文件要读取确认。

模型可以参与诊断，但不能成为唯一裁判。

### 11.3 验证失败后的恢复

验证失败时，Harness 不应该直接结束。它应根据错误类型选择恢复策略：

- 格式错误：重新格式化。
- 测试失败：读取失败日志，定位相关代码，修复后重跑。
- 权限失败：请求用户授权。
- 网络超时：退避重试。
- 数据不一致：回滚或停止。
- 安全策略失败：拒绝执行并解释原因。

恢复也要有次数限制。无限重试会浪费成本并可能造成副作用。

## 12. 评测体系

验证关注单次任务结果，评测关注系统长期质量。Harness 需要 evals 来衡量不同模型、提示词、工具、上下文策略和编排方式的效果。

### 12.1 为什么需要 evals

没有评测，团队只能凭感觉判断 Agent 变好还是变坏。一次 demo 成功不代表生产可靠。Agent 系统尤其需要评测，因为它的失败可能发生在多步骤链路中。

评测能回答：

- 新模型是否比旧模型更适合当前任务。
- 新提示词是否降低失败率。
- 新工具描述是否减少误调用。
- 新检索策略是否提高准确率。
- 新 guardrail 是否误伤正常请求。
- 成本和延迟是否可接受。

### 12.2 评测集设计

评测集应该来自真实任务，而不是只写理想样例。一个好的评测集包含：

- 正常任务。
- 边界任务。
- 模糊任务。
- 恶意输入。
- 工具失败场景。
- 权限不足场景。
- 长上下文场景。
- 需要人类审批场景。

例如代码 Agent 的评测集可以包含：

- 小 bug 修复。
- 多文件重构。
- 测试失败修复。
- 用户需求不清，需要提问。
- 工作区有未提交改动。
- 需要拒绝危险命令。
- 修改后必须更新文档。

### 12.3 评测指标

常见指标包括：

- 任务成功率。
- 一次通过率。
- 工具误调用率。
- 平均步骤数。
- 平均耗时。
- 平均成本。
- 验证通过率。
- 人类接管率。
- 安全违规率。
- 用户满意度。

不同业务要有不同指标。客服系统关注解决率和合规，代码系统关注测试通过和回归风险，数据分析系统关注准确性和可追溯。

### 12.4 评测不能只看最终答案

Agent 的中间过程也要评测。例如：

- 是否读取了正确文件。
- 是否调用了必要工具。
- 是否跳过验证。
- 是否在高风险操作前请求审批。
- 是否错误重试过多。

只看最终答案会漏掉很多 Harness 问题。

## 13. 可观测性与 Trace

可观测性是生产 Agent 必须具备的能力。没有 trace，Agent 出错后很难定位。

### 13.1 Trace 应记录什么

一次 Agent run 应记录：

- 用户输入。
- 系统配置版本。
- 模型版本。
- prompt 模板版本。
- 注入的上下文摘要。
- 工具列表和工具 schema。
- 模型输出。
- 工具调用参数。
- 工具返回结果。
- 状态转移。
- 验证结果。
- 错误和重试。
- 成本和延迟。
- 人类审批记录。

OpenAI Agents SDK 等框架提供了 tracing 能力，可以记录模型生成、工具调用、handoff、guardrail 和自定义事件。LangGraph 和 LangSmith 也强调 durable execution、persistence、human-in-the-loop、tracing 和 evaluation 等能力。具体技术选型会变，但核心思想不变：Agent 运行过程必须可追踪。

### 13.2 Trace 的价值

Trace 用于：

- 调试失败任务。
- 分析成本。
- 优化工具描述。
- 发现上下文污染。
- 复盘安全事件。
- 构建评测样本。
- 比较模型版本。
- 给用户解释系统行为。

没有 trace，团队只能看最终输出，无法知道模型为什么这么做。

## 14. Guardrails 与安全边界

Guardrails 是 Harness 中的护栏。它们限制模型输入、输出和工具行为，防止系统越界。

### 14.1 输入护栏

输入护栏检查用户请求是否允许处理。比如：

- 是否包含恶意指令。
- 是否请求违法或危险操作。
- 是否试图绕过权限。
- 是否包含敏感数据。
- 是否属于系统能力范围。

输入护栏不是简单关键词过滤，而是结合策略、用户角色、任务上下文和风险等级。

### 14.2 输出护栏

输出护栏检查模型回复是否合规。比如：

- 是否泄露敏感信息。
- 是否生成不允许的内容。
- 是否包含未授权数据。
- 是否有错误格式。
- 是否缺少免责声明或证据。

输出护栏适合客服、医疗、法律、金融、教育等高风险场景。

### 14.3 工具护栏

工具护栏检查工具调用是否安全。比如：

- SQL 是否只读。
- 文件路径是否在允许目录。
- 命令是否在白名单。
- API 操作是否需要审批。
- 参数是否超过限制。
- 是否可能产生不可逆副作用。

工具护栏特别重要，因为工具调用会产生真实影响。模型文字错误可能只是回答错，工具错误可能删库、转账、发错邮件或泄露数据。

### 14.4 Prompt Injection 防护

Prompt Injection 是 Agent 的常见风险。攻击者可能把恶意指令藏在网页、邮件、文档、代码注释、工单、PDF 里，让模型忽略系统规则或泄露信息。

防护策略包括：

- 区分系统指令、用户指令和外部内容。
- 给外部内容打标签，不把它当指令。
- 对工具结果进行净化和摘要。
- 高风险操作需要独立策略判断。
- 不允许外部内容修改安全规则。
- 对敏感工具使用审批。

Harness 必须假设外部数据不可信。

## 15. Human-in-the-loop

Human-in-the-loop 指人类参与 Agent 流程。它不是 Agent 能力不足的补丁，而是生产系统的重要控制机制。

### 15.1 什么时候需要人类审批

常见需要审批的情况：

- 写入数据库。
- 删除文件。
- 发送邮件或消息。
- 触发退款或支付。
- 修改生产配置。
- 部署上线。
- 访问敏感数据。
- 任务目标不明确。
- 模型置信度低。
- 验证失败但仍想继续。

LangChain/LangGraph 的 human-in-the-loop 文档中也强调，可以对高风险工具调用设置策略，在模型提出操作时暂停执行，等待人工批准或修改。

### 15.2 审批不是简单 yes/no

审批可以有多种结果：

- approve：同意执行。
- reject：拒绝执行。
- edit：修改工具参数后执行。
- request_more_info：要求补充信息。
- delegate：交给另一个人或系统。

Harness 应记录审批人、审批时间、原始参数、修改后参数和执行结果。

### 15.3 人类反馈进入记忆

人类审批和纠正是高价值反馈。系统可以从中学习：

- 哪类操作经常被拒绝。
- 哪些参数经常被修改。
- 哪些任务需要更早提问。
- 哪些 guardrail 太严格或太宽松。

但反馈进入长期记忆前也要过滤，不能把敏感内容直接保存。

## 16. 错误处理与恢复

生产 Agent 必然会失败。Harness 的成熟度体现在失败后怎么处理。

### 16.1 错误分类

常见错误包括：

- 模型理解错误。
- 上下文缺失。
- 工具参数错误。
- 工具超时。
- 权限不足。
- 外部服务失败。
- 验证失败。
- 安全策略拒绝。
- 用户需求冲突。
- 成本或步骤超限。

错误分类决定恢复策略。

### 16.2 重试策略

不是所有错误都能重试。网络超时可以重试，参数格式错误可以让模型修正后重试，权限不足不能盲目重试，安全策略拒绝更不能绕过。

重试要有上限、退避、日志和原因记录。多次失败后应该停止并向用户说明。

### 16.3 回滚策略

有副作用的任务需要回滚设计。例如：

- 修改文件前保存 diff。
- 数据库写入前开启事务。
- 部署前保留旧版本。
- 批量操作前生成预览。
- 删除前进入回收站或创建备份。

没有回滚策略的 Agent 不适合执行高风险操作。

## 17. 成本、延迟与性能

Harness 不只关心正确性，也关心成本和速度。

### 17.1 成本来源

成本来自：

- 模型调用 token。
- 工具调用费用。
- 检索和向量数据库。
- 外部 API。
- 运行环境。
- 人类审批时间。
- 失败重试。

长链路 Agent 如果没有预算控制，很容易成本失控。

### 17.2 降低成本的方法

- 用小模型处理简单分类、路由、格式校验。
- 只在关键步骤使用强模型。
- 缓存稳定上下文。
- 减少无效检索。
- 压缩历史状态。
- 限制最大步骤数。
- 对工具结果做摘要。
- 使用确定性脚本替代模型推理。

### 17.3 降低延迟的方法

- 并行检索。
- 并行只读工具调用。
- 预加载常用上下文。
- 流式输出。
- 让模型先返回计划或部分结果。
- 对长任务使用后台执行和通知。
- 对高频任务使用模板。

性能优化不能牺牲安全。不要为了快而跳过验证和审批。

## 18. 多 Agent Harness

多 Agent 是 Harness 的高级形态。多个 Agent 可以分工协作，但也会引入复杂性。

### 18.1 常见角色

- Router Agent：判断任务类型，分配给不同专家。
- Planner Agent：拆解任务，生成计划。
- Executor Agent：执行工具调用。
- Critic Agent：审查结果。
- Research Agent：收集资料。
- Coding Agent：修改代码。
- QA Agent：运行测试和验证。
- Safety Agent：检查风险。

### 18.2 多 Agent 的优势

优势包括：

- 分工清晰。
- 专家提示词更聚焦。
- 可独立评测。
- 审查和执行分离。
- 高风险任务可增加监督。

### 18.3 多 Agent 的风险

风险包括：

- 成本上升。
- 延迟上升。
- 状态同步困难。
- 责任边界不清。
- 多个 Agent 互相放大错误。
- 调试困难。

不要一开始就多 Agent。只有当单 Agent 的角色冲突明显，或确实需要独立审查、路由、并行处理时，才引入多 Agent。

## 19. MCP 在 Harness 中的位置

MCP，即 Model Context Protocol，是一种让 AI 应用连接外部工具、数据和提示模板的开放协议。它把外部能力以标准方式暴露给 AI 应用。MCP 中常见能力包括 Tools、Resources 和 Prompts。

在 Harness 视角下，MCP 是工具系统层和上下文管理层的一个标准化接口。它不等于 Harness 全部。MCP 解决“如何连接工具和资源”，Harness 还要解决权限、编排、状态、评测、观测、安全、恢复等更大的系统问题。

使用 MCP 时要特别注意安全。MCP Server 可能访问本地文件、数据库和命令行，因此 Harness 必须限制可信来源、权限范围、工具描述、参数校验和审计日志。不要把未知来源 MCP Server 直接接入高权限环境。

## 20. 从玩具 Agent 到生产 Agent

玩具 Agent 的特点：

- 只靠 prompt。
- 工具很少。
- 没有权限模型。
- 没有状态持久化。
- 没有评测。
- 没有 trace。
- 没有回滚。
- 成功靠演示。

生产 Agent 的特点：

- 有清晰任务边界。
- 有工具 schema 和权限。
- 有上下文管理。
- 有状态持久化。
- 有验证闭环。
- 有 trace 和指标。
- 有评测集。
- 有人类审批。
- 有失败恢复。
- 有成本控制。
- 有安全策略。

两者差别不在模型，而在 Harness。

## 21. Harness Engineering 实施路线

### 21.1 第一阶段：单任务闭环

先选一个高频、低风险、价值明确的任务。例如：

- 整理会议纪要。
- 修改博客样式。
- 生成周报。
- 查询只读数据。
- 修复简单代码 bug。

建立最小闭环：

```text
输入 -> 上下文 -> 模型 -> 工具 -> 验证 -> 输出
```

不要一开始做万能 Agent。

### 21.2 第二阶段：工具和权限

把工具做清楚。定义工具 schema、权限、错误输出、超时和日志。区分只读工具和写入工具。高风险工具先不要自动执行。

### 21.3 第三阶段：状态和恢复

加入任务状态。记录计划、步骤、工具结果、验证结果。失败后能继续或回滚。

### 21.4 第四阶段：评测和观测

收集真实任务，建立评测集。接入 trace，记录工具调用和状态转移。用数据而不是感觉改进系统。

### 21.5 第五阶段：生产治理

加入审批、安全策略、成本预算、版本管理、灰度发布、用户反馈和运维告警。

## 22. Harness 设计清单

设计一个 Agent Harness 时，可以按下面清单检查。

任务边界：

- 这个 Agent 解决什么任务？
- 不解决什么任务？
- 输入和输出是什么？
- 成功标准是什么？

上下文：

- 需要哪些信息？
- 信息从哪里来？
- 如何检索和压缩？
- 如何防止上下文污染？

工具：

- 有哪些工具？
- 每个工具是否职责单一？
- 参数 schema 是否严格？
- 输出是否结构化？
- 是否有权限和审计？

编排：

- 是否需要计划？
- 是否需要状态机？
- 是否需要多 Agent？
- 是否需要人工审批？

状态和记忆：

- 当前任务状态保存在哪里？
- 哪些内容进入长期记忆？
- 记忆如何更新和删除？

验证：

- 如何验证结果？
- 验证失败如何处理？
- 是否需要回滚？

观测：

- 是否记录 trace？
- 是否能复盘失败？
- 是否统计成本和延迟？

安全：

- 是否有输入、输出、工具护栏？
- 是否防 prompt injection？
- 是否有敏感数据隔离？

评测：

- 是否有真实任务评测集？
- 指标是什么？
- 如何比较版本？

## 23. 学习 Harness Engineering 的路线

第一步，理解 Agent 基础。学习 LLM、工具调用、ReAct、函数调用、RAG、记忆、规划。

第二步，理解工程闭环。做一个能读取文件、调用工具、验证结果的小 Agent。

第三步，学习上下文工程。练习检索、分块、摘要、引用、上下文压缩。

第四步，学习工具设计。写清楚工具 schema，处理错误、超时、权限和结构化输出。

第五步，学习状态机和工作流。用简单状态机管理任务阶段，再学习图编排和 durable execution。

第六步，学习评测。建立小型任务集，统计成功率、成本、延迟和失败类型。

第七步，学习可观测性。记录 trace，复盘每次工具调用和状态转移。

第八步，学习安全。研究 prompt injection、数据隔离、工具权限、人类审批和审计。

第九步，做真实项目。不要只写 demo，选择一个日常高频任务，让 Agent 真正帮你工作。

## 24. 一个代码 Agent Harness 示例

以代码 Agent 为例，完整 Harness 可以这样设计：

任务输入：

```text
用户描述 bug 或改动需求
```

流程：

```text
理解需求
-> 搜索相关文件
-> 阅读实现和测试
-> 制定修改计划
-> 编辑文件
-> 运行格式化
-> 运行相关测试
-> 如果失败，读取日志并修复
-> 构建或运行更大范围测试
-> 总结修改和验证结果
```

工具：

- `search_files`
- `read_file`
- `apply_patch`
- `run_tests`
- `run_build`
- `git_diff`

状态：

- 当前目标。
- 已读文件。
- 修改文件。
- 测试命令。
- 测试结果。
- 剩余风险。

护栏：

- 不回退用户未授权改动。
- 不执行危险删除命令。
- 不提交密钥。
- 修改前先读相关代码。
- 完成前必须验证或说明无法验证。

评测：

- 小 bug 修复成功率。
- 测试通过率。
- 平均轮数。
- 用户返工率。
- 误改无关文件次数。

这个例子说明，代码 Agent 的可靠性主要来自 Harness，而不是只来自模型。

## 25. 一个知识问答 Harness 示例

企业知识问答 Agent 的 Harness 可以这样设计：

输入：

```text
用户问题
```

流程：

```text
识别用户权限
-> 判断问题类型
-> 检索知识库
-> 重排候选文档
-> 注入最相关片段
-> 生成答案
-> 检查答案是否有来源
-> 输出引用和不确定性
```

工具：

- 文档检索。
- 权限检查。
- 文档读取。
- 引用校验。

护栏：

- 不回答无权限内容。
- 没有来源时说明无法确定。
- 不把外部文档中的指令当系统指令。
- 不泄露检索不到的内部资料。

评测：

- 答案准确率。
- 引用正确率。
- 拒答正确率。
- 权限违规率。
- 用户追问率。

这个场景中，Harness 的关键是权限、检索、引用和拒答。

## 26. Harness Engineering 常见误区

误区一：把 Harness 等同于 Prompt。Prompt 只是 Harness 的一部分。只改提示词无法解决状态、工具、安全、验证和观测问题。

误区二：一开始就做通用 Agent。通用 Agent 最难评测和控制。应该从具体任务开始。

误区三：工具越多越好。工具太多会增加选择难度和安全风险。工具要少而准。

误区四：模型自己会验证。模型可以辅助判断，但验证应尽量依赖外部信号。

误区五：记忆越多越好。记忆过多会污染上下文并带来隐私风险。

误区六：多 Agent 一定更强。多 Agent 增加成本和复杂度，只有明确收益时才使用。

误区七：只做离线评测。生产 trace 和用户反馈同样重要。

误区八：忽略失败恢复。真实系统一定会失败，恢复策略必须提前设计。

## 27. 总结

Harness Engineering 是 AI 应用从“能回答”走向“能做事”的关键工程能力。它关心模型之外的一切：上下文、工具、状态、编排、记忆、验证、观测、安全、评测和运维。

Prompt Engineering 让模型听懂任务，Context Engineering 让模型拿到事实，Harness Engineering 让模型在真实环境中稳定执行。越是长链路、高风险、生产级的 AI 系统，Harness 越重要。

一个成熟的 Harness 不是一次性堆功能，而是围绕具体任务逐步建立闭环。先做单任务，再加工具权限，再加状态恢复，再加评测观测，最后进入生产治理。它的核心原则是：最小必要上下文、窄工具接口、明确状态、外部验证、完整 trace、严格权限、可恢复失败、持续评测。

未来 AI 系统的竞争，不只是谁的模型更强，也是谁能把模型更可靠地接入真实工作流。Harness Engineering 正是这条路线上的核心能力。
