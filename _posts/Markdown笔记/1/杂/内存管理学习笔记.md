内存管理是计算机系统、操作系统、编程语言运行时和应用性能优化之间的交汇点。很多上层问题表面看是业务 bug、卡顿、崩溃、接口变慢，根因却可能是内存泄漏、越界访问、频繁分配、缓存局部性差、GC 暂停、页错误、内存碎片或对象生命周期设计不合理。

学习内存管理不能只背概念。更重要的是形成一条完整链路：程序如何被加载到进程地址空间，变量和对象分别放在哪里，虚拟地址如何映射到物理内存，堆分配器如何工作，垃圾回收如何判断对象是否存活，为什么有些内存问题会导致崩溃，有些只会导致性能慢慢变差。

## 1. 内存管理要解决什么问题

内存管理的核心目标是让程序安全、高效、可控地使用有限的存储资源。

主要问题包括：

- **空间分配**：程序需要内存时，从哪里分配，分配多大，如何对齐。
- **空间回收**：内存不再使用时，什么时候回收，由谁回收。
- **地址隔离**：不同进程之间不能随意访问彼此的内存。
- **权限控制**：代码段、数据段、栈、堆、内核空间需要不同读写执行权限。
- **地址转换**：程序看到的是虚拟地址，硬件实际访问的是物理地址。
- **性能优化**：减少分配次数、提升缓存命中率、降低页错误和 GC 暂停。
- **可靠性保障**：避免越界、悬空指针、双重释放、内存泄漏、栈溢出。

一句话概括：

> 内存管理就是在硬件、操作系统、运行时和程序之间，建立一套关于地址、空间、生命周期、权限和性能的管理机制。

## 2. 内存层次结构

程序员平时说的内存通常指主内存 RAM，但真实系统是分层的。

| 层级 | 典型容量 | 速度 | 特点 |
| --- | --- | --- | --- |
| CPU 寄存器 | 极小 | 极快 | 保存当前指令直接使用的数据和地址 |
| L1 Cache | 很小 | 很快 | 每个核心私有，延迟极低 |
| L2 Cache | 较小 | 快 | 通常每个核心私有或半共享 |
| L3 Cache | 较大 | 中等偏快 | 多核心共享，容量更大 |
| 主内存 RAM | GB 级 | 中等 | 程序运行的主要空间 |
| SSD/HDD | TB 级 | 慢 | 持久化存储 |
| 网络存储 | 更大 | 更慢 | 跨机器访问，延迟高 |

### 2.1 局部性原理

现代 CPU 依赖局部性提高性能。

- **时间局部性**：刚访问过的数据，短时间内可能再次访问。
- **空间局部性**：访问某个地址后，附近地址也可能被访问。

数组顺序遍历通常比链表遍历快，原因不只是数组下标简单，更重要的是数组元素连续存储，CPU cache 命中率更高。链表节点通常分散在堆上，访问下一个节点可能跳到完全不同的内存位置，引发 cache miss。

### 2.2 Cache Line

CPU cache 通常按 cache line 加载，常见大小是 64 字节。访问数组中的一个元素时，相邻元素也会被加载进缓存。

这会影响数据结构设计：

- 顺序访问连续数组通常很快。
- 大量小对象随机分布会降低缓存命中率。
- 结构体字段顺序会影响对齐、填充和缓存利用。
- 多线程写入相邻字段可能出现伪共享。

伪共享示例：两个线程分别更新两个不同变量，但这两个变量位于同一个 cache line。虽然逻辑上没有共享同一变量，硬件缓存一致性协议仍然会让 cache line 在核心之间来回失效，导致性能下降。

## 3. 程序运行时的内存布局

一个程序启动后，操作系统会为它创建进程地址空间。典型布局如下：

```text
高地址
+-----------------------------+
| 栈 Stack                    |
| 函数调用、局部变量、返回地址 |
+-----------------------------+
| mmap 区                     |
| 动态库、文件映射、匿名映射   |
+-----------------------------+
| 堆 Heap                     |
| 动态分配对象                |
+-----------------------------+
| BSS 段                      |
| 未初始化全局/静态变量        |
+-----------------------------+
| Data 段                     |
| 已初始化全局/静态变量        |
+-----------------------------+
| rodata 段                   |
| 只读常量、字符串字面量       |
+-----------------------------+
| Text 段                     |
| 程序机器指令                |
+-----------------------------+
低地址
```

不同操作系统、CPU 架构、编译器和安全机制会导致细节差异，但核心区域基本类似。

### 3.1 Text 段

Text 段保存机器指令，通常是只读且可执行的。把代码段设置成只读可以防止普通 bug 修改程序指令，也能提高攻击难度。

### 3.2 rodata 段

rodata 保存只读数据，例如字符串字面量、常量表等。

```c
const char *name = "memory";
```

字符串 `"memory"` 通常位于只读区域，尝试修改可能导致段错误。

### 3.3 Data 段和 BSS 段

Data 段保存已初始化的全局变量和静态变量。

```c
int global_count = 10;
static int status = 1;
```

BSS 段保存未显式初始化或初始化为 0 的全局变量和静态变量。

```c
int global_buffer[1024];
static int flag;
```

BSS 不需要在可执行文件中保存大量 0 数据，加载时由操作系统初始化为 0。

### 3.4 堆

堆用于动态分配对象。C 里通过 `malloc/free` 管理，C++ 里通过 `new/delete` 或容器、智能指针管理，Java/Kotlin/Go 等语言通常由运行时和 GC 管理。

堆的特点：

- 生命周期灵活，不随函数返回自动销毁。
- 适合保存大小和生命周期运行时才确定的数据。
- 管理成本高，容易出现泄漏、碎片、悬空引用。
- 频繁分配和释放会影响性能。

### 3.5 栈

栈用于函数调用，保存栈帧、局部变量、返回地址、部分寄存器上下文等。

栈的特点：

- 分配和释放速度快，只需要移动栈指针。
- 生命周期清晰，函数返回后栈帧自动销毁。
- 空间有限，递归过深或大对象放栈上可能导致栈溢出。
- 每个线程通常有自己的栈。

## 4. 栈内存

函数调用时会创建栈帧。栈帧中通常包含：

- 函数参数。
- 返回地址。
- 保存的寄存器。
- 局部变量。
- 对齐填充。

示例：

```c
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

调用 `add` 时，参数、返回地址和局部变量会参与栈帧构造。函数返回后，这个栈帧失效。

### 4.1 不要返回局部变量地址

错误示例：

```c
int* bad() {
    int x = 10;
    return &x;
}
```

`x` 是栈变量，函数返回后它所在的栈帧已经失效。返回它的地址会产生悬空指针。

正确做法取决于需求：

```c
int good_value() {
    int x = 10;
    return x;
}
```

或者由调用者传入输出参数：

```c
void good_output(int *out) {
    *out = 10;
}
```

或者在堆上分配，但要明确释放责任：

```c
int* good_heap() {
    int *p = malloc(sizeof(int));
    if (p != NULL) {
        *p = 10;
    }
    return p;
}
```

### 4.2 栈溢出

常见原因：

- 无限递归。
- 递归深度过大。
- 在栈上创建巨大数组。

错误示例：

```c
void dfs() {
    int buffer[1024 * 1024];
    dfs();
}
```

优化方向：

- 改成迭代。
- 控制递归深度。
- 大对象放到堆上。
- 合理设置线程栈大小。

## 5. 堆内存

堆是动态对象的主要空间。它解决了栈生命周期过短的问题，但引入了更复杂的管理。

### 5.1 堆分配的基本流程

以 `malloc(size)` 为例，分配器大致会做几件事：

1. 根据请求大小选择合适的空闲块。
2. 如果空闲块过大，可能切分成已用块和剩余空闲块。
3. 标记块元数据，例如大小、状态、前后关系。
4. 返回可用内存地址给程序。
5. 如果现有堆空间不足，向操作系统申请更多内存。

释放 `free(ptr)` 时，分配器会：

1. 根据指针找到块元数据。
2. 标记为空闲。
3. 尝试与相邻空闲块合并。
4. 必要时把大块内存归还给操作系统。

### 5.2 内存碎片

内存碎片分两类。

**内部碎片**：分配器实际给出的块比请求更大，多出来的空间不能被程序有效使用。

例如申请 13 字节，分配器为了对齐给了 16 字节，多出的 3 字节就是内部碎片。

**外部碎片**：总空闲空间足够，但被切成很多小块，无法满足一次大的连续分配。

外部碎片在长时间运行的服务、游戏、嵌入式系统里尤其值得关注。

### 5.3 堆分配的成本

堆分配比栈分配贵，原因包括：

- 需要查找合适空闲块。
- 需要维护元数据。
- 多线程下可能涉及锁或线程缓存。
- 可能触发系统调用。
- 会影响缓存局部性。
- 在 GC 语言中会增加垃圾回收压力。

优化原则：

- 避免在热点循环中频繁创建临时对象。
- 预分配容器容量。
- 复用缓冲区。
- 使用对象池要谨慎，避免把生命周期问题变得更复杂。
- 用 profiling 数据证明瓶颈，不要凭感觉优化。

## 6. 虚拟内存

现代操作系统通常让每个进程看到独立的虚拟地址空间。程序中看到的地址不是物理地址，而是虚拟地址。

```text
虚拟地址 -> MMU -> 页表 -> 物理地址
```

虚拟内存提供三个核心能力：

| 能力 | 说明 |
| --- | --- |
| 隔离 | 一个进程不能直接访问另一个进程的内存 |
| 抽象 | 程序看到连续地址，底层物理页可以不连续 |
| 管理 | 操作系统可以按页分配、回收、换出、共享和保护 |

### 6.1 页和页表

虚拟内存通常按页管理，常见页大小是 4KB。虚拟地址被拆成页号和页内偏移：

```text
虚拟地址 = 虚拟页号 + 页内偏移
```

页表记录虚拟页到物理页框的映射，以及权限信息：

- 是否存在。
- 是否可读。
- 是否可写。
- 是否可执行。
- 是否属于用户态。
- 是否脏页。
- 是否被访问过。

### 6.2 TLB

页表查询本身也要访问内存，如果每次内存访问都完整查页表，成本很高。TLB 是地址转换缓存，用来缓存最近的虚拟页到物理页映射。

TLB 命中时，地址转换很快。TLB 未命中时，需要查询页表，成本更高。

### 6.3 缺页异常

当访问的虚拟页没有有效映射时，会触发缺页异常。缺页不一定是错误，它可能是操作系统正常工作的机制。

常见情况：

- 第一次访问懒分配内存。
- 访问被换出的页面，需要从磁盘换入。
- 访问文件映射区域，需要从文件读取数据。
- 写入 copy-on-write 页面，需要复制新物理页。
- 访问非法地址，导致段错误或访问违规。

### 6.4 按需分配

很多系统中，申请大块内存不一定立即占用物理内存。只有真正访问到某个页时，操作系统才分配物理页。

例如：

```c
char *p = malloc(1024 * 1024 * 1024);
```

这行代码可能只是建立虚拟地址范围，并不代表立刻占用 1GB 物理内存。真正写入时才会逐步触发物理页分配。

## 7. mmap 与文件映射

`mmap` 可以把文件或匿名内存映射到进程地址空间。

典型用途：

- 动态库加载。
- 大文件读取。
- 进程间共享内存。
- 分配大块匿名内存。
- 数据库、搜索引擎、缓存系统中的文件映射。

文件映射的优势：

- 可以像访问内存一样访问文件内容。
- 操作系统负责按需加载页面。
- 多进程可以共享同一份只读映射。
- 对大文件随机访问比较方便。

需要注意：

- mmap 不是把整个文件立即读入内存。
- 顺序读大文件不一定比普通 read 更快。
- 映射文件被截断时可能触发异常。
- 修改映射内容的落盘时机需要理解 `msync` 和操作系统写回策略。

## 8. C/C++ 内存管理

C/C++ 提供了很强的控制能力，也把很多责任交给程序员。

### 8.1 常见错误

#### 8.1.1 内存泄漏

分配后没有释放。

```c
void leak() {
    int *p = malloc(sizeof(int));
    *p = 1;
}
```

修复：

```c
void no_leak() {
    int *p = malloc(sizeof(int));
    if (p == NULL) {
        return;
    }
    *p = 1;
    free(p);
}
```

#### 8.1.2 Use After Free

释放后继续使用。

```c
int *p = malloc(sizeof(int));
free(p);
*p = 10;
```

这种问题非常危险，可能立刻崩溃，也可能污染其他对象，过很久才暴露。

#### 8.1.3 Double Free

同一块内存释放两次。

```c
int *p = malloc(sizeof(int));
free(p);
free(p);
```

修复方式不是简单地到处写 `p = NULL`，更重要的是明确所有权，保证只有一个地方负责释放。

#### 8.1.4 越界访问

```c
int a[3] = {1, 2, 3};
a[3] = 4;
```

C/C++ 不会自动检查数组边界，越界写可能破坏相邻内存、返回地址、对象元数据或分配器元数据。

#### 8.1.5 malloc/free 和 new/delete 混用

错误示例：

```cpp
int *p = new int(1);
free(p);
```

规则：

- `malloc` 对应 `free`。
- `new` 对应 `delete`。
- `new[]` 对应 `delete[]`。

### 8.2 RAII

RAII 是 C++ 资源管理的核心思想：资源在对象构造时获取，在对象析构时释放。

```cpp
#include <fstream>

void write_file() {
    std::ofstream file("data.txt");
    file << "hello";
}
```

`file` 离开作用域时析构，文件句柄自动关闭。RAII 不只适用于内存，也适用于文件、锁、socket、数据库连接等资源。

### 8.3 智能指针

#### unique_ptr

独占所有权，不能复制，可以移动。

```cpp
#include <memory>

std::unique_ptr<int> p = std::make_unique<int>(10);
```

适合表示“这个对象只有一个拥有者”。

#### shared_ptr

引用计数共享所有权。

```cpp
#include <memory>

auto p1 = std::make_shared<int>(10);
auto p2 = p1;
```

当最后一个 `shared_ptr` 销毁时，对象释放。

注意：`shared_ptr` 不是默认选择。共享所有权会让生命周期变得不清晰，也有引用计数开销。

#### weak_ptr

弱引用，不增加引用计数，常用于打破循环引用。

```cpp
struct Node {
    std::weak_ptr<Node> parent;
    std::vector<std::shared_ptr<Node>> children;
};
```

### 8.4 C++ 内存管理原则

- 优先使用值语义和标准容器。
- 优先使用 `std::vector`、`std::string`、`std::array`。
- 需要动态对象时优先使用 `std::unique_ptr`。
- 少用裸 `new/delete`。
- 不要让多个模块隐式共享释放责任。
- 对外暴露指针时说明是否拥有所有权。
- 不要返回悬空引用或悬空指针。
- 使用 AddressSanitizer、Valgrind、静态分析工具辅助发现问题。

## 9. 垃圾回收

GC 语言把释放内存的工作交给运行时，但这不代表没有内存管理问题。GC 主要解决“什么时候释放对象”的问题，但程序仍然需要控制对象生命周期、引用关系和内存峰值。

### 9.1 对象可达性

大多数 GC 会从 GC Roots 出发，遍历所有可达对象。不可达对象可以被回收。

常见 GC Roots：

- 当前线程栈上的局部变量。
- 静态变量。
- JNI 或 native 引用。
- 运行时内部引用。
- 活跃线程对象。

如果对象从 GC Roots 可达，即使业务上已经“不需要”，GC 也不会回收它。

### 9.2 常见 GC 算法

#### 标记清除

先标记可达对象，再清除不可达对象。

优点：实现思路简单。

缺点：可能产生碎片。

#### 标记整理

标记后把存活对象移动到一起，减少碎片。

优点：空间连续。

缺点：移动对象有成本，需要更新引用。

#### 复制算法

把内存分成两块，每次只使用其中一块，GC 时把存活对象复制到另一块。

优点：适合存活率低的区域，分配快。

缺点：空间利用率较低。

#### 分代回收

基于经验假设：大部分对象朝生夕死，少数对象存活很久。

常见分代：

- 新生代：保存新创建对象，回收频繁。
- 老年代：保存长期存活对象，回收频率较低。

### 9.3 GC 暂停

GC 可能导致 Stop The World，也就是暂停应用线程。不同 GC 器会用并发、增量、分区等方式降低暂停时间。

性能关注点：

- 吞吐量：应用线程运行时间占比。
- 延迟：单次暂停时长。
- 内存占用：为了降低暂停可能需要更多内存。
- CPU 开销：并发 GC 会消耗额外 CPU。

没有绝对最好的 GC，只有适合场景的取舍。

## 10. JVM 内存模型和运行时区域

JVM 运行时内存区域常见划分：

| 区域 | 线程共享 | 作用 |
| --- | --- | --- |
| 程序计数器 | 否 | 当前线程执行位置 |
| Java 虚拟机栈 | 否 | 方法调用栈帧 |
| 本地方法栈 | 否 | Native 方法调用 |
| Java 堆 | 是 | 对象实例主要存放区域 |
| 方法区/元空间 | 是 | 类元数据、常量、方法信息 |
| 直接内存 | 是 | NIO、Native 分配的堆外内存 |

### 10.1 Java 堆

Java 对象大多在堆上分配。堆大小可以通过参数控制：

```bash
java -Xms512m -Xmx512m -jar app.jar
```

- `-Xms`：初始堆大小。
- `-Xmx`：最大堆大小。

线上服务常把两者设为相同，减少运行时扩缩容带来的抖动。

### 10.2 栈和 StackOverflowError

每个 Java 线程都有自己的虚拟机栈。递归过深会导致 `StackOverflowError`。

```java
public class Demo {
    static void f() {
        f();
    }

    public static void main(String[] args) {
        f();
    }
}
```

线程太多也会占用大量栈空间，导致内存压力。

### 10.3 堆外内存

堆外内存不受 Java 堆大小直接限制，常见来源：

- `ByteBuffer.allocateDirect`。
- Netty direct buffer。
- JNI/native 库。
- mmap 文件映射。

堆外内存泄漏时，堆 dump 里不一定能直接看到大对象，需要结合 native memory tracking、进程 RSS、系统工具分析。

### 10.4 Java 内存泄漏

Java 中的泄漏通常不是“没有 free”，而是“不再需要的对象仍然被引用”。

常见原因：

- 静态集合无限增长。
- 缓存没有淘汰策略。
- 监听器注册后没有注销。
- ThreadLocal 使用后没有 remove。
- 长生命周期对象持有短生命周期对象。
- 队列积压导致对象无法释放。

示例：

```java
private static final List<byte[]> CACHE = new ArrayList<>();

public void handle() {
    CACHE.add(new byte[1024 * 1024]);
}
```

只要 `CACHE` 一直可达，里面的数据就不会被 GC 回收。

## 11. Android 内存管理

Android 应用运行在受限环境中，内存管理直接影响流畅度和稳定性。

### 11.1 Android 常见内存压力来源

- Bitmap 占用过大。
- Activity 或 Fragment 泄漏。
- Context 被长生命周期对象持有。
- RecyclerView 缓存和图片加载策略不合理。
- 协程、线程、Handler 没有随生命周期取消。
- WebView、地图、相机等重资源组件释放不及时。
- 大列表一次性加载过多数据。

### 11.2 Activity 泄漏

错误示例：

```kotlin
object Holder {
    var activity: Activity? = null
}
```

`object` 是进程级单例，如果它持有 Activity，Activity 退出后仍然无法回收。

更合理的方式：

- 不要在单例中保存 Activity。
- 需要 Context 时优先使用 `applicationContext`。
- 生命周期相关任务绑定到 `lifecycleScope`。
- 监听器、回调、广播接收器及时注销。

### 11.3 Bitmap 优化

Bitmap 内存估算：

```text
宽度 * 高度 * 每像素字节数
```

例如 ARGB_8888 每像素 4 字节，一张 4000x3000 图片约占：

```text
4000 * 3000 * 4 = 48,000,000 字节，约 45.8 MB
```

优化方向：

- 按显示尺寸采样解码。
- 使用成熟图片库，如 Coil、Glide。
- 控制缓存大小。
- 避免在主线程解码大图。
- 列表滑动时合理暂停和恢复图片请求。

### 11.4 Android 排查工具

- Android Studio Memory Profiler。
- LeakCanary。
- Perfetto。
- `adb shell dumpsys meminfo <package>`。
- heap dump。
- logcat OOM 信息。

## 12. 常见内存问题分类

### 12.1 内存泄漏

表现：

- 内存随时间持续上涨。
- GC 后内存无法回落。
- 服务运行一段时间后 OOM。
- Android 页面退出后 Activity 仍然存在。

排查思路：

1. 观察内存趋势，不只看单点。
2. 触发 GC 后看存活对象。
3. 获取 heap dump。
4. 找到大对象或数量异常对象。
5. 追踪从 GC Roots 到目标对象的引用链。
6. 判断哪个长生命周期对象错误持有了它。

### 12.2 内存溢出

内存溢出是申请内存失败，常见表现：

- C/C++ `malloc` 返回 NULL。
- Java `OutOfMemoryError`。
- Android 应用被系统杀死。
- Linux 触发 OOM Killer。

泄漏可能导致溢出，但溢出不一定由泄漏导致。一次性加载过大文件、配置内存过小、并发过高、缓存策略错误，都可能导致溢出。

### 12.3 悬空指针

指针指向的对象已经释放或生命周期结束。

典型来源：

- 返回局部变量地址。
- 释放后继续保存指针。
- 容器扩容后旧引用失效。
- 对象移动后仍使用旧地址。

### 12.4 野指针

未初始化的指针指向随机地址。

```c
int *p;
*p = 10;
```

指针变量声明后应该初始化：

```c
int *p = NULL;
```

### 12.5 越界读写

越界读可能读到错误数据，越界写可能破坏程序状态。越界写是安全漏洞的重要来源。

防范方式：

- 使用带边界检查的容器或接口。
- 统一长度和容量语义。
- 对外部输入严格校验。
- 使用 sanitizer 检测。

### 12.6 内存碎片

表现：

- 可用内存看起来很多，但大块分配失败。
- 进程长期运行后性能下降。
- 分配耗时抖动。

缓解方式：

- 减少不同大小对象混合频繁分配释放。
- 使用分区分配、对象池、arena allocator。
- 批量分配和批量释放。
- 对长期对象和短期对象分开管理。

## 13. 内存性能优化

内存性能优化的第一原则是先测量，再优化。

### 13.1 减少不必要分配

低效示例：

```java
for (int i = 0; i < n; i++) {
    String s = prefix + i;
    list.add(s);
}
```

如果在热点路径中频繁创建临时对象，会增加 GC 压力。

优化方向：

- 复用对象或缓冲区。
- 预估集合容量。
- 避免不必要的装箱。
- 避免在渲染帧、网络热路径、日志热路径中创建大量临时对象。

### 13.2 预分配容量

```java
List<Integer> list = new ArrayList<>(10000);
```

如果知道大致元素数量，预分配可以减少扩容和复制。

C++ 中：

```cpp
std::vector<int> values;
values.reserve(10000);
```

### 13.3 改善数据局部性

优化方向：

- 使用连续内存结构。
- 避免过度指针跳转。
- 把热点字段放在一起。
- 冷热数据分离。
- 批量处理数据。

示例：

```cpp
struct Player {
    float x;
    float y;
    float vx;
    float vy;
    std::string name;
    std::string description;
};
```

如果物理更新只需要位置和速度，大量字符串字段会污染缓存。可以把热数据和冷数据拆开。

### 13.4 控制内存峰值

内存峰值过高比平均值更容易导致 OOM。

常见错误：

- 一次性读取整个大文件。
- 一次性把数据库结果全部加载到内存。
- 图片先解码原图再缩放。
- 多阶段处理时同时保留所有中间结果。

优化方式：

- 流式处理。
- 分页加载。
- 分块处理。
- 及时释放中间结果。
- 使用迭代器、Sequence、Flow 等惰性处理方式。

### 13.5 避免伪共享

多线程频繁写入相邻变量时，可能产生伪共享。可以通过填充、对齐或拆分数据结构减少影响。

伪共享通常需要性能计数器或基准测试验证，不建议为了理论上的伪共享过早改复杂。

## 14. 内存排查工具

### 14.1 Linux 常用命令

查看进程地址空间：

```bash
cat /proc/<pid>/maps
```

查看进程内存统计：

```bash
cat /proc/<pid>/status

```


```bash

```

常用于发现：

- use-after-free。
- double-free。
- 部分内存泄漏。

```bash

```

适合发现泄漏和非法访问，但运行开销较大。

- `jstat`：查看 GC 统计。

- 哪类对象占用最大。

## 15. 常见场景分析

### 15.1 服务运行几小时后越来越慢

可能原因：

- 缓存无上限。
- 队列积压。
- 内存泄漏导致 GC 越来越频繁。
- 大对象进入老年代。
- 堆外内存泄漏。
- swap 或 page fault 增多。

排查：

- 观察 RSS、heap used、GC 次数和暂停时间。
- 检查队列长度、缓存命中率和缓存大小。
- 获取 heap dump 对比不同时间点。
- 查看系统是否发生 swap。

### 15.2 接口偶发高延迟

可能原因：

- Full GC。
- 大对象分配。
- 缺页异常。
- 内存分配器锁竞争。
- 日志或序列化创建大量临时对象。
- 容器扩容导致复制。

排查：

- 对齐接口慢日志和 GC 日志时间。
- profiling 查看分配热点。
- 检查对象大小和分配频率。
- 观察 page fault、cache miss。

### 15.3 Android 页面退出后内存不下降

可能原因：

- Activity 被单例、静态变量、线程、Handler、协程、监听器持有。
- 图片请求没有取消。
- WebView 没有释放。
- ViewModel 或 Repository 持有 View。

排查：

- 使用 LeakCanary 查看泄漏链。
- 手动进入退出页面多次观察 heap。
- 检查长生命周期对象是否持有短生命周期 Context。

### 15.4 C++ 程序偶发崩溃

可能原因：

- 越界写破坏内存。
- use-after-free。
- 数据竞争。
- 对象生命周期不清晰。
- ABI 或内存分配器混用。

排查：

- 开启 ASan、UBSan、TSan。
- 用 core dump 和 gdb 查看崩溃点。
- 检查最近释放和最近写入位置。
- 缩小复现用例。

## 16. 内存管理设计原则

### 16.1 明确所有权

任何资源都要回答三个问题：

1. 谁创建它。
2. 谁拥有它。
3. 谁负责释放它。

C++ 中尽量用类型表达所有权：

- 值对象：当前作用域拥有。
- `std::unique_ptr`：唯一拥有。
- `std::shared_ptr`：共享拥有。
- 裸指针或引用：通常只观察，不拥有。

### 16.2 区分生命周期

常见生命周期：

- 临时变量：函数调用期间。
- 请求级对象：一次请求期间。
- 页面级对象：一个页面可见期间。
- 会话级对象：用户会话期间。
- 进程级对象：应用整个生命周期。

泄漏常常来自长生命周期对象错误持有短生命周期对象。

### 16.3 控制缓存

缓存不是越多越好。缓存必须有策略：

- 最大容量。
- 过期时间。
- 淘汰策略。
- 命中率统计。
- 内存压力下释放机制。

没有上限的缓存，本质上就是延迟爆炸的内存泄漏。

### 16.4 避免隐藏分配

一些代码表面简单，内部可能频繁分配：

- 字符串拼接。
- 正则表达式临时编译。
- 自动装箱。
- Stream/Iterator 中间对象。
- 日志参数构造。
- 容器自动扩容。

热点路径中要特别关注隐藏分配。

### 16.5 用工具验证

内存问题很容易被误判。建议用证据闭环：

```text
现象 -> 假设 -> 指标 -> 工具 -> 证据 -> 修复 -> 回归验证
```

例如怀疑内存泄漏：

1. 观察进程内存长期上升。
2. 强制 GC 后仍不下降。
3. 获取 heap dump。
4. 找到异常对象。
5. 分析引用链。
6. 修改生命周期或释放逻辑。
7. 压测验证内存曲线稳定。

## 17. 面试和复习重点

### 17.1 必会概念

- 栈和堆的区别。
- 虚拟地址和物理地址的区别。
- 分页、页表、TLB、缺页异常。
- 内部碎片和外部碎片。
- 内存泄漏、内存溢出、悬空指针、野指针。
- malloc/free 与 new/delete 的区别。
- RAII 和智能指针。
- GC Roots 和可达性分析。
- 新生代、老年代、Minor GC、Full GC。
- Java 堆、栈、方法区、直接内存。
- Android 中 Context、Activity、Bitmap 泄漏。

### 17.2 高频问题

**问题：栈和堆有什么区别？**

栈由编译器和运行时按函数调用自动管理，分配释放快，生命周期随作用域结束，空间较小。堆用于动态分配，生命周期灵活，需要程序员或 GC 管理，分配释放成本更高，也更容易产生泄漏和碎片。

**问题：什么是内存泄漏？**

内存泄漏是已经不再需要的内存仍然无法被回收。C/C++ 中常见原因是分配后没有释放；Java/Kotlin 中常见原因是不再需要的对象仍然被长生命周期引用持有。

**问题：虚拟内存有什么好处？**

虚拟内存提供进程隔离、统一地址空间、按需分配、权限保护、文件映射、共享库映射和更灵活的内存管理。

**问题：为什么 GC 语言仍然会内存泄漏？**

GC 只能回收不可达对象。如果对象虽然业务上没用了，但仍然被静态变量、缓存、线程、监听器等引用，GC 会认为它仍然存活。

**问题：如何排查线上内存上涨？**

先区分是堆内、堆外还是系统缓存。观察 RSS、堆使用量、GC 日志、线程数、直接内存、缓存大小和队列长度。必要时抓 heap dump，对比对象数量和引用链，再结合业务流量验证修复。

## 19. 速查清单

写代码前思考：

- 这个对象应该活多久？
- 谁拥有它？
- 谁负责释放或取消它？
- 是否可能被长生命周期对象持有？
- 是否可能在循环或热路径中频繁分配？
- 是否需要预分配容量？
- 是否可能一次性加载过多数据？
- 是否需要缓存上限？
- 是否有线程共享和可见性问题？

排查问题时思考：

- 是内存泄漏、内存溢出、碎片、GC、页错误还是缓存局部性问题？
- 内存上涨是堆内、堆外、线程栈、mmap、文件缓存还是 native？
- 是持续上涨，还是周期性上涨后回落？
- GC 后是否能回落？
- 大对象是谁，数量异常的对象是谁？
- 引用链从哪里来？
- 最近改动是否改变了生命周期或缓存策略？

## 20. 总结

内存管理不是某一种语言的局部知识，而是一套贯穿硬件、操作系统、编译器、运行时和应用架构的系统能力。

学习时可以按这条主线理解：

```text
内存层次 -> 进程地址空间 -> 栈和堆 -> 虚拟内存 -> 分页和 TLB
-> 分配器 -> GC -> 常见问题 -> 性能优化 -> 工具排查
```

真正掌握内存管理，意味着不仅能回答概念题，还能解释程序为什么崩溃、为什么变慢、为什么内存不释放，并能用工具拿到证据，最后把生命周期、所有权和资源边界设计清楚。
