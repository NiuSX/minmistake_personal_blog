# 嵌入式学习笔记

## 讲义导读：把概念落到可验证实践

这一章讲的是 **嵌入式学习笔记**，属于 **嵌入式系统**。阅读时不要把它当成零散资料堆叠，而要把它当成一份讲义：先弄清它解决什么问题，再看核心概念和流程，最后做一个能复现、能观察、能排错的小练习。

### 一句话先懂

嵌入式学习的核心，是让软件、硬件、外设、实时性和调试手段在资源受限的设备上稳定协作。

初学时先问三个问题：它的输入或前提是什么；它内部按什么规则工作；结果该用什么命令、日志、测试、图纸、波形或指标来证明。

### 通俗类比

嵌入式设备像一台小型专用机器：程序不是跑在无限资源的电脑上，而是在固定芯片、固定引脚和固定时序里工作。

类比只是入门扶手。真正掌握时，要回到准确术语、配置、接口、版本、边界条件、错误信息和验证证据上。能解释失败原因，比只会照着步骤跑通更重要。

### 本章学习主线

1. **先看场景**：这个知识点通常在什么项目、岗位或问题里出现？
2. **再看结构**：它有哪些核心对象、配置、文件、命令、接口或流程？
3. **然后看路径**：一次完整使用从哪里开始，到哪里结束，中间有哪些状态变化？
4. **接着看边界**：版本差异、平台差异、权限、性能、安全、兼容性和资源限制在哪里？
5. **最后看验证**：用最小样例、测试、日志、调试工具或实物结果证明理解是对的。

C/C++、MCU、寄存器、外设、启动、时钟、中断、DMA、通信协议、RTOS、驱动、低功耗和调试。

### 最小实践任务

做一个 MCU 小项目：GPIO、定时器、中断、串口日志、传感器读取和异常复位记录。

建议把练习记录成固定格式：目标、环境版本、最小示例、执行步骤、预期结果、实际结果、错误信息、定位过程和复盘。以后遇到真实项目问题时，这些记录会比单纯收藏教程更有用。

- 只会调用 HAL，不懂外设时序。
- 没有串口日志、断点和示波器配合。
- 忽略上电顺序、低功耗和异常恢复。

嵌入式开发是把软件运行在具体硬件上，让设备完成控制、采集、通信、显示、存储、运动控制等任务。它和普通上位机开发最大的区别是：代码必须面对真实硬件、时序、资源限制、异常状态和现场环境。

这份笔记适合嵌入式初学者、自动化电气工程师、硬件工程师、机器人开发人员、测试人员和想系统补基础的人阅读。目标不是只记概念，而是建立能写代码、能看原理图、能调板子、能定位问题的知识框架。

## 1. 嵌入式是什么

### 1.1 基本定义

嵌入式系统是以微控制器、微处理器、DSP、SoC、FPGA 等芯片为核心，嵌入到设备内部，用于完成特定功能的计算机系统。

常见例子：

- 家电控制板。
- 工业 PLC 模块。
- 电机驱动器。
- 机器人控制器。
- 传感器采集模块。
- 汽车 ECU。
- 医疗设备。
- 智能门锁。
- 物联网终端。
- 无人机飞控。
- 仪器仪表。

### 1.2 嵌入式开发关注点

- **功能正确**：设备要按需求工作。
- **实时性**：某些动作必须在规定时间内响应。
- **可靠性**：长时间运行不能轻易死机。
- **资源受限**：CPU、RAM、Flash、电源、引脚数量有限。
- **硬件相关**：代码依赖寄存器、外设、电路和传感器。
- **异常处理**：断电、掉线、传感器异常、通信丢包、看门狗复位都要考虑。
- **可维护性**：现场问题要能定位，版本要能追溯。

### 1.3 常见开发层级

- **裸机开发**：没有操作系统，主循环加中断。
- **RTOS 开发**：使用实时操作系统，如 FreeRTOS、RT-Thread、ThreadX。
- **嵌入式 Linux 开发**：运行 Linux，涉及驱动、文件系统、进程、网络。
- **SoC/应用处理器开发**：如 ARM Cortex-A、RISC-V Linux 平台。
- **FPGA/SoC 协同开发**：硬件逻辑和软件协同。

## 2. 学习路线

### 2.1 推荐顺序

1. 学 C 语言基础和指针。
2. 学数字电路、常见电子元件和原理图阅读。
3. 学单片机基本结构：CPU、Flash、RAM、GPIO、中断、时钟。
4. 学常用外设：UART、I2C、SPI、ADC、PWM、定时器。
5. 学 Keil、STM32CubeIDE、VS Code、GCC、Make、CMake 等工具。
6. 学调试方法：串口日志、断点、逻辑分析仪、示波器、J-Link/ST-Link。
7. 学 RTOS：任务、队列、信号量、互斥锁、定时器。
8. 学工程架构：驱动层、服务层、应用层、状态机、事件机制。
9. 学通信协议和现场总线：Modbus、CAN、RS485、Ethernet、MQTT。
10. 学低功耗、Bootloader、OTA、文件系统、可靠性设计。

### 2.2 入门项目建议

- LED 闪烁。
- 按键检测和消抖。
- 串口打印和命令解析。
- 定时器周期任务。
- PWM 调光或电机调速。
- ADC 采集电压。
- I2C 读取温湿度传感器。
- SPI 驱动 OLED 或 Flash。
- RS485 Modbus 通信。
- FreeRTOS 多任务控制。
- Bootloader 固件升级。

## 3. C 语言核心知识

C 语言是嵌入式最重要的基础。嵌入式 C 不只是语法，还要理解内存、寄存器、位操作、编译链接和硬件访问。

### 3.1 基本类型

常见类型：

- `char`：字符或 8 位数据。
- `short`：通常 16 位。
- `int`：通常 32 位，但和平台有关。
- `long`：32 位或 64 位，和平台有关。
- `float`：单精度浮点。
- `double`：双精度浮点。

嵌入式更推荐使用固定宽度类型：

```c
#include <stdint.h>

uint8_t  a;   // 无符号 8 位
int8_t   b;   // 有符号 8 位
uint16_t c;   // 无符号 16 位
uint32_t d;   // 无符号 32 位
int32_t  e;   // 有符号 32 位
```

这样能避免不同编译器和平台下类型长度不一致。

### 3.2 指针

指针保存地址，是嵌入式访问内存和寄存器的基础。

```c
uint32_t value = 10;
uint32_t *p = &value;

*p = 20;  // 修改 value
```

常见用途：

- 访问数组。
- 传递缓冲区。
- 操作结构体。
- 访问外设寄存器。
- 函数回调。

常见错误：

- 野指针：指针没有初始化。
- 空指针：没有检查 `NULL`。
- 越界访问：访问数组边界之外。
- 指针生命周期错误：返回局部变量地址。

### 3.3 `const`

`const` 表示只读。

```c
const uint8_t table[] = {1, 2, 3, 4};
```

常见用途：

- 查表数据。
- 固定配置。
- 字符串常量。
- 防止函数误修改输入数据。

```c
void send_data(const uint8_t *data, uint16_t len);
```

### 3.4 `volatile`

`volatile` 告诉编译器：这个变量可能被当前代码之外的因素改变，不能随意优化读取。

常见场景：

- 中断中修改的变量。
- 外设寄存器。
- DMA 共享缓冲区。
- 多任务共享标志。

```c
volatile uint8_t rx_done = 0;

void USART_IRQHandler(void)
{
    rx_done = 1;
}
```

如果没有 `volatile`，主循环可能因为编译器优化而读不到变量变化。

### 3.5 `static`

`static` 常见含义：

- 修饰局部变量：变量生命周期变成整个程序运行期间。
- 修饰全局变量或函数：限制作用域在当前 `.c` 文件内。

```c
static uint32_t counter = 0;

static void private_func(void)
{
}
```

工程中建议：不需要被其他文件访问的全局变量和函数都加 `static`，减少命名冲突。

### 3.6 位操作

嵌入式大量使用位操作。

```c
#define BIT(n) (1U << (n))

reg |= BIT(3);      // 置位 bit3
reg &= ~BIT(3);     // 清除 bit3
reg ^= BIT(3);      // 翻转 bit3
if (reg & BIT(3)) { // 判断 bit3
}
```

设置多位字段：

```c
reg &= ~(0x3U << 4);      // 清除 bit5:bit4
reg |=  (0x2U << 4);      // 设置 bit5:bit4 为 2
```

### 3.7 结构体

结构体用于组织相关数据。

```c
typedef struct {
    uint16_t voltage_mv;
    int16_t temperature_c;
    uint8_t status;
} SensorData;
```

常见用途：

- 设备配置。
- 传感器数据。
- 通信帧。
- 驱动句柄。
- 状态机上下文。

### 3.8 函数指针和回调

回调用于驱动层通知应用层。

```c
typedef void (*button_callback_t)(void);

static button_callback_t cb = NULL;

void button_register_callback(button_callback_t callback)
{
    cb = callback;
}

void button_on_pressed(void)
{
    if (cb != NULL) {
        cb();
    }
}
```

常见用途：

- 按键事件。
- 串口接收完成。
- 定时器超时。
- 状态变化通知。
- 协议栈事件。

### 3.9 内存区域

嵌入式常见内存区域：

- **Flash/ROM**：存放程序代码、只读常量。
- **RAM**：存放运行时变量、堆、栈。
- **栈 stack**：局部变量、函数调用现场。
- **堆 heap**：动态内存分配区域。
- **BSS**：未初始化的全局变量和静态变量。
- **Data**：已初始化的全局变量和静态变量。

注意：

- 小 MCU RAM 很小，不能随便定义大数组。
- 中断和任务都有栈，栈太小会溢出。
- 嵌入式中慎用 `malloc/free`，避免内存碎片和不可控失败。

## 4. 编译、链接和启动过程

### 4.1 编译流程

C 程序通常经历：

1. 预处理：处理 `#include`、`#define`、条件编译。
2. 编译：把 C 文件转成汇编或目标文件。
3. 汇编：生成机器码目标文件。
4. 链接：把多个目标文件、库、启动文件链接成可执行文件。
5. 转换：生成 `.hex`、`.bin` 等烧录文件。

### 4.2 常见文件

- `.c`：源文件。
- `.h`：头文件。
- `.s`：汇编文件。
- `.o/.obj`：目标文件。
- `.elf/.axf`：带调试信息的可执行文件。
- `.hex`：Intel HEX 格式烧录文件。
- `.bin`：纯二进制镜像。
- `.map`：链接映射文件，可查看函数和变量占用空间。
- `.ld/.sct`：链接脚本或分散加载文件。

### 4.3 启动过程

MCU 上电后通常执行：

1. 取复位向量，跳转到启动代码。
2. 设置栈指针。
3. 拷贝已初始化数据从 Flash 到 RAM。
4. 清零 BSS 区。
5. 初始化系统时钟。
6. 调用 C 库初始化。
7. 进入 `main()`。

### 4.4 启动文件

启动文件通常包含：

- 中断向量表。
- 复位入口。
- 默认中断处理函数。
- 栈和堆配置。

如果中断函数名称写错，可能不会进入你的中断函数，而是进入默认中断处理。

## 5. 单片机基础

### 5.1 MCU 组成

典型 MCU 包含：

- CPU 内核。
- Flash。
- SRAM。
- GPIO。
- 时钟系统。
- 中断控制器。
- 定时器。
- UART、I2C、SPI、CAN、USB、Ethernet 等通信外设。
- ADC、DAC、比较器。
- DMA。
- 看门狗。
- 电源管理。

### 5.2 常见内核

- ARM Cortex-M0/M0+：低成本、低功耗。
- ARM Cortex-M3：通用控制。
- ARM Cortex-M4：带 DSP 指令，部分支持 FPU。
- ARM Cortex-M7：高性能 MCU。
- ARM Cortex-M33：支持 TrustZone 安全特性。
- RISC-V MCU：开放指令集，生态在发展。

### 5.3 STM32 系列简单认识

常见系列：

- STM32F0：入门低成本。
- STM32F1：经典系列。
- STM32F4：性能较强，应用广。
- STM32F7/H7：高性能。
- STM32G0/G4：新一代通用和电机控制。
- STM32L 系列：低功耗。
- STM32WB/WL：无线相关。

选择 MCU 时关注：

- Flash 和 RAM 大小。
- 主频。
- GPIO 数量。
- 外设资源。
- 封装。
- 工作温度。
- 供货和成本。
- 开发工具和资料。

## 6. GPIO

GPIO 是通用输入输出引脚。

### 6.1 常见模式

- 输入浮空。
- 输入上拉。
- 输入下拉。
- 模拟输入。
- 推挽输出。
- 开漏输出。
- 复用功能。

### 6.2 上拉和下拉

上拉让引脚默认高电平，下拉让引脚默认低电平。

按键输入常见接法：

- 按键一端接地，引脚配置上拉，按下为低电平。
- 按键一端接电源，引脚配置下拉，按下为高电平。

### 6.3 推挽和开漏

- **推挽输出**：能主动输出高电平和低电平。
- **开漏输出**：只能主动拉低，输出高电平需要外部或内部上拉。

I2C 常用开漏输出，因为多个设备共享总线，任何设备都可以拉低总线。

### 6.4 按键消抖

机械按键按下时会抖动，软件需要消抖。

简单做法：

```c
if (read_key() == 0) {
    delay_ms(20);
    if (read_key() == 0) {
        // 确认按下
    }
}
```

工程中更推荐定时扫描状态机，避免阻塞延时。

## 7. 中断

中断用于让 CPU 响应外部或内部事件。

常见中断：

- GPIO 外部中断。
- 定时器中断。
- 串口接收中断。
- DMA 完成中断。
- ADC 转换完成中断。
- CAN 接收中断。

### 7.1 中断注意事项

- 中断函数要短，不要做耗时操作。
- 中断里不要长时间打印日志。
- 中断里不要调用可能阻塞的函数。
- 中断和主循环共享变量要加 `volatile`。
- 多字节共享数据要考虑原子性。
- RTOS 中中断调用系统 API 要使用 ISR 专用接口。

### 7.2 中断优先级

优先级决定多个中断同时发生时谁先执行。

注意：

- 高优先级中断可以打断低优先级中断。
- 优先级配置错误可能导致实时性问题。
- RTOS 中可调用系统 API 的中断优先级有约束。

## 8. 定时器

定时器是嵌入式最常用外设之一。

常见用途：

- 周期中断。
- PWM 输出。
- 输入捕获。
- 输出比较。
- 编码器接口。
- 脉冲计数。
- 超时检测。

### 8.1 周期定时

定时器通过时钟分频和自动重装值产生周期中断。

核心概念：

- 计数时钟。
- 预分频。
- 自动重装值。
- 计数模式。
- 中断频率。

### 8.2 PWM

PWM 是脉宽调制。

主要参数：

- 频率。
- 占空比。
- 分辨率。

常见用途：

- LED 调光。
- 蜂鸣器。
- 舵机控制。
- 电机调速。
- 加热功率控制。

## 9. UART 串口

UART 是最常用调试和通信接口之一。

### 9.1 基本参数

- 波特率，如 9600、115200。
- 数据位，常见 8 位。
- 校验位，无校验、奇校验、偶校验。
- 停止位，常见 1 位。
- 流控，硬件流控或无流控。

常见配置：`115200 8N1`，表示 115200 波特率、8 数据位、无校验、1 停止位。

### 9.2 串口接收方式

- 查询接收。
- 中断接收。
- DMA 接收。
- 空闲中断加 DMA 接收。

工程中高吞吐串口推荐 DMA 加空闲中断，减少 CPU 占用。

### 9.3 串口日志

串口日志是嵌入式调试的重要手段。

注意：

- 日志不能影响关键实时任务。
- 中断中不要直接大量打印。
- 日志等级要可配置。
- 现场版本建议保留关键错误日志。

## 10. I2C

I2C 是两线串行总线，使用 SCL 和 SDA。

### 10.1 特点

- 两根线连接多个设备。
- 每个从设备有地址。
- 开漏输出，需要上拉电阻。
- 适合低速传感器、EEPROM、RTC、IO 扩展芯片。

### 10.2 常见问题

- 地址 7 位和 8 位混淆。
- 上拉电阻不合适。
- 总线太长导致波形差。
- 从设备拉死总线。
- 时序不满足。
- 电平不匹配。

### 10.3 调试建议

- 用逻辑分析仪看 SCL/SDA。
- 确认 ACK/NACK。
- 确认设备地址。
- 确认读写方向位。
- 确认电源和上拉电阻。

## 11. SPI

SPI 是高速同步串行总线。

### 11.1 常见信号

- SCK：时钟。
- MOSI：主机输出从机输入。
- MISO：主机输入从机输出。
- CS/NSS：片选。

### 11.2 关键参数

- 时钟频率。
- CPOL：时钟空闲电平。
- CPHA：采样边沿。
- 数据位宽。
- MSB/LSB first。

### 11.3 常见用途

- OLED/TFT 屏。
- Flash。
- ADC/DAC。
- 传感器。
- 网卡芯片。
- 无线模块。

### 11.4 常见问题

- SPI 模式不匹配。
- 片选时序不对。
- MISO/MOSI 接反。
- 电平不匹配。
- 线太长导致信号质量差。
- 多从设备片选冲突。

## 12. ADC 和 DAC

### 12.1 ADC

ADC 把模拟电压转换成数字值。

关键参数：

- 分辨率，如 12 位、16 位。
- 参考电压。
- 采样率。
- 输入范围。
- 采样时间。
- 输入阻抗。

换算示例：

```c
uint16_t adc_raw = 2048;
float vref = 3.3f;
float voltage = adc_raw * vref / 4095.0f;
```

12 位 ADC 数值范围是 0 到 4095。

### 12.2 ADC 常见问题

- 参考电压不稳定。
- 模拟地和数字地干扰。
- 输入阻抗过高导致采样不准。
- 没有滤波。
- 采样时间太短。
- 通道切换后第一次采样不稳定。

### 12.3 DAC

DAC 把数字值转换为模拟电压，常用于模拟输出、波形生成、参考电压等。

## 13. DMA

DMA 允许外设和内存之间直接搬运数据，减少 CPU 参与。

常见用途：

- UART DMA 接收。
- ADC DMA 连续采样。
- SPI DMA 刷屏。
- I2S 音频数据搬运。

注意：

- DMA 缓冲区生命周期必须足够长。
- DMA 和 CPU 同时访问缓冲区要处理同步。
- Cache 平台要注意缓存一致性。
- DMA 完成中断里只做标记或轻量处理。

## 14. 看门狗

看门狗用于在程序跑飞或卡死时自动复位系统。

常见类型：

- 独立看门狗。
- 窗口看门狗。

使用原则：

- 不要在中断里无条件喂狗。
- 不要只在一个任务里喂狗。
- 应该确认关键任务都正常后再喂狗。
- 复位后要记录复位原因，方便定位。

## 15. 常用通信和现场总线

### 15.1 RS232

RS232 常用于点对点串口通信。

特点：

- 距离较短。
- 电平不是 TTL 电平，需要电平转换芯片。
- 常见于老设备、仪器仪表。

### 15.2 RS485

RS485 常用于工业现场。

特点：

- 差分传输，抗干扰较好。
- 支持较长距离。
- 支持多节点总线。
- 半双工常见。

注意：

- A/B 线不要接反。
- 总线两端通常需要终端电阻。
- 需要偏置电阻保证空闲电平。
- 多节点通信要避免同时发送。

### 15.3 Modbus

Modbus 是工业中非常常见的协议。

常见形式：

- Modbus RTU：常跑在 RS485 上。
- Modbus TCP：跑在以太网上。

常见功能码：

- `0x01`：读线圈。
- `0x02`：读离散输入。
- `0x03`：读保持寄存器。
- `0x04`：读输入寄存器。
- `0x05`：写单个线圈。
- `0x06`：写单个寄存器。
- `0x0F`：写多个线圈。
- `0x10`：写多个寄存器。

### 15.4 CAN

CAN 常用于汽车、工业控制、电机驱动、机器人。

特点：

- 多主总线。
- 仲裁机制。
- 抗干扰强。
- 有错误检测和错误处理。

关键概念：

- 标准帧 11 位 ID。
- 扩展帧 29 位 ID。
- 数据长度。
- 波特率。
- 终端电阻。
- 过滤器。

### 15.5 CANopen

CANopen 是基于 CAN 的高层协议。

常见概念：

- NMT：网络管理。
- PDO：过程数据对象，实时数据。
- SDO：服务数据对象，配置参数。
- Object Dictionary：对象字典。
- Heartbeat：心跳。

### 15.6 Ethernet

以太网用于高速通信、上位机连接、远程维护和工业协议。

常见协议：

- TCP。
- UDP。
- HTTP。
- MQTT。
- Modbus TCP。
- EtherNet/IP。
- Profinet。

### 15.7 MQTT

MQTT 常用于物联网。

特点：

- 发布/订阅模式。
- Broker 中转消息。
- 适合设备上报数据和云端控制。

## 16. RTOS 基础

RTOS 是实时操作系统，适合任务多、逻辑复杂、需要并发处理的嵌入式项目。

常见 RTOS：

- FreeRTOS。
- RT-Thread。
- ThreadX。
- Zephyr。
- uC/OS。

### 16.1 任务

任务是 RTOS 调度的基本单位。

每个任务通常有：

- 任务函数。
- 优先级。
- 栈空间。
- 任务状态。

常见状态：

- 运行。
- 就绪。
- 阻塞。
- 挂起。

### 16.2 优先级

高优先级任务可以抢占低优先级任务。

注意：

- 高优先级任务不能一直占用 CPU。
- 低优先级任务可能饿死。
- 优先级设置要和实时性匹配。

### 16.3 队列

队列用于任务之间传递数据。

常见用途：

- 串口接收数据传给协议任务。
- 传感器数据传给控制任务。
- 按键事件传给 UI 任务。

### 16.4 信号量

信号量用于事件同步或资源计数。

类型：

- 二值信号量。
- 计数信号量。

常见用途：

- 中断通知任务。
- DMA 完成通知。
- 资源数量控制。

### 16.5 互斥锁

互斥锁用于保护共享资源。

常见共享资源：

- 串口打印。
- I2C 总线。
- SPI 总线。
- 全局数据结构。
- 文件系统。

注意：中断中通常不能使用普通互斥锁。

### 16.6 软件定时器

软件定时器用于周期或一次性超时处理。

适合：

- 状态超时。
- 周期检测。
- 延迟执行。

不适合：

- 高精度硬实时波形输出。
- 高频中断级控制。

### 16.7 RTOS 常见问题

- 栈溢出。
- 优先级反转。
- 死锁。
- 队列满。
- 忘记释放互斥锁。
- 中断优先级配置错误。
- 任务里使用阻塞延时导致响应慢。

## 17. 驱动开发

驱动负责把硬件细节封装成稳定接口。

### 17.1 驱动层设计原则

- 对上提供清晰 API。
- 对下隔离寄存器或 HAL 细节。
- 初始化、读写、状态查询、错误处理要完整。
- 不把业务逻辑写进底层驱动。
- 支持超时和错误返回。

### 17.2 常见驱动接口

```c
typedef enum {
    DRIVER_OK = 0,
    DRIVER_ERR,
    DRIVER_TIMEOUT,
    DRIVER_INVALID_ARG,
} DriverStatus;

DriverStatus sensor_init(void);
DriverStatus sensor_read(int16_t *temperature, uint16_t *humidity);
```

### 17.3 分层建议

- **BSP 层**：板级支持包，引脚、时钟、底层外设初始化。
- **Driver 层**：芯片或设备驱动，如 EEPROM、温度传感器、Flash。
- **Service 层**：业务服务，如数据采集、通信协议、参数管理。
- **Application 层**：产品逻辑、状态机、流程控制。

## 18. 状态机

状态机非常适合嵌入式流程控制。

### 18.1 基本概念

- 状态。
- 事件。
- 转移条件。
- 动作。

示例：

```c
typedef enum {
    SYS_IDLE,
    SYS_RUNNING,
    SYS_ERROR,
} SystemState;

static SystemState state = SYS_IDLE;

void system_step(void)
{
    switch (state) {
    case SYS_IDLE:
        if (start_cmd_received()) {
            state = SYS_RUNNING;
        }
        break;

    case SYS_RUNNING:
        if (fault_detected()) {
            state = SYS_ERROR;
        }
        break;

    case SYS_ERROR:
        if (reset_cmd_received()) {
            state = SYS_IDLE;
        }
        break;
    }
}
```

### 18.2 状态机优点

- 流程清晰。
- 方便处理异常。
- 方便调试。
- 方便扩展。
- 避免大量嵌套 `if`。

## 19. 硬件基础

嵌入式软件至少要能看懂常见原理图和硬件接口。

### 19.1 电源

常见电压：

- 24V：工业设备常用。
- 12V：电机、模块、车载常见。
- 5V：USB、部分传感器、继电器。
- 3.3V：MCU、通信芯片、传感器常见。
- 1.8V/1.2V：高速芯片、内核电压。

关注点：

- 电压范围。
- 电流能力。
- 纹波。
- 上电顺序。
- 欠压复位。
- 反接保护。
- 过流保护。
- ESD 和浪涌保护。

### 19.2 电平匹配

不同芯片电平可能不同。

常见组合：

- 5V MCU 和 3.3V 传感器。
- 3.3V MCU 和 5V 模块。
- TTL 串口和 RS232。
- MCU GPIO 和 24V 工业输入。

不匹配可能烧坏芯片或导致通信不稳定。

### 19.3 输入接口

常见输入：

- 按键。
- 限位开关。
- NPN/PNP 传感器。
- 光电开关。
- 编码器。
- 模拟量输入。

工业输入常见 24V，需要光耦、限流、电平转换和滤波。

### 19.4 输出接口

常见输出：

- LED。
- 蜂鸣器。
- 继电器。
- MOS 管。
- 电磁阀。
- 电机驱动。
- 24V 数字输出。

感性负载要注意续流二极管、TVS、RC 吸收或驱动芯片保护。

### 19.5 常见元件

- 电阻：限流、分压、上拉、下拉。
- 电容：滤波、去耦、储能、延时。
- 二极管：单向导通、续流、保护。
- 稳压管/TVS：过压保护。
- 三极管：小电流开关、放大。
- MOS 管：大电流开关。
- 光耦：隔离输入输出。
- 继电器：隔离和大电流开关。
- 运放：信号放大和滤波。

### 19.6 去耦电容

MCU 和数字芯片电源脚附近通常需要 0.1uF 去耦电容。

作用：

- 抑制瞬态电流造成的电压波动。
- 降低高频噪声。
- 提高芯片工作稳定性。

### 19.7 原理图阅读顺序

1. 先看电源输入和电源树。
2. 看 MCU 型号、供电、复位、晶振、下载接口。
3. 看每个外设连接到哪些引脚。
4. 看通信接口和电平转换。
5. 看输入输出保护。
6. 看接口连接器定义。
7. 对照 PCB 和实物确认方向、脚位和丝印。

### 20.1 常用工具

- 测电阻。
- 测通断。
- 测电流。

## 21. 常见故障排查

### 21.1 程序下载失败

可能原因：

- 芯片没有供电。
- SWDIO/SWCLK 接反或断线。
- 复位脚异常。
- BOOT 配置错误。
- 芯片被读保护。
- 调试接口被代码复用。
- 时钟配置错误导致芯片跑飞。

### 21.2 串口没有输出

可能原因：

- TX/RX 接反。
- 波特率不一致。
- 电平不匹配。
- 串口号选错。
- GPIO 复用配置错误。
- 没有共地。
- 串口打印被阻塞。

### 21.3 I2C 读不到设备

可能原因：

- 设备地址错误。
- 上拉电阻缺失。
- SDA/SCL 接反。
- 电源不对。
- 总线被拉低。
- 读写时序不对。
- 7 位地址和 8 位地址混淆。

### 21.4 SPI 通信异常

可能原因：

- CPOL/CPHA 配错。
- 片选时序错误。
- MOSI/MISO 接反。
- SPI 频率太高。
- 数据位序错误。
- 多设备片选冲突。

### 21.5 程序偶发死机

可能原因：

- 栈溢出。
- 数组越界。
- 野指针。
- 中断里耗时太长。
- 任务死锁。
- 电源干扰。
- 看门狗配置不合理。
- EMI 干扰。

### 21.6 ADC 数据不稳定

可能原因：

- 参考电压噪声。
- 输入没有滤波。
- 地线干扰。
- 采样时间太短。
- 信号源阻抗太高。
- 附近有高频或大电流开关信号。

## 22. 工程结构建议

一个较清晰的嵌入式工程可以这样组织：

```text
Project
├── App
│   ├── app_main.c
│   ├── app_state.c
│   └── app_config.h
├── BSP
│   ├── bsp_gpio.c
│   ├── bsp_uart.c
│   └── bsp_timer.c
├── Drivers
│   ├── sensor_xxx.c
│   ├── eeprom.c
│   └── motor_driver.c
├── Middleware
│   ├── modbus
│   ├── protocol
│   └── ring_buffer
├── RTOS
├── Utils
│   ├── log.c
│   ├── crc.c
│   └── filter.c
└── README.md
```

### 22.1 头文件原则

- `.h` 文件放公开接口。
- `.c` 文件放具体实现。
- 不要在头文件里定义普通全局变量。
- 头文件要加 include guard。

```c
#ifndef SENSOR_H
#define SENSOR_H

void sensor_init(void);

#endif
```

### 22.2 命名建议

- 模块名统一前缀，如 `uart_`、`sensor_`、`motor_`。
- 宏使用大写，如 `UART_RX_BUF_SIZE`。
- 类型名清晰，如 `MotorState`、`SensorData`。
- 函数表达动作，如 `motor_start()`、`sensor_read()`。

### 22.3 错误处理

函数不要只返回 `void`，关键操作要返回状态。

```c
typedef enum {
    APP_OK = 0,
    APP_ERR_TIMEOUT,
    APP_ERR_INVALID_PARAM,
    APP_ERR_COMM,
} AppStatus;
```

### 22.4 日志

日志建议分级：

- ERROR：错误。
- WARN：警告。
- INFO：普通信息。
- DEBUG：调试信息。

现场版本建议保留 ERROR 和关键 INFO。

## 23. Bootloader 和固件升级

Bootloader 是上电后先运行的一段程序，用于启动应用程序或更新固件。

### 23.1 常见功能

- 判断是否进入升级模式。
- 接收新固件。
- 校验固件完整性。
- 写入 Flash。
- 跳转到应用程序。
- 回滚或恢复出厂固件。

### 23.2 升级方式

- 串口升级。
- USB 升级。
- CAN 升级。
- RS485 升级。
- 以太网升级。
- 无线 OTA。

### 23.3 关键问题

- Flash 分区。
- 中断向量表重定位。
- 固件 CRC 或签名校验。
- 断电保护。
- 版本管理。
- 防止刷错型号。

## 24. 低功耗设计

低功耗常见于电池设备和物联网终端。

### 24.1 软件手段

- 降低主频。
- 不用的外设关闭时钟。
- 使用睡眠、停止、待机模式。
- 减少无线通信时间。
- 批量采集和上报。
- 使用中断唤醒，避免忙等。

### 24.2 硬件手段

- 选择低功耗芯片。
- 选择低静态电流 LDO/DC-DC。
- 关闭传感器电源。
- 减少分压电阻长期耗电。
- 控制 LED 和蜂鸣器耗电。

## 25. 安全性和可靠性

### 25.1 软件可靠性

- 参数范围检查。
- 通信超时处理。
- 状态机异常分支。
- 看门狗。
- 断电数据保护。
- CRC 校验。
- 错误日志。
- 默认安全状态。

### 25.2 硬件可靠性

- 电源保护。
- 反接保护。
- ESD 保护。
- 浪涌保护。
- 隔离。
- 滤波。
- 散热。
- 连接器防松。

### 25.3 工业现场注意事项

- 大电流电机、电磁阀、接触器会带来干扰。
- 通信线和动力线要分开布线。
- 屏蔽线要按规范接地。
- 传感器输入要滤波和防抖。
- 重要输出要有反馈检测。
- 故障时输出要进入安全状态。

## 26. 常用算法和工具函数

### 26.1 环形缓冲区

环形缓冲区常用于串口接收。

核心概念：

- 写指针。
- 读指针。
- 缓冲区满。
- 缓冲区空。

### 26.2 CRC

CRC 用于数据校验。

常见协议：

- Modbus RTU 使用 CRC16。
- 固件升级常用 CRC32。

### 26.3 滤波

常见滤波方法：

- 滑动平均。

## 32. 一句话原则

- 嵌入式调试先看电源，再看时钟，再看引脚，再看代码。
- 能用仪器看到的信号，不要只靠猜。
- 中断做通知，任务做处理。
- 驱动解决硬件访问，应用解决业务逻辑。
- 通信必须有超时，状态机必须有异常分支。
- 现场设备默认要考虑断电、掉线、误触发、干扰和重启。
- 好的嵌入式代码不是能跑一次，而是能长期稳定运行、容易定位问题、方便后续维护。

#### 典型场景拆解

在真实场景中，$topic 通常会经历“需求出现、方案选择、实现落地、问题暴露、持续优化”几个阶段。需求出现时，要先判断这个需求属于基础能力、性能优化、体验改进、可靠性建设还是长期架构演进。不同类型的需求对方案的评价标准不同：基础能力看正确性，性能优化看指标，体验改进看路径是否顺滑，可靠性建设看故障时能否降级和恢复，架构演进看未来变化是否容易吸收。

方案选择阶段，最容易犯的错误是直接套用熟悉工具。更稳妥的方式是列出约束：数据规模、时延要求、资源预算、团队熟悉度、运维能力、安全要求、可测试性和长期维护成本。只有把约束列清楚，才能解释为什么选择当前方案。否则方案看似高级，实际可能只是增加了复杂度。

问题暴露阶段，要优先区分“理解错误、实现错误、环境错误、数据错误、依赖错误、边界条件错误”。很多复杂问题之所以难排，是因为一开始就把问题归因到错误层级。例如把配置问题当成算法问题，把权限问题当成代码问题，把数据分布变化当成模型失效，把硬件噪声当成软件逻辑错误。好的排查顺序应该从可观测事实开始，而不是从猜测开始。

#### 常见误区与纠偏

第一个误区是只记结论，不记前提。$topic 中很多结论都是有条件的：适用于小规模，不一定适用于大规模；适用于离线处理，不一定适用于实时系统；适用于单机环境，不一定适用于分布式环境；适用于教学案例，不一定适用于生产项目。纠偏方法是在每个重要结论后面补一句“适用条件”和“不适用情况”。

### 精讲扩展 3：嵌入式学习笔记 的实践方法、常见问题 与工程化理解

学习 $topic 时，不能只把它当成一个孤立知识点来背诵，而要把它放到 $category 的完整问题链条里理解。一个知识点通常同时包含概念定义、适用边界、输入输出、运行过程、常见异常和工程取舍。真正掌握它，意味着看到一个具体场景时，能够判断它解决什么问题、依赖哪些前提、失败时会出现什么现象，以及应该用什么手段验证自己的判断。

从 $a 的角度看，最重要的是先建立清晰的对象模型。也就是明确系统里有哪些参与者、它们之间如何连接、数据或控制信号如何流动、哪些环节是同步的、哪些环节是异步的、哪些状态是临时状态、哪些状态需要长期保存。很多初学问题并不是公式不会、API 不熟，而是对象边界不清：把配置当成状态，把结果当成过程，把局部现象当成全局规律。写笔记时建议始终追问：这个概念的主体是谁，输入是什么，输出是什么，中间约束是什么，错误会在哪里暴露。

从 $b 的角度看，流程比单点知识更关键。一个成熟方案通常不是单个技巧，而是一组步骤：先确定目标，再拆分约束，然后选择工具，最后通过测试和复盘确认效果。比如在实际项目中，不能只问“怎么实现”，还要问“为什么要这样实现”“有没有更简单的替代方案”“边界条件是什么”“数据量、并发量、实时性、可靠性变化后还能不能工作”。这种流程意识能够避免把学习停留在教程层面，也能让后续排错有明确路线。

$topic 的 $c 往往决定它在真实项目中的稳定性。理论上可行的方案，到了工程环境中会受到数据质量、硬件条件、依赖版本、网络环境、团队协作、部署方式和维护成本影响。写代码或做设计时，应该把正常路径和异常路径同时考虑：正常情况下如何运行，输入为空怎么办，超时怎么办，重复执行怎么办，部分成功怎么办，版本升级后兼容性怎么办，日志和指标如何证明系统确实按预期工作。
