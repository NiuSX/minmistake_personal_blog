# 01-总览：做一款游戏需要学什么

## 1. 一款游戏由哪些部分组成

一款完整游戏通常包含：

- **玩法规则**：玩家能做什么，系统如何响应。
- **程序实现**：输入、角色、敌人、物理、UI、存档、网络等。
- **美术资产**：角色、场景、道具、图标、特效、动画。
- **音频资产**：音乐、音效、环境声、语音。
- **关卡和内容**：地图、任务、敌人配置、剧情事件、奖励。
- **UI/UX**：菜单、HUD、设置、背包、结算、引导。
- **工程工具**：编辑器工具、资源管理、配置表、版本控制。
- **测试优化**：Bug 修复、性能优化、兼容性、平衡性。
- **发布运营**：平台打包、商店页、更新、数据分析、社区反馈。

个人开发可以使用现成素材和工具降低工作量，但这些模块的基本概念都需要理解。

## 2. 学习方向总表

| 方向 | 需要学习 | 典型产出 |
| --- | --- | --- |
| 编程基础 | 变量、函数、面向对象、数据结构、调试 | 可运行的游戏逻辑 |
| 游戏引擎 | 场景、节点/Actor、组件、资源、打包 | 可玩的工程 |
| 游戏程序 | 输入、角色控制、AI、物理、存档、工具 | 核心功能和系统 |
| 游戏设计 | 核心循环、机制、数值、反馈、奖励 | 设计文档和规则 |
| 关卡设计 | 空间、节奏、敌人、任务、引导 | 地图和可玩内容 |
| 美术 | 构图、色彩、建模、贴图、像素画、UI 图标 | 游戏资产 |
| 动画特效 | 骨骼动画、状态机、粒子、Shader | 动态表现 |
| 音频 | 音效、音乐、混音、触发规则 | 听觉反馈 |
| UI/UX | 信息层级、流程、控件、引导 | 界面和交互体验 |
| 网络后端 | 同步、协议、账号、存档、排行榜 | 在线功能 |
| 测试发布 | Bug 管理、性能、平台规范、商店页 | 可发布版本 |

## 3. 独立开发者需要达到什么程度

独立开发不代表每项都专业级，但至少要做到：

- 能用引擎做出完整可玩的流程。
- 能修改和组织代码。
- 能制作或整合基础素材。
- 能设计明确的核心玩法。
- 能制作基本 UI 和反馈。
- 能测试、打包、修复问题。
- 能把游戏发布给别人玩。

## 4. 团队开发中各岗位

- **游戏策划/设计师**：玩法、系统、关卡、数值、文档。
- **客户端程序**：游戏逻辑、表现、UI、资源加载、性能。
- **服务器程序**：账号、匹配、战斗同步、数据库、运营工具。
- **技术美术 TA**：Shader、特效、管线、工具、性能。
- **美术**：角色、场景、UI、动画、特效。
- **音频设计**：音乐、音效、混音、实现。
- **测试 QA**：功能测试、兼容性、平衡、回归。
- **制作人/项目管理**：范围、排期、风险、资源协调。

## 5. 不建议一开始做的项目

初学者不建议一开始做：

- 开放世界。
- MMO。
- 大型多人竞技。
- 大型 3D 动作 RPG。
- 复杂沙盒建造。
- 高拟真物理模拟。
- 大量剧情分支的长篇 RPG。

更适合入门：

- 2D 平台跳跃。
- 俯视角射击。
- 塔防。
- 回合制小 RPG。
- 益智解谜。
- Roguelike 小原型。
- 简单模拟经营。
- 卡牌战斗原型。

## 6. 最小完整游戏标准

一个最小完整游戏至少包含：

- 开始界面。
- 明确目标。
- 可控制角色或核心交互。
- 成功和失败条件。
- 计分或奖励。
- 至少一个完整关卡或一轮游戏。
- 结算界面。
- 重新开始。
- 音效或视觉反馈。
- 可打包运行。

能完成一个小而完整的游戏，比做一个庞大但永远未完成的项目更有价值。

## 讲义级补充：如何真正学懂《01-总览：做一款游戏需要学什么》

> 适用位置：游戏开发学习路线\01-总览：做一款游戏需要学什么.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

游戏知识要同时服务于玩家体验和工程实现。一个机制是否成立，不只看代码能不能跑，还要看玩家是否理解、是否有选择、反馈是否及时、成本是否可控。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 体验目标：玩家应该感到什么、学会什么、追求什么。
- 核心循环：行动、反馈、奖励、成长和再次行动。
- 系统约束：资源、数值、关卡、难度和节奏。
- 制作落地：引擎、资源、工具、测试、优化和发布。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把游戏系统类比成一组相互咬合的规则齿轮：移动、战斗、资源、关卡、奖励和 UI 任何一环过强或过弱，都会改变玩家的节奏和动机。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《01-总览：做一款游戏需要学什么》时，可以做一个纸面或引擎原型：只保留一个核心动作、一个反馈、一个奖励和一个失败条件。观察玩家是否不用解释也能理解目标，再决定是否增加系统。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 先描述玩家目标、核心循环和关键反馈，而不是先堆功能列表。
- 再把机制拆成输入、规则、状态变化、奖励和失败条件。
- 最后用原型测试验证玩家是否真的理解并愿意重复游玩。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。
6. 反例与排错：展示错误做法会导致什么现象，再说明如何定位和修复。
7. 总结迁移：最后说明它和相邻知识点的区别、联系以及后续该学什么。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。
- 记录遇到的第一个错误、定位过程和最终修复方法。
- 完成后写 5 行复盘：我原来以为是什么，实际是什么，下次会如何更快处理。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 功能很多但核心循环不清晰，玩家不知道为什么继续玩。
- 只凭主观感觉调数值，没有记录版本和测试反馈。
- 忽视新手引导、反馈节奏和失败后的恢复路径。

遇到这些问题时，不要急着背更多资料。更有效的办法是回到一个最小例子，把输入、状态变化、输出和验证方式重新走一遍。

### 9. 自测题

1. 用一句话说明本讲主题解决的核心问题。
2. 列出本讲最重要的 3 个概念，并说明它们的关系。
3. 举一个生活类比，再指出这个类比在哪些地方不严谨。
4. 写出一个最小实践任务的验收标准。
5. 如果结果不符合预期，你会优先检查哪 3 个环节？为什么？
6. 本讲和相邻章节的知识边界是什么？哪些问题应该交给其他章节解决？

### 10. 复习口诀

先问场景，再看输入；先拆结构，再走流程；先做小例，再谈优化；先会排错，再做规模化。
