## 讲义导读：从原型到可玩的游戏

这一章讲的是 **02-编程基础与计算机基础**，属于 **游戏编程与计算机基础**。学习游戏开发时，不要只收集引擎教程、API 和工具名，而要把每个知识点放进一条作品闭环：先做出可操作的原型，再补反馈、内容、UI、测试、优化和发布，最后用真实试玩记录决定下一步改什么。

### 一句话先懂

编程基础要服务于游戏循环：读取输入、更新状态、检测碰撞、播放反馈、保存数据，并稳定跑在目标帧率。

初学时先问三个问题：玩家能做什么，系统怎样回应，玩家为什么愿意继续玩。只要这三个问题不清楚，技术越堆越多，项目越容易变成半成品。

### 通俗类比

游戏程序像舞台后台：每一帧都要安排演员位置、灯光音效、规则判定和观众看到的画面，任何一步卡住都会破坏体验。

类比只是入口。真正做游戏时，要回到游戏循环、输入延迟、反馈、状态机、关卡节奏、资源规范、性能预算、构建版本和玩家测试这些可执行对象上。

### 本章学习主线

1. **先做最小可玩原型**：角色能动，目标清楚，有失败和胜利，能重开。
2. **再补核心反馈**：输入、命中、受伤、奖励、错误、加载和过场都要有及时反馈。
3. **然后扩展内容**：关卡、敌人、道具、任务和数值要服务核心循环，而不是单纯堆量。
4. **接着做工程化**：资源命名、版本管理、存档、配置、调试工具、性能分析和构建流程要跟上。
5. **最后做试玩复盘**：观察玩家哪里不懂、哪里无聊、哪里卡住，再按证据调整。

### 本章重点抓手

变量、函数、面向对象、数据结构、事件、状态机、时间步、资源管理、调试和版本控制。

### 最小实践任务

不用复杂引擎，先做一个纯 2D 小循环：移动、碰撞、计分、失败、重开和调试日志。

建议每个学习阶段都产出一个可运行版本，并记录：目标、核心循环、操作方式、已完成内容、已知问题、试玩反馈和下一版改动。游戏开发最怕长期“学习中”却没有作品；小而完整的游戏，比庞大但不可玩的工程更有价值。

### 常见误区

- 语法会写但不会组织游戏状态。
- 把 Update 里堆满逻辑，后期难改。
- 不会用调试器和日志定位问题。

### 推荐工具与资料

Unity Learn/Unity Docs、Unreal Engine Documentation、Godot Docs、MDN Game Development、Git、itch.io/Steamworks 文档、性能分析器。

### 读完本章应该能做到

- 用自己的话说明本章内容服务于游戏作品的哪个环节。
- 做出一个能试玩的最小版本，而不是只完成孤立功能。
- 根据玩家反馈或性能数据提出下一版改动。
- 把引擎教程转化成自己的项目能力和作品集证据。

## 1. 为什么游戏开发必须学编程

游戏是大量规则和状态变化组成的交互系统。编程负责把规则变成可运行的行为：

- 玩家输入。
- 角色移动。
- 碰撞检测。
- 敌人 AI。
- 技能和伤害。
- UI 状态。
- 存档读写。
- 关卡事件。
- 网络通信。

即使使用可视化脚本，也需要理解编程思维。

## 2. 推荐语言

| 语言 | 常见用途 |
| --- | --- |
| C# | Unity 主力语言 |
| C++ | Unreal、引擎底层、高性能模块 |
| GDScript | Godot 常用脚本 |
| Lua | 热更新、配置脚本、部分商业项目 |
| Python | 工具脚本、资源处理、自动化 |
| JavaScript/TypeScript | Web 游戏、工具、小游戏 |

初学者建议：

- 学 Unity：先学 C#。
- 学 Unreal：先蓝图，再逐步学 C++。
- 学 Godot：先学 GDScript。
- 做 Web 小游戏：学 JavaScript 或 TypeScript。

## 3. 编程基础

必须掌握：

- 变量。
- 条件判断。
- 循环。
- 函数。
- 数组和列表。
- 字典/映射。
- 类和对象。
- 继承和组合。
- 接口或抽象。
- 事件和回调。
- 调试。

游戏例子：

```csharp
public class Player
{
    public int hp = 100;

    public void TakeDamage(int damage)
    {
        hp -= damage;
        if (hp <= 0)
        {
            Die();
        }
    }

    private void Die()
    {
        // 玩家死亡逻辑
    }
}
```

## 4. 数据结构

常用数据结构：

- **数组/List**：存放敌人、道具、关卡列表。
- **Dictionary/Map**：通过 ID 查找配置、物品、角色。
- **Queue**：任务队列、消息队列。
- **Stack**：撤销、状态回退。
- **Set**：去重，如已收集物品。
- **Tree**：行为树、技能树、UI 层级。
- **Graph**：地图寻路、节点关系。

## 7. 调试能力

必须学会：

- 看报错。
- 打断点。
- 单步执行。
- 查看变量。
- 打日志。
- 复现 Bug。
- 缩小问题范围。
- 读调用栈。

调试习惯：

- 不要盲改。
- 先复现，再定位。
- 一次只改一个变量。
- 记录原因和修复方式。

## 8. 版本控制

游戏项目必须学习 Git。

常用操作：

- `git status`
- `git add`
- `git commit`
- `git branch`
- `git checkout`

## 2026 游戏开发资料与项目核对补充

游戏开发路线建议按“官方文档 + 小项目 + 玩家反馈”三层学习。

- **引擎资料**：Unity 以 Unity Learn 和 Unity Documentation 为准；Unreal 以 Epic Developer Community / Unreal Engine Documentation 为准；Godot 以 Godot stable documentation 为准。
- **Web 游戏**：HTML5、Canvas、WebGL、WebGPU、Web Audio 和浏览器兼容性优先查 MDN Web Docs。
- **项目方法**：引擎学习以 Unity、Unreal、Godot 官方文档为准；Web 游戏查 MDN；发布平台和多人后端查对应官方文档。 学一个系统就做一个小版本，避免只看教程不交付。
- **发布核对**：目标平台、输入设备、分辨率、性能、存档、崩溃日志、商店素材、许可证和第三方资源授权都要提前确认。
- **作品集要求**：展示最终视频和截图之外，还要写清核心玩法、技术实现、工具链、遇到的问题、如何测试和如何迭代。

通俗地说，官方文档告诉你工具怎么用，小项目证明你能做出来，玩家反馈告诉你哪里真的需要改。三者结合，才是有效的游戏开发学习路线。
