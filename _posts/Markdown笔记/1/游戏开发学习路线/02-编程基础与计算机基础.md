# 02-编程基础与计算机基础

## 1. 为什么游戏开发必须学编程

游戏是大量规则和状态变化组成的交互系统。编程负责把规则变成可运行的行为：

- 玩家输入。
- 角色移动。
- 碰撞检测。
- 敌人 AI。
- 技能和伤害。
- UI 状态。
- 存档读写。
- 关卡事件。
- 网络通信。

即使使用可视化脚本，也需要理解编程思维。

## 2. 推荐语言

| 语言 | 常见用途 |
| --- | --- |
| C# | Unity 主力语言 |
| C++ | Unreal、引擎底层、高性能模块 |
| GDScript | Godot 常用脚本 |
| Lua | 热更新、配置脚本、部分商业项目 |
| Python | 工具脚本、资源处理、自动化 |
| JavaScript/TypeScript | Web 游戏、工具、小游戏 |

初学者建议：

- 学 Unity：先学 C#。
- 学 Unreal：先蓝图，再逐步学 C++。
- 学 Godot：先学 GDScript。
- 做 Web 小游戏：学 JavaScript 或 TypeScript。

## 3. 编程基础

必须掌握：

- 变量。
- 条件判断。
- 循环。
- 函数。
- 数组和列表。
- 字典/映射。
- 类和对象。
- 继承和组合。
- 接口或抽象。
- 事件和回调。
- 调试。

游戏例子：

```csharp
public class Player
{
    public int hp = 100;

    public void TakeDamage(int damage)
    {
        hp -= damage;
        if (hp <= 0)
        {
            Die();
        }
    }

    private void Die()
    {
        // 玩家死亡逻辑
    }
}
```

## 4. 数据结构

常用数据结构：

- **数组/List**：存放敌人、道具、关卡列表。
- **Dictionary/Map**：通过 ID 查找配置、物品、角色。
- **Queue**：任务队列、消息队列。
- **Stack**：撤销、状态回退。
- **Set**：去重，如已收集物品。
- **Tree**：行为树、技能树、UI 层级。
- **Graph**：地图寻路、节点关系。

## 5. 算法基础

游戏中常见算法：

- 排序。
- 搜索。
- 路径查找 A*。
- 随机权重。
- 碰撞检测。
- 插值。
- 状态机。
- 行为树。
- 简单概率和期望。

不需要一开始学成算法竞赛水平，但要能理解常见问题如何建模。

## 6. 数学基础

游戏开发常用数学：

- 向量。
- 坐标系。
- 三角函数。
- 矩阵。
- 四元数。
- 插值。
- 概率。
- 曲线。

2D 游戏重点：

- 坐标。
- 向量方向。
- 距离。
- 速度。
- 碰撞区域。

3D 游戏还要理解：

- 摄像机。
- 法线。
- 光照。
- 旋转。
- 空间变换。

## 7. 调试能力

必须学会：

- 看报错。
- 打断点。
- 单步执行。
- 查看变量。
- 打日志。
- 复现 Bug。
- 缩小问题范围。
- 读调用栈。

调试习惯：

- 不要盲改。
- 先复现，再定位。
- 一次只改一个变量。
- 记录原因和修复方式。

## 8. 版本控制

游戏项目必须学习 Git。

常用操作：

- `git status`
- `git add`
- `git commit`
- `git branch`
- `git checkout`
- `git merge`
- `git pull`
- `git push`

注意：

- Unity/Unreal 项目要正确配置 `.gitignore`。
- 大型美术资源可能需要 Git LFS。
- 提交信息要说明修改内容。
- 不要把临时缓存和构建产物提交进去。

## 9. 学习练习

- 写一个猜数字游戏。
- 写一个背包系统。
- 写一个回合制战斗模拟。
- 写一个技能冷却系统。
- 写一个简单 A* 寻路。
- 写一个存档读写 JSON。

## 讲义级补充：如何真正学懂《02-编程基础与计算机基础》

> 适用位置：游戏开发学习路线\02-编程基础与计算机基础.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

游戏知识要同时服务于玩家体验和工程实现。一个机制是否成立，不只看代码能不能跑，还要看玩家是否理解、是否有选择、反馈是否及时、成本是否可控。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 体验目标：玩家应该感到什么、学会什么、追求什么。
- 核心循环：行动、反馈、奖励、成长和再次行动。
- 系统约束：资源、数值、关卡、难度和节奏。
- 制作落地：引擎、资源、工具、测试、优化和发布。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把游戏系统类比成一组相互咬合的规则齿轮：移动、战斗、资源、关卡、奖励和 UI 任何一环过强或过弱，都会改变玩家的节奏和动机。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《02-编程基础与计算机基础》时，可以做一个纸面或引擎原型：只保留一个核心动作、一个反馈、一个奖励和一个失败条件。观察玩家是否不用解释也能理解目标，再决定是否增加系统。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 先描述玩家目标、核心循环和关键反馈，而不是先堆功能列表。
- 再把机制拆成输入、规则、状态变化、奖励和失败条件。
- 最后用原型测试验证玩家是否真的理解并愿意重复游玩。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。
6. 反例与排错：展示错误做法会导致什么现象，再说明如何定位和修复。
7. 总结迁移：最后说明它和相邻知识点的区别、联系以及后续该学什么。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。
- 记录遇到的第一个错误、定位过程和最终修复方法。
- 完成后写 5 行复盘：我原来以为是什么，实际是什么，下次会如何更快处理。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 功能很多但核心循环不清晰，玩家不知道为什么继续玩。
- 只凭主观感觉调数值，没有记录版本和测试反馈。
- 忽视新手引导、反馈节奏和失败后的恢复路径。

遇到这些问题时，不要急着背更多资料。更有效的办法是回到一个最小例子，把输入、状态变化、输出和验证方式重新走一遍。

### 9. 自测题

1. 用一句话说明本讲主题解决的核心问题。
2. 列出本讲最重要的 3 个概念，并说明它们的关系。
3. 举一个生活类比，再指出这个类比在哪些地方不严谨。
4. 写出一个最小实践任务的验收标准。
5. 如果结果不符合预期，你会优先检查哪 3 个环节？为什么？
6. 本讲和相邻章节的知识边界是什么？哪些问题应该交给其他章节解决？

### 10. 复习口诀

先问场景，再看输入；先拆结构，再走流程；先做小例，再谈优化；先会排错，再做规模化。
