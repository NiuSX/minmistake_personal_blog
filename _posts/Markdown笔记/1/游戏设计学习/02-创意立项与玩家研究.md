# 02-创意立项与玩家研究

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **02-创意立项与玩家研究**，属于 **创意立项与玩家研究**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

立项不是证明自己点子很酷，而是确认目标玩家是谁、他们为什么会玩、你的游戏能提供什么差异化体验。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

立项像开店选址：喜欢做什么很重要，但还要知道顾客是谁、旁边已有谁、你凭什么让人进来。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

目标玩家、竞品拆解、体验关键词、创意约束、市场定位、风险假设和验证问题。

### 最小实践任务

选一个想法，写出 3 个目标玩家假设、3 个竞品对比和 5 个需要用原型验证的问题。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 先做大世界观，再想玩家为什么玩。
- 只问朋友喜不喜欢，不观察真实行为。
- 目标玩家太泛，导致设计取舍摇摆。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

> 本节是讲义化改写后的阅读入口。后续正文中的设计方法、GDD、清单和模板，都应围绕“体验目标 + 可玩规则 + 试玩证据”来理解。
## 1. 创意不是立项

一个点子只是起点，不等于项目可以成立。立项需要回答：

- 目标玩家是谁。
- 玩家为什么会想玩。
- 核心体验是什么。
- 玩法是否能做出差异。
- 团队是否有能力完成。
- 开发周期和成本是否可控。
- 商业模式是否成立。
- 是否有足够内容支撑。

例如“做一款开放世界修仙游戏”只是想法。真正的立项要明确平台、规模、核心循环、操作方式、美术规格、内容范围、团队配置和风险。

## 2. 玩家画像

玩家画像用于帮助团队统一目标受众。

常见维度：

- 年龄段。
- 游戏平台。
- 每天游戏时间。
- 付费习惯。
- 喜欢的游戏类型。
- 讨厌的体验。
- 社交需求。
- 操作能力。
- 对复杂系统的接受度。
- 对剧情、竞技、养成、探索的偏好。

示例：

```text
目标玩家：20-35 岁移动端休闲策略玩家
游戏时间：每天碎片化 10-30 分钟
核心需求：轻度策略、持续成长、低压力竞争
不能接受：强制长时间在线、复杂操作、过重付费压迫
```

## 3. 市场与竞品分析

竞品分析不是复制，而是理解玩家预期和市场空位。

需要分析：

- 游戏类型和平台。
- 核心玩法。
- 首日体验。
- 新手引导。
- 成长系统。
- 付费结构。
- 内容更新方式。
- 社区评价。
- 差评集中点。
- 留存和长期目标。

## 4. 核心体验陈述

核心体验要用一句话表达清楚。

格式：

```text
让 [目标玩家] 在 [场景/平台] 中，通过 [核心行为] 获得 [核心感受]。
```

示例：

- 让休闲玩家通过简单合成和订单规划获得轻松成长感。
- 让硬核动作玩家通过精准闪避和连招获得高压战斗成就感。
- 让合作玩家通过分工和沟通解决混乱厨房任务获得欢乐感。

## 5. 立项评估

### 5.1 玩法风险

- 核心机制是否真的有趣。
- 玩家是否容易理解。
- 操作是否适合目标平台。
- 是否有足够变化空间。
- 是否能支撑长期内容。

### 5.2 技术风险

- 引擎是否适合。
- 网络同步是否复杂。
- AI、物理、渲染、地图生成是否有难点。
- 性能是否可控。
- 是否依赖团队没有经验的技术。

### 5.3 内容风险

- 是否需要大量关卡。
- 是否需要大量剧情文本。
- 是否需要大量美术资产。
- 内容生产管线是否成熟。
- 是否能持续更新。

### 5.4 商业风险

- 玩家是否愿意付费。
- 付费点是否影响公平和体验。
- 是否适合买断、广告、内购、订阅或 DLC。
- 获客成本是否可能过高。

## 6. 立项输出物

立项阶段常见输出：

- 项目一句话介绍。
- 目标玩家画像。
- 竞品分析。
- 核心体验说明。
- 核心玩法草案。
- 美术风格参考。
- 平台和商业模式。
- 项目范围。
- 风险列表。
- 原型验证计划。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
