# 03-核心玩法与机制设计

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **03-核心玩法与机制设计**，属于 **核心玩法与机制**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

核心玩法是玩家最常重复、最能产生乐趣的动作和决策，机制则是支撑这些动作的规则系统。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

核心玩法像一道菜的主味：配菜可以很多，但主味不清楚，玩家就不知道这游戏到底好玩在哪里。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

核心循环、动词、目标、规则、选择、风险、奖励、反馈、失败条件和可扩展性。

### 最小实践任务

为一个 5 分钟原型定义 1 个核心动词、2 个资源、3 种选择和明确的胜负反馈。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 机制很多但互不支撑。
- 玩家选择没有代价。
- 反馈太弱，玩家不知道自己做对还是做错。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

## 1. 核心玩法是什么

核心玩法是玩家在游戏中最频繁、最重要、最能代表游戏体验的行为组合。

例如：

- 射击游戏：瞄准、移动、射击、换弹、躲避、击败敌人。
- 平台跳跃：奔跑、跳跃、躲避陷阱、收集物品、到达终点。
- 卡牌游戏：抽牌、出牌、构筑、资源管理、击败对手。
- 模拟经营：生产、收集、建造、升级、扩大规模。
- Roguelike：探索、战斗、选择奖励、构筑流派、挑战随机关卡。

核心玩法必须尽早用原型验证。

## 2. 核心循环

核心循环是玩家反复执行的行为闭环。

示例：

```text
进入关卡 -> 战斗 -> 获得资源 -> 强化角色 -> 解锁新关卡 -> 再次挑战
```

核心循环要回答：

- 玩家做什么。
- 系统给什么反馈。
- 玩家获得什么奖励。
- 奖励如何推动下一轮行为。

## 3. 机制设计

机制是游戏规则中的基本动作和反应。

常见机制：

- 移动。
- 攻击。
- 防御。
- 闪避。
- 跳跃。
- 收集。
- 合成。
- 建造。
- 交易。
- 探索。
- 解谜。
- 升级。
- 抽卡。
- 匹配。

每个机制需要定义：

- 输入方式。
- 生效条件。
- 消耗资源。
- 成功结果。
- 失败结果。
- 冷却或限制。
- 反馈表现。
- 和其他机制的关系。

## 4. 目标与动机

玩家需要短期、中期、长期目标。

- **短期目标**：几秒到几分钟内完成，如击败敌人、完成一次合成。
- **中期目标**：几十分钟到几小时，如通关章节、解锁角色。
- **长期目标**：几天到几个月，如收集全图鉴、达到高排名。

目标设计要避免：

- 目标过多，玩家不知道做什么。
- 目标太远，玩家没有动力。
- 目标太简单，缺少成就感。
- 目标和核心玩法脱节。

## 5. 挑战设计

挑战来自限制和压力。

常见挑战来源：

- 时间限制。
- 资源限制。
- 敌人压力。
- 空间限制。
- 信息不完全。
- 操作精度。
- 策略选择。
- 风险收益权衡。

好的挑战应该让玩家觉得“我能做到，但需要更好地理解或操作”。

## 6. 奖励设计

奖励用于强化行为。

常见奖励：

- 数值奖励：金币、经验、材料。
- 能力奖励：技能、装备、属性。
- 内容奖励：关卡、剧情、角色。
- 表现奖励：皮肤、称号、特效。
- 社交奖励：排名、点赞、合作成果。
- 情绪奖励：爽感、惊喜、成就。

奖励要和核心玩法绑定。玩家通过主要行为获得主要成长，游戏才会形成正向循环。

## 7. 反馈设计

反馈让玩家知道自己做了什么、系统发生了什么。

反馈类型：

- 视觉：动画、闪光、颜色、UI 变化。
- 听觉：音效、音乐变化。
- 触觉：震动。
- 数值：伤害数字、经验条、分数。
- 规则反馈：敌人后退、门打开、技能冷却。

反馈不足会让游戏显得迟钝；反馈过度会造成噪音和疲劳。

## 8. 机制组合

有深度的玩法通常来自机制之间的组合。

示例：

- 火焰机制 + 草地机制 = 蔓延燃烧。
- 冰冻机制 + 水面机制 = 冻结路径。
- 潜行机制 + 声音机制 = 引诱敌人。
- 卡牌机制 + 能量机制 = 出牌顺序策略。

机制组合要注意：

- 玩家是否能理解。
- 组合是否有可预期结果。
- 是否产生过强套路。
- 是否让系统复杂度失控。

## 9. 核心玩法检查

- 玩家一分钟内主要在做什么。
- 这个行为是否有趣。
- 操作是否清晰。
- 成功和失败是否明确。
- 玩家是否知道下一步目标。
- 奖励是否驱动下一轮行为。
- 是否有成长空间。
- 是否能扩展出更多内容。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
