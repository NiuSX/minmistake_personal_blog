# 05-关卡设计与内容节奏

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **05-关卡设计与内容节奏**，属于 **关卡设计与内容节奏**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

关卡设计是在空间、敌人、资源、目标和节奏中安排玩家的学习、挑战、喘息和惊喜。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

关卡像一次导览路线：先让玩家看到目标，再给他练手机会，然后提高挑战，最后用奖励或反转收束。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

教学节奏、空间引导、视线、路径、挑战递进、资源投放、检查点、复用机制和内容节奏。

### 最小实践任务

用灰盒做 3 个房间：教学、变化、考验，并记录玩家在哪停顿、迷路或失败。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 只堆内容，不设计学习顺序。
- 引导全靠文字。
- 检查点和失败惩罚破坏节奏。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

## 1. 关卡设计是什么

关卡设计是把游戏规则、空间、敌人、资源、任务、谜题、剧情和节奏组织成玩家实际体验的过程。

关卡不是地图本身，而是玩家在空间和规则中经历的一段体验。

## 2. 关卡目标

每个关卡都要有明确目标：

- 教会一个新机制。
- 测试一个已学机制。
- 提供一次强战斗。
- 展示剧情事件。
- 提供探索奖励。
- 制造节奏变化。
- 作为 Boss 前铺垫。

如果关卡没有设计目标，很容易变成无意义填充。

## 3. 关卡基本结构

常见结构：

```text
进入 -> 学习/观察 -> 小挑战 -> 节奏提升 -> 高潮 -> 奖励 -> 退出
```

另一种结构：

```text
安全区 -> 教学区 -> 练习区 -> 组合挑战区 -> 终局挑战
```

## 4. 难度递进

经典设计方法：

1. 引入机制。
2. 让玩家安全练习。
3. 提高压力。
4. 与旧机制组合。
5. 用变化方式测试掌握程度。

示例：

- 先教玩家跳过小坑。
- 再加入移动平台。
- 再加入敌人干扰。
- 最后要求玩家在限时内连续跳跃。

## 5. 空间设计

空间影响玩家行为。

常见空间类型：

- 开阔空间：适合移动、躲避、群战。
- 狭窄通道：适合紧张感、伏击、限制视野。
- 高低差：创造路线选择和战斗优势。
- 分叉路线：提供探索和选择。
- 循环路线：减少回头路疲劳。
- 安全区：让玩家休息、整理和理解目标。

## 6. 引导设计

引导不一定是文字提示，也可以通过空间和视觉完成。

常见方法：

- 光线引导。
- 道路朝向。
- 颜色对比。
- 运动物体吸引注意。
- 远处地标。
- 敌人或奖励摆放。
- 摄像机角度。
- 声音来源。

好的引导让玩家觉得是自己发现的。

## 7. 战斗关卡

战斗关卡要关注：

- 敌人种类。
- 敌人数量。
- 出怪顺序。
- 战斗空间。
- 掩体和障碍。
- 补给位置。
- 逃生路线。
- 高潮时刻。

敌人配置原则：

- 不同敌人承担不同职责。
- 远程、近战、控制、坦克、召唤等角色要形成组合。
- 不要同时引入过多新敌人。
- 失败原因要清晰。

## 8. 解谜关卡

解谜关卡要关注：

- 玩家是否知道目标。
- 线索是否足够。
- 规则是否一致。
- 反馈是否明确。
- 是否有误导但不欺骗。
- 是否有卡死恢复方案。

解谜难度来自理解和推理，不应该来自界面不清或操作别扭。

## 9. 内容节奏

玩家不能一直高压，也不能一直平淡。

常见节奏：

- 教学。
- 探索。
- 小挑战。
- 奖励。
- 强挑战。
- 休息。
- 剧情。
- Boss。

节奏安排示例：

```text
轻松探索 -> 小战斗 -> 新机制教学 -> 组合挑战 -> 补给休息 -> Boss 战 -> 大奖励
```

## 10. 关卡文档需要包含

- 关卡名称。
- 设计目标。
- 目标玩家状态。
- 地图草图。
- 玩家路线。
- 敌人和资源配置。
- 触发器和事件。
- 剧情对白。
- 奖励。
- 失败条件。
- 复活点。
- 特殊规则。
- 测试重点。

## 11. 关卡检查

- 玩家是否知道去哪里。
- 玩家是否知道为什么失败。
- 新机制是否有安全练习空间。
- 难度是否逐步提升。
- 是否有节奏变化。
- 奖励是否和挑战匹配。
- 地图是否存在迷路或卡死。
- 关卡是否服务整体体验。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
