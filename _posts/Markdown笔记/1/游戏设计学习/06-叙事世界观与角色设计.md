## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **06-叙事世界观与角色设计**，属于 **叙事、世界观与角色**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

游戏叙事要服务玩家行动：世界观、角色和剧情最好能通过目标、选择、场景、道具和反馈被玩家体验到。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

游戏叙事像主题乐园故事线：玩家不是坐着听讲，而是在场景里走、拿道具、做选择，从行动中理解世界。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

主题、角色动机、世界规则、环境叙事、任务文本、对白、分支、节奏和玩家代理感。

### 最小实践任务

把一段设定改写成一个可玩的任务：目标、场景线索、NPC 反应、选择和后果。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 设定很多但玩法不体现。
- 剧情强迫玩家停下来听。
- 角色动机和玩家目标冲突。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

## 1. 游戏叙事的特点

游戏叙事不是只写故事，而是通过玩家行为、规则、场景、任务、角色、道具和反馈共同讲述体验。

游戏叙事可以来自：

- 主线剧情。
- 支线任务。
- 环境细节。
- 角色对白。
- 道具描述。
- 系统规则。
- 玩家选择。
- 关卡事件。
- 音乐和美术氛围。

## 2. 世界观设计

世界观回答“这个世界如何运转”。

需要定义：

- 世界背景。
- 历史事件。
- 地理区域。
- 势力组织。
- 科技或魔法规则。
- 社会结构。
- 资源和冲突来源。
- 普通人的生活状态。

世界观不是越大越好。它应该服务玩法和情绪。

## 3. 角色设计

角色设计需要包含：

- 名字。
- 身份。
- 外形特征。
- 目标。
- 动机。
- 性格。
- 能力。
- 弱点。
- 和主角关系。
- 角色弧光。

角色要能推动玩家行为，而不是只存在于设定文档。

## 4. 主角设计

主角常见类型：

- 明确人格主角。
- 玩家自定义角色。
- 沉默主角。
- 群像主角。

设计主角要考虑：

- 玩家能否代入。
- 主角能力是否符合玩法。
- 主角目标是否和玩家目标一致。
- 主角成长是否能在系统中体现。

## 5. 任务叙事

任务不是让玩家跑腿，而是给玩家明确动机和行为目标。

任务结构：

```text
触发原因 -> 目标说明 -> 行动过程 -> 阻碍变化 -> 结果反馈 -> 奖励或后果
```

常见任务类型：

- 收集。
- 击败。
- 护送。
- 探索。
- 解谜。
- 选择。
- 防守。
- 潜入。

任务设计要避免：

- 目标重复。
- 文案很长但行为单一。
- 玩家不知道为什么做。
- 奖励和任务难度不匹配。

## 6. 环境叙事

环境叙事通过场景讲故事。

示例：

- 破损的门说明这里曾发生战斗。
- 桌上的信件说明人物关系。
- 被封锁的道路暗示灾难范围。
- 墙上的涂鸦表达阵营立场。

环境叙事优点：

- 不打断玩家控制。
- 增强沉浸。
- 让玩家主动发现。
- 可以服务关卡引导。

## 7. 分支和选择

选择要有意义，需要带来：

- 不同结果。
- 不同代价。
- 不同信息。
- 不同角色关系。
- 不同资源收益。
- 不同后续关卡或任务。

伪选择会降低信任。如果资源有限，可以做小范围差异，但要让玩家感受到选择被承认。

## 8. 文案设计

游戏文案要简洁、准确、符合语气。

不同文本要求：

- UI 文案：短、清楚、无歧义。
- 任务文本：说明目标和动机。
- 道具描述：提供功能和世界观信息。
- 角色对白：体现身份和性格。
- 系统提示：告诉玩家发生了什么和下一步怎么做。

## 9. 叙事和玩法结合

好的结合方式：

- 剧情目标转化为关卡目标。
- 角色能力来自故事设定。
- 世界规则影响玩法规则。
- 玩家选择影响系统和内容。
- Boss 机制体现角色性格。

差的结合方式：

- 玩法和剧情互不相关。
- 大段剧情打断核心体验。
- 文案解释规则，但玩家实际体验不到。
- 世界观设定复杂但对游戏没有影响。

## 10. 叙事检查

- 玩家是否知道自己为什么行动。
- 角色目标是否清晰。
- 世界规则是否一致。
- 文案是否过长。
- 剧情是否服务玩法。
- 关键情绪是否通过玩法表达。
- 选择是否有反馈。
- 任务是否避免重复跑腿感。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
