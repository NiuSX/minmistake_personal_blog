# 07-UX、UI、音频与反馈设计

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **07-UX、UI、音频与反馈设计**，属于 **UX、UI、音频与反馈**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

反馈设计让玩家理解系统状态：我按了什么、命中了什么、损失了什么、下一步该去哪。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

反馈像和玩家对话：按钮、音效、闪光、震动和文字都在回答玩家“刚才发生了什么”。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

信息层级、HUD、菜单、教程、可访问性、音效、音乐、触觉反馈、动效和状态提示。

### 最小实践任务

给一个原型补 5 类反馈：输入、命中、危险、奖励、错误，并做一次无说明试玩。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- UI 解释系统，而不是系统本身可读。
- 音频和视觉反馈互相抢注意。
- 忽略色弱、字幕、音量和输入重映射。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

## 1. UX 和 UI 的区别

UX 关注玩家完成目标的整体体验，UI 是具体界面呈现和交互控件。

UX 问题：

- 玩家是否知道现在能做什么。
- 玩家是否知道下一步去哪里。
- 操作路径是否顺畅。
- 信息是否足够但不过载。
- 错误是否能恢复。

UI 问题：

- 按钮放哪里。
- 文本怎么写。
- 信息如何排版。
- 图标是否清楚。
- 状态如何显示。

## 2. 信息层级

界面中信息要有优先级。

常见层级：

- 必须立即看到：生命、弹药、关键倒计时。
- 当前决策需要：任务目标、资源数量、技能冷却。
- 可延后查看：背包详情、说明文本、历史记录。
- 装饰信息：氛围、背景、非必要统计。

不要把所有信息都同等强调。

## 3. HUD 设计

HUD 是游戏中持续显示的信息界面。

常见元素：

- 血条。
- 蓝条/能量条。
- 弹药。
- 小地图。
- 技能栏。
- 任务目标。
- 交互提示。
- 状态效果。

HUD 设计原则：

- 不遮挡关键游戏区域。
- 信息稳定，减少跳动。
- 危险信息要明显。
- 非关键状态可以弱化。
- 支持不同分辨率和平台。

## 4. 菜单和流程

常见界面流程：

- 主菜单。
- 设置。
- 存档。
- 角色。
- 背包。
- 商店。
- 任务。
- 地图。
- 结算。

流程设计要关注：

- 常用功能路径短。
- 返回逻辑一致。
- 危险操作有确认。
- 加载和等待有反馈。
- 错误提示可理解。

## 5. 新手引导

引导要让玩家学会游戏，而不是读完说明。

常见方式：

- 分步教学。
- 情境教学。
- 任务引导。
- 高亮提示。
- 限制操作范围。
- 示例演示。
- 失败后提示。

引导原则：

- 一次只教一个核心点。
- 教完马上让玩家实践。
- 不要在玩家还没需要时提前讲复杂系统。
- 支持跳过或回看。

## 6. 操作手感

手感来自输入、响应、动画、音效、镜头和规则共同作用。

影响因素：

- 输入延迟。
- 动画前摇和后摇。
- 移动加速度。
- 碰撞盒大小。
- 镜头跟随。
- 命中暂停。
- 屏幕震动。
- 音效力度。
- 特效清晰度。

动作游戏尤其要重视“按下按钮到角色响应”的时间。

## 7. 反馈设计

反馈要告诉玩家：

- 操作已发生。
- 是否成功。
- 影响了什么。
- 下一步可以做什么。

示例：

- 攻击命中：敌人受击动画、音效、伤害数字、血条变化。
- 技能不可用：按钮变灰、冷却数字、提示音。
- 获得奖励：飞行动画、音效、数量变化。
- 任务完成：弹窗、音效、地图标记变化。

## 8. 音频设计

音频包括：

- 背景音乐。
- 环境音。
- UI 音效。
- 战斗音效。
- 角色语音。
- 提示音。

音频作用：

- 强化情绪。
- 提供反馈。
- 提示危险。
- 区分状态。
- 增强空间感。

注意：

- 重要提示音要容易辨认。
- 高频重复音效不能刺耳。
- 音量层级要合理。
- 音乐变化可以提示阶段变化。

## 9. 无障碍设计

常见考虑：

- 字体大小可调。
- 色盲模式。
- 关键提示不只依赖颜色。
- 按键可重映射。
- 字幕。
- 降低镜头晃动选项。
- 难度选项。

无障碍不是额外装饰，它能扩大玩家范围并减少挫败。

## 10. UX 检查

- 玩家是否知道当前目标。
- 操作是否符合平台习惯。
- 常用功能是否容易找到。
- 重要信息是否明显。
- 提示是否能解决问题。
- 失败后是否知道原因。
- UI 是否遮挡关键内容。
- 音效是否强化而不是干扰体验。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
