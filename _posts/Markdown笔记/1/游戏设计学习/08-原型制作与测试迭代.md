# 08-原型制作与测试迭代

## 1. 原型的目的

原型不是做完整游戏，而是用最小成本验证关键问题。

常见验证目标：

- 核心玩法是否有趣。
- 操作是否顺手。
- 规则是否能被理解。
- 难度是否合理。
- 机制是否能扩展。
- 技术方案是否可行。
- 美术风格是否成立。

## 2. 原型类型

### 2.1 纸面原型

适合验证：

- 桌游规则。
- 卡牌机制。
- 资源流。
- 回合制战斗。
- 关卡流程。

优点是快，缺点是不适合验证手感和实时操作。

### 2.2 灰盒原型

使用简单方块、占位图和临时 UI 验证玩法。

适合验证：

- 移动手感。
- 战斗节奏。
- 关卡空间。
- 镜头。
- 交互流程。

### 2.3 垂直切片

垂直切片是一小段接近最终品质的完整体验。

用于验证：

- 美术、音频、玩法、UI 是否能形成完整体验。
- 团队管线是否能跑通。
- 项目品质目标是否现实。

## 3. 原型制作原则

- 只验证一个或少数关键问题。
- 使用占位资源。
- 不要过早做完整系统。
- 保留可调整参数。
- 快速试玩。
- 记录结论。

不要把原型代码直接当正式工程长期扩展，除非一开始就按工程质量写。

## 4. 测试方法

### 4.1 内部测试

适合早期快速发现明显问题。

风险：

- 团队成员知道设计意图，容易自动补全信息。
- 熟悉游戏后会低估新手难度。

### 4.2 外部测试

让真实目标玩家测试。

观察重点：

- 玩家第一反应。
- 卡在哪里。
- 是否理解目标。
- 是否重复犯错。
- 是否自发继续玩。
- 情绪变化。

### 4.3 静默观察

测试时不要一直解释，让玩家自然游玩。设计师要观察玩家行为，而不是急着为设计辩护。

## 5. 测试问题设计

不要只问“好不好玩”。可以问：

- 你觉得当前目标是什么。
- 你为什么这样选择。
- 哪里让你困惑。
- 哪里最有成就感。
- 哪里最想退出。
- 你愿意继续玩多久。
- 你觉得下一步会发生什么。

## 6. 数据记录

记录内容：

- 测试版本。
- 玩家信息。
- 游玩时长。
- 失败次数。
- 卡点位置。
- 放弃位置。
- 功能使用频率。
- 反馈原话。
- 观察结论。

## 7. 迭代流程

```text
提出假设 -> 制作原型 -> 测试 -> 记录问题 -> 分析原因 -> 修改设计 -> 再测试
```

注意：

- 玩家提出的是症状，不一定是正确解决方案。
- 多个玩家都卡同一处，优先处理。
- 不要为了一个玩家的偏好推翻整体设计。
- 迭代要保留版本记录。

## 8. 常见测试发现

- 玩家不知道目标。
- 操作反馈不明显。
- UI 文字太多。
- 难度曲线过陡。
- 奖励不吸引人。
- 机制有趣但节奏拖沓。
- 教学太早或太晚。
- 失败惩罚过重。

## 9. 原型检查

- 原型验证的问题是否明确。
- 是否真的需要做成可玩版本。
- 是否避免了无关细节。
- 参数是否方便调整。
- 测试对象是否接近目标玩家。
- 测试结论是否有记录。
- 修改后是否再次验证。
