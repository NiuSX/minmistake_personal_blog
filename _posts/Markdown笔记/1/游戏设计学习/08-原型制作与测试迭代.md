# 08-原型制作与测试迭代

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **08-原型制作与测试迭代**，属于 **原型、测试与迭代**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

原型的目的不是做漂亮，而是用最小成本验证核心假设：玩家是否理解、是否愿意重复、哪里出现问题。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

原型像试菜：先用小份测试味道和流程，再决定要不要投入完整装修和批量采购。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

纸面原型、灰盒、垂直切片、测试问题、观察记录、问卷、迭代优先级和失败复盘。

### 最小实践任务

做一次 3 人试玩测试：不解释规则，只观察行为，记录卡点、误解、情绪和改版清单。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 把原型当正式内容打磨。
- 测试时一直解释，污染结果。
- 只听玩家建议，不分析背后的问题。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

## 1. 原型的目的

原型不是做完整游戏，而是用最小成本验证关键问题。

常见验证目标：

- 核心玩法是否有趣。
- 操作是否顺手。
- 规则是否能被理解。
- 难度是否合理。
- 机制是否能扩展。
- 技术方案是否可行。
- 美术风格是否成立。

## 2. 原型类型

### 2.1 纸面原型

适合验证：

- 桌游规则。
- 卡牌机制。
- 资源流。
- 回合制战斗。
- 关卡流程。

优点是快，缺点是不适合验证手感和实时操作。

### 2.2 灰盒原型

使用简单方块、占位图和临时 UI 验证玩法。

适合验证：

- 移动手感。
- 战斗节奏。
- 关卡空间。
- 镜头。
- 交互流程。

### 2.3 垂直切片

垂直切片是一小段接近最终品质的完整体验。

用于验证：

- 美术、音频、玩法、UI 是否能形成完整体验。
- 团队管线是否能跑通。
- 项目品质目标是否现实。

## 3. 原型制作原则

- 只验证一个或少数关键问题。
- 使用占位资源。
- 不要过早做完整系统。
- 保留可调整参数。
- 快速试玩。
- 记录结论。

不要把原型代码直接当正式工程长期扩展，除非一开始就按工程质量写。

## 4. 测试方法

### 4.1 内部测试

适合早期快速发现明显问题。

风险：

- 团队成员知道设计意图，容易自动补全信息。
- 熟悉游戏后会低估新手难度。

### 4.2 外部测试

让真实目标玩家测试。

观察重点：

- 玩家第一反应。
- 卡在哪里。
- 是否理解目标。
- 是否重复犯错。
- 是否自发继续玩。
- 情绪变化。

### 4.3 静默观察

测试时不要一直解释，让玩家自然游玩。设计师要观察玩家行为，而不是急着为设计辩护。

## 5. 测试问题设计

不要只问“好不好玩”。可以问：

- 你觉得当前目标是什么。
- 你为什么这样选择。
- 哪里让你困惑。
- 哪里最有成就感。
- 哪里最想退出。
- 你愿意继续玩多久。
- 你觉得下一步会发生什么。

## 6. 数据记录

记录内容：

- 测试版本。
- 玩家信息。
- 游玩时长。
- 失败次数。
- 卡点位置。
- 放弃位置。
- 功能使用频率。
- 反馈原话。
- 观察结论。

## 7. 迭代流程

```text
提出假设 -> 制作原型 -> 测试 -> 记录问题 -> 分析原因 -> 修改设计 -> 再测试
```

注意：

- 玩家提出的是症状，不一定是正确解决方案。
- 多个玩家都卡同一处，优先处理。
- 不要为了一个玩家的偏好推翻整体设计。
- 迭代要保留版本记录。

## 8. 常见测试发现

- 玩家不知道目标。
- 操作反馈不明显。
- UI 文字太多。
- 难度曲线过陡。
- 奖励不吸引人。
- 机制有趣但节奏拖沓。
- 教学太早或太晚。
- 失败惩罚过重。

## 9. 原型检查

- 原型验证的问题是否明确。
- 是否真的需要做成可玩版本。
- 是否避免了无关细节。
- 参数是否方便调整。
- 测试对象是否接近目标玩家。
- 测试结论是否有记录。
- 修改后是否再次验证。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
