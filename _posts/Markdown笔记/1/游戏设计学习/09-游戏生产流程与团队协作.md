# 09-游戏生产流程与团队协作

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把体验目标变成可测试规则

这一章讲的是 **09-游戏生产流程与团队协作**，属于 **生产流程与团队协作**。学习游戏设计时，不要把它当成“想点子”或“写设定”，而要把它当成把体验目标落地的工程：玩家是谁，想让他产生什么感受，他能做什么选择，系统如何反馈，失败和奖励怎样出现，最后用试玩证明设计是否成立。

### 一句话先懂

团队协作要让设计、程序、美术、音频、测试和运营对同一个体验目标工作，并能处理范围、变更和风险。

初学时先问三个问题：玩家的目标是什么；玩家每 10 秒会做什么选择；系统怎样让玩家知道选择的结果。回答不了这三个问题，说明设计还没有从想法落到玩法。

### 通俗类比

游戏团队像拍电影加造机器：创意、技术和制作要同步，否则剧本改了、道具没改、现场就会停工。

类比只是入口。真正做设计时，要回到目标玩家、核心循环、机制、规则、资源、反馈、难度曲线、关卡节奏、UI 信息和 playtest 记录这些可讨论、可实现、可验证的对象上。

### 本章学习主线

1. **先定义体验目标**：想让玩家紧张、放松、好奇、掌控、社交、探索，还是获得成长感？
2. **再拆核心循环**：玩家看到目标，做出行动，系统反馈结果，玩家获得新目标或新资源。
3. **然后设计选择与代价**：好的选择要有信息、风险、收益和后果，不只是点哪个都一样。
4. **接着安排节奏**：教学、练习、变化、挑战、奖励和休息要交替出现。
5. **最后测试迭代**：观察玩家真实行为，而不是只听他口头说“挺好”。

### 本章重点抓手

里程碑、任务拆分、评审、需求变更、跨职能沟通、资产流程、风险管理和版本发布。

### 最小实践任务

把一个功能拆成设计说明、程序任务、美术需求、音频需求、测试点和验收标准。

建议每次设计练习都保留三份材料：一页规则说明、一段可玩的原型或流程图、一份试玩观察记录。游戏设计不是证明自己想法正确，而是尽早发现玩家怎样误解、绕过或不关心你的设计。

### 常见误区

- 设计文档写完就不更新。
- 需求没有验收标准。
- 改动范围失控，团队不知道优先级。

### 推荐资料与方法

Game Design Workshop、GDC/GDC Vault、Unity GDD 与 QA 资料、NN/g 用户测试方法、电子表格、流程图、原型工具、问卷和观察记录。

### 读完本章应该能做到

- 用体验目标解释设计，而不是只描述功能列表。
- 把一个想法拆成核心循环、规则、反馈和测试问题。
- 根据试玩观察提出修改，而不是只按个人喜好争论。
- 写出团队能实现、测试能验收的设计说明。

> 本节是讲义化改写后的阅读入口。后续正文中的设计方法、GDD、清单和模板，都应围绕“体验目标 + 可玩规则 + 试玩证据”来理解。
## 1. 游戏项目阶段

常见阶段：

1. 概念阶段。
2. 原型阶段。
3. 预生产阶段。
4. 正式生产阶段。
5. Alpha。
6. Beta。
7. 上线准备。
8. 上线运营。

## 2. 概念阶段

目标：

- 明确项目方向。
- 定义目标玩家。
- 验证市场机会。
- 提出核心体验。

输出：

- 一句话介绍。
- 玩家画像。
- 竞品分析。
- 核心玩法草案。
- 风格参考。
- 风险列表。

## 3. 原型阶段

目标：

- 验证核心玩法是否成立。
- 验证关键技术是否可行。
- 验证操作和节奏。

输出：

- 可玩原型。
- 测试记录。
- 迭代结论。
- 是否继续开发的判断。

## 4. 预生产阶段

目标：

- 建立完整设计框架。
- 跑通技术和内容管线。
- 制作垂直切片。
- 明确项目范围和排期。

输出：

- GDD。
- 技术方案。
- 美术风格规范。
- 关卡样板。
- UI 样板。
- 数值框架。
- 制作规范。
- 项目计划。

## 5. 正式生产阶段

目标：

- 批量制作内容。
- 完成系统功能。
- 持续集成和测试。

设计师任务：

- 维护设计文档。
- 输出配置表。
- 跟进实现质量。
- 验收功能。
- 调整数值。
- 处理反馈。
- 参与版本规划。

## 6. Alpha 阶段

Alpha 通常表示主要功能和内容已接近完整，但仍有大量问题。

重点：

- 功能完整性。
- 主流程跑通。
- 重大缺陷修复。
- 内容缺口确认。
- 性能初步优化。

## 7. Beta 阶段

Beta 更接近上线质量。

重点：

- 平衡性。
- 稳定性。
- 新手体验。
- 兼容性。
- 本地化。
- 运营活动。
- 商业化验证。

## 8. 上线准备

需要确认：

- 版本号。
- 打包流程。
- 平台审核材料。
- 隐私协议。
- 服务器容量。
- 日志和监控。
- 客服工具。
- 新手活动。
- 支付测试。
- 崩溃收集。
- 热更新方案。

## 9. 团队协作

### 9.1 策划和程序

策划要提供：

- 清晰规则。
- 状态流转。
- 边界条件。
- 配置字段。
- 异常处理。
- 验收标准。

程序需要反馈：

- 实现成本。
- 技术限制。
- 性能风险。
- 更合理的数据结构。

### 9.2 策划和美术

策划要提供：

- 功能需求。
- 角色定位。
- 场景用途。
- 交互状态。
- 资源规格。

美术需要反馈：

- 制作周期。
- 表现限制。
- 风格一致性。
- 资源复用方案。

### 9.3 策划和测试

策划要提供：

- 功能说明。
- 测试重点。
- 正常流程。
- 异常流程。
- 配置表说明。

测试需要反馈：

- Bug。
- 体验问题。
- 复现步骤。
- 风险范围。

## 10. 版本管理

设计文档和配置表必须有版本管理。

需要记录：

- 修改日期。
- 修改人。
- 修改内容。
- 影响范围。
- 是否已同步程序、美术、测试。

## 11. 需求变更

需求变更要问：

- 为什么改。
- 影响哪些系统。
- 是否影响排期。
- 是否影响已有资源。
- 是否需要重新测试。
- 是否有替代方案。

不要口头变更后不更新文档。

## 2026 游戏设计资料与试玩核对补充

游戏设计类笔记建议按“设计理论 + 行业案例 + 试玩证据”三层学习。

- **设计理论**：Game Design Workshop 适合建立 playcentric design 思路，把正式系统、戏剧系统和动态体验联系起来。
- **行业案例**：GDC 和 GDC Vault 可用于学习真实项目中的系统设计、关卡设计、叙事、UX、制作流程和复盘。
- **用户研究**：NN/g 的可用性测试、访谈、观察和启发式评估方法，可以迁移到游戏 UI、教程、引导和新手体验测试中。
- **项目文档**：Unity 的 GDD 与 QA 资料可作为入门模板，但具体 GDD 要服务团队沟通和可测试验收，不必追求厚文档。
- **实践要求**：游戏设计资料以可执行练习、设计拆解、GDC 经验分享和实际 playtest 记录为主，不把单个设计理论当万能公式。 每个设计结论最好配一个规则例子、原型、试玩记录或数据指标。

通俗地说，理论帮你提出假设，案例帮你扩大经验，试玩帮你面对现实。没有试玩证据的设计，很容易只是设计师自己的想象。
