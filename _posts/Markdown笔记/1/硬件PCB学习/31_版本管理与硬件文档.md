# 31 版本管理与硬件文档

## 学习目标

学完本章，你应该能：

- 理解硬件项目为什么也需要版本管理。
- 建立清晰的项目目录结构。
- 写硬件版本记录、测试记录和改版说明。
- 避免文件混乱导致生产或调试错误。

硬件不像软件一样可以随时在线更新。PCB 一旦打样，错误就变成实物成本。版本和文档管理能显著降低返工和沟通成本。

## 1. 为什么硬件需要版本管理

硬件版本管理用于追踪：

- 原理图改了什么。
- PCB 改了什么。
- BOM 改了什么。
- 哪一版已经打样。
- 哪一版有问题。
- 哪一版可以生产。
- 测试结果对应哪一版。

没有版本管理会导致：

- 用错 Gerber 下单。
- BOM 和 PCB 不匹配。
- 不知道哪版修复了问题。
- 调试记录无法对应实物。
- 团队成员拿错文件。

## 2. 推荐目录结构

示例：

```text
project_name/
  README.md
  CHANGELOG.md
  hardware/
    kicad/
    schematic_pdf/
    gerber/
    bom/
    cpl/
    assembly/
    datasheets/
    test/
    photos/
  firmware/
  docs/
```

小项目也可以简化，但至少要区分：

- 源文件。
- 生产文件。
- BOM。
- 测试记录。

## 3. 文件命名

推荐：

```text
projectname_hw_v0.1_2026-06-06.kicad_pro
projectname_schematic_v0.1.pdf
projectname_gerber_v0.1.zip
projectname_bom_v0.1.xlsx
projectname_test_report_v0.1.md
```

不要：

```text
最终版.zip
最终版2.zip
新最终版.zip
打样这个.zip
```

硬件文件必须能从名字看出版本。

## 4. 硬件版本号

常见：

- v0.1：第一版原型。
- v0.2：修复明显问题。
- v1.0：功能稳定版本。
- v1.1：小改版。
- v2.0：大改版。

也可以使用：

```text
EVT -> DVT -> PVT -> MP
```

含义：

- EVT：工程验证。
- DVT：设计验证。
- PVT：生产验证。
- MP：量产。

初学项目用 v0.1、v0.2、v1.0 就够。

## 5. CHANGELOG

改版记录应包含：

```text
版本：v0.2
日期：2026-06-06
修改内容：
- 修正 USB D+ / D- 接反问题
- 增加 3V3 测试点
- 将 LDO 封装从 SOT-23 改为 SOT-223
- LED 限流电阻从 1k 改为 330R
原因：
- v0.1 USB 无法枚举
- LDO 满载温度过高
验证：
- USB 枚举正常
- 3V3 满载 300mA 温度可接受
```

## 6. 硬件测试记录

测试记录包括：

- 板子版本。
- 焊接状态。
- 上电电流。
- 各路电源电压。
- 下载是否成功。
- 接口测试结果。
- 传感器数据。
- 温度。
- 发现问题。

示例：

| 测试项 | 结果 | 备注 |
| :--- | :--- | :--- |
| 输入 5V | 通过 | 5.02V |
| 3V3 输出 | 通过 | 3.31V |
| SWD 下载 | 通过 | 正常识别 |
| UART | 失败 | TX/RX 可能接反 |

## 7. 板上版本标识

PCB 丝印应包含：

- 项目名。
- 硬件版本。
- 日期或批次。
- 关键接口方向。

例如：

```text
SENSOR_NODE
HW v0.2
2026-06
```

调试多块板时非常有用。

## 8. Git 管理硬件

可以用 Git 管理：

- 原理图源文件。
- PCB 源文件。
- README。
- CHANGELOG。
- 生产文件。

注意：

- 大型二进制文件谨慎管理。
- 每次打样前打 tag。
- 生产文件和源文件都要对应版本。

示例 tag：

```text
hw-v0.1-fab
hw-v0.2-fab
```

## 9. 发布包

每次打样或生产建议生成发布包：

```text
release_hw_v0.2/
  schematic.pdf
  gerber.zip
  drill.zip
  bom.xlsx
  cpl.csv
  assembly.pdf
  readme.txt
```

这样板厂、贴片厂和自己都不会混淆。

## 实操练习

1. 为你的 PCB 项目建立目录结构。
2. 写一个 README，说明项目功能和供电方式。
3. 写一个 CHANGELOG。
4. 给 PCB 丝印添加项目名和版本号。
5. 为一次打样生成 release 文件夹。

## 检查清单

- PCB 上是否有版本号？
- Gerber 是否有明确版本？
- BOM 是否和 PCB 对应？
- 是否有测试记录？
- 是否记录改版原因？
- 是否能找到每一版生产文件？

## 常见误区

- 误区：个人项目不用文档。
  纠正：过几周后你也会忘记当时为什么这么改。

- 误区：只保存最新文件。
  纠正：硬件问题经常需要对比旧版本。

- 误区：Gerber 和源文件不用对应。
  纠正：生产文件必须能追溯到源文件版本。

## 本章总结

硬件版本管理是工程能力的一部分。清晰命名、版本记录、测试记录、生产发布包，会让项目更可靠，也让改版更有依据。
