# 02. 信息表示：二进制、整数、浮点与编码

最后调研时间：2026-06-11

## 1. 为什么从信息表示开始

计算机系统中所有东西最终都是位：

```text
程序 = 位
整数 = 位
浮点数 = 位
字符 = 位
图片 = 位
文件 = 位
网络包 = 位
```

系统学习的第一步是理解“同一串位可以被解释成不同含义”。

例如：

```text
01000001
```

可以解释为：

- 无符号整数 65。
- ASCII 字符 `A`。
- 某个机器指令的一部分。
- 图片像素的一部分。

数据没有天然意义，意义来自解释规则。

## 2. 进制

常见进制：

| 进制 | 基数 | 常见用途 |
|---|---:|---|
| 二进制 | 2 | 机器内部表示 |
| 八进制 | 8 | Unix 权限历史表示 |
| 十进制 | 10 | 人类日常计算 |
| 十六进制 | 16 | 地址、机器码、位模式 |

十六进制常用，因为 1 个十六进制位刚好表示 4 个二进制位：

```text
0xF = 1111
0x10 = 0001 0000
0xFF = 1111 1111
```

## 3. 位、字节、字

| 单位 | 含义 |
|---|---|
| bit | 位，0 或 1 |
| byte | 字节，通常 8 bit |
| word | 字，CPU 自然处理宽度，取决于架构 |

常见大小：

```text
1 KiB = 1024 bytes
1 MiB = 1024 KiB
1 GiB = 1024 MiB
```

注意：

- KB 有时表示 1000 bytes，有时表示 1024 bytes。
- KiB 明确表示 1024 bytes。

## 4. 无符号整数

n 位无符号整数范围：

```text
0 ~ 2^n - 1
```

例如 8 位：

```text
00000000 = 0
00000001 = 1
11111111 = 255
```

无符号整数运算通常是模运算：

```c
unsigned char x = 255;
x = x + 1; // 结果通常回到 0
```

理解溢出很重要，因为系统底层大量使用无符号类型表示大小、地址、位掩码。

## 5. 有符号整数与补码

现代计算机通常使用补码表示有符号整数。

n 位补码范围：

```text
-2^(n-1) ~ 2^(n-1)-1
```

8 位补码：

```text
00000000 = 0
00000001 = 1
01111111 = 127
10000000 = -128
11111111 = -1
```

### 5.1 为什么用补码

补码的优点：

- 加法和减法可以统一用同一套硬件。
- 0 只有一种表示。
- 符号扩展简单。

例如：

```text
  00000001   1
+ 11111111  -1
= 00000000   0
```

最高位不是单纯“符号位”，而是有负权重：

```text
10000000 = -128
```

### 5.2 有符号和无符号转换

同一位模式解释不同，值不同：

```text
11111111
```

- 作为 `uint8_t`：255。
- 作为 `int8_t`：-1。

C/C++ 中混用 signed 和 unsigned 容易出 bug：

```c
int x = -1;
unsigned y = 1;
printf("%d\n", x < y); // 可能输出 0
```

原因：比较时 `x` 可能转换为 unsigned，变成很大的数。

## 6. 整数溢出

溢出是系统安全和 bug 的高频来源。

```c
int a = 2147483647;
int b = a + 1;
```

C/C++ 中有符号整数溢出是未定义行为；无符号整数溢出按模 2^n 回绕。

常见风险：

- 数组长度计算溢出。
- 内存分配大小溢出。
- 边界检查绕过。
- 时间戳溢出。
- ID 自增回绕。

安全写法：

- 检查加法是否溢出。
- 使用更大类型。
- 使用编译器 sanitizer。
- 避免 signed/unsigned 混用。

## 7. 位运算

常见位运算：

| 运算 | 含义 |
|---|---|
| `&` | 按位与 |
| `|` | 按位或 |
| `^` | 按位异或 |
| `~` | 按位取反 |
| `<<` | 左移 |
| `>>` | 右移 |

用途：

- 位掩码。
- 权限标志。
- 网络协议字段。
- 压缩状态。
- 高性能计算。
- 硬件寄存器控制。

示例：

```c
#define READ  0x1
#define WRITE 0x2
#define EXEC  0x4

int perm = READ | WRITE;
if (perm & READ) {
    // has read permission
}
```

## 8. 字节序

字节序描述多字节数据在内存中的排列。

```text
数值：0x12345678
```

大端：

```text
12 34 56 78
```

小端：

```text
78 56 34 12
```

常见：

- x86 通常是小端。
- 网络字节序通常是大端。

网络编程中常用：

```c
htonl()
htons()
ntohl()
ntohs()
```

## 9. 浮点数

浮点数通常遵循 IEEE 754 标准思想，由三部分组成：

```text
符号位 + 阶码 + 尾数
```

以单精度 float 为例：

| 部分 | 位数 |
|---|---:|
| sign | 1 |
| exponent | 8 |
| fraction | 23 |

浮点数表示的是近似值，不是任意实数。

### 9.1 浮点误差

经典例子：

```python
0.1 + 0.2
```

结果不是精确的 `0.3`，因为 0.1 和 0.2 在二进制中无法有限表示。

不要用浮点直接比较相等：

```c
if (a == b) { } // 不推荐
```

应使用误差范围：

```c
if (fabs(a - b) < 1e-9) { }
```

### 9.2 特殊值

浮点数有特殊值：

- `+0`
- `-0`
- `+inf`
- `-inf`
- `NaN`

`NaN` 不等于任何值，包括它自己。

```c
double x = NAN;
printf("%d\n", x == x); // 通常为 0
```

## 10. 字符编码

字符不是字节本身，而是通过编码规则映射。

常见编码：

| 编码 | 特点 |
|---|---|
| ASCII | 7 位，英文基础字符 |
| GBK | 中文编码，兼容部分中文历史系统 |
| Unicode | 字符集，定义字符码点 |
| UTF-8 | Unicode 的变长编码，互联网常用 |
| UTF-16 | 常见于 Windows、Java 内部历史实现 |

### 10.1 UTF-8

UTF-8 是变长编码：

- 英文通常 1 字节。
- 常见中文通常 3 字节。
- emoji 可能 4 字节或更多组合。

常见问题：

- 文件编码不一致导致乱码。
- 字符数量不等于字节数量。
- 字符截断可能截在多字节中间。
- 大小写、排序、规范化在 Unicode 下很复杂。

## 11. 数据结构的内存布局

C 结构体可能有 padding：

```c
struct A {
    char c;
    int x;
};
```

`sizeof(struct A)` 可能是 8，而不是 5，因为 `int` 需要对齐。

对齐的原因：

- CPU 访问对齐地址更快。
- 某些架构不支持非对齐访问。
- ABI 需要固定规则。

优化结构体大小可以调整字段顺序：

```c
struct B {
    int x;
    char c;
};
```

但不要为了省几个字节破坏语义，除非结构体数量巨大。

## 12. 常见坑

| 坑 | 说明 |
|---|---|
| signed/unsigned 混用 | 可能导致比较结果反直觉 |
| 整数溢出 | 可能造成安全漏洞 |
| 浮点直接判等 | 精度误差导致失败 |
| 忽略字节序 | 网络协议和二进制文件解析错误 |
| 字符数等于字节数 | UTF-8 中文会出错 |
| 结构体大小手算错误 | 忽略 padding 和 alignment |
