# 02. 信息表示：二进制、整数、浮点与编码

最后调研时间：2026-06-11

## 1. 为什么从信息表示开始

计算机系统中所有东西最终都是位：

```text
程序 = 位
整数 = 位
浮点数 = 位
字符 = 位
图片 = 位
文件 = 位
网络包 = 位
```

系统学习的第一步是理解“同一串位可以被解释成不同含义”。

例如：

```text
01000001
```

可以解释为：

- 无符号整数 65。
- ASCII 字符 `A`。
- 某个机器指令的一部分。
- 图片像素的一部分。

数据没有天然意义，意义来自解释规则。

## 2. 进制

常见进制：

| 进制 | 基数 | 常见用途 |
|---|---:|---|
| 二进制 | 2 | 机器内部表示 |
| 八进制 | 8 | Unix 权限历史表示 |
| 十进制 | 10 | 人类日常计算 |
| 十六进制 | 16 | 地址、机器码、位模式 |

十六进制常用，因为 1 个十六进制位刚好表示 4 个二进制位：

```text
0xF = 1111
0x10 = 0001 0000
0xFF = 1111 1111
```

## 3. 位、字节、字

| 单位 | 含义 |
|---|---|
| bit | 位，0 或 1 |
| byte | 字节，通常 8 bit |
| word | 字，CPU 自然处理宽度，取决于架构 |

常见大小：

```text
1 KiB = 1024 bytes
1 MiB = 1024 KiB
1 GiB = 1024 MiB
```

注意：

- KB 有时表示 1000 bytes，有时表示 1024 bytes。
- KiB 明确表示 1024 bytes。

## 4. 无符号整数

n 位无符号整数范围：

```text
0 ~ 2^n - 1
```

例如 8 位：

```text
00000000 = 0
00000001 = 1
11111111 = 255
```

无符号整数运算通常是模运算：

```c
unsigned char x = 255;
x = x + 1; // 结果通常回到 0
```

理解溢出很重要，因为系统底层大量使用无符号类型表示大小、地址、位掩码。

## 5. 有符号整数与补码

现代计算机通常使用补码表示有符号整数。

n 位补码范围：

```text
-2^(n-1) ~ 2^(n-1)-1
```

8 位补码：

```text
00000000 = 0
00000001 = 1
01111111 = 127
10000000 = -128
11111111 = -1
```

### 5.1 为什么用补码

补码的优点：

- 加法和减法可以统一用同一套硬件。
- 0 只有一种表示。
- 符号扩展简单。

例如：

```text
  00000001   1
+ 11111111  -1
= 00000000   0
```

最高位不是单纯“符号位”，而是有负权重：

```text
10000000 = -128
```

### 5.2 有符号和无符号转换

同一位模式解释不同，值不同：

```text
11111111
```

- 作为 `uint8_t`：255。
- 作为 `int8_t`：-1。

C/C++ 中混用 signed 和 unsigned 容易出 bug：

```c
int x = -1;
unsigned y = 1;
printf("%d\n", x < y); // 可能输出 0
```

原因：比较时 `x` 可能转换为 unsigned，变成很大的数。

## 6. 整数溢出

溢出是系统安全和 bug 的高频来源。

```c
int a = 2147483647;
int b = a + 1;
```

C/C++ 中有符号整数溢出是未定义行为；无符号整数溢出按模 2^n 回绕。

常见风险：

- 数组长度计算溢出。
- 内存分配大小溢出。
- 边界检查绕过。
- 时间戳溢出。
- ID 自增回绕。

安全写法：

- 检查加法是否溢出。
- 使用更大类型。
- 使用编译器 sanitizer。
- 避免 signed/unsigned 混用。

## 7. 位运算

常见位运算：

| 运算 | 含义 |
|---|---|
| `&` | 按位与 |
| `|` | 按位或 |
| `^` | 按位异或 |
| `~` | 按位取反 |
| `<<` | 左移 |
| `>>` | 右移 |

用途：

- 位掩码。
- 权限标志。
- 网络协议字段。
- 压缩状态。
- 高性能计算。
- 硬件寄存器控制。

示例：

```c
#define READ  0x1
#define WRITE 0x2
#define EXEC  0x4

int perm = READ | WRITE;
if (perm & READ) {
    // has read permission
}
```

## 8. 字节序

字节序描述多字节数据在内存中的排列。

```text
数值：0x12345678
```

大端：

```text
12 34 56 78
```

小端：

```text
78 56 34 12
```

常见：

- x86 通常是小端。
- 网络字节序通常是大端。

网络编程中常用：

```c
htonl()
htons()
ntohl()
ntohs()
```

## 9. 浮点数

浮点数通常遵循 IEEE 754 标准思想，由三部分组成：

```text
符号位 + 阶码 + 尾数
```

以单精度 float 为例：

| 部分 | 位数 |
|---|---:|
| sign | 1 |
| exponent | 8 |
| fraction | 23 |

浮点数表示的是近似值，不是任意实数。

### 9.1 浮点误差

经典例子：

```python
0.1 + 0.2
```

结果不是精确的 `0.3`，因为 0.1 和 0.2 在二进制中无法有限表示。

不要用浮点直接比较相等：

```c
if (a == b) { } // 不推荐
```

应使用误差范围：

```c
if (fabs(a - b) < 1e-9) { }
```

### 9.2 特殊值

浮点数有特殊值：

- `+0`
- `-0`
- `+inf`
- `-inf`
- `NaN`

`NaN` 不等于任何值，包括它自己。

```c
double x = NAN;
printf("%d\n", x == x); // 通常为 0
```

## 10. 字符编码

字符不是字节本身，而是通过编码规则映射。

常见编码：

| 编码 | 特点 |
|---|---|
| ASCII | 7 位，英文基础字符 |
| GBK | 中文编码，兼容部分中文历史系统 |
| Unicode | 字符集，定义字符码点 |
| UTF-8 | Unicode 的变长编码，互联网常用 |
| UTF-16 | 常见于 Windows、Java 内部历史实现 |

### 10.1 UTF-8

UTF-8 是变长编码：

- 英文通常 1 字节。
- 常见中文通常 3 字节。
- emoji 可能 4 字节或更多组合。

常见问题：

- 文件编码不一致导致乱码。
- 字符数量不等于字节数量。
- 字符截断可能截在多字节中间。
- 大小写、排序、规范化在 Unicode 下很复杂。

## 11. 数据结构的内存布局

C 结构体可能有 padding：

```c
struct A {
    char c;
    int x;
};
```

`sizeof(struct A)` 可能是 8，而不是 5，因为 `int` 需要对齐。

对齐的原因：

- CPU 访问对齐地址更快。
- 某些架构不支持非对齐访问。
- ABI 需要固定规则。

优化结构体大小可以调整字段顺序：

```c
struct B {
    int x;
    char c;
};
```

但不要为了省几个字节破坏语义，除非结构体数量巨大。

## 12. 常见坑

| 坑 | 说明 |
|---|---|
| signed/unsigned 混用 | 可能导致比较结果反直觉 |
| 整数溢出 | 可能造成安全漏洞 |
| 浮点直接判等 | 精度误差导致失败 |
| 忽略字节序 | 网络协议和二进制文件解析错误 |
| 字符数等于字节数 | UTF-8 中文会出错 |
| 结构体大小手算错误 | 忽略 padding 和 alignment |

## 讲义级补充：如何真正学懂《02. 信息表示：二进制、整数、浮点与编码》

> 适用位置：计算机系统\02-数据表示.md  
> 说明：本补充用于把原始提纲扩展成课堂讲义式学习材料。阅读时建议先看原文，再用本节建立知识框架、例子、实践和自测闭环。

### 1. 这一讲要解决什么问题

计算机系统知识解释程序为什么这样运行。学习时要把高级语言、编译器、指令、内存、操作系统、文件、网络和硬件资源联系起来，避免只停留在 API 使用层。

学习本讲时，可以用三个问题检查自己是否真的理解：

1. 它解决的真实问题是什么？
2. 如果没有它，系统会出现什么具体麻烦？
3. 在真实项目中，应该用什么现象或指标判断它做得好不好？

### 2. 核心知识拆解

可以把本讲拆成几块来学：

- 表示：数据如何编码，指令如何表达操作。
- 执行：CPU、寄存器、内存和操作系统如何协作。
- 资源：进程、线程、文件、网络和设备如何被管理。
- 诊断：性能、崩溃、安全和可靠性如何分析。

拆解的好处是防止“整章都懂一点，但哪块都说不清”。复习时可以逐块追问：它的输入是什么、输出是什么、依赖什么、失败时有什么表现。

### 3. 通俗类比

可以把程序运行类比成工厂流水线：源代码经过编译链接变成机器能执行的指令，CPU 负责加工，内存负责临时摆放材料，磁盘负责长期仓储，操作系统负责调度工人和分配场地。

类比不是严格定义，但能帮助初学者先建立直觉。真正使用时，还要回到术语、公式、接口、数据结构、时序图或工程规范上，把“感觉理解”变成“可验证理解”。

### 4. 具体例子

学习《02. 信息表示：二进制、整数、浮点与编码》时，可以用一个最小程序做追踪：从源代码、编译产物、进程启动、内存占用到系统调用逐步观察。系统知识越底层，越需要用工具看到证据。

讲义级学习不能只停留在“概念解释”。至少要有一个能跑、能算、能画或能检查的例子。例子越小，越容易看清关键机制；等机制清楚后，再逐步扩展到复杂项目。

### 5. 学习路径

- 先从一个简单程序出发，追踪它如何被编译、加载、执行和退出。
- 再理解 CPU、内存、文件系统、进程线程、网络和安全机制如何配合。
- 最后用调试器、性能分析器和系统日志把抽象概念落到真实现象。

建议每学完一小节都做一次“复述练习”：不用看笔记，用自己的话讲清楚概念、输入、输出、关键步骤和常见错误。如果讲不清，通常说明还没有真正掌握。

### 6. 课堂讲解框架

可以按下面顺序讲解或复习本主题：

1. 背景：先讲这个知识为什么出现，它试图降低什么成本、解决什么风险或提升什么能力。
2. 基本概念：给出核心名词的准确定义，说明它们之间的关系。
3. 工作流程：按时间顺序描述一次完整过程，必要时画出流程图、状态机或数据流图。
4. 关键细节：解释最容易误解的机制，例如边界条件、异常处理、性能限制、资源生命周期或安全约束。
5. 实战例子：用一个足够小但完整的例子，把概念落到命令、代码、图纸、配置、数据或操作步骤上。

### 7. 最小实践任务

为了避免“看懂了但不会用”，建议为本讲配一个最小实践：

- 选一个可以在 30 到 90 分钟内完成的小任务。
- 明确输入、预期输出和验收标准。

如果本主题偏理论，实践可以是手算一个小例子、画一张流程图、推导一个简化公式或解释一段真实日志；如果偏工程，实践应该尽量落到可运行命令、可测试代码、可检查配置或可测量硬件现象上。

### 8. 常见误区

- 只记结论，不理解适用条件。
- 只看正常流程，不看异常、边界和失败恢复。
