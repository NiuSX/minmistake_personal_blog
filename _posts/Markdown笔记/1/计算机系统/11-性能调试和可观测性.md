# 11. 性能、调试与可观测性

最后调研时间：2026-06-13

## 1. 性能优化原则

性能优化最重要的原则：

```text
先测量，再优化。
```

不要凭感觉猜瓶颈。程序慢可能是：

- CPU 计算。
- 内存访问。
- 锁竞争。
- 系统调用太多。
- 磁盘 I/O。
- 网络等待。
- 数据库。
- GC。
- 日志过多。
- 线程调度。

推荐排查顺序：

```text
明确现象 -> 固定指标 -> 缩小范围 -> 找证据 -> 改一处 -> 复测
```

不要一边改代码一边猜。每次优化都应记录：

- 优化前指标。
- 改动内容。
- 优化后指标。
- 输入规模和环境。
- 是否引入新的成本。

## 2. 延迟与吞吐

延迟：

```text
单个请求花多长时间完成
```

吞吐：

```text
单位时间完成多少请求
```

二者不总是同时提升。

例如批处理可以提高吞吐，但可能增加单个请求延迟。

延迟还要看分布：

| 指标 | 含义 |
|---|---|
| P50 | 一半请求低于该延迟，代表典型体验 |
| P95 | 95% 请求低于该延迟，代表较慢用户体验 |
| P99 | 尾部延迟，常暴露排队、锁、GC、I/O 抖动 |
| max | 极端值，容易受个别异常影响 |

只看平均值会掩盖尾部问题。例如 99 个请求 10ms，1 个请求 5s，平均值可能看起来还能接受，但 P99 或 max 已经说明用户体验很差。

## 3. CPU 性能分析

常用工具：

```bash
top
htop
pidstat 1
perf stat ./app
perf record ./app
perf report
```

`perf stat` 可查看：

- cycles。
- instructions。
- cache misses。
- branch misses。
- context switches。
- page faults。

常见解释：

| 指标 | 可能含义 |
|---|---|
| cycles 高 | CPU 时间消耗高，但还要结合 instructions |
| instructions 高 | 执行指令多，可能算法或循环问题 |
| IPC 低 | 可能等待内存、分支失败、流水线停顿 |
| cache-misses 高 | 数据局部性差或工作集太大 |
| branch-misses 高 | 分支不可预测 |
| context-switches 高 | 线程过多、阻塞唤醒频繁或锁竞争 |
| page-faults 高 | 按需分页、内存映射、内存压力或访问模式问题 |

如果 CPU 忙，要判断：

- 是用户态计算忙？
- 还是内核态系统调用忙？
- 是否频繁上下文切换？
- 是否缓存 miss 多？
- 是否分支预测失败多？

## 4. 系统调用分析

`strace` 可以跟踪系统调用：

```bash
strace ./app
strace -f ./app
strace -c ./app
strace -e trace=file ./app
strace -e trace=network ./app
```

适合排查：

- 程序卡在哪个系统调用。
- 文件找不到。
- 权限错误。
- 网络连接失败。
- 系统调用次数过多。

`strace` 会改变程序时序，不适合长期挂在高负载生产进程上。需要谨慎使用采样时间和过滤条件：

```bash
strace -f -tt -T -e trace=network -p <pid>
strace -c -p <pid>
```

`-T` 能看到单次系统调用耗时，`-c` 能汇总调用次数和时间占比。次数极高的 `read/write/futex/stat` 往往值得进一步分析。

## 5. 调试器 gdb

常用命令：

```gdb
break main
run
next
step
continue
bt
frame 1
info locals
info registers
x/16gx $rsp
disassemble
```

调试崩溃：

```bash
gdb ./app core
```

查看调用栈：

```gdb
bt
```

多线程：

```gdb
info threads
thread apply all bt
```

调试优化构建时要注意：

- 编译优化会内联函数、消除变量、重排代码。
- `-g` 只是生成调试信息，不等于关闭优化。
- 线上 core 文件需要匹配同一版本的二进制和符号文件。
- 崩溃栈损坏时，可能需要结合寄存器、反汇编和内存查看。

## 6. 内存问题分析

工具：

```bash
valgrind --leak-check=full ./app
gcc -fsanitize=address -g app.c
gcc -fsanitize=undefined -g app.c
```

常见问题：

- 越界访问。
- use-after-free。
- double free。
- 内存泄漏。
- 未初始化读取。
- 栈溢出。

内存增长排查要区分：

| 现象 | 可能原因 |
|---|---|
| VSZ 大 | mmap、大地址空间预留，不一定占物理内存 |
| RSS 持续增长 | 泄漏、缓存增长、碎片、线程增加 |
| RSS 不下降 | 分配器未归还 OS、碎片、仍有活跃对象 |
| page fault 增多 | 工作集变大、mmap 按需加载、内存压力 |

不要只用 `free` 判断进程内存问题，要结合 `pmap`、`/proc/<pid>/smaps`、分配器统计和应用指标。

## 7. I/O 性能分析

命令：

```bash
iostat -x 1
pidstat -d 1
iotop
df -h
du -sh *
```

关注：

- 磁盘利用率。
- await。
- 队列长度。
- 每秒读写次数。
- 吞吐量。
- fsync 频率。

`iostat -x` 常看：

| 字段 | 直觉 |
|---|---|
| `r/s`、`w/s` | 读写 IOPS |
| `rkB/s`、`wkB/s` | 读写吞吐 |
| `await` | I/O 平均等待时间 |
| `aqu-sz` | 平均队列长度 |
| `%util` | 设备忙碌程度 |

高 `%util` 不一定说明设备已经达到极限，现代 SSD 和多队列设备需要结合延迟、队列深度和吞吐看。

优化方向：

- 批量写。
- 减少小随机 I/O。
- 使用缓存。
- 避免频繁 fsync。
- 选择合适文件格式。

## 8. 网络性能分析

命令：

```bash
ss -tanp
sar -n DEV 1
curl -w '@format.txt' -o /dev/null -s URL
tcpdump -i any -nn port 443
```

关注：

- DNS 时间。
- TCP 连接时间。
- TLS 握手时间。
- 首字节时间。
- 传输时间。
- 丢包。
- 重传。
- 连接数。

curl 分阶段计时示例 `format.txt`：

```text
time_namelookup:  %{time_namelookup}
time_connect:     %{time_connect}
time_appconnect:  %{time_appconnect}
time_starttransfer: %{time_starttransfer}
time_total:       %{time_total}
```

运行：

```bash
curl -w '@format.txt' -o /dev/null -s https://example.com
```

这些指标可以把慢请求拆成 DNS、TCP、TLS、服务端处理和传输阶段。

## 9. 可观测性

可观测性常由三部分组成：

- Logs：发生了什么。
- Metrics：系统现在怎样。
- Traces：请求经过哪些服务和阶段。

### 9.1 日志

好的日志应包含：

- 时间。
- 级别。
- 请求 ID。
- 用户或任务 ID。
- 关键参数。
- 错误码。
- 耗时。

不要：

- 记录敏感信息。
- 在热点路径打印大量日志。
- 只写“失败了”而没有上下文。

### 9.2 指标

常见指标：

- QPS。
- 延迟 P50/P95/P99。
- 错误率。
- CPU。
- 内存。
- 磁盘 I/O。
- 网络 I/O。
- 队列长度。
- 线程数。
- GC 时间。

平均值不够，要看分位数。

### 9.3 链路追踪

分布式系统中，一个请求可能经过多个服务。

Trace 记录：

- 请求路径。
- 每个阶段耗时。
- 错误位置。
- 上下游依赖。

三类数据要能通过同一个请求 ID 或 trace ID 串起来。否则日志里看到错误、指标里看到延迟、trace 里看到慢 span，却无法确认是不是同一批请求。

可观测性不是多打日志，而是回答问题：

- 哪个用户或任务受影响？
- 慢在本服务还是下游？
- 是所有请求慢还是某类请求慢？
- 是最近发布、流量变化还是资源瓶颈导致？
- 错误是否可重试，重试是否放大问题？

## 10. 性能优化常见方向

| 瓶颈 | 方向 |
|---|---|
| CPU | 算法优化、减少分支、SIMD、缓存友好 |
| 内存 | 减少分配、复用对象、改善局部性 |
| 锁 | 降低锁粒度、减少共享、分片 |
| I/O | 批量、异步、缓存、顺序访问 |
| 网络 | 连接复用、压缩、减少 round trip |
| 数据库 | 索引、查询优化、连接池、缓存 |

## 11. 常见误区

| 误区 | 正确认识 |
|---|---|
| 先优化看起来慢的代码 | 先 profiling |
| 平均延迟代表用户体验 | 看 P95/P99 |
| CPU 低说明没问题 | 可能在等锁、I/O、网络 |
| 加线程一定更快 | 可能增加调度和锁竞争 |
| 缓存越多越好 | 需要一致性、过期、内存成本 |
| 只看单机指标 | 分布式系统还要看下游、负载均衡和请求分布 |
| 压测 QPS 高就代表线上快 | 线上有真实数据分布、抖动、失败和依赖 |
| 优化后不复测 | 无法确认收益，也无法发现副作用 |
