# 12. 安全与可靠性

<!-- lecture-notes:integrated-v2 -->

## 讲义导读：把程序运行拆成系统链路

这一章讲的是 **12. 安全与可靠性**，属于 **安全与可靠性**。读计算机系统时，不要把它当成名词表，而要把它当成一条从源代码到真实机器行为的链路：代码写下去，编译器怎么翻译，链接器怎么装配，加载器怎么启动，进程怎么占用地址空间，CPU 怎么执行指令，内核怎么处理系统调用，文件和网络怎么把数据送出去。

### 一句话先懂

安全可靠性关注的是程序在错误输入、资源耗尽、并发故障和攻击场景下能否限制损害、暴露证据并恢复服务。

初学时可以先盯住三个问题：第一，程序此刻在哪一层运行；第二，这一层把什么输入转换成了什么输出；第三，如果结果不对，应该用哪个工具观察证据。

### 通俗类比

可靠系统像有消防门、监控、备用电源和演练流程的大楼；平时看起来麻烦，事故发生时决定损失边界。

类比只是入门扶手。真正考试、调试或做项目时，要回到准确术语：地址、字节、指令、寄存器、符号、页表、文件描述符、socket、锁、调度、缓存、协议字段和错误码。能把类比翻译回这些具体对象，才算真的懂。

### 本章学习主线

1. **先定位层级**：这是语言问题、编译链接问题、运行时问题、内核问题、硬件问题，还是网络/存储问题？
2. **再追踪路径**：一次完整过程从哪里开始，经过哪些对象，最后在哪里产生可见结果？
3. **然后看状态**：关键状态包括寄存器、栈、堆、页表、文件偏移、进程状态、锁状态、TCP 状态和缓存状态。
4. **接着找边界**：位宽、对齐、权限、资源限制、并发时序、平台差异和标准未定义行为，都会让直觉失效。
5. **最后做验证**：用小程序、命令、日志、反汇编、抓包、性能计数或 sanitizer 证明解释是对的。

### 本章重点抓手

内存安全、权限边界、输入验证、未定义行为、崩溃恢复、隔离、最小权限、日志审计、备份、幂等和故障注入。

### 最小实践任务

用 ASan/UBSan/TSan、模糊测试、资源限制和故障注入检查小程序，再写清楚失败时的恢复策略。

建议把每次实验记录成固定格式：目标、环境、最小代码、运行命令、观察结果、底层解释、容易误判的点、下一步问题。这样以后遇到段错误、性能抖动、死锁、网络超时或数据丢失时，不会只凭感觉猜。

### 常见误区

- 把没有崩溃当成安全。
- 只做 happy path 测试。
- 权限、输入、日志和恢复策略没有闭环。

### 推荐观察工具

gcc/clang、objdump/readelf、gdb、strace、perf、pmap、ss、tcpdump、/proc、日志和小基准测试。

### 读完本章应该能做到

- 用自己的话解释本章概念，并能指出它处在系统链路的哪一层。
- 画出一个最小路径图，说明数据或控制流从哪里来、到哪里去。
- 用至少一个命令或实验观察到真实现象，而不是只复述结论。
- 说清一个常见故障的表现、可能原因、验证方式和修复方向。

## 1. 为什么系统学习必须包含安全和可靠性

系统程序直接面对：

- 内存。
- 文件。
- 网络。
- 权限。
- 并发。
- 外部输入。
- 崩溃和断电。

因此错误往往不只是功能 bug，还可能导致：

- 数据丢失。
- 服务不可用。
- 权限绕过。
- 远程代码执行。
- 信息泄漏。
- 资源耗尽。

## 2. 内存安全

C/C++ 常见内存安全问题：

- 缓冲区溢出。
- use-after-free。
- double free。
- 越界读。
- 未初始化内存读取。
- 空指针解引用。
- 格式化字符串漏洞。

防御：

- 使用安全语言或安全抽象。
- 边界检查。
- RAII。
- 智能指针。
- sanitizers。
- fuzzing。
- 编译器保护。

安全的关键不是“不要犯错”，而是让错误更难被利用、更早被发现、影响范围更小：

| 层次 | 目标 | 例子 |
|---|---|---|
| 语言/抽象 | 减少内存错误 | Rust、RAII、智能指针、边界检查 |
| 编译期 | 暴露危险模式 | warnings、sanitizers、静态分析 |
| 运行时 | 阻止或发现利用 | ASLR、NX、canary、CFI |
| 架构 | 限制影响范围 | sandbox、最小权限、隔离 |
| 流程 | 持续发现问题 | fuzzing、代码审查、依赖扫描 |

编译选项示例：

```bash
gcc -Wall -Wextra -Werror -g \
    -fsanitize=address,undefined \
    main.c -o main
```

## 3. 权限与最小权限原则

最小权限原则：

```text
程序只应拥有完成任务所需的最小权限。
```

实践：

- 不要用 root 跑普通服务。
- 文件权限最小化。
- 密钥不要写进代码。
- 服务账号权限隔离。
- 网络端口只开放必要范围。
- 容器和沙箱限制能力。

权限设计要避免“启动需要高权限，所以运行也一直高权限”。常见做法是启动阶段完成绑定低端口、读取配置、打开必要文件后，尽快降权到普通用户，并限制后续能力。

敏感信息处理：

- 密钥不要进 Git。
- 不要把 token 打到日志。
- 配置文件权限最小化。
- 密钥轮换要有流程。
- 错误响应不要泄漏内部路径、SQL、栈信息。

## 4. 隔离

隔离减少故障和攻击影响范围。

常见隔离机制：

- 进程地址空间。
- 用户和组权限。
- chroot。
- namespace。
- cgroup。
- seccomp。
- 容器。
- 虚拟机。

隔离不是绝对安全，但能降低风险。

隔离机制有边界：

| 机制 | 主要限制 |
|---|---|
| chroot | 只改变根目录视图，不是完整安全边界 |
| namespace | 隔离某类资源视图，仍依赖内核安全 |
| cgroup | 限制资源，不直接提供权限隔离 |
| seccomp | 限制系统调用面，但策略需要维护 |
| 容器 | 共享宿主机内核，强隔离弱于虚拟机 |
| 虚拟机 | 隔离强，但资源成本更高 |

安全设计要假设隔离可能被配置错误或被绕过，因此仍然需要最小权限、输入验证和审计。

## 5. 输入验证

所有外部输入都不可信：

- HTTP 请求。
- 文件内容。
- 命令行参数。
- 环境变量。
- 网络包。
- 数据库内容。
- 用户上传文件。

需要验证：

- 类型。
- 长度。
- 范围。
- 编码。
- 格式。
- 权限。

不要把输入直接拼进命令或 SQL。

输入验证分两类：

| 类型 | 说明 |
|---|---|
| 语法验证 | 长度、字符集、格式、类型 |
| 语义验证 | 这个用户是否有权限、状态是否允许、业务约束是否满足 |

只做语法验证不够。例如请求里的 `user_id` 是合法整数，不代表当前用户有权限操作这个 `user_id`。

常见危险点：

- SQL 拼接。
- shell 命令拼接。
- 路径拼接导致目录穿越。
- 反序列化不可信数据。
- 正则表达式灾难性回溯导致 CPU DoS。
- 上传文件类型和路径处理不严。

## 6. 资源耗尽

攻击或 bug 可能耗尽：

- CPU。
- 内存。
- 磁盘。
- 文件描述符。
- 线程。
- 连接数。
- 队列。

防御：

- 限流。
- 超时。
- 最大请求体。
- 最大连接数。
- 队列长度限制。
- ulimit。
- cgroup。
- 熔断和降级。

资源限制要尽量靠近入口：

- HTTP 请求体最大大小。
- 上传文件大小。
- 每用户/每 IP 频率限制。
- 连接数限制。
- 队列长度限制。
- 单任务内存和 CPU 限制。
- 日志速率限制。

没有上限的队列、缓存、线程创建和重试都是可靠性风险。

## 7. 超时与重试

没有超时的系统是不可靠的。

所有外部调用都应该有：

- 连接超时。
- 读写超时。
- 总超时。
- 取消机制。

重试要谨慎：

- 只重试可重试错误。
- 使用指数退避。
- 加随机抖动。
- 限制最大次数。
- 确保幂等性。

错误重试可能放大故障。

重试前要判断操作是否幂等：

| 操作 | 重试风险 |
|---|---|
| 查询 | 通常可重试，但要注意放大读压力 |
| 创建订单 | 可能重复创建，需要幂等键 |
| 扣款 | 高风险，必须有事务、幂等和对账 |
| 发送通知 | 可能重复发送，需要去重或可接受重复 |

重试还应结合 deadline。一个请求总预算 1s，就不能每层都重试 3 次并各自等待 1s，否则尾部延迟会失控。

## 8. 崩溃一致性

写文件或数据库时，可能发生：

- 程序崩溃。
- 机器断电。
- 磁盘错误。
- 部分写入。

可靠写入需要考虑：

- 临时文件。
- 原子 rename。
- fsync 文件。
- fsync 目录。
- 日志或 WAL。

简单模式：

```text
写 temp 文件
fsync temp
rename temp -> target
fsync 目录
```

具体需求要根据文件系统和平台验证。

可靠写入还要考虑：

- partial write。
- `fsync` 返回错误。
- `close` 返回错误。
- 临时文件和目标文件是否在同一文件系统。
- 目录项是否持久化。
- 多进程并发写同一文件。

数据库、消息队列和文件系统的可靠性通常依赖 WAL、校验和、版本号、原子提交和恢复扫描。应用自己写“持久化小文件”时，也应该借鉴这些思想。

## 9. 可恢复设计

可靠系统应该能从错误中恢复：

- 进程崩溃后重启。
- 任务可重放。
- 状态可恢复。
- 数据有备份。
- 操作有幂等性。
- 日志可用于排查。

不要只设计“正常路径”，要设计失败路径。

恢复设计常见问题：

| 问题 | 设计方向 |
|---|---|
| 任务执行一半崩溃 | 任务状态机、幂等执行、可重放日志 |
| 消息处理后崩溃但未 ack | 至少一次投递 + 去重 |
| 依赖长时间不可用 | 熔断、降级、排队上限 |
| 配置错误发布 | 快速回滚、灰度、配置校验 |
| 数据损坏 | 校验和、备份、恢复演练 |

“有备份”不等于“可恢复”。必须定期演练恢复流程，确认恢复时间和恢复点是否满足要求。

## 10. 安全编译和运行时保护

常见保护：

- ASLR。
- NX。
- Stack canary。
- PIE。
- RELRO。
- Fortify Source。
- seccomp。

检查二进制可使用 `checksec` 等工具。

示例编译选项：

```bash
gcc -O2 -g -Wall -Wextra \
    -fstack-protector-strong -D_FORTIFY_SOURCE=2 \
    -fPIE -pie \
    -Wl,-z,relro,-z,now \
    main.c -o app
```

不同编译器、平台和发行版默认保护不同。安全选项可能影响兼容性和性能，应在目标环境验证。

## 11. Fuzzing

Fuzzing 通过大量随机或变异输入寻找崩溃和漏洞。

适合：

- 解析器。
- 协议实现。
- 文件格式处理。
- 编译器前端。
- 压缩/解压库。

fuzzing 的关键是构造一个稳定、快速、可重复的测试入口：

- 输入来自 fuzzer 提供的字节数组。
- 不依赖网络、时间、随机外部状态。
- 崩溃能被稳定复现。
- 每次执行尽量快。
- 配合 sanitizer 放大错误信号。

fuzzing 找到的是“有触发样本的问题”，不是所有问题。认证、授权、业务逻辑漏洞仍然需要设计审查和测试。

常见工具：

- libFuzzer。
- AFL++。
- honggfuzz。

## 12. 可靠性清单

- 所有外部输入是否验证。
- 所有外部调用是否有超时。
- 是否限制请求大小。
- 是否限制并发和队列长度。
- 是否处理部分读写。
- 是否处理 `EINTR`。
- 文件写入是否考虑崩溃一致性。
- 日志是否足以定位问题。
- 是否避免 root 运行。
- 密钥是否安全保存。
- 内存错误是否用 sanitizer 检查。
- 并发代码是否有线程安全策略。
- 是否有依赖版本和已知漏洞扫描。
- 是否有备份恢复演练。
- 是否限制日志中的敏感信息。
- 是否定义重试、超时、熔断和幂等策略。
- 是否对上传、解析、压缩、图片处理等入口做 fuzz 或边界测试。

## 2026 计算机系统资料与实验核对补充

这一组笔记建议按“教材主线 + 标准文档 + 本机实验”三层学习，不要只看二手总结。

- **教材主线**：CS:APP 适合建立程序员视角，OSTEP 适合按虚拟化、并发、持久化理解操作系统。
- **接口核对**：Linux man-pages 用来查系统调用、库函数和命令行为；Linux Kernel docs 用来查内核机制；POSIX.1-2024 用来核对可移植接口；ELF gABI 用来核对目标文件、符号、重定位和动态链接。
- **工具链核对**：GCC/LLVM/Clang 文档用于确认编译选项、优化、调试信息和 sanitizer 行为，不要只凭旧教程记命令。
- **网络核对**：HTTP 语义看 RFC 9110，HTTP 缓存看 RFC 9111，TCP 看 RFC 9293，QUIC 看 RFC 9000；抓包结果要和协议字段对应起来。
- **实验要求**：优先用 CS:APP 建立程序员视角，用 OSTEP 建立操作系统主线，再用 Linux man-pages、Linux Kernel docs、POSIX.1-2024、ELF gABI 和 RFC 原文核对接口细节。 每个结论最好配一个能复现的最小程序或命令输出。

通俗地说，教材负责“搭骨架”，标准负责“定规则”，实验负责“验真假”。系统知识最怕只会背概念；能把源码、命令输出、内核接口和协议原文对上，才真正能用于排错和优化。
