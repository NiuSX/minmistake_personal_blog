## 讲义导读：从像素到可用视觉系统

这一章讲的是 **01-CV 总览与学习路线**。阅读时不要只记 OpenCV 函数、模型名字或论文缩写，而要把它放进完整视觉链路：图像如何采集，像素和颜色空间是否正确，光照和噪声如何影响结果，相机几何是否成立，模型输入和标注是否一致，评价指标是否对应漏检误检成本，部署时前后处理、延迟和现场漂移是否可控。CV 的目标不是让一张样例图看起来成功，而是在真实图像分布中稳定输出可用信息。

### 一句话先懂

CV 学习路线要从图像本质和 OpenCV 开始，再进入传统算法、几何视觉、深度模型、任务评估和工程部署。

### 通俗类比

可以把 CV 系统想成流水线上的质检员：相机和光源负责把产品拍清楚，预处理像擦干净镜头和统一灯光，传统特征或神经网络负责看重点，检测和分割负责指出位置，后处理负责合并和过滤结果，评估指标负责统计漏检、误检和定位偏差，部署监控负责发现现场条件变化。

类比只是第一层直觉。回到技术上，要把每一步写成明确对象：输入图像的尺寸、通道、数据类型、坐标系、标注格式、模型输出、阈值、指标和运行环境。视觉问题很容易被表面效果迷惑，必须用可复现样本和量化指标来判断。

### 本章学习主线

1. **图像来源**：确认采集设备、光照、分辨率、帧率、压缩、畸变和场景覆盖。
2. **表示方式**：弄清像素数组、通道顺序、颜色空间、坐标系、数据类型和归一化。
3. **算法假设**：说明方法依赖什么条件，例如边缘明显、光照稳定、尺度不变、训练分布一致或标注可靠。
4. **输出解释**：区分类别、框、mask、关键点、文字、轨迹、三维点和置信度的含义。
5. **评估诊断**：用 IoU、mAP、Dice、OCR 准确率、MOT 指标、延迟和错误样本定位问题。
6. **工程闭环**：把训练、推理、后处理、可视化、日志、监控和现场迭代连起来。

### 概念怎么学才不容易忘

遇到视觉算法时，建议按 白话作用 -> 输入输出 -> 图像假设 -> 参数含义 -> 可视化结果 -> 失败样本 六步学习。比如 Canny 不只是一个函数，要看梯度、阈值和噪声；YOLO 不只是一个模型，要看输入尺寸、置信度、NMS、mAP 和小目标漏检；相机标定不只是拿到内参，要看重投影误差和畸变是否影响实际测量。

## 1. 什么是计算机视觉

计算机视觉 Computer Vision，简称 CV，是让计算机从图像、视频、深度图、点云、多光谱图像等视觉数据中理解世界的技术。它的目标不是简单“看见图片”，而是从视觉数据中提取结构化信息，用于识别、检测、定位、测量、跟踪、分割、重建、理解和决策。

常见任务：

- 图像分类：判断图里是什么。
- 目标检测：找出目标位置和类别。
- 图像分割：把图像按像素划分区域。
- 实例分割：区分同类目标的不同实例。
- 关键点检测：检测人体、手、脸、物体关键点。
- OCR：检测和识别图像中的文字。
- 目标跟踪：在视频中持续追踪目标。
- 行为识别：识别视频中的动作或事件。
- 三维重建：从图像恢复三维结构。
- 缺陷检测：发现工业产品表面异常。
- 视觉测量：测量尺寸、角度、位置、面积。
- 机器人视觉：识别物体、定位、抓取、导航。

## 2. CV 学习的知识地图

完整 CV 学习可分为七层：

### 2.1 图像基础层

需要理解：

- 像素、分辨率、通道。
- RGB、BGR、HSV、灰度图。
- 图像坐标系。
- 图像直方图。
- 卷积和滤波。
- 频域和空域。
- 噪声和模糊。

### 2.2 OpenCV 工具层

需要掌握：

- 图像读取、显示、保存。
- 色彩空间转换。
- 缩放、裁剪、旋转、仿射变换。
- 阈值、边缘、轮廓、形态学。
- 模板匹配、霍夫变换。
- 相机标定、特征匹配。

### 2.3 传统算法层

需要掌握：

- 滤波降噪。
- 阈值分割。
- 边缘检测。
- 轮廓分析。
- 连通域分析。
- 形态学处理。
- 角点检测。
- 特征描述子。
- 图像配准。

传统算法优势是可解释、速度快、数据需求低；缺点是对光照、背景、姿态、噪声敏感。

### 2.4 机器学习层

需要理解：

- 特征工程。
- SVM、KNN、随机森林。
- 聚类。
- PCA 降维。
- HOG + SVM。
- Haar + AdaBoost。

这部分是传统 CV 和深度学习之间的重要过渡。

### 2.5 深度学习视觉层

需要掌握：

- CNN。
- 卷积、池化、归一化、激活函数。
- ResNet、MobileNet、EfficientNet。
- Vision Transformer。
- 迁移学习。
- 数据增强。
- 损失函数。
- 优化器。

### 2.6 任务模型层

需要掌握：

- 分类模型。
- 检测模型：YOLO、Faster R-CNN、SSD、RetinaNet。
- 分割模型：FCN、U-Net、DeepLab、Mask R-CNN、SAM。
- 关键点模型：OpenPose、HRNet、MediaPipe。
- OCR 模型：CRNN、DBNet、PaddleOCR。
- 跟踪模型：SORT、DeepSORT、ByteTrack。

### 2.7 工程落地层

需要掌握：

- 数据采集。
- 标注规范。
- 模型训练。
- 评估指标。
- 推理加速。
- ONNX、TensorRT、OpenVINO。
- 边缘部署。
- 摄像头标定。
- 线上监控。
- 误报漏报分析。

## 3. CV 常见应用场景

### 3.1 工业视觉

任务：

- 缺陷检测。
- 尺寸测量。
- OCR 读码。
- 定位引导。
- 装配检测。
- 外观检测。

特点：

- 场景可控。
- 光源和相机很重要。
- 精度和稳定性要求高。
- 误报漏报成本明确。
- 传统算法和深度学习经常混合使用。

### 3.2 自动驾驶和机器人

任务：

- 车道线检测。
- 车辆行人检测。
- 语义分割。
- 深度估计。
- SLAM。
- 目标跟踪。
- 视觉导航。

特点：

- 环境复杂。
- 实时性要求高。
- 需要多传感器融合。
- 安全性要求极高。

### 3.3 安防和视频分析

任务：

- 人脸识别。
- 行人检测。
- 越界检测。
- 人群计数。
- 行为识别。
- 目标跟踪。

特点：

- 摄像头多。
- 光照变化大。
- 遮挡多。
- 隐私和合规重要。

### 3.4 医学影像

任务：

- 病灶检测。
- 器官分割。
- 影像分类。
- 三维重建。

特点：

- 数据标注昂贵。
- 误判影响大。
- 可解释性重要。
- 需要医生参与评估。

## 4. 学习路线

### 阶段 1：图像和 OpenCV

目标：

- 能读写图像。
- 能进行预处理。
- 能用阈值、边缘、轮廓解决简单问题。

练习：

- 提取图片中的红色区域。
- 统计目标面积。
- 检测圆形零件。
- 识别二维码区域。

### 阶段 2：传统算法

目标：

- 理解滤波、边缘、轮廓、特征。
- 能用传统方法做定位和测量。

练习：

- 用 Canny 提取边缘。
- 用霍夫变换检测直线和圆。
- 用轮廓计算物体中心和角度。
- 用 ORB 做图像匹配。

### 阶段 3：深度学习

目标：

- 能训练分类和检测模型。
- 理解数据增强、损失函数、指标。

练习：

- 训练猫狗分类。
- 训练 YOLO 检测自定义目标。
- 训练 U-Net 做分割。

### 阶段 4：工程部署

目标：

- 模型能在真实环境稳定运行。
- 能分析误报漏报。
- 能优化推理速度。

练习：

- 摄像头实时检测。
- ONNX 导出。
- TensorRT 加速。
- 写一个可视化检测工具。

## 5. CV 项目基本流程

```text
需求分析 -> 图像采集 -> 数据标注 -> 数据清洗 -> 基线算法 -> 模型训练 -> 指标评估 -> 误差分析 -> 部署优化 -> 现场迭代
```

关键原则：

- 先明确业务指标，再选择算法。
- 数据质量优先于模型复杂度。
- 先做简单基线，再上复杂模型。
- 评估指标要和实际场景一致。
- 现场光照、相机、安装位置会决定很多算法成败。

## 2026 CV 资料与工程核对补充

#### 典型场景拆解

在真实场景中，$topic 通常会经历“需求出现、方案选择、实现落地、问题暴露、持续优化”几个阶段。需求出现时，要先判断这个需求属于基础能力、性能优化、体验改进、可靠性建设还是长期架构演进。不同类型的需求对方案的评价标准不同：基础能力看正确性，性能优化看指标，体验改进看路径是否顺滑，可靠性建设看故障时能否降级和恢复，架构演进看未来变化是否容易吸收。

方案选择阶段，最容易犯的错误是直接套用熟悉工具。更稳妥的方式是列出约束：数据规模、时延要求、资源预算、团队熟悉度、运维能力、安全要求、可测试性和长期维护成本。只有把约束列清楚，才能解释为什么选择当前方案。否则方案看似高级，实际可能只是增加了复杂度。

实现落地阶段，要把 $a 和 $b 拆成可验证的小步骤。每一步都应该有明确的输入、输出和检查方式。对学习笔记而言，这意味着不能只有大段概念，还应该补充流程图式的文字描述、伪代码、命令示例、参数解释、错误现象和排查路径。这样以后复习时，笔记不仅能帮助理解，也能直接指导实践。

问题暴露阶段，要优先区分“理解错误、实现错误、环境错误、数据错误、依赖错误、边界条件错误”。很多复杂问题之所以难排，是因为一开始就把问题归因到错误层级。例如把配置问题当成算法问题，把权限问题当成代码问题，把数据分布变化当成模型失效，把硬件噪声当成软件逻辑错误。好的排查顺序应该从可观测事实开始，而不是从猜测开始。

持续优化阶段，不应只追求把当前问题压下去，还要沉淀成规则。比如记录触发条件、影响范围、定位方法、最终修复、预防措施和可监控指标。这样下一次出现类似问题时，团队可以复用经验，而不是重新从零排查。

#### 常见误区与纠偏
