## 一、为什么必须理解内存管理

内存管理是程序运行的底层基础。一个程序从源代码变成可执行文件，再被操作系统加载到进程空间中运行，过程中所有数据、指令、对象、函数调用、线程执行、系统调用、缓存、文件映射、网络缓冲都离不开内存。

很多上层问题表面看是业务 bug，根因却是内存问题。例如：

- 应用运行一段时间后越来越卡，可能是内存泄漏、缓存无界增长或频繁 GC。
- 服务偶发崩溃，可能是空指针、野指针、越界写、双重释放或栈溢出。
- 接口延迟抖动，可能是堆分配太频繁、内存局部性差、GC 暂停或 page fault。
- 大文件处理失败，可能是一次性加载导致内存峰值过高。
- 多线程数据错乱，可能与对象逃逸、共享可变状态、内存可见性有关。

现代语言和框架屏蔽了很多内存细节，但屏蔽不等于不存在。越靠近性能、稳定性、并发、移动端、嵌入式、游戏、数据库、虚拟机、服务端高并发，内存管理越重要。

理解内存管理至少能带来四类能力：

1. **解释程序如何运行**：变量在哪里，对象在哪里，函数调用如何保存上下文，线程如何拥有自己的栈。
2. **定位稳定性问题**：能区分泄漏、溢出、悬垂指针、越界访问、OOM、GC 抖动。
3. **做性能优化**：减少分配次数，改善局部性，降低内存峰值，避免不必要复制。
4. **设计更可靠的架构**：明确所有权、生命周期、缓存策略、资源释放时机。

内存管理不是某一种语言的特性，而是从硬件、操作系统到运行时共同组成的一套系统。

## 二、内存的分层模型

讨论内存时，不能只把内存理解成“RAM 条子”。计算机系统中存在一个层次化存储结构：

| 层级 | 典型容量 | 典型速度 | 作用 |
| --- | --- | --- | --- |
| CPU 寄存器 | 极小 | 极快 | 保存当前执行指令需要的操作数、地址、状态 |
| L1 Cache | 很小 | 很快 | CPU 最近访问的数据和指令 |
| L2/L3 Cache | 较小 | 快 | 多核心共享或半共享缓存 |
| 主内存 RAM | GB 级 | 中等 | 程序主要运行空间 |
| SSD/HDD | TB 级 | 慢 | 持久化存储 |
| 网络/远程存储 | 更大 | 更慢 | 分布式数据访问 |

程序员平时说的“内存”多数指主内存，但性能问题经常由缓存层级决定。CPU 访问寄存器和访问主内存的成本差距巨大。即使两个算法时间复杂度一样，内存访问模式不同，实际性能也可能相差很多。

### 1. 局部性原理

现代计算机依赖局部性原理提高性能：

- **时间局部性**：刚访问过的数据，很可能马上再次访问。
- **空间局部性**：访问某个地址后，很可能访问它附近的地址。

数组遍历通常比链表遍历快，原因不只是数组下标访问简单，更重要的是数组连续存储，空间局部性好，CPU cache 命中率高。链表节点分散在堆上，访问下一个节点可能跳到完全不同的内存页，导致 cache miss。

### 2. Cache Line

CPU cache 通常不是按单个字节加载，而是按 cache line 加载，常见大小为 64 字节。当访问数组中的一个元素时，它附近的一段数据会一起进入缓存。顺序遍历数组就能利用这一点。

这也解释了为什么结构体字段排列、数组结构设计会影响性能。例如：

- `Array of Structs` 适合每次都访问对象的大多数字段。
- `Struct of Arrays` 适合批量处理某一个字段，常见于游戏引擎、图形计算和数据分析。

### 3. 内存带宽与延迟

内存性能可以从两个维度理解：

- **延迟**：一次读取多久能返回。
- **带宽**：单位时间能传输多少数据。

随机小块访问主要受延迟影响；连续大块访问主要受带宽影响。高性能程序通常会尽量把随机访问改成连续访问，把小对象分散分配改成批量分配。

## 三、程序运行时的内存布局

一个可执行程序被操作系统加载后，会形成进程地址空间。典型进程内存区域包括：

```text
高地址
┌──────────────────────────┐
│ 栈 Stack                  │
│ 函数调用、局部变量         │
├──────────────────────────┤
│ 内存映射区 mmap           │
│ 动态库、文件映射、匿名映射 │
├──────────────────────────┤
│ 堆 Heap                   │
│ 动态分配对象              │
├──────────────────────────┤
│ BSS 段                    │
│ 未初始化全局/静态变量      │
├──────────────────────────┤
│ Data 段                   │
│ 已初始化全局/静态变量      │
├──────────────────────────┤
│ Text 段                   │
│ 程序机器指令、只读常量      │
└──────────────────────────┘
低地址
```

不同操作系统、CPU 架构、编译器和安全机制会导致细节差异，但核心思想相似。

### 1. Text 段

Text 段保存程序机器指令，通常是只读的，防止程序运行时修改自身指令。字符串字面量、只读常量也可能放在只读区域。

把代码段设置为只读，有助于安全：

- 防止普通 bug 破坏指令。
- 增加攻击者注入代码的难度。
- 多个进程运行同一个程序时，代码页可共享。

### 2. Data 段

Data 段保存已初始化的全局变量和静态变量，例如：

```c
int global_count = 10;
static int cached_size = 128;
```

这些变量的生命周期贯穿整个进程，从程序启动存在到程序退出。

### 3. BSS 段

BSS 段保存未显式初始化或初始化为 0 的全局/静态变量，例如：

```c
int global_zero;
static char buffer[4096];
```

BSS 段在可执行文件中不需要真实保存大量 0，只需要记录大小，加载时由操作系统初始化为 0。因此一个大数组放在 BSS 中，不一定显著增大可执行文件体积，但会增加运行时内存占用。

### 4. Heap 堆

堆用于动态分配内存。C 中的 `malloc/free`，C++ 中的 `new/delete`，Java/Kotlin 中大部分对象，JavaScript/Python 中大部分对象，都可以理解为在堆上管理。

堆的特点：

- 生命周期不由函数调用自动决定。
- 对象可以跨函数、跨线程、跨模块使用。
- 管理复杂，容易泄漏或碎片化。
- 通常由运行时或内存分配器维护。

### 5. Stack 栈

栈用于函数调用、局部变量、返回地址、保存寄存器等。每个线程通常有自己独立的栈。

栈的特点：

- 分配和释放非常快，只需要移动栈指针。
- 生命周期严格服从函数调用关系。
- 空间有限，递归太深或局部数组太大可能导致栈溢出。

### 6. mmap 区域

内存映射区用于：

- 加载动态链接库。
- 映射文件到内存。
- 分配大块匿名内存。
- 线程栈。
- 共享内存。

很多现代内存分配器会对小对象使用堆管理，对大对象直接使用 `mmap` 申请独立区域。

## 四、栈内存：函数调用的基础设施

栈是理解函数调用最关键的结构。函数调用不是抽象魔法，它需要保存返回地址、参数、局部变量和调用前的寄存器状态。

### 1. 栈帧

每次函数调用通常会创建一个栈帧。栈帧中可能包含：

- 函数参数。
- 返回地址。
- 上一个栈帧指针。
- 局部变量。
- 临时值。
- 保存的寄存器。

例如：

```c
int add(int a, int b) {
    int sum = a + b;
    return sum;
}

int main() {
    int result = add(1, 2);
    return result;
}
```

调用 `add` 时，`main` 的执行上下文需要暂时保存，`add` 的参数和局部变量被放入新的栈帧。`add` 返回后，其栈帧被销毁，控制流回到 `main`。

### 2. 栈分配为什么快

栈分配本质上是移动栈指针：

- 进入函数：栈指针移动，预留当前函数需要的空间。
- 退出函数：栈指针恢复，整个栈帧失效。

它不需要复杂搜索、合并空闲块，也不会产生普通意义上的堆碎片。只要函数调用关系符合后进先出，栈就是极高效的内存管理方式。

### 3. 栈变量的生命周期

局部变量通常在函数返回后失效：

```c
int* wrong() {
    int x = 42;
    return &x;
}
```

这里返回了局部变量地址。函数返回后 `x` 所在栈帧已经失效，指针指向的内存可能被后续函数调用覆盖。这类问题称为悬垂指针。

### 4. 栈溢出

栈空间有限。常见栈溢出原因：

- 无限递归。
- 递归深度过大。
- 在栈上创建超大数组或对象。
- 每个线程栈过大导致线程数量受限。

示例：

```c
void crash() {
    int huge[10000000];
    huge[0] = 1;
}
```

这类大对象应考虑放到堆上，或使用流式处理、分块处理。

### 5. 递归与栈

递归天然消耗栈。每一层递归都有自己的参数和局部变量。如果递归深度无法可靠限制，就要考虑改成迭代，或者显式维护一个堆上的栈结构。

尾递归在某些语言或编译器中可以优化为循环，但不能默认依赖。尤其在 JVM、普通 Java/Kotlin 环境中，尾递归优化不是普遍自动保证的。Kotlin 需要满足条件并显式使用 `tailrec`。

## 五、堆内存：动态对象的主战场

堆解决了栈无法解决的问题：对象生命周期不一定服从函数调用关系。

例如，一个函数创建对象并返回给调用方：

```cpp
std::unique_ptr<User> createUser() {
    return std::make_unique<User>();
}
```

对象创建于函数内部，但函数返回后对象仍然需要存在。这种场景适合堆。

### 1. 堆的核心问题

堆管理比栈复杂得多，主要因为它要回答三个问题：

1. **哪里有足够大的空闲空间？**
2. **释放后的空间如何复用？**
3. **如何判断一块内存是否还能被访问？**

手动内存管理语言把第三个问题交给程序员；垃圾回收语言把它交给运行时；引用计数语言使用引用数量近似判断；Rust 使用所有权和借用规则在编译期约束。

### 2. 堆对象生命周期

堆对象生命周期通常经历：

1. 分配空间。
2. 初始化对象。
3. 被引用和使用。
4. 不再需要。
5. 释放空间或等待回收。

问题最多发生在第 4 到第 5 步：

- 该释放没释放，产生内存泄漏。
- 还在使用就释放，产生 use-after-free。
- 释放两次，产生 double free。
- 持有过期地址，产生悬垂指针。
- 保留无用引用，导致 GC 无法回收。

### 3. 小对象与大对象

小对象数量多，分配频繁，管理重点是速度和碎片控制。大对象占用明显，管理重点是避免峰值过高、避免复制、及时释放。

常见优化策略：

- 小对象池化。
- 批量分配。
- 避免循环中创建临时对象。
- 大对象使用流式处理。
- 使用内存映射处理大文件。
- 对缓存设置容量上限和淘汰策略。

### 4. 对象逃逸

对象是否需要放到堆上，关键看它是否逃逸出当前作用域。比如：

```java
void foo() {
    Point p = new Point(1, 2);
    use(p);
}
```

如果 `p` 没有逃出方法，JIT 可能通过逃逸分析进行优化，例如标量替换、栈上分配或消除分配。虽然 Java 语义上对象在堆上，但运行时可以在不改变语义的前提下优化。

## 六、虚拟内存：现代操作系统的核心抽象

现代程序看到的地址通常不是物理内存地址，而是虚拟地址。操作系统和 CPU 的内存管理单元 MMU 会把虚拟地址转换为物理地址。

虚拟内存解决了几个关键问题：

- **隔离**：每个进程拥有自己的地址空间，不能随意访问其他进程内存。
- **简化编程模型**：程序以为自己拥有连续地址空间。
- **安全**：页面可以设置读、写、执行权限。
- **共享**：多个进程可以映射同一个物理页，例如共享库。
- **懒加载**：需要时才真正分配物理页。
- **换页**：物理内存不足时可把页面换出到磁盘。

### 1. 虚拟地址空间

在 64 位系统中，虚拟地址空间非常大，但这不代表真实占用同等物理内存。程序可以保留一段虚拟地址，但只有实际访问时，操作系统才可能分配物理页。

这解释了为什么有些程序的虚拟内存看起来很大，但常驻内存不一定大。

常见指标：

- **Virtual Size / VIRT**：进程虚拟地址空间大小。
- **RSS / Resident Set Size**：实际驻留在物理内存中的页面大小。
- **PSS / Proportional Set Size**：共享页面按比例分摊后的内存大小。

分析内存占用时不能只看虚拟内存，要结合 RSS、PSS、堆大小、匿名映射、文件映射等指标。

### 2. 地址空间隔离

每个进程拥有独立页表。进程 A 中的地址 `0x1000` 和进程 B 中的地址 `0x1000` 可以映射到完全不同的物理内存。这样一个进程崩溃通常不会直接破坏另一个进程。

内核空间通常映射在进程地址空间的高地址区域，但用户态代码不能随意访问。进入内核需要系统调用、中断或异常。

### 3. Copy-on-Write

写时复制是一种重要优化。比如 Unix 系统中 `fork()` 创建子进程时，并不会立即复制父进程所有物理内存，而是让父子进程共享页面，并把页面标记为只读。当某一方写入时，才复制对应页面。

这样 `fork()` 的成本大幅降低，尤其是子进程很快 `exec()` 加载新程序时，根本不需要复制大量内存。

### 4. 内存映射文件

`mmap` 可以把文件映射到内存地址空间，程序像访问内存一样访问文件内容。操作系统负责按页加载、回写和缓存。

适用场景：

- 大文件随机访问。
- 多进程共享数据。
- 数据库存储引擎。
- 零拷贝或少拷贝 I/O。

但 `mmap` 不等于总是更快。它可能带来 page fault、地址空间占用、回写时机复杂、错误处理复杂等问题。是否使用要看访问模式和数据规模。

## 七、分页、页表、TLB 与缺页异常

虚拟内存通常以页为单位管理。常见页大小为 4KB，也有大页如 2MB、1GB。

### 1. 页与页框

- **虚拟页**：虚拟地址空间中的固定大小块。
- **物理页框**：物理内存中的固定大小块。
- **页表**：记录虚拟页到物理页框的映射关系。

虚拟地址可以拆成：

```text
虚拟页号 + 页内偏移
```

地址转换时，MMU 根据虚拟页号查页表得到物理页框，再加上页内偏移得到物理地址。

### 2. 多级页表

如果每个进程维护一张线性大页表，空间开销会非常大。多级页表通过分层结构只为实际使用的地址范围分配页表页。

64 位系统中虚拟地址空间巨大，多级页表尤其重要。

### 3. TLB

页表查询本身也要访问内存，会很慢。CPU 使用 TLB 缓存最近的虚拟页到物理页映射。TLB 命中时地址转换很快；TLB 未命中时需要查页表。

影响 TLB 的因素：

- 工作集大小。
- 内存访问是否连续。
- 页面大小。
- 上下文切换频率。

大页可以减少 TLB 压力，因为同样 TLB 项能覆盖更大内存范围。但大页也可能增加内部碎片和分配难度。

### 4. 缺页异常

当进程访问的虚拟页没有有效物理映射时，会发生缺页异常。缺页不一定是错误，可能是正常机制：

- 第一次访问懒分配页面。
- 从磁盘加载被换出的页面。
- 加载文件映射的一页。
- Copy-on-Write 触发复制。

但如果访问非法地址，缺页异常会被操作系统转化为错误，例如 Linux 的 segmentation fault。

### 5. Page Fault 的性能影响

轻微缺页只需要建立映射，成本相对可控；严重缺页需要磁盘 I/O，成本极高。程序内存超出物理内存导致频繁换页时，会进入 thrashing 状态，CPU 大量时间耗在换页上，应用响应极慢。

避免严重缺页的策略：

- 控制工作集大小。
- 分块处理数据。
- 避免同时加载过多大对象。
- 使用合理缓存上限。
- 监控 RSS、swap 使用、major page faults。

## 八、内存分配器原理

内存分配器负责在堆上分配和释放内存。不同语言和运行时有不同实现，但许多思想相通。

### 1. 分配器的基本职责

分配器需要完成：

- 管理空闲内存块。
- 查找满足大小要求的块。
- 必要时向操作系统申请更多内存。
- 释放后回收并复用空间。
- 合并相邻空闲块。
- 控制碎片。
- 在多线程下保证安全和性能。

### 2. 常见分配策略

**First Fit**：找到第一个足够大的空闲块。实现简单，速度较快，但可能在前部产生碎片。

**Best Fit**：找到最接近需求大小的空闲块。理论上减少浪费，但搜索成本高，也可能产生很多小碎片。

**Worst Fit**：使用最大的空闲块，试图保留中等大小块。实际并不总是好。

**Segregated Free Lists**：按大小分类维护多个空闲链表，小对象从对应大小类中快速分配。现代分配器常用类似思想。

**Buddy System**：把内存按 2 的幂拆分和合并，便于快速找到伙伴块并合并，常见于操作系统内核页分配。

### 3. 分配元数据

分配器通常需要为每个块维护元数据，例如：

- 块大小。
- 是否空闲。
- 前后块指针。
- 校验信息。

元数据可能放在用户内存块前面，也可能集中管理。元数据损坏会导致严重问题。例如 C 程序越界写可能破坏分配器元数据，最终在 `free()` 时崩溃。

### 4. 多线程分配

多线程程序中，如果所有线程都竞争同一个全局堆锁，分配性能会很差。现代分配器通常会使用：

- 线程本地缓存。
- 多个 arena。
- 按 CPU 或线程分片。
- 无锁或低锁数据结构。
- 小对象批量申请。

这能显著降低锁竞争，但也可能增加总内存占用，因为每个线程/arena 都可能保留自己的空闲块。

### 5. 对象池

对象池是一种应用层分配优化。适合：

- 对象创建销毁频繁。
- 对象大小相似。
- 初始化成本高。
- 生命周期短但数量大。

但对象池不是万能优化。风险包括：

- 池子无界增长导致泄漏式占用。
- 对象复用前状态未清理导致脏数据。
- 增加代码复杂度。
- 与现代 GC 或分配器优化冲突。

在 Java/Kotlin 等托管语言中，小短命对象分配通常很快，过早对象池化反而可能让对象活得更久，增加老年代压力。

## 九、内存碎片问题

内存碎片是堆管理中的经典问题。它分为内部碎片和外部碎片。

### 1. 内部碎片

内部碎片指分配给程序的块大于实际需要，块内部未使用的部分浪费。

例如程序申请 33 字节，分配器按 48 字节大小类分配，多出的 15 字节就是内部碎片。

内部碎片常见于：

- 对齐要求。
- 固定大小类。
- Buddy System 按 2 的幂分配。

### 2. 外部碎片

外部碎片指总空闲内存足够，但分散成很多小块，无法满足大块连续分配。

例如空闲块总共 100MB，但最大连续块只有 2MB，那么申请 20MB 仍然失败。

外部碎片常见于：

- 长短生命周期对象混合分配。
- 大小差异很大的对象交错分配。
- 释放顺序不可预测。

### 3. 碎片治理

常见方法：

- 按大小分类分配。
- 相同生命周期对象一起分配。
- 使用 arena/region 分配，整体释放。
- 对大对象单独管理。
- GC 中使用复制或整理算法移动对象。
- 减少长期对象与短期对象混杂。

托管运行时可以移动对象，因此有机会整理堆；C/C++ 中指针直接暴露给程序，对象移动很困难，碎片治理更多依赖分配器和编程策略。

## 十、C/C++ 内存管理

C/C++ 给程序员很高自由度，也要求程序员承担内存生命周期责任。

### 1. malloc/free

C 中动态内存常用：

```c
int* arr = malloc(sizeof(int) * 10);
if (arr == NULL) {
    return;
}

arr[0] = 1;
free(arr);
arr = NULL;
```

关键点：

- `malloc` 只分配字节，不调用构造函数。
- 返回 `void*`，C++ 中不应优先使用。
- 释放必须使用 `free`。
- 释放后指针值本身不会自动变空。

### 2. new/delete

C++ 中：

```cpp
User* user = new User("Tom");
delete user;
user = nullptr;
```

`new` 完成两件事：

1. 分配内存。
2. 调用构造函数。

`delete` 完成两件事：

1. 调用析构函数。
2. 释放内存。

数组要匹配数组形式：

```cpp
int* arr = new int[10];
delete[] arr;
```

`new/delete`、`new[]/delete[]`、`malloc/free` 必须匹配使用，混用会导致未定义行为。

### 3. 常见错误

**内存泄漏**：

```cpp
void leak() {
    int* p = new int(1);
    // 忘记 delete
}
```

**重复释放**：

```cpp
int* p = new int(1);
delete p;
delete p;
```

**释放后使用**：

```cpp
int* p = new int(1);
delete p;
int x = *p;
```

**越界访问**：

```cpp
int* arr = new int[3];
arr[3] = 10;
delete[] arr;
```

这些问题都是未定义行为。未定义行为最危险之处在于它不保证立即崩溃，可能表现为偶发错误、数据损坏、线上随机故障。

### 4. 野指针与悬垂指针

野指针通常指未初始化的指针：

```cpp
int* p;
*p = 10;
```

悬垂指针指曾经有效，但指向的对象已经失效：

```cpp
int* p = new int(10);
delete p;
// p 现在是悬垂指针
```

把释放后的指针设为 `nullptr` 可以降低重复释放和误用风险，但只能处理当前变量。其他拷贝出去的指针仍然悬垂。

### 5. 内存对齐

CPU 对数据访问通常有对齐要求或性能偏好。结构体可能因对齐产生 padding：

```cpp
struct A {
    char c;
    int i;
};
```

`A` 的大小可能不是 5，而是 8。字段顺序会影响结构体大小：

```cpp
struct B {
    int i;
    char c;
};
```

复杂数据结构中，合理排列字段可以减少内存占用。但不要为了微小节省牺牲可读性，除非对象数量巨大或性能敏感。

## 十一、RAII、智能指针与所有权

C++ 中现代内存管理的核心不是到处手写 `delete`，而是 RAII。

### 1. RAII 思想

RAII 即 Resource Acquisition Is Initialization，资源获取即初始化。对象构造时获取资源，析构时释放资源。资源包括内存、文件句柄、锁、socket、数据库连接等。

```cpp
void foo() {
    std::vector<int> data(1000);
    // 离开作用域自动释放
}
```

RAII 的优势：

- 自动释放，减少泄漏。
- 异常安全。
- 生命周期清晰。
- 所有权绑定到对象作用域。

### 2. unique_ptr

`std::unique_ptr` 表示独占所有权：

```cpp
std::unique_ptr<User> user = std::make_unique<User>();
```

特点：

- 不能复制，只能移动。
- 离开作用域自动释放。
- 适合表达“这个对象只有一个拥有者”。

优先使用 `std::make_unique`，避免裸 `new`。

### 3. shared_ptr

`std::shared_ptr` 表示共享所有权，通过引用计数管理生命周期：

```cpp
auto user = std::make_shared<User>();
auto copy = user;
```

当最后一个 `shared_ptr` 销毁时，对象释放。

问题：

- 引用计数有开销。
- 所有权关系容易变模糊。
- 循环引用会泄漏。

### 4. weak_ptr

`std::weak_ptr` 不增加引用计数，用于打破循环引用：

```cpp
struct Node {
    std::shared_ptr<Node> child;
    std::weak_ptr<Node> parent;
};
```

使用前需要 `lock()` 得到临时 `shared_ptr`：

```cpp
if (auto p = weak.lock()) {
    p->doSomething();
}
```

### 5. 所有权设计原则

优秀 C++ 代码要让所有权一眼可见：

- 默认使用值对象和标准容器。
- 需要动态对象时优先 `unique_ptr`。
- 确实共享生命周期时才用 `shared_ptr`。
- 观察关系用裸指针、引用或 `weak_ptr`，但不要误表达所有权。
- 避免接口返回需要调用方手动释放的裸指针。

## 十二、垃圾回收机制

垃圾回收解决的是“自动判断哪些对象不再可达并回收”的问题。它降低了手动释放的复杂度，但并不消灭内存问题。

### 1. 可达性分析

现代 GC 通常从 GC Roots 出发，遍历对象引用图。能从 Roots 到达的对象是活对象，不能到达的是垃圾。

常见 GC Roots：

- 线程栈中的局部变量。
- 静态变量。
- JNI/native 引用。
- 运行时内部引用。
- 活跃线程对象。

如果一个无用对象仍被静态集合、单例、线程、回调持有，它仍然可达，GC 不会回收。这就是托管语言中的内存泄漏。

### 2. 标记-清除

Mark-Sweep 分两步：

1. 标记所有可达对象。
2. 清除未标记对象。

优点是实现相对直接，不需要移动对象。缺点是清除后容易产生碎片，分配时需要查找空闲块。

### 3. 标记-整理

Mark-Compact 在标记后把活对象向一端移动，整理出连续空闲空间。

优点：

- 减少碎片。
- 分配可以更快。

缺点：

- 移动对象需要更新引用。
- 整理成本高。
- 可能产生暂停。

### 4. 复制算法

复制算法把内存分成两块，每次只使用一块。GC 时把活对象复制到另一块，然后整块清空旧区域。

适合新生代，因为大部分新对象很快死亡，活对象少，复制成本低。

缺点：

- 需要额外空间。
- 老年代活对象多时复制成本高。

### 5. 分代收集

分代假说：

- 大多数对象朝生夕死。
- 熬过多次 GC 的对象更可能继续存活。

因此堆常分为：

- 新生代：短命对象，GC 频繁，回收快。
- 老年代：长寿对象，GC 较少，回收成本高。

这就是为什么减少短命对象不一定总是优化重点，真正麻烦的往往是对象意外晋升到老年代，或者缓存把大量对象长期持有。

### 6. Stop-The-World

GC 为了保证对象图一致性，某些阶段需要暂停应用线程，这叫 Stop-The-World。不同 GC 算法会尽量减少暂停时间，例如并发标记、增量收集、分区收集。

GC 优化常见目标：

- 降低平均吞吐损失。
- 降低最大暂停时间。
- 降低内存占用。
- 提高分配速度。

这些目标彼此冲突。低延迟 GC 往往需要更多 CPU 或内存；高吞吐 GC 可能允许较长暂停。

### 7. 引用计数

引用计数在对象被引用时加一，引用消失时减一，为 0 时释放。

优点：

- 回收及时。
- 实现思路直观。
- 不一定需要长时间 STW。

缺点：

- 每次引用变化都有成本。
- 多线程下引用计数更新需要同步。
- 无法单独处理循环引用。

Swift、Objective-C ARC、C++ `shared_ptr` 都使用引用计数思想。实际系统通常还会结合弱引用或额外循环检测。

## 十三、JVM 内存模型与运行时区域

Java/Kotlin 开发中，常说 JVM 内存，通常包括运行时数据区和 Java 内存模型两个相关但不同的概念。

### 1. JVM 运行时数据区

JVM 运行时常见区域：

- 程序计数器。
- Java 虚拟机栈。
- 本地方法栈。
- Java 堆。
- 方法区/元空间。
- 直接内存。

### 2. 程序计数器

每个线程都有自己的程序计数器，用于记录当前线程执行到哪条字节码指令。线程切换后，JVM 依靠它恢复执行位置。

程序计数器是线程私有的，通常不会发生 OOM。

### 3. Java 虚拟机栈

每个线程有自己的 Java 栈。方法调用会创建栈帧，栈帧中包含：

- 局部变量表。
- 操作数栈。
- 动态链接。
- 方法返回地址。

局部变量表中保存基本类型值和对象引用。对象本身通常在堆上。

栈相关错误：

- `StackOverflowError`：栈深度过深，常见于递归。
- `OutOfMemoryError: unable to create new native thread`：线程太多，无法为新线程分配栈或系统资源。

### 4. Java 堆

Java 堆是大多数对象实例所在区域，是 GC 管理重点。堆大小通常由参数控制：

```text
-Xms 初始堆大小
-Xmx 最大堆大小
```

堆中可能进一步划分新生代、老年代，也可能因不同 GC 实现而采用 region 分区。

常见错误：

```text
java.lang.OutOfMemoryError: Java heap space
```

原因可能是：

- 实际数据量超过堆容量。
- 内存泄漏。
- 缓存无界增长。
- 一次性加载大文件。
- 大量对象意外长期存活。

### 5. 方法区与元空间

方法区保存类元数据、常量、方法信息等。Java 8 以后 HotSpot 使用 Metaspace 元空间，位于本地内存。

常见错误：

```text
java.lang.OutOfMemoryError: Metaspace
```

可能原因：

- 动态生成大量类。
- 类加载器泄漏。
- 热部署框架未正确释放旧类加载器。

### 6. 直接内存

`ByteBuffer.allocateDirect` 可以分配堆外直接内存。它不在 Java 堆内，但仍占用进程内存。

优点：

- 可减少 Java 堆与 native I/O 之间复制。
- 适合高性能网络和文件 I/O。

风险：

- 不受普通堆大小直接约束。
- 释放依赖 Cleaner 或显式机制，不如堆对象直观。
- 可能出现进程 RSS 很高但 Java heap dump 看不出原因。

### 7. Java 内存模型 JMM

Java 内存模型主要讨论多线程下的可见性、有序性和原子性，不等于 JVM 运行时内存区域。

核心概念：

- 线程有自己的工作内存抽象。
- 共享变量位于主内存抽象中。
- `volatile` 保证可见性和一定有序性。
- `synchronized`、锁、并发工具建立 happens-before 关系。

内存管理与并发经常交织。例如一个对象构造后发布给其他线程，如果没有正确同步，其他线程可能看到未完全初始化的状态。

## 十四、Android 内存管理要点

Android 应用运行在资源受限设备上，内存问题对用户体验影响非常直接。应用占用过高可能导致卡顿、后台进程被杀、页面重建、OOM。

### 1. Android 进程与内存限制

每个应用进程有自己的内存上限。不同设备、系统版本、应用配置的 heap size 不同。可以通过运行时 API 获取大致限制，但不能把上限当作可随意用满的预算。

移动端内存优化更强调：

- 页面关闭后释放引用。
- 图片按需解码。
- 避免 Activity/Fragment 泄漏。
- 控制缓存大小。
- 避免在主线程做大内存分配。

### 2. Activity 泄漏

常见泄漏来源：

- 静态变量持有 Activity。
- 单例持有 View 或 Context。
- Handler 延迟消息持有外部类。
- 回调、监听器未注销。
- 协程、线程、Rx 订阅生命周期超出页面。
- Dialog、PopupWindow 持有窗口上下文。

错误示例：

```kotlin
object UserManager {
    var context: Context? = null
}
```

如果这里保存的是 Activity context，页面销毁后仍被单例持有，Activity 无法回收。应尽量使用 `applicationContext`，并避免全局持有界面对象。

### 3. Handler 泄漏

非静态内部 Handler 会隐式持有外部 Activity：

```kotlin
class MainActivity : Activity() {
    private val handler = Handler(Looper.getMainLooper())
}
```

如果有延迟消息，消息队列持有 Handler，Handler 持有 Activity，导致 Activity 延迟回收。现代 Android 更推荐结合生命周期的协程、`lifecycleScope`、`repeatOnLifecycle` 等方式管理异步任务。

### 4. 图片内存

图片是 Android 内存大户。图片内存大小主要由像素数量和像素格式决定，而不是文件大小。

例如一张 4000x3000 的 ARGB_8888 图片：

```text
4000 * 3000 * 4 = 48,000,000 bytes ≈ 45.8 MB
```

优化策略：

- 按目标 View 尺寸采样解码。
- 使用成熟图片库处理缓存和生命周期。
- 列表中及时取消无用请求。
- 避免把大图转成 Base64 字符串。
- 谨慎使用超大 Bitmap。

### 5. Compose 与内存

Jetpack Compose 中也会有内存问题：

- `remember` 保存了过大的对象。
- 状态提升不当导致页面长期持有数据。
- 列表 item key 不稳定导致重复创建。
- 协程 effect 未随生命周期取消。
- lambda 捕获了不该长期持有的对象。

Compose 不是自动免疫泄漏。它改变了 UI 构建方式，但对象生命周期、引用链、缓存策略仍然需要清晰设计。

### 6. Android 排查工具

常用工具：

- Android Studio Memory Profiler。
- Heap Dump。
- LeakCanary。
- Perfetto / System Trace。
- `adb shell dumpsys meminfo <package>`。
- `adb shell am dumpheap`。

排查时重点看：

- 页面退出后 Activity 是否仍存在。
- Dominator tree 中谁持有大对象。
- Bitmap、byte[]、集合、缓存是否异常。
- native 内存是否异常增长。
- GC 是否频繁。

## 十五、常见内存问题与排查思路

### 1. 内存泄漏

内存泄漏指不再需要的对象仍然无法释放。

C/C++ 中常见原因：

- 忘记 `free/delete`。
- 异常路径没有释放。
- 所有权不清晰。
- 容器中保存指针但未释放对象。

Java/Kotlin 中常见原因：

- 静态集合持有对象。
- 单例持有 Context/View。
- 监听器未移除。
- 线程或协程生命周期过长。
- 缓存没有上限。
- ThreadLocal 未清理。

排查方法：

1. 观察内存是否持续增长且不回落。
2. 触发 GC 后看活对象是否仍增长。
3. 生成 heap dump。
4. 找占用最大的对象类型。
5. 看引用链，定位谁持有它。
6. 判断持有是否符合业务生命周期。

### 2. OOM

OOM 是结果，不是原因。原因可能很多：

- 真正需要的数据超过可用内存。
- 内存泄漏。
- 单次分配过大。
- 碎片导致连续空间不足。
- native 内存超限。
- 线程过多。
- 元空间或类加载器泄漏。

排查 OOM 要先看错误类型和上下文。例如 JVM 中：

- `Java heap space`：Java 堆不足。
- `GC overhead limit exceeded`：GC 花费大量时间但回收很少。
- `Metaspace`：类元数据区域不足。
- `Direct buffer memory`：直接内存不足。
- `unable to create new native thread`：线程或系统资源不足。

### 3. 栈溢出

常见原因：

- 无限递归。
- 数据结构有环但递归遍历未检测。
- 递归深度与输入规模线性相关。
- 栈上大对象。

解决：

- 增加递归终止条件。
- 使用 visited 集合处理图结构。
- 改成迭代。
- 减小栈帧大小。
- 必要时调整线程栈大小，但这通常不是首选。

### 4. 内存越界

C/C++ 中越界访问是高危问题：

- 写坏相邻对象。
- 破坏分配器元数据。
- 导致安全漏洞。
- 让程序在离真正错误很远的地方崩溃。

工具：

- AddressSanitizer。
- Valgrind。
- Dr. Memory。
- 静态分析器。
- 编译器警告。

### 5. Use-After-Free

释放后继续使用内存。它可能导致：

- 读取旧数据。
- 修改已经被其他对象复用的内存。
- 崩溃。
- 安全漏洞。

解决思路：

- 明确所有权。
- 使用 RAII 和智能指针。
- 减少裸指针跨作用域传播。
- 使用工具检测。
- 避免异步回调中引用已销毁对象。

### 6. Double Free

同一块内存释放两次。第二次释放时，分配器状态可能已经改变，容易破坏堆结构。

根因通常是所有权不清晰：多个地方都以为自己负责释放。解决方向仍是所有权建模，而不是到处加判空。

### 7. 频繁 GC

频繁 GC 通常说明：

- 分配速率太高。
- 短时间创建大量临时对象。
- 堆设置过小。
- 对象晋升过快。
- 老年代空间紧张。

优化：

- 减少热点路径临时对象。
- 复用必要缓冲区。
- 分批处理。
- 调整堆参数。
- 找出异常对象来源。
- 优化缓存策略。

## 十六、性能优化中的内存视角

内存优化不是单纯“少用内存”，而是在性能、复杂度、可维护性之间取平衡。

### 1. 降低分配频率

频繁分配会带来：

- 分配器开销。
- GC 压力。
- cache 污染。
- 内存峰值上升。

热点循环中尤其要注意：

```java
for (int i = 0; i < n; i++) {
    String s = prefix + i;
    list.add(s);
}
```

这类代码是否有问题取决于规模、频率和运行时优化。优化前应先测量，不要凭直觉改复杂。

### 2. 减少不必要复制

复制大对象会增加 CPU 和内存带宽压力。常见场景：

- 字节数组反复拷贝。
- 图片多次转换格式。
- 字符串拼接产生中间对象。
- 网络数据从内核态到用户态多次复制。
- 集合转换链过长。

优化方式：

- 使用流式 API。
- 使用缓冲区。
- 使用切片/view。
- 避免无意义中间集合。
- 批量处理。

### 3. 控制内存峰值

内存峰值比平均值更容易导致 OOM。例如：

```text
读取 500MB 文件 -> 解析成对象 1GB -> 生成结果 500MB
```

如果三个阶段的数据同时存在，峰值可能超过 2GB。即使最终结果不大，中间峰值也可能击穿内存。

降低峰值的方式：

- 流式读取。
- 分块解析。
- 及时释放中间引用。
- 避免同时保留原始数据和解析结果。
- 管道化处理。

### 4. 改善内存局部性

局部性对性能影响巨大。优化方法：

- 使用连续数组代替指针链结构。
- 热字段集中存放。
- 冷字段拆出去。
- 批量处理同类数据。
- 避免对象过度拆分。

在高性能场景中，数据布局往往比面向对象层级更重要。

### 5. 缓存不是越多越好

缓存的本质是用内存换时间。好的缓存必须回答：

- 最大容量是多少？
- 淘汰策略是什么？
- key 的生命周期是什么？
- value 是否持有大对象？
- 是否需要弱引用？
- 是否线程安全？
- 是否有命中率监控？

没有上限的缓存经常变成内存泄漏。

### 6. 内存池与 Arena

Arena 思想是把一批生命周期相近的对象从同一区域分配，然后整体释放。适合：

- 编译器 AST。
- 请求级临时对象。
- 游戏一帧内临时数据。
- 批处理任务。

优势：

- 分配快。
- 释放快。
- 减少碎片。

代价：

- 单个对象不能单独释放。
- 生命周期边界必须清晰。
- 容易因一个长寿对象拖住整个 arena。

## 十七、高质量代码中的内存管理原则

### 1. 生命周期要明确

每个对象都应该能回答：

- 谁创建它？
- 谁拥有它？
- 谁可以引用它？
- 什么时候不再需要？
- 谁负责释放或取消引用？

生命周期不清晰，内存问题迟早出现。

### 2. 所有权要单一或显式共享

单一所有权最简单。共享所有权成本高，必须有明确理由。

常见坏味道：

- 到处传裸指针并约定谁释放。
- 全局变量保存业务对象。
- 回调保存页面对象但不注销。
- 缓存保存对象但没有淘汰。
- 多个模块都能延长对象生命周期。

### 3. 资源释放要异常安全

代码不能只在正常路径释放资源，还要考虑异常、提前 return、取消、超时。

好做法：

- C++ 使用 RAII。
- Java/Kotlin 使用 `try-with-resources`、`use`。
- 协程绑定生命周期。
- finally 中释放必要资源。

### 4. 大对象要谨慎

大对象包括：

- 大数组。
- 大字符串。
- 大 Bitmap。
- 大集合。
- 大 ByteBuffer。
- 大 JSON DOM。

处理原则：

- 能流式就流式。
- 能分页就分页。
- 能采样就采样。
- 能压缩就压缩。
- 用完尽快断开引用。

### 5. 不要过早优化

内存优化必须基于数据：

- 内存曲线。
- 分配热点。
- heap dump。
- GC 日志。
- profiler。
- 线上监控。

没有证据的优化容易增加复杂度，还可能让性能变差。

### 6. 建立内存预算

重要系统应有内存预算：

- 单个请求最大内存。
- 单个页面最大 Bitmap 数量。
- 缓存最大容量。
- 队列最大长度。
- 单批任务最大数据量。
- 单进程内存上限。

没有预算的系统只能靠 OOM 来发现边界。

### 7. 用工具而不是猜

常见工具按语言和场景不同：

- C/C++：AddressSanitizer、LeakSanitizer、Valgrind、heaptrack、perf。
- Java/JVM：JFR、jcmd、jmap、jstat、VisualVM、MAT、async-profiler。
- Android：Memory Profiler、LeakCanary、Perfetto、dumpsys meminfo。
- Linux：top、htop、pmap、smem、vmstat、perf、/proc。

工具的价值不是替代理解，而是验证假设。

## 十八、面试与复习提纲

### 1. 必会概念

- 栈和堆的区别。
- 进程地址空间布局。
- 虚拟内存和物理内存的关系。
- 页表、分页、TLB、缺页异常。
- malloc/free 与 new/delete 区别。
- 内存泄漏、野指针、悬垂指针。
- 内部碎片与外部碎片。
- GC Roots 和可达性分析。
- 标记清除、标记整理、复制算法。
- 分代 GC 的原因。
- JVM 堆、栈、方法区、直接内存。
- Android Activity 泄漏原因。

### 2. 高频问题简答

**栈和堆有什么区别？**

栈主要服务函数调用，自动分配释放，速度快，空间有限，生命周期服从调用关系。堆用于动态对象，生命周期更灵活，但分配释放成本更高，容易产生泄漏和碎片。

**为什么局部变量不能返回地址？**

局部变量通常位于当前函数栈帧中。函数返回后栈帧失效，地址指向的内存可能被复用，继续访问就是悬垂指针。

**为什么有 GC 还会内存泄漏？**

GC 只能回收不可达对象。如果无用对象仍被静态变量、缓存、线程、回调、单例等引用，它仍然可达，GC 不会回收。

**为什么数组通常比链表快？**

数组连续存储，空间局部性好，CPU cache 命中率高。链表节点分散，遍历时可能频繁 cache miss 和 TLB miss。

**什么是 OOM？**

OOM 是内存分配失败的结果。可能是堆不足、直接内存不足、元空间不足、线程太多、单次分配太大、泄漏或碎片等原因。

**什么是 Copy-on-Write？**

多个执行实体先共享同一份数据，并把页面标记为只读。当某一方写入时再复制，避免不必要的提前复制。

**引用计数有什么问题？**

引用变化有额外成本，多线程同步复杂，循环引用无法单靠普通引用计数回收。

### 3. 排查题思路

遇到“内存不断上涨”：

1. 区分是 Java 堆、native、线程栈、文件映射还是缓存。
2. 强制或等待 GC 后看是否回落。
3. 采集多份 heap dump 做对比。
4. 找增长最快的类型和最大 retained size。
5. 分析引用链。
6. 回到业务生命周期判断是否异常持有。

遇到“偶发 OOM”：

1. 查看 OOM 类型和日志。
2. 找发生时的输入规模和并发量。
3. 看是否有大对象单次分配。
4. 看峰值是否由多个阶段数据叠加。
5. 检查缓存、队列、批处理是否有上限。
6. 增加监控和保护性限制。

遇到“GC 频繁卡顿”：

1. 看分配速率。
2. 看 young GC 和 full GC 频率。
3. 看老年代占用和晋升情况。
4. 找对象分配热点。
5. 判断是业务创建过多、缓存策略错误还是堆参数不合理。

## 总结

内存管理的核心不是记住某个语言的 API，而是理解对象和资源的生命周期。栈解决规则明确的短生命周期问题，堆解决灵活生命周期问题，虚拟内存提供隔离和抽象，分配器负责空间复用，GC 和引用计数试图自动化回收，RAII 和所有权模型则把释放责任显式编码进程序结构。

真正可靠的内存管理能力来自三个层面：

1. **底层机制清楚**：知道数据在哪里，地址如何转换，分配释放发生了什么。
2. **生命周期清楚**：知道对象由谁拥有，何时释放，为什么仍然存活。
3. **工具验证清楚**：用 profiler、heap dump、GC 日志和系统指标验证判断。

当你能从“这块内存是谁申请的、谁持有的、为什么还没释放、访问模式是否高效”这几个问题出发分析程序，就已经具备了内存管理的核心思维。
