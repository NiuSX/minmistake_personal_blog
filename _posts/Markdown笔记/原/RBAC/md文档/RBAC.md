
# RBAC 介绍
RBAC（Role-Based Access Control，基于角色的访问控制），让用户通过角色与权限进行关联。简单地说，一个用户拥有若干角色，每一个角色拥有若干权限。这样，就构造成“用户-角色-权限”的授权模型。在这种模型中，用户与角色之间，角色与权限之间，一般是多对多的关系

- 角色（Role）：角色是指在系统中具有一组相关权限的抽象概念，代表了用户在特定上下文中的身份或职能，例如管理员、普通用户等。

- 权限（Permission）：权限是指对系统资源进行操作的许可，如读取、写入、修改等。权限可以被分配给角色。

- 用户（User）：用户是指系统的实际使用者，每个用户可以被分配一个或多个角色。

- 分配（Assignment）：分配是指将角色与用户关联起来，以赋予用户相应的权限。

RBAC模型广泛应用于系统安全、数据库 管理、网络管理等领域，它提供了一种可扩展、可管理的访问控制机制，有助于保护系统资源免受未经授权的访问和潜在的安全威胁。

# RBAC 四级分级

1. RBAC0（基础模型）：最简单的RBAC形式，是RBAC的核心思想，员工使用角色来获取权限（使用最多）。

- 三个元素：用户、角色和权限。
模型设计基于“多对多”原则，即多个用户可以具有相同的角色，一个用户可以具有多个角色。同样，您可以将同一权限分配给多个角色，也可以将同一角色分配给多个权限。

用户1 ──┐
用户2 ──┼── 角色A ──┬── 权限1
用户3 ──┘          ├── 权限2
                   └── 权限3

2. RBAC1（角色继承）：分层，建立在FlatRBAC规则之上，增加角色分层。

-增加第四个组件-层次结构，定义了不同角色之间的资历关系。
通过允许高级角色自动获取下级角色的权限，可以消除冗余，例如在角色重叠时必须指定某些权限。

总经理
          │
    ┌─────┼─────┐
    │     │     │
  副总A  副总B  副总C
    │     │
  经理A  经理B

继承类型：

- 一般继承：角色间有明显层级

- 受限继承：加入限制条件

- 多继承：一个角色可继承多个父角色

3. RBAC2（约束模型）：受约束的，建立在分层RBAC0之上，并增加职责分离。

为了在存在利益冲突策略的情况下提供帮助，将根据用户分配添加角色之间的关系。例如，作为一个角色的成员的用户将无法被指派为具有利益冲突的角色的成员。

增加各种约束规则：

|约束类型|	说明|	示例|
|---|---|---|
|互斥角色	|同一用户不能同时拥有	|出纳和会计不能是同一个人|
|基数约束	|角色拥有的用户数限制	|一个部门只能有一个经理|
|先决条件	|获得角色B必须先有角色A	|先当普通员工才能当主管|
|时间约束	|角色只在特定时间有效	|夜班值班员角色只在22:00-6:00生效|


4. RBAC3（统一模型）：RBAC3=RBAC1+RBAC2

与SSD 一样，DSD限制了可用的用户权限，但基于不同的上下文。例如，根据会话期间执行的任务，用户可能需要不同级别的访问，DSD限制会话期间激活的权限。


# RBAC 优缺点

**优点**
- 简化权限管理（批量操作）
- 权限结构清晰，易于理解
- 符合企业组织结构
- 便于审计和合规
- 支持职责分离
- 可扩展性强

**缺点**
- 角色数量膨胀后管理复杂
- 对细粒度权限（如数据级）支持不够
- 角色定义需要深入的业务理解
- 可能产生角色爆炸问题
- 动态权限场景处理复杂
