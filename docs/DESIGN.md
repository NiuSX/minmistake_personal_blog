# 个人技术博客设计规范

> 本文档是个人技术博客的视觉与交互约束，服务于作者、未来维护者和 AI 协作者。
> 项目采用 Jekyll + Minimal Mistakes 生成静态站点，部署到 GitHub Pages。
> 设计的第一目标是让技术内容容易阅读、检索和长期沉淀，而不是制造营销转化。

## 1. 产品定位

这是一个以个人知识沉淀为核心的技术博客，主要发布：

- 编程语言、框架、工具和工程实践笔记。
- 项目复盘、问题排查和解决方案。
- UI 设计、前端开发和 AI 编程实践。
- 对书籍、文章、课程或技术概念的结构化总结。
- 可供未来自己快速回看的学习记录。

网站是静态内容站点，不包含复杂后台、交易流程或高频实时数据。所有页面都应围绕“发现文章、理解文章、继续阅读、回到归档”展开。

## 2. 目标用户

- **作者本人**：快速记录、整理和回顾学习过程。
- **技术学习者**：按照主题查找教程、笔记和经验总结。
- **同行访客**：通过搜索、分类和标签定位具体问题。
- **未来维护者或 AI**：在不重新猜测设计意图的情况下新增页面和组件。

## 3. 核心任务与优先级

页面设计按以下优先级决策：

1. 阅读文章。
2. 找到相关文章或同一主题的其他文章。
3. 通过搜索、分类、标签和归档定位内容。
4. 了解作者、博客方向和项目来源。
5. 访问代码仓库、演示地址或参考资料。

不要为了增加视觉效果而牺牲正文宽度、目录可见性、代码可读性或移动端操作空间。

## 4. 设计原则

### 4.1 内容优先

标题、正文、代码、图表和结论是页面的视觉主体。装饰只用于建立层次，不与内容争夺注意力。

### 4.2 清晰可扫读

使用明确的标题层级、短段落、列表、表格、引用和代码块组织长文。读者应能在几秒内判断文章主题、难度和更新时间。

### 4.3 克制而有个性

整体保持安静、专业、耐看；允许通过少量品牌色、头像、插图和首页介绍体现作者个性，但不使用强烈的营销式视觉语言。

### 4.4 设计可复用

相同语义必须使用相同的颜色、字号、间距和组件。新增页面优先复用已有模式，不为单篇文章创建独有的视觉系统。

### 4.5 静态站点友好

页面不依赖复杂客户端状态才能阅读。即使 JavaScript 被禁用、图片加载失败或网络较慢，正文、导航和文章链接仍应可用。

## 5. 品牌气质

### 5.1 关键词

清晰、专注、诚实、温和、理性、持续积累。

### 5.2 应该呈现的感觉

- 像一份认真维护的个人技术手册。
- 像作者在与读者分享真实的学习过程。
- 页面有秩序，但不显得僵硬或企业化。
- 技术细节有足够空间，阅读过程轻松连续。

### 5.3 避免呈现的感觉

- 营销落地页、产品官网或数据大屏。
- 过度炫技的玻璃拟态、霓虹渐变和复杂动效。
- 只有漂亮标题、缺乏实际内容的空页面。
- 通过“震撼”“颠覆”“终极”等词制造夸张感。

## 6. 色彩系统

颜色以浅色阅读为默认模式。颜色必须服务于层级、链接、状态和可读性，不得为了装饰随意增加新的高饱和颜色。

### 6.1 基础颜色

| Token | 色值 | 用途 |
| --- | --- | --- |
| `--color-page` | `#F7F8FA` | 全局页面背景 |
| `--color-surface` | `#FFFFFF` | 文章、卡片和导航表面 |
| `--color-surface-muted` | `#F0F3F5` | 代码块、次级区域和悬停背景 |
| `--color-text` | `#1F2933` | 正文和主要信息 |
| `--color-heading` | `#17202A` | 标题和高优先级文字 |
| `--color-text-muted` | `#66727E` | 日期、摘要、辅助说明 |
| `--color-border` | `#D9E0E5` | 分隔线、表格和输入边框 |
| `--color-primary` | `#256D85` | 主链接、主按钮和当前导航 |
| `--color-primary-hover` | `#1B5266` | 链接和按钮悬停状态 |
| `--color-primary-soft` | `#E6F2F5` | 标签、提示背景和选中背景 |

### 6.2 状态颜色

| Token | 色值 | 用途 |
| --- | --- | --- |
| `--color-success` | `#247A52` | 成功或已完成状态 |
| `--color-warning` | `#A56616` | 注意事项和兼容性提醒 |
| `--color-danger` | `#B54747` | 错误、失效链接和危险提示 |
| `--color-info` | `#356AA0` | 一般信息和补充说明 |

状态色只能出现在有明确语义的提示、标签或图标中，不能用来给普通文章卡片染色。

### 6.3 对比度

- 正文与背景必须达到适合长时间阅读的对比度。
- 不使用浅灰色承载关键内容。
- 颜色不是唯一的状态表达方式；状态还必须通过文字、图标或结构表达。
- 深色模式如未来加入，必须单独检查代码块、链接、引用和图片边界。

## 7. 字体与排版

### 7.1 字体

- 中文正文优先使用系统无衬线字体，保证 Windows、macOS 和移动端都有可接受的 fallback。
- 英文、数字和代码使用清晰的系统字体或等宽字体。
- 不在文章正文中使用装饰性字体。
- 字体选择应优先考虑字形清晰、加载稳定和跨平台一致性。

推荐字体栈：

```css
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC",
  "Hiragino Sans GB", "Microsoft YaHei", sans-serif;
```

代码字体栈：

```css
font-family: "SFMono-Regular", Consolas, "Liberation Mono", Menlo, monospace;
```

### 7.2 字号与行高

- 正文桌面端建议 `16px`，移动端不得低于 `15px`。
- 正文行高建议 `1.7`，代码行高建议 `1.5` 至 `1.7`。
- 文章标题应明显高于正文，但不占据首屏过多空间。
- 文章内标题最多使用 `h1`、`h2`、`h3` 三个主要层级；更深层级只在确有必要时使用。
- 日期、分类、标签等元信息应小于正文，但仍保持可读，不得使用极小字号制造“信息噪声”。

### 7.3 标题规则

- 文章标题准确描述主题，不使用无意义的标题党表达。
- 标题尽量使用中文；必要的技术名词保留官方英文名称。
- 标题长度过长时允许自然换行，不使用横向滚动或强制截断。
- 每个页面只能有一个视觉上的主标题。

### 7.4 正文规则

- 每段控制在适合连续阅读的长度，避免整屏只有一个超长段落。
- 重要结论可以使用短句、列表或提示块突出，不使用大字号喊话。
- 行内代码、链接、数字和英文单词不得因为样式导致布局溢出。
- 代码、表格和图片都必须在窄屏下有可控的处理方式。

## 8. 间距与布局

### 8.1 间距基准

使用 `4px` 为基础单位，优先使用以下间距：

`4px`、`8px`、`12px`、`16px`、`24px`、`32px`、`48px`、`64px`。

不得在相似组件中随意出现大量接近但不相同的间距值。

### 8.2 页面版心

- 桌面端页面内容最大宽度建议为 `1120px` 至 `1200px`。
- 文章正文阅读列建议控制在 `680px` 至 `780px`。
- 文章详情页可以保留目录或作者信息侧栏，但侧栏不能压缩正文到难以阅读。
- 页面左右边距在桌面端保持稳定，在移动端至少保留 `16px`。
- 页面顶部留白应足够建立呼吸感，但不能让读者滚动很久才能看到文章内容。

### 8.3 页面分区

页面通常由以下区域组成：

1. 顶部导航：站点名称、主要栏目和搜索入口。
2. 页面标题区：页面标题、简短说明或面包屑。
3. 主内容区：文章列表、文章正文或关于信息。
4. 辅助区：目录、分类、标签、相关文章或作者信息。
5. 页脚：版权、站点说明和必要的外部链接。

## 9. 页面类型

### 9.1 首页

首页的任务是让新访客理解“这是一个什么博客”，并快速进入内容。

必须包含：

- 一句清晰的博客定位。
- 最近文章或精选文章入口。
- 主要分类或技术方向。
- 通往归档、关于页和搜索的路径。

首页可以有一个简洁的个人介绍区域，但不应使用全屏英雄区、复杂轮播或多个并列 CTA。

### 9.2 文章列表页与归档页

- 文章标题是第一视觉重点。
- 每项显示日期，并在有价值时显示分类、标签和摘要。
- 列表按时间倒序，排序逻辑稳定。
- 分页、年份归档和分类归档的交互方式保持一致。
- 空归档需要说明“暂时没有文章”，不能显示空白页面。

### 9.3 文章详情页

文章详情页是整个网站的核心页面，推荐结构：

1. 文章标题。
2. 发布日期、更新时间、分类、标签和预计阅读时间。
3. 文章摘要或导语。
4. 文章目录（长文显示，短文可隐藏）。
5. 正文内容。
6. 文章末尾的相关文章、参考资料和返回归档入口。

正文不应被广告、浮动组件或无关推荐打断。

### 9.4 关于页

关于页应回答：作者是谁、关注什么方向、为什么写这个博客、如何联系或查看代码。内容以真实、简洁为主，不做个人简历式的过度包装。

### 9.5 搜索页

- 搜索入口在桌面端和移动端都容易找到。
- 搜索结果显示标题、日期和能够帮助判断相关性的摘要。
- 搜索无结果时给出清晰说明和替代路径，例如浏览分类或归档。
- 搜索结果不能只显示 URL、内部 ID 或无意义的模板文本。

### 9.6 404 页

404 页需要明确说明页面不存在，并提供返回首页、查看归档和搜索的入口。可以有轻量的个性化文案，但不能让错误页面变成纯装饰插画。

## 10. 导航规范

- 主要导航控制在 `4` 至 `6` 项。
- 导航名称使用短而明确的中文，如“文章”“归档”“分类”“关于”。
- 当前页面必须有可感知的选中状态，不能只依赖颜色。
- 外部链接使用明确的外链提示，并在新窗口打开时保持行为一致。
- 移动端导航默认收起，展开后不能遮挡关闭入口和主要内容。
- 页面返回路径优先使用站点导航和面包屑，不依赖浏览器后退按钮。

## 11. 组件规范

### 11.1 链接

- 普通正文链接使用主色，并提供悬停、焦点和访问后的可区分状态。
- 链接文字应说明目标，不使用大量“点击这里”。
- 同一段正文中不应堆叠过多颜色不同的链接。

### 11.2 按钮

静态博客中的按钮数量应少而明确，主要用于“阅读文章”“查看归档”“返回首页”等导航动作。

- 主按钮：用于页面最重要的一个动作。
- 次按钮：用于次要导航动作。
- 文本链接：用于低强调度的补充入口。
- 同一视觉区域最多使用一个主按钮。
- 按钮高度建议不小于 `36px`，移动端可增大到 `40px` 以上。
- 不把普通文章链接全部做成按钮。

### 11.3 文章卡片

- 卡片优先展示标题、日期和摘要。
- 卡片之间使用稳定的间距或细分隔线，不依赖厚重阴影区分。
- 卡片圆角建议 `6px` 至 `10px`，同一页面保持一致。
- 标签数量过多时只显示最相关的少数标签。
- 卡片整体可点击时，内部不要再嵌套行为冲突的多个可点击区域。

### 11.4 标签与分类

- 分类表达稳定的内容领域，标签表达文章中的具体技术或关键词。
- 标签使用浅色背景和正文色，不使用彩虹色分类。
- 标签文字简短，避免把整句描述放进标签。
- 分类和标签必须能回到对应的聚合页面。

### 11.5 目录

- 目录只展示当前文章的标题层级。
- 长文使用粘性目录，短文不强行显示目录。
- 当前阅读位置可以有轻量高亮，但不能使用强烈动画。
- 移动端目录应折叠或放在正文之前，不能固定占据屏幕边缘。

### 11.6 代码块

- 代码块必须使用等宽字体、独立背景和足够内边距。
- 显示语言时使用准确的语言名称或缩写。
- 长代码允许横向滚动，不强制折行破坏代码结构。
- 复制操作应有清晰的成功反馈；复制失败时不能静默。
- 代码块中不使用过低对比度的注释色。

### 11.7 表格

- 表头与正文有清晰层级，边框保持轻量。
- 文本默认左对齐，数字和尺寸数据可右对齐。
- 宽表格在移动端横向滚动，不压缩到无法阅读。
- 表格前应有必要的上下文说明，不能只丢出一张无标题表格。

### 11.8 图片、插图与图表

- 图片用于解释概念、展示结果或提供上下文，不作为无意义装饰。
- 每张有信息价值的图片都需要准确的替代文本或图注。
- 图片默认响应式缩放，禁止溢出正文列。
- 截图应尽量裁剪到关键区域，并在正文中说明读者应该观察什么。
- 图表颜色使用同一套色彩系统，不能引入与站点无关的强烈配色。

### 11.9 引用与提示块

- 引用用于原文摘录、观点补充或资料来源，不用于普通正文强调。
- 提示块分为信息、注意、成功和错误四种语义，颜色和图标保持固定。
- 提示块内容应短，复杂解释回到正文段落。

## 12. 内容与文案规范

### 12.1 语气

整体语气真诚、具体、平实，像作者在分享经过验证的学习过程。

- 可以说明“我如何遇到问题、如何验证、哪里仍不确定”。
- 优先写事实、步骤、限制和结论。
- 不用夸张承诺替代技术细节。
- 不把个人经验写成对所有项目都成立的绝对结论。

### 12.2 文章标题

标题应包含主题和必要的限定条件，例如版本、场景或问题类型。避免标题只写“学习笔记”“踩坑记录”而没有具体对象。

### 12.3 元信息

文章建议维护以下 front matter 信息：

- `title`：准确、可检索的文章标题。
- `date`：首次发布或记录日期。
- `categories`：一至两个稳定分类。
- `tags`：少量具体技术标签。
- `excerpt`：帮助读者判断是否值得阅读的摘要。
- `toc`：长文按需开启目录。

日期、分类和标签的展示顺序在所有文章页保持一致。

### 12.4 参考资料

外部资料应尽量放在文章末尾的“参考资料”部分，说明资料名称和使用目的。不要在正文中散落大量无法判断用途的裸链接。

## 13. 交互与状态

### 13.1 基本状态

所有可操作元素至少考虑：默认、悬停、键盘焦点、已访问、禁用或不可用状态。

### 13.2 静态站点状态

- 页面加载失败时，已生成的正文结构仍应尽量可见。
- 图片失败时显示替代文本、图注或合理的占位区域。
- 搜索无结果时显示原因和下一步。
- 归档或分类没有内容时显示空状态。
- 文章不存在时使用 404 页面，而不是空白模板。
- 外部资源不可用时，不影响本地文章正文和站点导航。

### 13.3 动效

- 动效只用于导航展开、焦点变化、复制反馈和轻量状态转换。
- 动效时间建议在 `150ms` 至 `250ms`。
- 不使用自动播放视频、强制滚动、持续闪烁或大范围视差。
- 尊重 `prefers-reduced-motion` 设置。

## 14. 响应式规则

### 14.1 移动端

- 单列布局为默认方案。
- 正文左右内边距至少 `16px`。
- 导航、目录和搜索入口必须容易触达。
- 代码块和表格允许横向滚动，但页面本身不能出现意外横向滚动。
- 卡片内容自然堆叠，不通过缩小字体塞入多列。
- 点击目标建议不小于 `40px` 的可操作区域。

### 14.2 平板端

- 可以使用文章正文与辅助栏的有限双栏布局。
- 侧栏内容优先级低于正文，空间不足时先隐藏或移动到正文后。

### 14.3 桌面端

- 允许使用宽版心、目录和辅助导航，但正文阅读列仍保持适宜宽度。
- 文章标题、目录和正文在视觉上保持清晰的主次关系。

## 15. 可访问性

- 使用语义化 HTML：标题、导航、主内容、文章、页脚和列表结构正确。
- 标题层级不得为了字号跳级。
- 图片提供替代文本；纯装饰图片使用空替代文本。
- 所有键盘可操作元素都有清晰焦点样式。
- 不以颜色作为唯一的信息表达方式。
- 链接和按钮文字应能脱离上下文理解目标。
- 代码、表格和长链接不能破坏键盘导航。
- 页面语言和文章语言设置准确。

## 16. SEO 与静态部署约束

- 每个页面拥有唯一、准确的页面标题。
- 文章 URL 稳定、可读，避免无意义的随机参数。
- 页面摘要服务于搜索结果和社交分享，不堆砌关键词。
- 图片使用压缩后的合适尺寸，避免把大图直接作为正文资源。
- 站点核心阅读路径不依赖第三方脚本。
- 不在文档中假设存在数据库、登录系统或后台管理功能。
- 任何新增的外部脚本都需要说明用途、失败影响、隐私影响和静态部署兼容性。

## 17. 禁止事项

- 禁止把博客首页设计成强营销风格的落地页。
- 禁止大面积使用渐变、玻璃拟态、霓虹色和高强度背景纹理。
- 禁止在每张文章卡片上使用不同颜色、不同圆角或不同阴影。
- 禁止为了“高级感”降低正文对比度或字号。
- 禁止用多个无意义的卡片包裹普通文本。
- 禁止使用自动轮播、自动播放音视频和强制弹窗打断阅读。
- 禁止新增没有语义的装饰图标、徽章或数字。
- 禁止隐藏发布日期、更新时间、文章分类等有助于判断内容时效的信息。
- 禁止为了展示技术而引入不必要的前端框架或客户端状态。
- 禁止在没有更新本文档的情况下引入一套新的颜色、字号或组件风格。

## 18. 给 AI 协作者的执行规则

当 AI 新增或修改页面时，必须遵守以下顺序：

1. 先判断页面属于首页、列表页、文章页、关于页、搜索页还是错误页。
2. 优先复用 Minimal Mistakes 已有布局、归档、导航和文章组件。
3. 只使用本文档定义的颜色、字体、间距、圆角和阴影。
4. 先保证正文、导航、代码和移动端可用，再考虑装饰。
5. 为新增的搜索、筛选或异步行为补充空状态、错误状态和键盘焦点状态。
6. 不添加与静态站点定位不匹配的后台、登录、交易或营销模块。
7. 如果需求与本文档冲突，先指出冲突，再选择最小范围的例外，并同步更新本文档。

推荐提示词：

```text
请根据项目根目录的 DESIGN.md 修改这个 Jekyll 技术博客页面。
优先保证长文阅读、代码可读性、信息层级和移动端适配。
复用 Minimal Mistakes 的已有结构，不要把页面改造成营销落地页。
不要引入未定义的颜色、渐变、复杂动效或与静态部署无关的功能。
完成后检查默认、空状态、错误状态、键盘焦点和窄屏布局。
```

## 19. 设计验收清单

新增页面或组件合并前，至少检查：

- [ ] 页面类型和核心阅读任务明确。
- [ ] 标题、正文、代码和辅助信息层级清晰。
- [ ] 颜色、字号、间距、圆角和阴影来自本文档。
- [ ] 桌面端版心和文章正文宽度适合阅读。
- [ ] 移动端没有意外横向滚动，代码和表格可处理。
- [ ] 链接、按钮和导航有悬停、焦点和当前状态。
- [ ] 图片有替代文本，提示块有明确语义。
- [ ] 搜索无结果、归档为空、页面不存在等状态有反馈。
- [ ] 页面不依赖第三方脚本才能阅读核心内容。
- [ ] 文案准确、克制，没有夸张的营销表达。

## 20. 维护方式

本文档不是愿望清单，而是随着真实页面演进的设计契约。

- 新增一种稳定页面模式时，先在本文档中定义，再实现代码。
- 删除或替换颜色、字体或组件时，同时更新对应章节。
- 如果某条规则连续被违反，说明规则不够具体或组件抽象不合理，应优先修正规则或组件。
- 每次较大的视觉改版后，至少复查首页、文章列表、文章详情、搜索和 404 页面。
- 任何例外都应写明适用页面、原因和边界，避免例外扩散成新的默认风格。
