    gradle基础

    介绍

    Gradle 是一个基于 Apache Ant 和 Apache Maven 概念的项目自动化构建工具。它使用一种基于 Groovy 或 Kotlin 的特定领域语言(DSL)来声明项目配置，而不是传统的 XML。

    Gradle 构建工具是一款快速、可靠且适应性的开源构建自动化工具，采用优雅且可扩展的声明式构建语言
    Gradle 支持 Android、Java、Kotlin 多平台、Groovy、Scala、Javascript 和 C/C++。

    主要特点：

    • 灵活且高性能的构建工具

    • 支持多项目构建

    • 强大的依赖管理

    • 支持增量构建

    • 丰富的插件生态系统

    • 与 Maven 和 Ivy 仓库兼容

    核心概念

    Gradle 通过构建脚本中的信息自动化软件的构建、测试和部署。

    Gradle 构建以项目和任务为定义，使用用 Groovy 或 Kotlin 编写的构建脚本进行配置。 概念

    • 建造： 产出过程和环境。构建包含一个或多个项目及其构建脚本。
    • 项目：一个可以构建的软件，比如应用程序或库。一个构建可以包含单个根项目或多个子项目。
    • 任务：一个基本的工作单元，比如编译代码或运行测试。任务可以在构建脚本中声明，或由插件添加。
    • 构建脚本：一个配置文件（），定义任务、依赖以及其他指示Gradle如何构建项目的指令。build.gradle(.kts)
    • 插件：用来扩展 Gradle 的功能（比如插件）。插件通常会为项目添加任务和约定。Java
    • 依赖性：项目所需的外部或内部资源。Gradle在构建过程中会自动解决这些问题。

    1. .gradle：由 Gradle 生成的项目专用缓存目录。
    2. libs.versions.toml：版本目录定义了中心位置依赖的一组版本。
    3. gradle-wrapper.jar：gradle-wrapper.jar：这是一个包含Gradle Wrapper代码的小型JAR文件。如果 Gradle 尚未安装，它负责下载并安装正确的项目版本
    4. gralde-wrapper.properties：包含Gradle包装器的配置属性，如分发网址（下载Gradle的来源）和分发类型（ZIP或TARBALL）
    5. gradlew：用于使用 Gradle Wrapper 执行构建的macOS 和 Linux 脚本。
    6. gradlew.bat：用于使用 Gradle 包装器执行构建的 Windows 脚本。
    7. settings.gradle.kts：项目的设置文件，定义了子项目列表。
    8. build.gradle.kts：项目构建脚本
    9. gradle.properties：构建过程的行为控制
    10. local.properties：个人电脑环境强相关的本地配置
    11. app：应用的源代码和构建配置。

    gradle 包装器（wrapper）

    包装具有以下好处：

    1. 自动下载并使用特定的Gradle版本。
    2. 在某个Gradle版本上标准化项目。
    3. 为不同用户和环境（IDE、CI服务器等）提供相同的Gradle版本。
    4. 这样运行 Gradle 构建就很简单，无需手动安装。

    Gradle的两种运行方式：

    1. 使用系统安装的 Gradle 发行版——通过运行命令。gradle build
    2. 使用 Gradle 包装器——通过运行 Gradle 项目中包含的 or 脚本。
      • Linux 或 OSX 机器上的封装调用：$ ./gradlew build
      • Windows PowerShell 上的 Wrapper 调用：$ gradlew.bat build

    gradle 操作

    常用操作

    编译安装 assembleDebug 编译 Debug APK，不自动安装 installDebug 编译并安装到设备/模拟器 clean 清理构建缓存，解决奇怪问题时必用

    测试相关 testDebugUnitTest 运行本地单元测试（快速） connectedDebugAndroidTest 运行设备测试（UI 测试） lintDebug 代码质量检查

    诊断和调试 dependencies 查看依赖树，排查依赖冲突 signingReport 获取签名指纹（配置地图/推送时用） tasks 查看所有可用任务 properties 查看项目属性

    最常用 运行 App ./gradlew installDebug 清理项目 ./gradlew clean 跑单元测试 ./gradlew testDebugUnitTest 看依赖树 ./gradlew dependencies 获取签名 ./gradlew signingReport 代码检查 ./gradlew lintDebug 查看所有任务 ./gradlew tasks 打 Release 包 ./gradlew assembleRelease

    android

    androiddependencies： 查看 Android 项目的依赖关系

    • 显示项目所有的 Android 相关依赖（包括直接依赖和传递依赖）

    • 输出格式清晰，会列出每个依赖的配置名称（如 implementation、api）及其对应的库

    • 常用于排查依赖冲突、确认某个库是否被正确引入

    signingReport: 查看签名信息

    • 显示所有构建变体（debug/release 等）的签名配置信息

    • 输出内容包括：MD5、SHA-1、SHA-256 指纹、密钥别名、密钥位置等

    • 常用于获取调试签名指纹（如配置 Google Maps API、Firebase 时需要）

    sourceSets:查看源码集配置

    • 显示项目中每个构建变体的源码目录结构

    • 输出包括：java 源码目录、res 资源目录、manifest 文件位置、assets 目录等

    • 常用于确认不同构建类型/产品风味使用的源码文件是否正确

    build

    assemble： 编译并打包所有构建变体的 APK/AAB

    • 变体：会根据项目配置生成 assembleDebug、assembleRelease 等具体任务
    • 输出：生成最终的安装包文件

    assembleAndroidTest：编译 Android 测试（Instrumentation Tests）的 APK

    • 输出：生成测试 APK，用于在设备或模拟器上运行 UI 测试

    assembleUnitTest：编译单元测试（JUnit tests）的代码

    • 输出：生成测试类文件，用于在本地 JVM 上运行

    build: 完整的构建流程 = assemble + check

    • 包含：编译代码、运行所有测试、打包、生成报告等
    • 常用：构建完整项目时使用

    buildDependents: 构建当前项目及其依赖该项目的其他项目

    • 场景：在多模块项目中，修改了底层模块后，确保依赖它的上层模块也重新构建

    buildKotlinToolingMetadata: 生成 Kotlin 工具元数据

    • 说明：Kotlin Gradle 插件的内部任务，用于管理 Kotlin 版本兼容性
    • 通常：不需要手动执行，由 Kotlin 插件自动管理

    buildNeeded: 构建当前项目及其所依赖的所有项目

    • 场景：确保所有依赖模块都更新到最新状态

    bundle: 打包所有构建变体的 Android App Bundle（.aab）文件

    • 输出：生成 AAB 格式，用于上传 Google Play
    • 变体：bundleDebug、bundleRelease 等

    clean: 删除所有构建产物

    • 删除内容：build/ 目录、生成的 APK/AAB、中间文件等
    • 常用：遇到构建缓存问题时，先执行 clean 再重新构建

    compileDebugAndroidTestSources: 编译 Android 测试（Instrumentation Tests）的源代码

    • 构建变体：Debug
    • 测试类型：AndroidTest（运行在设备/模拟器上的测试）
    • 编译内容：src/androidTest/java/ 目录下的测试代码
    • 输出：生成 .class 文件，用于后续打包成测试 APK

    compileDebugSources

    • 作用：编译主应用程序的源代码（Debug 变体）
    • 构建变体：Debug
    • 测试类型：无（主代码）
    • 编译内容：src/main/java/ + src/debug/java/ 目录下的代码
    • 输出：生成 Debug 版本的字节码

    compileDebugUnitTestSources：编译单元测试（JUnit Tests）的源代码

    • 构建变体：Debug
    • 测试类型：UnitTest（运行在本地 JVM 上的测试）
    • 编译内容：src/test/java/ 目录下的测试代码
    • 输出：生成测试类的 .class 文件

    compileReleaseSources： 编译主应用程序的源代码（Release 变体）

    • 构建变体：Release
    • 测试类型：无（主代码）
    • 编译内容：src/main/java/ + src/release/java/ 目录下的代码
    • 输出：生成 Release 版本的字节码

    compileReleaseUnitTestSources：编译单元测试的源代码（针对 Release 变体）

    • 构建变体：Release
    • 测试类型：UnitTest
    • 编译内容：src/test/java/ 目录下的测试代码
    • 场景：测试 Release 版本的代码逻辑

    help

    项目信息类:

    projects: 列出项目中所有的子项目（模块）

    • 输出：显示项目树状结构
    • 用途：了解多模块项目的组织结构

    tasks:列出当前项目所有可执行的任务

    • 输出：按任务组分类显示（build、help、verification等）
    • 用途：查看有哪些任务可用，支持搜索过滤

    properties：显示项目的所有 Gradle 属性

    • 输出：包括项目属性、系统属性、环境变量等
    • 用途：调试属性配置，查看当前所有可用属性

    依赖和配置类: dependencies：显示项目的依赖关系树

    • 输出：列出所有配置（implementation、api、testImplementation等）的依赖
    • 用途：排查依赖冲突、查看传递依赖

    dependencyInsight：深入分析特定依赖的详情

    • 输出：显示为什么引入该依赖、从哪里引入、版本冲突等信息
    • 用途：排查依赖版本问题

    resolvableConfigurations：显示所有可解析的配置（resolvable configurations）

    • 输出：列出可以被解析的依赖配置
    • 用途：高级依赖分析，了解配置的解析状态

    环境和工具类: buildEnvironment：显示构建脚本的依赖和环境信息

    • 输出：包括 Gradle 版本、Groovy/Kotlin 版本、buildSrc 依赖等
    • 用途：排查构建脚本自身的问题

    javaToolchains：显示可用的 Java 工具链（JDK）

    • 输出：列出系统中检测到的 JDK 及其位置、版本
    • 用途：确认项目使用的 JDK 是否正确

    outgoingVariants：显示模块输出的变体信息（Variants）

    • 输出：列出当前模块对外暴露的构建变体及其属性
    • 用途：高级功能，用于调试变体选择和依赖匹配

    辅助类

    help：显示 Gradle 帮助信息

    • 输出：基本命令用法和常用任务
    • 用途：快速查看 Gradle 命令帮助

    kotlinDslAccessorsReport：生成 Kotlin DSL 访问器报告

    • 输出：列出所有可用的 DSL 扩展和访问器
    • 用途：Kotlin DSL 开发调试，了解可用的扩展方法

    install

    installDebug：安装 Debug 版本的 APK 到连接的设备/模拟器

    • 特性：如果设备上已安装相同包名的应用，会先卸载再安装；支持多设备时需指定设备
    • 用途：快速将调试版应用安装到设备进行手动测试

    installDebugAndroidTest：安装 Android 测试 APK 到设备/模拟器

    • 前置条件：需要先安装主应用 APK
    • 用途：在设备上安装测试 APK，用于运行 Instrumentation Tests

    uninstallAll：卸载所有构建变体的应用

    • 特性：根据包名卸载，会移除所有已安装的变体版本
    • 用途：清理设备上的所有测试版本应用

    uninstallDebug：卸载 Debug 版本的应用

    • 场景： 清理设备上的调试版本 遇到签名冲突时先卸载旧版本 准备安装新版本前清理环境

    uninstallDebugAndroidTest ：卸载 Android 测试 APK

    • 场景： 清理测试残留 重新安装测试 APK 前清理 避免测试缓存导致的异常

    uninstallRelease：卸载 Release 版本的应用

    • 场景： 清理生产版本 准备安装新正式版本 切换不同签名版本时

    verification

    check ：运行所有验证任务（包括测试、代码检查等）

    • 用途：完整的项目验证，通常在提交代码或 CI 流程中运行

    connectedCheck ：运行所有需要在设备/模拟器上执行的测试

    • 用途：验证应用在真实设备上的行为

    deviceCheck ：运行所有需要在设备上执行的测试（与 connectedCheck 类似）

    • 用途：用于设备测试的通用任务

    connectedAndroidTest：在所有连接的设备上运行 Android 测试（Instrumentation Tests）

    • 输出：生成测试报告在 build/reports/androidTests/
    • 场景：运行 UI 测试、集成测试

    connectedDebugAndroidTest：仅针对 Debug 变体运行 Android 测试

    • 用途：专门测试 Debug 版本的功能

    checkJetifier：检查 Jetifier 迁移状态

    • 输出：显示哪些依赖需要迁移，迁移状态如何
    • 用途：确认 AndroidX 迁移是否完整

    checkKotlinGradlePluginConfigurationErrors: 检查 Kotlin Gradle 插件配置错误

    • 检查内容： Kotlin 版本兼容性 插件配置是否正确 依赖是否冲突
    • 用途：在构建前发现 Kotlin 配置问题

    deviceAndroidTest 设备测试的通用任务（与 connectedAndroidTest 类似）

    • 用途：在设备上运行测试的另一种入口

    deviceCheck：运行所有需要在设备上执行的检查任务

    • 这是一个聚合任务，会触发 connectedAndroidTest 等设备测试

    lint 对所有构建变体运行 Lint 检查

    • 输出：生成完整的 Lint 报告
    • 检查内容： 性能问题（如过度绘制、内存泄漏） 安全性问题（如未加密的网络请求） 正确性问题（如缺少权限检查） 国际化问题（如硬编码字符串） API 兼容性（如使用高版本 API 未做版本判断）

    lintDebug：仅对 Debug 变体运行 Lint 检查

    • 用途：开发阶段快速检查，避免运行所有变体节省时间
    • 场景：日常开发中的代码质量检查

    lintRelease：仅对 Release 变体运行 Lint 检查

    • 用途：发布前对正式版本进行完整检查
    • 场景：CI/CD 流程中，确保 Release 版本质量

    lintFix：自动修复 Lint 发现的部分问题

    • 添加缺失的 @NonNull/@Nullable 注解
    • 移除未使用的资源
    • 修复简单的代码风格问题
    • 添加缺失的权限检查 注意：不是所有问题都能自动修复

    lintVitalRelease：对 Release 版本运行关键（Vital）Lint 检查 特点

    • 更严格：只检查严重影响应用质量的严重问题 更快：跳过非关键检查项
    • 集成：在生成 Release APK/AAB 时会自动运行
    • 触发时机：执行 assembleRelease 时自动触发
    • 失败策略：如果发现严重问题，构建会失败

    test 运行所有构建变体的单元测试

    • 用途：一次性运行所有本地 JVM 测试

    testDebugUnitTest：运行 Debug 变体的单元测试

    • 运行环境：本地 JVM（不需要设备或模拟器）
    • 特点： 速度快 适合纯逻辑测试
    • 用途：日常开发中的快速测试验证

    testReleaseUnitTest ：运行 Release 变体的单元测试 场景：

    • 验证 Release 版本的代码逻辑
    • 检查 ProGuard/R8 混淆后的行为
    • CI/CD 流程中的完整测试

    updateLintBaseline：更新 Lint 基线文件（lint-baseline.xml） 功能：

    • 将当前所有 Lint 问题记录为”已接受”状态
    • 生成基线文件后，后续 Lint 检查只报告新增的问题
    • 帮助团队逐步改进代码质量

    场景：

    • 接手遗留项目，想逐步修复问题
    • 暂时接受现有问题，只关注新增问题
    • 团队约定代码质量改进计划
