# 研究结论：个人技术博客基础站点

**功能**：个人技术博客基础站点

**日期**：2026-07-19

## 决策 1：继续使用 Jekyll 静态生成

- **决策**：文章、页面、归档、分类、标签、搜索和站点元数据继续由 Jekyll 源文件生成静态输出。
- **理由**：项目当前已经使用 Jekyll、Markdown、Liquid、Sass 和 Bundler；个人博客的核心需求是长期内容沉淀，不需要运行时数据库或服务端接口。
- **依据**：`Gemfile`、`Gemfile.lock`、`_config.yml`、`README.md` 和现有 `_posts/`、`_pages/` 目录。
- **备选方案**：迁移到单页应用或带后台的 CMS。
- **放弃原因**：会增加构建、运行、认证和部署复杂度，且不解决当前博客的核心阅读问题。

## 决策 2：以 Minimal Mistakes 作为页面基础

- **决策**：复用 Minimal Mistakes 的 `single`、`posts`、`search`、`categories`、`archive`、`tag`
  和 `default` 等页面能力，并通过项目覆盖文件扩展外观和结构。
- **理由**：主题已经提供文章详情、归档、分类、标签、搜索、目录、相关内容、导航和页脚能力。
- **依据**：`_layouts/`、`_includes/`、`_sass/` 和 `_config.yml` 已存在完整主题扩展结构。
- **备选方案**：从零编写页面模板和文章渲染系统。
- **放弃原因**：重复实现成熟主题能力，增加长期维护和主题升级成本。

## 决策 3：使用 front matter 作为文章元数据契约

- **决策**：文章至少维护 `title`、`date`、`categories`、`tags`（有需要时）和 `excerpt`；站点默认值集中在 `_config.yml`。
- **理由**：这些字段同时服务于文章详情、首页列表、归档、分类、标签、搜索、SEO 和相关文章。
- **依据**：现有 `_posts/` 文件的 front matter 形式和 `_config.yml` 的 `defaults` 配置。
- **备选方案**：把元数据维护在独立数据库或单独 JSON 索引中。
- **放弃原因**：引入同步问题，并削弱“新增一个 Markdown 文件即可发布”的内容流程。

## 决策 4：保留现有静态搜索与归档路径

- **决策**：搜索继续使用 Minimal Mistakes/Jekyll 当前配置，归档使用 `year-archive`，分类和标签使用 Liquid 聚合页面。
- **理由**：现有 `_pages/year-archive.md`、`_pages/categories.md`、`_pages/search.md` 和 `_config.yml` 已形成完整内容发现路径。
- **备选方案**：增加 Algolia、独立搜索 API 或外部内容索引服务。
- **放弃原因**：对个人博客属于非必要外部依赖，并可能使核心搜索在网络或服务失败时不可用。

## 决策 5：以 `/blog` 作为生产部署路径

- **决策**：所有模板、导航和资源都必须通过 Jekyll URL 过滤器生成链接；生产工作流显式使用 `--baseurl "/blog"`。
- **理由**：`_config.yml` 配置了 `url: https://NiuSX.github.io` 和 `baseurl: /blog`，工作流会把 `_site/` 同步到发布仓库的 `blog/` 目录。
- **依据**：`README.md` 和 `.github/workflows/pages.yml`。
- **备选方案**：假设站点始终部署在域名根路径。
- **放弃原因**：会导致本地看似正常、生产资源和站内链接失效。

## 决策 6：不建立外部 API 契约

- **决策**：项目不创建 API、数据库 schema 或服务端接口契约；以 `contracts/site-routes.md` 描述公共页面路由、文章 front matter 和静态资源行为。
- **理由**：网站的公共接口是读者访问的页面和内容链接，不是运行时 JSON API。
- **备选方案**：创建空的 REST/GraphQL 契约目录。
- **放弃原因**：没有真实接口，空契约会制造误导性维护负担。

## 决策 7：以构建和静态输出检查作为主验证方式

- **决策**：计划阶段定义 `bundle exec jekyll build`、生产 `baseurl` 构建、生成路由检查、资源检查、响应式和可访问性检查。
- **理由**：静态博客最关键的失败模式是构建失败、路径错误、资源缺失、移动端溢出和结构不可访问。
- **备选方案**：引入复杂的端到端测试或后端测试框架。
- **放弃原因**：当前范围不包含运行时服务，过早引入会增加复杂度而不能覆盖主要风险。

## 决策 8：将视觉规范作为实现门禁

- **决策**：所有页面和组件变更必须遵守 `docs/DESIGN.md` 中的内容优先、排版、色彩、间距、组件、响应式和可访问性规则。
- **理由**：视觉一致性是长期内容站点可维护性的一部分，不能在实现阶段依赖临时审美判断。
- **备选方案**：每个页面自行选择颜色、字号和布局。
- **放弃原因**：会造成页面风格漂移，增加未来维护和评审成本。
