# 功能规格：个人技术博客基础站点

**功能分支**：`001-personal-tech-blog`

**创建日期**：2026-07-19

**状态**：草稿

**输入**：用户描述：生成一份完整的中文个人技术博客规格文档。项目是使用 Minimal Mistakes 主题的 Jekyll 静态个人技术博客，部署到 GitHub Pages；设计与内容约束以 `docs/DESIGN.md` 为准。

## 用户场景与测试（必填）

本项目面向长期技术内容沉淀，核心用户包括博客作者、技术学习者和偶然访问的同行读者。所有用户故事都必须能够通过发布后的站点页面独立验证。

### 用户故事 1 - 阅读和理解技术文章（优先级：P1）

作为技术学习者，我希望从首页、文章列表或搜索结果进入一篇文章，并在清晰的标题、摘要、目录和正文结构中完成阅读，从而理解一个技术主题或解决一个具体问题。

**优先级原因**：文章阅读是博客存在的核心价值，其他功能都必须服务于发现和理解内容。

**独立测试**：从首页进入任意文章详情页，验证读者可以看到文章标题、发布日期、分类或标签、正文结构、代码块、图片和参考资料，并能在移动端完成阅读。

**验收场景**：

1. **给定**读者打开首页，**当**读者选择最近文章或精选文章，**则**站点进入对应文章详情页，并显示准确的标题和文章元信息。
2. **给定**读者打开文章详情页，**当**读者浏览正文，**则**标题层级、段落、列表、引用、代码块、表格和图片均保持清晰可读。
3. **给定**文章包含较长目录、代码块或宽表格，**当**读者使用移动设备阅读，**则**目录可以折叠，代码块和表格可以在局部横向查看，页面本身不发生意外横向滚动。
4. **给定**读者关闭 JavaScript 或外部资源加载失败，**当**读者访问文章，**则**文章正文和主要导航仍然可用。

---

### 用户故事 2 - 发现和定位文章（优先级：P1）

作为技术学习者，我希望通过导航、文章归档、分类、标签和搜索快速定位感兴趣的文章，从而不必逐页浏览整个博客。

**优先级原因**：随着文章数量增长，可靠的内容发现能力直接决定博客是否可用。

**独立测试**：从站点导航进入归档、分类、标签和搜索路径，分别使用时间、主题、技术关键词定位已知文章，并验证无结果时有明确反馈。

**验收场景**：

1. **给定**读者打开文章归档，**当**读者按年份或时间浏览，**则**文章按照稳定的时间顺序展示标题、日期和必要摘要。
2. **给定**文章配置了分类和标签，**当**读者点击分类或标签，**则**站点展示对应的内容聚合页，并可返回文章详情页。
3. **给定**读者输入文章标题、技术名词或正文关键词，**当**读者提交搜索，**则**结果显示相关标题、日期和摘要，并支持继续进入文章。
4. **给定**搜索关键词没有匹配结果，**当**搜索完成，**则**页面说明没有结果，并提供浏览归档、分类或返回首页的替代路径。
5. **给定**某个分类、标签或年份暂时没有文章，**当**读者进入聚合页，**则**页面显示清晰的空状态，而不是空白页面。

---

### 用户故事 3 - 发布和维护技术内容（优先级：P1）

作为博客作者，我希望通过统一的文章格式、分类标签和页面约定发布新内容，并让新文章自动出现在首页、归档、搜索和相关内容入口中，从而降低长期维护成本。

**优先级原因**：博客必须支持持续写作；如果新增文章需要重复修改多个页面，内容规模增长后就难以维护。

**独立测试**：新增一篇符合约定的 Markdown 文章，完成本地构建后验证其详情页、首页列表、归档、分类/标签和搜索入口均能正确反映新内容。

**验收场景**：

1. **给定**作者新增一篇包含标题、日期、分类、标签和摘要的文章，**当**站点重新发布，**则**文章详情页和文章聚合入口自动出现该内容。
2. **给定**作者更新文章标题、摘要或分类，**当**站点重新发布，**则**文章详情页、列表和搜索结果展示一致的新信息。
3. **给定**作者提交一篇包含代码、图片、表格或外部参考资料的文章，**当**读者访问文章，**则**内容按照统一的博客设计规范展示。
4. **给定**文章或图片路径配置错误，**当**作者执行发布前检查，**则**能够发现构建、链接或资源问题，而不是将明显的损坏页面发布到生产环境。

---

### 用户故事 4 - 可靠地访问静态站点（优先级：P1）

作为读者，我希望无论使用桌面浏览器、移动设备或不同网络环境访问博客，都能获得稳定的页面、正确的链接和可理解的错误反馈。

**优先级原因**：静态部署的核心优势是稳定和可复现；部署路径、资源路径或错误页面失效会直接破坏所有用户故事。

**独立测试**：在本地预览和 GitHub Pages 生产路径下分别访问首页、文章、归档、搜索、关于页和 404 页面，检查链接、资源、标题、订阅入口和移动端布局。

**验收场景**：

1. **给定**站点部署在配置的 `url` 和 `baseurl` 下，**当**读者从任意入口访问页面，**则**内部链接、图片、样式和脚本都指向正确路径。
2. **给定**读者访问不存在的页面，**当**站点返回错误页，**则**页面说明资源不存在，并提供首页、归档和搜索入口。
3. **给定**外部评论、分析、字体或嵌入资源不可用，**当**读者访问文章，**则**本地正文、导航和必要资源不被阻断。
4. **给定**读者使用键盘操作站点，**当**读者在导航、搜索、目录和文章链接之间移动，**则**焦点顺序合理且当前焦点清晰可见。

---

### 边界情况

- 文章没有摘要、标签、图片或相关文章时如何保持列表和详情页结构完整？
- 文章标题、标签、代码行、表格列或外部 URL 很长时如何防止布局破坏？
- 文章集合、分类、标签、年份归档或搜索结果为空时如何展示空状态？
- 图片、字体、评论、分析或其他第三方资源加载失败时如何降级？
- 文章路径、资源路径或 `baseurl` 配置错误时如何在发布前发现？
- 读者使用窄屏、键盘、低带宽环境或禁用 JavaScript 时如何继续阅读？
- 文章包含较深标题层级、复杂表格或很长代码时如何保持目录和正文可用？
- 访问不存在的文章、分类或静态页面时如何提供 404 引导？

## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供一个能够说明博客定位、作者方向和主要内容入口的首页。
- **FR-002**：系统必须提供文章详情页，展示标题、发布日期、更新时间（如有）、分类、标签、摘要和正文。
- **FR-003**：系统必须支持 Markdown 文章中的标题、段落、列表、引用、代码块、表格、图片、链接和参考资料。
- **FR-004**：系统必须提供文章列表和归档入口，并按稳定的时间顺序展示文章。
- **FR-005**：系统必须提供分类聚合和标签聚合入口，使读者可以按主题或技术关键词浏览文章。
- **FR-006**：系统必须提供搜索入口，并在结果中展示标题、日期、摘要和文章链接。
- **FR-007**：系统必须为搜索无结果、空分类、空标签和空归档提供明确的空状态。
- **FR-008**：系统必须提供关于页面，说明作者、关注方向、博客用途和可访问的代码或联系方式。
- **FR-009**：系统必须提供 404 页面，说明页面不存在，并提供首页、归档和搜索入口。
- **FR-010**：系统必须提供站点级导航，至少覆盖首页、文章/归档、分类或标签、搜索和关于入口。
- **FR-011**：系统必须在文章末尾提供适用的相关文章、参考资料或返回归档入口，帮助读者继续阅读。
- **FR-012**：系统必须支持统一的文章 front matter，至少能够表达标题、日期、分类、标签和摘要。
- **FR-013**：系统必须根据文章 front matter 自动更新首页文章列表、归档、分类、标签和搜索内容，不要求作者手动同步多个列表页面。
- **FR-014**：系统必须支持站点标题、描述、规范链接、订阅源和站点地图等基础发布信息。
- **FR-015**：系统必须在链接或资源发生错误时提供可理解的反馈，不得静默显示损坏或空白页面。
- **FR-016**：系统必须允许作者通过新增或修改单篇 Markdown 文章完成常规内容发布，不要求修改模板代码。

### 静态站点与部署需求（必填）

- **SD-001**：核心页面和文章内容必须由静态文件提供，不得依赖服务端、数据库、登录或运行时 API。
- **SD-002**：项目必须使用 Jekyll 和 Minimal Mistakes 作为站点生成与主题基础；新增页面优先复用主题的布局、include、配置或 Sass 扩展点。
- **SD-003**：站点必须部署到 GitHub Pages，并支持配置的 `url` 和 `baseurl`，当前生产路径为 `/blog`。
- **SD-004**：内部链接、文章链接、图片、样式、脚本、订阅源和站点地图必须在生产部署路径下解析正确。
- **SD-005**：发布前必须能够通过 `bundle exec jekyll build` 完成可重复构建，并在构建失败时阻止发布。
- **SD-006**：生成目录 `_site/` 和缓存目录不得作为内容源进行手动维护或修复。
- **SD-007**：任何新增插件、外部脚本、嵌入服务或构建依赖都必须说明用途、失败影响、隐私影响和 GitHub Pages 兼容性。
- **SD-008**：外部服务不可用时，文章正文、站点导航、归档和本地资源必须继续可用。

### 视觉与内容需求

- **VD-001**：页面必须遵循 `docs/DESIGN.md` 定义的“内容优先、清晰可扫读、克制而有个性、设计可复用、静态站点友好”原则。
- **VD-002**：默认视觉必须使用浅色阅读背景、清晰正文颜色、稳定的主色、轻量边框和有限的状态颜色。
- **VD-003**：正文桌面端建议使用不低于 `16px` 的字号，移动端不得低于 `15px`，正文行高建议为 `1.7`。
- **VD-004**：文章正文阅读列应控制在适合长文阅读的宽度，桌面端页面版心建议控制在 `1120px` 至 `1200px`。
- **VD-005**：禁止使用营销落地页风格、大面积渐变、玻璃拟态、霓虹色、复杂轮播和无意义装饰。
- **VD-006**：文章卡片、标签、目录、代码块、表格、图片、引用和提示块必须遵循统一的组件规则。
- **VD-007**：同一语义不得在不同页面使用不同颜色、字号、圆角、阴影或交互方式。

### 可访问性与响应式需求（修改界面时必填）

- **UI-001**：页面必须使用语义化 HTML 和正确的标题层级，且每页只有一个主要页面标题。
- **UI-002**：所有键盘可操作元素必须有清晰可见的焦点状态，导航、搜索、目录和文章链接必须可通过键盘访问。
- **UI-003**：有信息价值的图片必须提供准确替代文本或图注，纯装饰图片不得承载必要信息。
- **UI-004**：颜色不得作为唯一的状态表达方式，正文、链接、代码和提示块必须保持足够对比度。
- **UI-005**：站点必须适配移动端、平板和桌面端；页面不得出现意外的整体横向滚动。
- **UI-006**：代码块和宽表格可以局部横向滚动，但不得通过压缩文字破坏可读性。
- **UI-007**：动效必须轻量，并尊重 `prefers-reduced-motion`；核心阅读不得依赖动效。
- **UI-008**：核心文章和导航在 JavaScript 不可用、图片失败或第三方资源失败时仍必须可用。

### 关键实体（涉及结构化内容）

- **文章**：一篇可独立阅读的技术内容，包含标题、日期、摘要、分类、标签、正文、图片、代码和参考资料等属性。
- **分类**：用于表达稳定内容领域的聚合维度，例如前端开发、工具实践或 UI 设计。
- **标签**：用于表达文章涉及的具体技术、工具、概念或场景。
- **站点页面**：首页、文章列表、归档、搜索、关于和 404 等非文章页面。
- **导航入口**：指向首页、文章、归档、分类、标签、搜索、关于或外部代码仓库的链接。
- **静态资源**：图片、样式表、脚本、字体、订阅源和站点地图等发布资源。

## 成功标准（必填）

### 可度量结果

- **SC-001**：新读者可以在首页最多经过 2 次明确操作进入一篇最新文章或精选文章。
- **SC-002**：读者可以通过搜索、分类、标签或归档在最多 3 次明确操作内定位一篇已知文章。
- **SC-003**：文章详情页首屏能够显示文章标题和必要元信息，读者不需要先关闭弹窗、广告或轮播才能开始阅读。
- **SC-004**：100% 的核心页面（首页、文章详情、文章列表、归档、搜索、关于和 404）在移动端和桌面端均可访问，且不出现页面级横向滚动。
- **SC-005**：作者新增一篇符合约定的 Markdown 文章后，不需要手动修改首页、归档、分类、标签和搜索列表即可在这些入口看到文章。
- **SC-006**：每次发布前，站点都能完成一次可重复的构建检查；构建失败时不得将不完整产物发布到生产环境。
- **SC-007**：核心文章正文、站点导航和本地资源在 JavaScript 或任一可选外部资源失败时仍可访问。
- **SC-008**：所有核心页面的主要操作都可以通过键盘完成，并且焦点位置始终可见。
- **SC-009**：读者能够通过标题、摘要、日期、分类或标签判断文章是否与当前需求相关，不需要打开每一篇文章逐一确认。
- **SC-010**：新增文章、修改文章元信息和新增静态页面均有明确的目录、命名和 front matter 约定，维护者可以按照文档完成操作。

## 假设

- 项目继续使用 Jekyll、Minimal Mistakes、Bundler 和 GitHub Pages 作为基础技术与部署方式。
- 项目面向公开访问，不包含用户登录、权限管理、付费内容、订单、实时消息或后台管理系统。
- 文章以中文为主要语言，技术名词、代码和官方 API 名称可以保留英文。
- 文章内容由作者通过版本控制提交，发布流程由现有 GitHub Actions 工作流完成。
- 搜索、评论、统计和外部嵌入属于可选能力，不得成为核心阅读路径的前置条件。
- `docs/DESIGN.md` 是视觉、组件、内容语气和响应式规则的设计依据。
- 站点默认以浅色模式为主；深色模式不属于本规格的首期范围。
- 本规格不要求新增独立后台或客户端应用；任何超出静态博客范围的能力都需要单独规格和宪章评估。
